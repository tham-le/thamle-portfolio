WTF are Scancodes, Keycodes, and Virtual Keys?

Why pressing ‘A’ is the most complicated thing you’ll do today.

You press a button on your keyboard. A letter appears on your screen. Simple, right?

Wrong.

Between your finger hitting plastic and the letter ‘A’ appearing in your text editor, that signal passes through a gauntlet of translation layers that are holding onto 40 years of backward compatibility debt. It’s a game of “telephone” played by three different systems that barely speak the same language.

If you’ve ever tried to write a keylogger, a game engine, or an embedded driver, you’ve likely stared at 0x1E, 30, and 0x41 and wondered why they all claim to be the letter ‘A’.

Let’s decode the mess.

The Core Analogy

Think of your computer input system like an International Diplomacy Chain:

  1. The Scancode (The Foreign Diplomat): The keyboard hardware speaks a weird, proprietary dialect. It screams “Button at Row 2, Column 1 was pressed!” It doesn’t know what an “A” is. It only knows physical geography.

  2. The Keycode (The Translator): The Kernel driver hears the diplomat and translates it into a standard military code. “Subject #30 initiated action.” This code is uniform across the OS, regardless of whether the diplomat is using a USB, Bluetooth, or ancient PS/2 dialect.

  3. The Virtual Key (The Protocol Officer): The Windowing System (Windows/X11) takes that military code and applies context (Layouts). “Ah, the user is in US-English mode, so Code #30 means ‘A’. But if they were in French mode, it would mean ‘Q’.”

The Problem

Why do we need three layers?

Hardware Fragmentation.

Keyboards are dumb. Old PS/2 keyboards use Scan Code Set 1 or Set 2. USB keyboards use HID Usage Tables. Bluetooth keyboards do their own thing. If the OS had to understand every hardware variation directly, your kernel would be 90% keyboard drivers.

We need a way to say “Physical Key 3” → “Standard Key ID” → “The Letter A”.

The Breakdown

1. Scancodes (The Hardware Truth)

This is the raw data sent by the keyboard controller. It represents a physical location on the board.

  • The WTF Factor: It is largely hardware-dependent.
  • Example: On a standard US keyboard, the ‘A’ key might send 0x1E (USB HID) or 0x1C (PS/2 Set 2).
  • Reality Check: If you are reading scancodes, you are writing a device driver or working with raw USB packets. Turn back now.

2. Keycodes (The OS Standard)

The Kernel (specifically the Input Subsystem) sanitizes the scancode. It maps the hardware-specific garbage to a known constant.

  • On Linux: The kernel atkbd or hid-generic driver converts the scancode to a Linux Input Event Code defined in input-event-codes.h.
  • The Code: KEY_A is always 30 on Linux, regardless of your keyboard brand.
  • Important: This is still not the letter ‘A’. It’s just an identifier for “The key next to CapsLock.”

3. Virtual Keys / Keysyms (The Application Layer)

This is what your browser or IDE sees. The OS applies a Keyboard Layout (mapping table) to the Keycode.

  • On Windows: VK_A (Virtual Key).
  • On Linux (X11/Wayland): XK_a (Keysym).
  • The Magic: This is where Shift + Keycode 30 becomes ‘A’, and Keycode 30 alone becomes ‘a’. This is also where Keycode 30 magically becomes ‘Q’ if you switch your OS language to French (AZERTY).

Game Dev Trap

If you bind movement to “Virtual Key W”, a French player will have to twist their fingers to press ‘Z’ (where ‘W’ is physically located on AZERTY). Pro Tip: For movement (WASD), read Scancodes or raw Keycodes to get physical location. For chat, read Virtual Keys to get characters.
