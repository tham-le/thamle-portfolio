WTF is the Linux Input Core?

WTF is the Linux Input Core?

Nov 28, 20255 min read

  • linux
  • kernel
  • drivers
  • architecture

The universal translator for your hardware.

In the last article, we watched a keyboard send a raw Scancode and the OS turn it into a Keycode. But we skipped a massive step: Who actually does that work?

If you plug in a 5mousefrom1999anda200 pro-gaming mouse, they both move the cursor. If you plug in a PS5 controller, it looks just like a joystick from 2005 to your code.

This miracle is performed by the Linux Input Subsystem (or Input Core). It is the kernel’s “Diplomatic Corps”—a massive machinery that forces thousands of weird, proprietary hardware devices to speak one single, boring language.

The Core Analogy

Think of the Input Core as a Funnel and a Translator.

  1. The Wide Top (Hardware Drivers): Thousands of drivers listening to chaotic noise. The USB driver hears binary packets; the PS/2 driver reads voltage pulses on a wire; the Bluetooth driver decrypts radio waves. They all speak different languages.

  2. The Neck (The Input Core): This is the middleman. It forces every driver to convert their specific noise into a standard format. It doesn’t care how the button was pressed, only that it was pressed.

  3. The Spout (Event Handlers): The standardized data flows out to specific “Handlers” (like evdev) that write the data to files in /dev/input/. This is what your applications actually read.

The Architecture (The Layer Cake)

The subsystem is split into three strict layers. Understanding this changes how you debug hardware forever.

1. The Device Drivers (The Grunt Work)

Located in: drivers/input/mouse, drivers/input/keyboard, etc.

These guys talk to the metal. When you click a mouse button, a specific driver (like usbhid or psmouse) wakes up.

  • Job: Catch the hardware interrupt.
  • Translation: “Hey, the hardware sent 0x01 on the wire. I know that means ‘Left Click’.”
  • Action: Call input_report_key(dev, BTN_LEFT, 1).

Crucial Point: The driver never talks to userspace. It just reports facts to the Core.

2. The Input Core (The Manager)

Located in: drivers/input/input.c

This is the kernel module that defines the “truth.” It provides the definitions for BTN_LEFT, KEY_A, and ABS_X.

  • Job: It manages the list of all input devices.
  • Normalization: It filters noise (debouncing) and handles auto-repeat (if you hold a key down).
  • Routing: It takes the report from the driver and pushes it to the appropriate Handler.

3. The Event Handlers (The Delivery Service)

Located in: drivers/input/evdev.c, drivers/input/mousedev.c

These modules bridge the gap to “User Space” (where your programs live).

  • evdev (The Modern Standard): It takes the event and writes it to /dev/input/eventX. This is what 99% of modern Linux software uses.
  • mousedev (The Retro Hack): It pretends to be an ancient PS/2 mouse (/dev/input/mice) so that legacy software (and X11 in fallback mode) can still work even if you’re using a USB mouse.

The Problem It Solves

Without the Input Core, you would need a different code path to read from a USB keyboard vs. a Bluetooth keyboard.

Instead, the Input Core guarantees that every input device looks exactly the same to your program: a file that spits out struct input_event packets.

Code Example: “Who are you?”

In the previous article, we read events. But the Input Core also lets us query the device. “Are you a mouse? Do you have an ‘A’ key? Do you support Force Feedback?”

This is done via ioctl (Input/Output Control) calls on the device file. It allows us to peek at the struct input_dev bits stored in the kernel.

What’s happening here:

  1. EVIOCGNAME: We ask the Input Core “What string did the driver register?”
  2. EVIOCGBIT(0, ...): We ask “Which event types (EV_*) do you support?”
  3. The Result: If you run this on your mouse, you’ll see EV_KEY (buttons) and EV_REL (movement). If you run it on your keyboard, you’ll see EV_KEY and maybe EV_LED.

Common Misconceptions

Confusing evdev with libinput

evdev is the Kernel interface (raw events from the kernel). libinput is a Userspace library that reads evdev and fixes bugs (like touchpad gestures or broken hardware reports). Most GUI apps talk to libinput, which talks to evdev.

What’s Next

Now we know how the Kernel standardizes hardware into events. But raw events are still too low-level for a browser. A browser wants “Text,” not “Keycode 30 pressed.”
