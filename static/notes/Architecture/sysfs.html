The Core Analogy

The Problem

Before Linux 2.6 (early 2000s), we had /proc. /proc was designed for Process information (like ps data), but lazy developers started dumping hardware configs in there too. It became a messy dumpster fire where files contained unstructured text, weird formats, and unrelated settings.

The kernel developers realized they needed a structured, rigorous way to expose the Device Model (the topology we discussed in the last article).

Enter Sysfs.

How It Works

Sysfs is mounted at /sys. It is a Virtual Filesystem. When you run cat /sys/class/net/eth0/speed, the kernel doesn‚Äôt read a disk. It triggers a C function inside the network driver, reads a variable from RAM, converts it to a string (‚Äú1000‚Äù), and hands it to you.

The Three Golden Rules of Sysfs

  1. One File, One Value: unlike /proc (which might contain a table of data), a sysfs file should contain exactly one piece of data (e.g., just the integer 50).
  2. Text Based: Everything is ASCII. No binary blobs.
  3. Strict Hierarchy: The folder structure maps 1:1 to the internal C structures (kobjects) inside the kernel code.

The Neighborhood Map

  • /sys/class (The Functional View): Grouped by what it does.
    • Want to find your network card? Look in /sys/class/net/.
    • Want to find your mouse? Look in /sys/class/input/.
    • It doesn‚Äôt care if it‚Äôs PCI or USB; it just cares that it‚Äôs a ‚ÄúNetwork Device‚Äù.
  • /sys/devices (The Physical View): Grouped by how it‚Äôs connected.
    • This is the deep topology tree (PCI ‚Üí USB ‚Üí Device) we saw in Device Topology.
  • /sys/block: Hard drives and partitions.
  • /sys/power: Sleep and hibernation controls.

Pro Tip: sysfs vs ioctl

ioctl is the old school way: specific, efficient, but opaque (you need C headers to know the magic numbers). sysfs is the modern way: discoverable, scriptable, but slightly slower (string parsing overhead). For high-performance data (like audio or 3D graphics), we still use ioctl or mmap. for configuration, we use sysfs.
