WTF is Device Topology on Linux?

WTF is Device Topology on Linux?

Nov 28, 20257 min read

  • linux
  • sysfs
  • hardware
  • architecture
  • deep-dive

It’s not a list. It’s a massive, dysfunctional corporate org chart.

When you type ls /dev, you see a flat list of files: sda, ttyUSB0, event4. It looks like all devices are created equal, sitting next to each other in a happy pool.

This is a lie. Untitled In reality, your hardware is arranged in a strict, deep hierarchy. Your mouse isn’t just “connected to the computer.” It is connected to a specific Port, on a specific USB Hub, driven by a specific USB Controller, which hangs off a specific PCIe Lane, which is managed by the Northbridge (or CPU).

This hierarchy is the Device Topology. Understanding it is the difference between “I plugged it in and it works” and “I know exactly why moving this USB dongle to the adjacent port changed its device name from ttyUSB0 to ttyUSB1 and broke my deployment script.”

The Core Analogy

Think of your computer hardware like a Massive Corporation (“Kernel Corp”).

  1. The CPU (The CEO): sits at the top. It doesn’t talk to customers (devices) directly. It only talks to its direct reports.

  2. The Buses (Middle Management): These are the PCIe Bridges and USB Controllers. They manage departments.

    • The PCIe Bus is the Senior VP.
    • The USB Controller is a Regional Manager reporting to the Senior VP.

  3. The Hubs (Team Leads): They take one connection from the manager and split it into multiple ports for workers.

  4. The Devices (The Workers): Your mouse, WiFi card, and NVMe drive. They are the leaf nodes doing the actual work.

Crucial Lesson: If you fire a Regional Manager (disable a USB Controller), every single worker under them (mouse, keyboard, webcam) gets fired instantly. The topology dictates the power structure.

The Problem

Why do we need this tree structure? Why not just a flat list?

  1. Addressing: The CPU has limited pins. It can’t wire directly to 127 USB devices. It wires to one Controller, which handles the addressing for sub-devices.
  2. Power Management: If you want to put the computer to sleep, you must walk the tree from the bottom up. You can’t power down the PCIe bus (VP) until the WiFi card (Worker) has finished its shutdown tasks.
  3. Bandwidth: Topology tells you who shares a pipe. If you plug two high-speed webcams into the same USB Hub (Team Lead), they fight for the same bandwidth. If you plug them into different root ports, they don’t.

How It Works: Sysfs

Linux exposes this entire tree via Sysfs, mounted at /sys.

While /dev is the Storefront (easy access for customers), /sys/devices is the Warehouse Blueprint.

The Path of Madness

If you have a mouse at /dev/input/event5, its true location in the topology might look like this:

Let’s decode this address:

  1. pci0000:00: The PCI Root Complex (The CEO’s office).
  2. 0000:00:14.0: The USB Controller attached to the PCI bus (The Regional Manager).
  3. usb1: The USB Root Hub created by that controller.
  4. 1-2: Physical Port 2 on that Root Hub.
  5. 1-2:1.0: Interface 0 on the device plugged into Port 2.
  6. input5: The abstract input device created by the kernel driver.

This path is stable(ish). If you reboot, the mouse is still at PCI address 00:14.0 Port 2. This is how OS tools can remember “The mouse on the left port” versus “The mouse on the right port.”

Visualizing the Tree

You don’t need to cd through 20 directories to see this. Linux has built-in tools to visualize the topology.

1. lsusb -t (The USB Tree)

Shows the hierarchy of USB buses and devices.

Translation: Devices 2 and 3 share the bandwidth of Bus 01.

2. lspci -tv (The PCI Tree)

Shows how your graphics card, NVMe drives, and USB controllers connect to the CPU.

Translation: The NVMe SSD is behind a bridge (1d.0), sitting on secondary bus 02.

The "Parent" Concept

In the Linux Kernel struct device, there is a pointer struct device *parent. When you see 1-2 is the parent of 1-2:1.0, that is literally a pointer dereference in C code inside the kernel. Sysfs just visualizes these pointers as folders.

What’s Next

You now understand that devices aren’t just a list; they are a tree. The OS uses this tree to identify hardware based on where it is plugged in, not just what it is.

This is exactly how udev rules work. When you write a rule saying “The keyboard in USB Port 2 should be named my_keyboard”, you are matching against this topology path.
