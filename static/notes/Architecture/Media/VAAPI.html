WTF is VAAPI?

Your CPU can encode H.264. Your GPU can do it 10x faster while using a fraction of the power. VAAPI is how Linux makes that happen.

You’re encoding a 4K video with FFmpeg. On CPU (libx264), it takes 45 minutes and pegs all cores at 100%. With VAAPI, the same encode takes 4 minutes and the CPU barely notices — the GPU’s dedicated media engine handles it.

VAAPI (Video Acceleration API) is the Linux abstraction layer between applications and GPU video hardware. It provides a vendor-neutral interface for hardware-accelerated video encoding, decoding, and post-processing. Think of it as the video equivalent of OpenGL — one API, multiple GPU backends.

The Core Analogy

Imagine a factory with two production lines:

  • CPU encoding is the handcraft line. Skilled workers (CPU cores) do everything manually — motion estimation, transform coding, entropy coding. High quality, very slow, the workers are exhausted.
  • VAAPI encoding sends the work to a specialized machine (the GPU’s media engine). It’s built to do one thing — encode video — and it does it at 10x the speed while the workers are free to do other tasks.

VAAPI is the instruction manual that tells your application how to talk to that specialized machine, regardless of who manufactured it (Intel, AMD, or NVIDIA).

The Architecture

VAAPI is a three-layer stack:

Application (FFmpeg, GStreamer, mpv)
       |
    libva          ← The API library (vendor-neutral)
       |
  VA-API driver    ← GPU-specific backend
       |
    GPU hardware   ← Media engine on the chip

libva

The core library that defines the API. Applications link against libva and call functions like vaCreateSurfaces, vaBeginPicture, vaRenderPicture. The library loads the appropriate backend driver at runtime based on the GPU present.

VA-API Drivers

Each GPU vendor provides a driver that implements the libva interface:

Intel has the most mature VAAPI support — their iHD driver covers the full API. AMD support through Mesa is solid for decode and improving for encode. NVIDIA’s native stack uses NVENC/NVDEC instead of VAAPI, though a compatibility layer exists.

The DRI Device

Applications access the GPU through a DRI (Direct Rendering Infrastructure) device node:

/dev/dri/renderD128    ← Render-only access (no display)
/dev/dri/card0         ← Full GPU access (display + render)

renderD128 is the preferred device for headless encoding — it doesn’t require a display server. This is what you’d mount into a container for GPU rendering in Docker.

Checking Your Hardware

The vainfo command shows what your GPU supports:

Reading vainfo output:

  • Profile = codec + feature level (e.g., H264Main, HEVCMain10 for 10-bit HEVC)
  • Entrypoint = operation type:
    • VLD = decode (Variable Length Decode)
    • EncSlice = encode
    • VideoProc = post-processing (scaling, color conversion, deinterlacing)

If a profile/entrypoint pair is listed, your GPU can do it in hardware. If not, you’re stuck with CPU.

Using VAAPI with FFmpeg

FFmpeg is the most common way to use VAAPI. The integration uses FFmpeg’s hwaccel framework.

Hardware Decoding

What happens:

  1. FFmpeg opens the DRI device
  2. Input is sent to the GPU’s decoder (VLD entrypoint)
  3. Decoded frames stay in GPU memory (VAAPI surfaces)
  4. Frames are re-encoded on the GPU (EncSlice entrypoint)
  5. Output is written — the data never touched the CPU for pixel processing

Hardware Encoding

The -vf filter chain:

  • format=nv12 — converts to NV12 pixel format (what VAAPI expects)
  • hwupload — transfers frames from CPU memory to GPU memory

Scaling on the GPU

scale_vaapi does the downscale on the GPU — no CPU involvement for resizing.

If you have an NVIDIA GPU, NVENC is generally better supported than VAAPI’s NVIDIA compatibility layer. Use h264_nvenc directly.

If you have Intel or AMD, VAAPI is the native choice. Intel’s iHD driver is particularly mature for server-side encoding workloads.

Common Pitfalls

”No VAAPI support” even though the GPU supports it — check that the right driver is installed. AMD needs mesa-va-drivers. Intel needs intel-media-va-driver (or intel-vaapi-driver for older hardware). The LIBVA_DRIVER_NAME environment variable can force a specific driver.

Permission denied on /dev/dri — your user needs to be in the video and/or render groups: sudo usermod -aG video,render $USER.

Encode quality is worse than libx264 — hardware encoders trade quality for speed. At the same bitrate, libx264 typically produces better visual quality. Hardware encoders win on speed and power consumption. Increase the bitrate or lower the QP to compensate.

NV12 format required — VAAPI expects NV12 (or P010 for 10-bit) input. Forgetting the format=nv12,hwupload filter chain causes “unsupported pixel format” errors.

Summary

VAAPI is Linux’s vendor-neutral API for hardware video encoding and decoding. It sits between applications (FFmpeg, GStreamer) and GPU-specific drivers (Intel iHD, AMD radeonsi). Use vainfo to check what your GPU supports. With FFmpeg, the typical encode command uses -vaapi_device /dev/dri/renderD128 and a -vf 'format=nv12,hwupload' filter chain. Hardware encoding is 5-10x faster than CPU encoding at the cost of slightly lower quality per bitrate. Intel has the most mature support; AMD is solid and improving; NVIDIA users should prefer NVENC directly.
