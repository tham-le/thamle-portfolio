WTF is evdev?

WTF is evdev?

Nov 28, 20255 min read

The Esperanto of hardware input.

In the dark ages of Linux, reading input was a nightmare. A mouse spoke “PS/2 stream”. A joystick spoke “Gameport”. A USB keyboard spoke “HID”. If you wanted to write a game, you had to write five different drivers just to let the player move forward.

Then came evdev (Event Device).

evdev is the great normalizer. It is the kernel interface that says: “I don’t care if you are a $500 flight stick or a barcode scanner; you will all speak the exact same language.”

If you are writing a Wayland compositor, a keylogger, or an embedded GUI, you aren’t talking to hardware. You are talking to evdev.

The Core Analogy

Think of evdev like a Stock Ticker Tape.

  1. The Events: It doesn’t give you the “state” of the keyboard (e.g., “A is currently held down”). It gives you a stream of changes.
  2. The Atomicity: When you move your mouse diagonally, the hardware actually sends two values: X change and Y change. evdev groups these into a “packet” so your cursor doesn’t zigzag.
  3. The Format: Every single event, whether it’s a button press, a mouse wiggle, or a touchpad swipe, fits into the exact same C struct structure.

The Problem

Hardware is chaotic.

  • A mouse sends relative data (“I moved 5 units left”).
  • A touchscreen sends absolute data (“Finger is at pixel 500, 300”).
  • A keyboard sends binary state (“Key 30 is DOWN”).

Without a unified layer, your application would need if (is_mouse) { ... } else if (is_touch) { ... }.

evdev abstracts this by providing a generic file at /dev/input/eventX that streams standard packets.

How It Works: The Protocol

Every time you touch hardware, the kernel generates a struct input_event. This is the “byte on the wire” of the evdev protocol.

It looks like this (simplified):

The Big Three Types

  1. EV_KEY (0x01): Buttons and Keys.
    • value 1: Pressed.
    • value 0: Released.
    • value 2: Auto-repeat (holding it down).
  2. EV_REL (0x02): Relative Axes (Mice).
    • Used for things that don’t have a start or end point. You can scroll down forever.
    • code: REL_X, REL_Y, REL_WHEEL.
  3. EV_ABS (0x03): Absolute Axes (Joysticks, Touchscreens).
    • Used for things with physical limits. A joystick can only go so far left.
    • code: ABS_X, ABS_Y, ABS_PRESSURE.

The Synchronization (EV_SYN)

This is the part that trips up beginners. When you move your mouse diagonally, the driver sends:

  1. EV_REL, REL_X, +5
  2. EV_REL, REL_Y, -3

If you processed these one by one, your cursor would move Right, then Up. That looks jagged. So evdev adds a punctuation mark: EV_SYN.

  1. EV_REL, REL_X, +5
  2. EV_REL, REL_Y, -3
  3. EV_SYN, SYN_REPORT, 0 (Translation: “Batch Complete. Draw the frame now.“)

The Library Wrapper: libevdev

Directly reading structs from file descriptors is the “Unix Way,” but it’s fragile (struct padding, endianness, versioning).

In production (e.g., in X.Org or Westond), nobody reads raw files. They use libevdev. It is a wrapper library that:

  1. Handles ioctl calls to query device names/capabilities.
  2. Syncs state (if you sleep and wake up, it queries the current key states).
  3. Provides nice string names for constants (libevdev_event_type_get_name(EV_REL) returns “EV_REL”).

Pro Tip

If you are writing a simple hack, read the file directly. If you are writing shipping software, use libevdev.

Common Pitfalls

  1. Permissions: /dev/input/event* is owned by root (and usually the input group). Your user cannot read it by default. You need to add your user to the input group or use sudo.
  2. Grab Mode: If X11 or Wayland is reading the mouse, they might “grab” the device (using EVIOCGRAB), preventing other apps from seeing events. This is a security feature so keyloggers can’t sniff passwords easily.
  3. The “Keyboard” is not one device: On modern laptops, you might have event3 (Keyboard), event4 (Video Bus Keys), and event5 (Power Button). They all emit key events. You have to listen to all of them to catch everything.

What’s Next
