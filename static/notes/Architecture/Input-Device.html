WTF is an Input Device?

The journey from mechanical switches to software events

You press a key. Text appears on screen. WTF just happened?

Between your finger and that character is a complex chain of hardware, electrical signals, kernel drivers, and abstraction layers. This article breaks down what an “input device” actually is, and why every operating system needs a sophisticated stack to handle them.

The Core Analogy

Think of an input device like a translator at the United Nations.

Your keyboard doesn’t speak “computer language” natively. When you press the ‘A’ key:

  • The keyboard speaks “electrical signal language” (voltage changes)
  • The USB controller speaks “USB protocol language” (data packets)
  • The OS kernel needs “event language” (structured data)
  • Your application wants “character language” (‘a’ or ‘A’)

Each layer translates between two languages. An input device is the starting point of this translation chain.

The Problem This Solves

Why Not Just Read Hardware Directly?

Imagine if every application had to know:

  • Is this a PS/2 keyboard or USB keyboard?
  • What’s the scan code for ‘A’ on this specific keyboard model?
  • How do I read from I/O port 0x60 vs USB endpoint 0x81?
  • Does this mouse send 3-byte packets or 4-byte packets?

This would be a nightmare. Every app would need hardware-specific code for every input device model ever made.

The Abstraction Solution

Operating systems solve this with abstraction layers:

[Physical Hardware]
       ↓
[Hardware Protocol Layer]  ← USB, PS/2, Bluetooth, I²C
       ↓
[Kernel Driver Layer]      ← Device-specific translation
       ↓
[Input Subsystem]          ← Normalized events
       ↓
[User-Space API]           ← Applications read here

Each layer speaks two languages:

  • Input: The layer below’s protocol
  • Output: A more abstract, standardized format

How It Actually Works

The Three Core Concepts

1. Physical Connection

An input device connects via a bus/transport layer:

  • USB (Universal Serial Bus) - Modern keyboards, mice, game controllers
  • PS/2 (Personal System/2) - Legacy keyboards/mice, still present on some motherboards
  • I²C (Inter-Integrated Circuit) - Internal laptop touchpads
  • Bluetooth - Wireless keyboards/mice
  • SPI (Serial Peripheral Interface) - Embedded touchscreens

Each bus has its own protocol for communication.

2. Device Protocol

On top of the transport, the device speaks a data protocol:

  • HID (Human Interface Device) - Universal standard for keyboards, mice, gamepads
  • Proprietary protocols - Gaming mice with custom features
  • Touch protocols - Multitouch-specific formats

Most modern devices use HID because it’s standardized and works everywhere.

3. Kernel Abstraction

The OS kernel provides:

  • Linux: Input Core subsystem (/drivers/input/)
  • Windows: HID miniport driver model
  • macOS: IOHIDFamily framework
  • Android: InputManagerService (built on Linux kernel)

This is where hardware diversity becomes software uniformity.

Windows Equivalent (Raw Input)

Key differences from Linux:

  1. Windows uses message-based architecture (WM_INPUT message)
  2. Must create a window (even hidden) to receive messages
  3. RAWINPUT structure instead of input_event
  4. Registration required: RegisterRawInputDevices()

Similarity: Both provide access to raw device data, bypassing higher-level processing.

The Big Picture

What Makes an Input Device Universal

An input device becomes “universal” when it has:

  1. Standard physical interface (USB, Bluetooth)
  2. Standard protocol (HID)
  3. Kernel driver support (built into OS)
  4. Abstracted events (normalized format for apps)

This is why your $10 keyboard works instantly on any modern OS.
