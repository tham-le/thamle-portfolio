WTF is hwdb?

The “Cheat Sheet” for lying hardware.

We’ve established that hardware is chaotic. Keyboards send random scancodes, mice lie about their DPI resolution, and trackpads pretend to be generic mice.

The Kernel tries its best to standardize this, but it can’t know every single quirk of every $10 generic keyboard manufactured in a basement in Shenzhen.

Enter hwdb (Hardware Database).

It is a massive, static lookup table that says: “I know this device claims to be a standard keyboard, but when it sends code 0x12, it actually means ‘Volume Up’, not ‘Launch Calculator’.”

It is the patch layer for hardware reality.

The Core Analogy

Think of your Linux system as an Strict Examining Board:

  1. The Hardware: A student taking a test. They are mostly correct, but they have terrible handwriting and sometimes just make things up.
  2. The Kernel: The strict proctor. It records exactly what the student wrote. “Student wrote ‘Glarb’.“
  3. udev: The grading machine. It needs standardized answers.
  4. hwdb: The Grading Key / Errata Sheet. It contains notes like: “If Student #4521 writes ‘Glarb’, mark it as ‘Answer C’.”

Without hwdb, the system blindly accepts the hardware’s gibberish. With hwdb, we can correct mistakes before applications see them.

The Problem

Why do we need a database? Why doesn’t the driver just fix it?

Scale and Laziness.

  1. Rebranding: A manufacturer releases a “Gaming Keyboard” that is electrically identical to their “Office Keyboard” but has a different plastic shell and Product ID. They don’t update the firmware, so the “Sniper Button” sends the letter P.
  2. Mistakes: A laptop manufacturer wires the “WiFi Toggle” button to the scancode for “Eject CD-ROM”.
  3. Missing Info: Cheap mice don’t report their DPI (Dots Per Inch). libinput needs to know the DPI to calculate smooth acceleration.

We cannot patch the Linux Kernel C source code every time a company releases a buggy mouse. That would be insanity. instead, we push a text update to hwdb.

How It Works

hwdb is part of the systemd ecosystem (specifically systemd-hwdb).

  1. Source Files: Text files located in /usr/lib/udev/hwdb.d/ (system default) and /etc/udev/hwdb.d/ (user overrides).
  2. Compilation: You run systemd-hwdb update. This compiles the text files into a binary B-tree database (/etc/udev/hwdb.bin) for fast lookups.
  3. Lookup: When a device is plugged in, udev constructs a “Modalias” string (a long ID string identifying the hardware) and queries the binary database.
  4. Application: If a match is found, udev applies the properties (like remapping a key) to the device node.

The Format: The Modalias Nightmare

The hardest part of hwdb is the Match String. It looks like a cat walked on a keyboard.

It usually follows the format: bus:idVendor:idProduct:version:modalias_garbage

Beyond Keyboards: Mice and DPI

hwdb isn’t just for keys. It’s critical for high-performance mouse usage.

Generic mice don’t tell the OS their sensor resolution. Without this, libinput assumes a default (usually 1000 DPI). If you have a 4000 DPI gaming mouse, the cursor will fly across the screen at light speed.
