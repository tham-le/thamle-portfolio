WTF is udev?

WTF is udev?

Nov 28, 20255 min read

  • linux
  • system-administration
  • automation
  • hotplug

The HR Department of your Linux Kernel.

In the previous articles, we learned that the Kernel sees hardware as a Topology tree, and Sysfs exposes that tree as files.

But when you plug in a USB camera, who actually creates /dev/video0? Who decides that this specific camera should belong to the video group? Who decides that your network card is named enp3s0 instead of eth0?

That is udev (Userspace /dev).

It is the background daemon that listens to the kernel shouting “NEW HARDWARE!” and scrambles to organize the mess before you notice.

The Core Analogy

If the Kernel is the CEO and the Hardware are the Employees:

udev is the HR Department.

  1. Onboarding: When a new device shows up (plugged in), the Kernel just assigns it a number. udev steps in, checks its ID badge (Vendor ID), gives it a desk (creates /dev/node), and assigns it a name tag (Symlink).
  2. Policy Enforcement: HR decides who gets keys to the server room. udev decides if a device is read-only or if regular users can touch it (chmod).
  3. Standardization: HR ensures everyone’s email follows the first.last format. udev ensures network cards are named based on their physical slot, so they don’t change names after a reboot.

The Problem: The Bad Old Days

Before udev (and 2.6 kernels), we had a static /dev directory.

It was just a folder on your hard drive with thousands of empty files: /dev/sda1 through /dev/sda15, /dev/ttyS0 through /dev/ttyS99.

  • Bloat: You had entries for hardware you didn’t own.
  • Race Conditions: If you had two USB printers, the one you turned on first became lp0. If you turned them on in reverse order tomorrow, your printing script would send the payroll spreadsheet to the label printer.

udev solves this by creating device nodes dynamically and deterministically.

How It Works: The Pipeline

This happens in milliseconds every time you plug something in:

  1. The Event: The Kernel detects a USB insertion.
  2. The Signal: The Kernel sends a uevent (a structured text message) over a Netlink socket. It basically yells: ADD /devices/pci.../usb1/1-1.
  3. The Listener: systemd-udevd (the userspace daemon) catches this message.
  4. The Match: udev scans its rule files (/etc/udev/rules.d/) looking for matches. “If VendorID is 0x1234, do X.”
  5. The Action: udev creates the file in /dev, sets permissions, creates symlinks, or runs a script.

Common Misconceptions

”udev loads drivers”

False. The Kernel detects hardware and loads the module (driver) using internal aliases. udev can trigger module loading in some edge cases, but usually, the driver is already loaded by the time udev sees the device. udev manages the Device Node (the file), not the Driver (the code).

”I can use udev to launch my GUI app when I insert a USB drive”

Mostly False. udev runs as root in a restricted environment. It cannot easily talk to your desktop session (Wayland/X11).

  • The Wrong Way: Trying to run DISPLAY=:0 firefox inside a udev rule. It will fail or be killed for taking too long.
  • The Right Way: Have udev send a signal (via Systemd or D-Bus), and have your user session listen for that signal.

WTF is enp3s0?

Why did Linux get rid of eth0? Because eth0 is a lie.

In the old days, eth0 was “the first network card found.” If you added a new card, or the kernel enumerated PCIe buses differently after an update, your WAN card might suddenly become eth1 and your LAN card eth0. Your firewall rules would then apply to the wrong interfaces. Security Nightmare.

udev now uses Predictable Network Interface Names:

  • en: Ethernet
  • p3: Bus 3 (PCI location)
  • s0: Slot 0

enp3s0 effectively means “The card plugged into physical slot 3.” It will never change, even if you buy a new card or update the kernel.
