WTF is HID?

WTF is HID?

How one protocol made input devices universal

You plug in a USB keyboard. It works instantly on Linux, Windows, macOS, your PlayStation, even your TV. WTF makes this possible?

The answer: HID (Human Interface Device) - a protocol so successful you’ve probably never heard of it, because it just works.

The Core Analogy

Think of HID like a resume for input devices.

When you apply for a job, you send a resume that says:

  • “I can do these things: typing, spreadsheets, presentations"
  • "Here’s my experience level for each skill"
  • "Call me at this number when you need me”

A HID device sends a similar “resume” (Report Descriptor) that says:

  • “I have these capabilities: 5 buttons, 2 axes, a scroll wheel"
  • "Here’s the format of data I’ll send you"
  • "Read from this USB endpoint when you want my input”

The operating system reads this resume and says “Got it, you’re a mouse” - without needing a custom driver.

The Problem This Solves

Before HID: Driver Hell

In the 1990s, every input device needed its own driver:

Microsoft Mouse → Needs MOUSE.SYS driver
Logitech Mouse → Needs LOGI.SYS driver  
Generic Mouse → Needs GMOUSE.DRV driver
Joystick Brand X → Needs JOY_X.DRV driver

Problems:

  • ❌ New device = find and install driver
  • ❌ No driver for your OS = device doesn’t work
  • ❌ Driver updates break things
  • ❌ Manufacturers had to write drivers for Windows, Mac, Linux, etc.

After HID: Universal Recognition

With HID (standardized in 1996):

ANY HID Keyboard → Works instantly
ANY HID Mouse → Works instantly
ANY HID Gamepad → Works instantly

Benefits:

  • ✅ Plug and play on all major operating systems
  • ✅ One driver (HID driver) handles thousands of devices
  • ✅ Manufacturers don’t need to write OS-specific drivers
  • ✅ Devices work on future operating systems

This is why your 2025 keyboard works on a Windows XP virtual machine from 2001.

How It Actually Works

The Three Core Components

HID has three key concepts:

1. Report Descriptor (The Device Resume)

When a HID device connects, it sends a Report Descriptor - a binary structure that describes:

  • What kind of device is this? (keyboard, mouse, gamepad, etc.)
  • What inputs does it have? (keys, buttons, axes, etc.)
  • What’s the format of data it will send?

Think of it as self-documentation.

2. Reports (The Actual Data)

Once the OS understands the descriptor, the device sends Reports - small packets of input data:

  • Input Reports: Device → Host (key presses, mouse movement)
  • Output Reports: Host → Device (LED states, rumble commands)
  • Feature Reports: Bidirectional configuration

3. HID Usage Tables (The Common Language)

HID defines standard codes for everything:

  • Usage Page 0x01 = Generic Desktop
    • Usage 0x02 = Mouse
    • Usage 0x06 = Keyboard
  • Usage Page 0x09 = Buttons
    • Usage 0x01 = Button 1
    • Usage 0x02 = Button 2

This is the universal dictionary that makes cross-platform work possible.

What this says in plain English:

“I’m a mouse. I have 3 buttons (each 1 bit: pressed/not pressed), plus 5 bits of padding. Then I have X and Y movement, each 8 bits, ranging from -127 to +127, and they’re relative movements (deltas), not absolute positions.”

The resulting Input Report will be 4 bytes:

Byte 0: [Btn3][Btn2][Btn1][Pad][Pad][Pad][Pad][Pad]
Byte 1: X movement (-127 to +127)
Byte 2: Y movement (-127 to +127)
Byte 3: Wheel movement (if present)

How the OS Uses This

Linux Kernel Flow:

USB device connects
  ↓
drivers/hid/usbhid/hid-core.c detects HID class device
  ↓
Requests and parses Report Descriptor
  ↓
drivers/hid/hid-input.c translates HID to Linux input events
  ↓
Creates /dev/input/eventX via evdev

Key files in Linux kernel:

  • drivers/hid/hid-core.c - Main HID parser
  • drivers/hid/hid-input.c - Translates HID → Linux input events
  • drivers/hid/usbhid/ - USB transport for HID
  • drivers/hid/hid-generic.c - Generic fallback driver

Windows Kernel Flow:

USB device connects
  ↓
USBPORT.SYS (USB bus driver) detects device
  ↓
HIDCLASS.SYS queries Report Descriptor
  ↓
HIDUSB.SYS (HID miniport) handles USB communication
  ↓
HID.DLL exposes to user-space applications

Windows uses these drivers:

  • HIDCLASS.SYS - HID class driver
  • HIDUSB.SYS - USB transport for HID
  • HIDPARSE.SYS - Report Descriptor parser
  • HID.DLL - User-mode library

05 01 09 02 a1 01 09 01 a1 00 05 09 19 01 29 08
15 00 25 01 95 08 75 01 81 02 05 01 09 30 09 31
09 38 15 81 25 7f 75 08 95 03 81 06 c0 c0

What’s happening:

  1. Line 19: HIDIOCGRAWINFO gets vendor/product IDs
  2. Line 31: HIDIOCGRDESCSIZE gets descriptor size
  3. Line 37: HIDIOCGRDESC retrieves the actual Report Descriptor
  4. Line 43: We print it in hexadecimal (what you see above)

  • HidD_GetHidGuid() - Get HID device class GUID

  • HidP_GetCaps() - Get device capabilities

Common Misconceptions

❌ “HID is only for USB devices”

Wrong. HID is transport-agnostic:

  • USB HID - Most common (USB keyboards, mice)
  • Bluetooth HID - Wireless keyboards, mice, gamepads
  • I²C HID - Internal laptop touchpads
  • SPI HID - Some embedded touchscreens

HID is a protocol, not a physical connection.

❌ “All USB devices are HID”

Wrong. USB has many device classes:

  • HID (0x03) - Keyboards, mice, gamepads
  • Mass Storage (0x08) - USB drives, external hard disks
  • Audio (0x01) - USB microphones, speakers
  • Video (0x0E) - Webcams
  • CDC (0x02) - USB serial adapters

Only devices in the HID class (0x03) use HID protocol.

❌ “HID means it works without drivers”

Mostly true, but… The OS still needs:

  • HID class driver (built into all modern OSes)
  • Transport driver (USB, Bluetooth, I²C)

What you DON’T need:

  • Device-specific driver (for most devices)

Exceptions: Gaming mice with 20 buttons, flight sim controllers, etc., might need vendor-specific drivers for advanced features.

The Big Picture

Why HID Changed Everything

Before HID (1990s):

1 device = 1 custom driver × 3 operating systems
New device = months of driver development

After HID (1996+):

1 device = works on ALL operating systems instantly
New device = plug and play

HID’s Success Story

Adoption:

  • ✅ Windows 98+ (native support)
  • ✅ Linux 2.6+ (native support)
  • ✅ macOS (native support since OS X)
  • ✅ Android (native support)
  • ✅ PlayStation, Xbox, Nintendo Switch
  • ✅ Smart TVs, set-top boxes

Result: You can use the same keyboard on any device made in the last 25 years.

What HID Doesn’t Solve

HID handles device description and data format, but NOT:

  • Higher-level interpretation (What does “Button 1” mean in your game?)
  • Device-specific features (RGB lighting, macro keys)
  • Performance optimization (Polling rate, latency)

These are handled by:

  • Linux: libinput, game engines, desktop environments
  • Windows: DirectInput, XInput, Raw Input
  • Applications: Game-specific button mapping
