WTF is the X11 Clipboard?

You copy text. You close the app. You paste. Nothing. Welcome to X11.

On Windows and macOS, the clipboard is a system service — copy puts data in the OS, paste reads it back. The source app can exit and paste still works. Simple.

X11 does it completely differently. There is no central clipboard storage. When you “copy” text, the data stays in the source application’s memory. When another app wants to “paste,” it sends a request to the source app, which responds with the data. If the source app is dead, the data is gone.

This is not a bug. It’s a deliberate design decision from the 1980s that prioritizes lazy evaluation and network efficiency over convenience. Understanding how it works explains a whole category of clipboard weirdness on Linux.

The Core Analogy

On Windows, copying is like photocopying a document and putting it in a shared filing cabinet. Anyone can pull it out later, even if you leave the building.

On X11, copying is like raising your hand and saying “I have this document.” When someone wants it, they walk over and ask you for a copy. If you’ve already left the building, they get nothing.

The X server is just the receptionist — it tracks who raised their hand, but never holds the document itself.

Three Selections, Not One

X11 doesn’t have “a clipboard.” It has three independent selections:

PRIMARY — the implicit selection. Highlight text with your mouse and it’s automatically available. Paste with middle-click. No Ctrl+C needed. This is the Unix way.

CLIPBOARD — the explicit selection. Ctrl+C copies, Ctrl+V pastes. This is the Windows/macOS model, bolted onto X11 later.

SECONDARY — reserved for “two-argument” operations. Almost nothing uses it. Pretend it doesn’t exist.

Most modern Linux desktops primarily use CLIPBOARD. But terminal emulators, Vim, and many X11-native apps still use PRIMARY heavily. If you’ve ever highlighted text in a terminal and middle-clicked to paste in another window, that’s PRIMARY at work.

The Exchange

The X server translates this into a SelectionRequest event sent to the owner:

SelectionRequest {
    owner:     source_window     // who has the data
    requestor: dest_window       // who wants it
    selection: CLIPBOARD         // which selection
    target:    UTF8_STRING       // desired format
    property:  MY_PROPERTY       // where to put the result
    time:      ...
}

Format Negotiation with TARGETS

The owner might have data in multiple formats — plain text, HTML, an image. The requestor needs to know what’s available before asking.

The requestor first asks for TARGETS:

The owner responds with a list of atoms (format identifiers) it supports:

TARGETS response: [UTF8_STRING, STRING, text/html, image/png, TIMESTAMP]

The requestor picks the best format and requests it. This is why you can paste rich HTML from a browser into a word processor, or paste plain text into a terminal — the requestor chooses which format to ask for.

Common format atoms:

  • UTF8_STRING — modern standard for text
  • STRING — legacy ISO Latin-1 text
  • COMPOUND_TEXT — locale-aware text (legacy)
  • text/html — HTML content
  • image/png — image data
  • MULTIPLE — request several formats in one round trip

The INCR Protocol (Large Transfers)

X11 has a maximum request size (~256KB by default). For larger data, the INCR (incremental) protocol kicks in:

  1. Owner responds with property type INCR and a size estimate
  2. Owner deletes the property (signals start of chunking)
  3. Owner writes the first chunk (~256KB)
  4. Requestor reads it, then deletes the property (signals “ready for more”)
  5. Owner writes the next chunk
  6. Repeat until owner writes a zero-length chunk (signals completion)

Both sides synchronize via PropertyNotify events — each property deletion is a “go ahead” signal.

INCR Support Is Spotty

Many apps implement INCR for receiving but not for sending. Copying large data (images, big text blocks) silently fails if the owner doesn’t support INCR. Tools like xclip and xsel don’t support INCR sending either. For large data, use files instead of the clipboard.

Why Paste Fails

Now you understand the protocol, the failure modes make sense:

Source app closed — the owner exited, the X server has no one to route SelectionRequest to. The data was in the app’s memory and is now gone.

Source app is busy — the owner’s event loop is blocked (maybe a dialog box is open). It never processes the SelectionRequest. The requestor waits, then times out.

Format mismatch — the requestor asks for UTF8_STRING, but the owner only has STRING. The owner sets property to None in SelectionNotify. Some apps don’t fall back to alternative formats.

Race condition — user copies in app A, immediately copies in app B. App B claims CLIPBOARD ownership. App A’s data was never persisted anywhere.

Clipboard Managers: The Fix

Since the X server doesn’t store clipboard data, clipboard managers exist to fill the gap:

  1. A daemon watches for CLIPBOARD ownership changes
  2. When a new owner appears, the daemon immediately requests the data
  3. The daemon stores the data in its own memory
  4. The daemon then claims CLIPBOARD ownership itself
  5. The original app can now exit — the daemon has the data
App copies "hello"  →  App claims CLIPBOARD
                    →  Clipboard manager detects change
                    →  Manager requests data from app
                    →  Manager stores "hello" in its memory
                    →  Manager claims CLIPBOARD ownership
                    →  App can now exit safely
                    →  Later paste requests served by manager

Common clipboard managers: Klipper (KDE), GPaste (GNOME), Parcellite, CopyQ. Most desktop environments run one by default — which is why clipboard usually works fine on modern Linux desktops and only breaks in minimal WM setups.

Wayland: The Redesign

Wayland fixes the clipboard architecture fundamentally:

X11Wayland
StorageOwner’s memoryOwner’s memory (same)
TransferWindow properties + eventsKernel pipes (fd passing)
SecurityAny app can read any selectionOnly focused client gets clipboard events
Large dataINCR protocolPipe buffering (no size limit)
Format negotiationTARGETS atomMIME types

The key improvement is security: on X11, any application can request any selection at any time — a clipboard sniffer is trivial. On Wayland, only the focused window receives clipboard offers. The compositor mediates everything.

Data transfer uses file descriptors instead of window properties — the compositor pipes data directly from source to destination through the kernel. No INCR protocol needed, no size limits, no round-trip events.

The fundamental “owner holds the data” model is the same, though. If the source app dies before the paste, the data is still gone — clipboard managers are still needed.

Summary

X11 has no central clipboard. When you copy, the data stays in the source app’s memory. Paste sends a request to the source, which responds with the data on demand. This means closing the source app kills the clipboard data. Three selections exist (PRIMARY for mouse selection, CLIPBOARD for Ctrl+C, SECONDARY for nothing). Format negotiation happens via the TARGETS protocol. Large transfers use the INCR chunking protocol. Clipboard managers fix the “close app, lose data” problem by eagerly requesting and caching clipboard contents. Wayland keeps the same owner-based model but fixes security (focused-window-only access) and replaces the event-based transfer with kernel pipes.
