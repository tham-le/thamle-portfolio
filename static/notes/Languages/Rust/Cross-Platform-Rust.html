WTF is Cross-Platform Rust?

”Write once, run everywhere” is a lie. But Rust makes the truth manageable.

Rust compiles to native code on every major platform. The language is the same everywhere. The standard library is (mostly) the same everywhere. But the moment you touch the OS — file paths, TLS, GPU access, clipboard, input handling — everything diverges.

Cross-platform Rust isn’t about avoiding platform-specific code. It’s about organizing it: isolating the differences, sharing what’s common, and making the compiler enforce that you haven’t forgotten a platform.

The Core Analogy

Think of cross-platform development like a restaurant chain. The menu (your public API) is the same everywhere. The recipes (core logic) are the same. But the kitchen equipment differs by location — one restaurant has a gas stove, another has induction, another has a wood-fired oven.

Rust’s cfg system is like a kitchen manual that says: “If gas stove, do step 3a. If induction, do step 3b.” And the compiler checks that you’ve written instructions for every kitchen before the restaurant opens.

cfg Attributes: Compile-Time Platform Detection

The #[cfg(...)] attribute conditionally compiles code based on the target:

This isn’t runtime branching — the compiler excludes non-matching code entirely. When targeting Linux, the Windows and macOS versions don’t exist in the binary.

Available cfg Predicates

PredicateValuesExample
target_oslinux, windows, macos, ios, android#[cfg(target_os = "linux")]
target_archx86_64, aarch64, arm, wasm32#[cfg(target_arch = "aarch64")]
target_familyunix, windows, wasm#[cfg(unix)]
target_envgnu, musl, msvc#[cfg(target_env = "musl")]
target_pointer_width32, 64#[cfg(target_pointer_width = "64")]

Shorthand: #[cfg(unix)] matches Linux, macOS, FreeBSD, Android, iOS — anything POSIX-like. #[cfg(windows)] matches Windows.

The Platform Complexity Spectrum

Not all platforms are equal in difficulty:

PlatformComplexityWhy
LinuxLowNative target, full toolchain, everything is available
macOSLowSimilar to Linux, Rust works natively
WindowsMediumDifferent APIs, different linking (MSVC vs GNU), different conventions
iOSMediumNeeds Xcode, but Rust FFI is straightforward via C ABI
AndroidHighJNI bridge, AAR dependencies, Gradle build integration, NDK toolchain

Android is consistently the hardest platform. The JNI boundary between Rust and Java/Kotlin introduces a whole category of problems: runtime classpath issues, JNIEnv thread-locality, garbage collector interactions, and AAR packaging. See JNI for the details.

iOS is surprisingly simple in comparison. Rust talks to iOS frameworks through the C ABI — no bridge language, no runtime dependency management.

Testing Across Platforms

You can’t just test on Linux and ship to Windows. Platform-specific bugs hide in:

  • Path separators: / vs \. Use std::path::PathBuf everywhere.
  • Line endings: \n vs \r\n. Be explicit when parsing.
  • Case sensitivity: Linux filesystems are case-sensitive, Windows and macOS (default) are not.
  • Dynamic linking: .so vs .dll vs .dylib. Different search paths, different naming.
  • Signals vs messages: Unix uses signals; Windows uses the message queue.

The Honest Truth

Cross-platform Rust is not “free.” The language gives you powerful tools — cfg, conditional dependencies, cross-compilation — but the platform differences are real. TLS works differently. Input works differently. Clipboard works differently. Display servers work differently.

The goal isn’t to hide these differences behind a perfect abstraction. The goal is to isolate them so that 90% of your codebase is shared and the platform-specific 10% is well-organized and explicitly marked.

my_app/
├── src/
│   ├── core/           # Platform-independent logic (90%)
│   │   ├── protocol.rs
│   │   ├── encoding.rs
│   │   └── config.rs
│   ├── platform/       # Platform-specific code (10%)
│   │   ├── linux.rs
│   │   ├── windows.rs
│   │   ├── android.rs
│   │   └── ios.rs
│   └── main.rs         # cfg-based dispatch

Summary

Cross-platform Rust uses #[cfg(target_os = "...")] for compile-time platform detection, target-specific dependencies in Cargo.toml, and build scripts for platform-specific linking. The compiler ensures you handle every target — unmatched cfg blocks are compile errors, not runtime surprises. The real challenge isn’t the Rust code — it’s the platform differences underneath: TLS verification, clipboard protocols, display servers, and native interop. Android (JNI + AAR + Gradle) is consistently the hardest target. iOS is surprisingly simple. Organize your codebase so 90% is shared and the platform-specific 10% is isolated in clearly marked modules.
