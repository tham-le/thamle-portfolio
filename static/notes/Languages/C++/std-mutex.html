WTF is std::mutex?

std::mutex is C++‚Äòs mutual exclusion primitive. It ensures only one thread can access shared data at a time.

Think of it as a lock on a bathroom door - only one person can be inside at once.

The Bathroom Analogy

The Shared Bathroom

Without Mutex (Chaos)

Multiple people try to use the bathroom simultaneously:

  • Person A opens door
  • Person B also opens door (no lock!)
  • Both try to use bathroom at same time
  • Chaos and disaster

With Mutex (Order)

One person enters and locks the door:

  • Person A locks door, uses bathroom
  • Person B tries to enter ‚Üí door locked ‚Üí waits
  • Person A unlocks door and leaves
  • Person B locks door, uses bathroom

Only one person inside at a time. No collisions.

That‚Äôs std::mutex - a lock that ensures only one thread accesses critical code at a time.

The Problem: Data Races

Without synchronization, threads can corrupt shared data:

Output: 143219 or 187654 or any random number! Never 200000.

Why? counter++ is three operations:

  1. Read counter (e.g., 0)
  2. Add 1 (result: 1)
  3. Write back (counter = 1)

Threads can interleave these steps, causing lost updates. This is a Data-Race.

The Solution: std::mutex

Now it works! Only one thread modifies counter at a time.

Better: RAII with std::lock_guard

Manually calling lock() and unlock() is dangerous - what if you forget unlock()?

Use RAII:

Benefits:

  • Can‚Äôt forget to unlock
  • Exception-safe (unlocks even if exception thrown)
  • More readable

Always prefer std::lock_guard over manual lock/unlock.

std::unique_lock - More Flexible

When to use:

  • Need to unlock before scope ends
  • Conditional locking
  • Transfer lock ownership
  • Use with condition variables

std::scoped_lock - Lock Multiple Mutexes (C++17)

Deadlock-free! std::scoped_lock uses a deadlock avoidance algorithm.

Always use std::scoped_lock when locking multiple mutexes.

Common Pitfalls

1. Deadlock

Problem:

  • Thread 1 holds mtx1, waits for mtx2
  • Thread 2 holds mtx2, waits for mtx1
  • Both stuck forever!

Solution: Use std::scoped_lock or always lock in same order.

2. Forgetting to Unlock

Solution: Always use RAII (std::lock_guard, std::unique_lock).

Performance Considerations

Cost of Locking

Locking has overhead (~20-50 nanoseconds). Minimize lock acquisitions.

Contention

Too much contention (many threads waiting) ‚Üí slow.

Solutions:

  • Reduce critical section size
  • Use lock-free data structures (std::atomic)
  • Shard data (multiple mutexes for different parts)

The Big Takeaway

std::mutex in a Nutshell

std::mutex provides mutual exclusion - ensures only one thread accesses shared data at a time.

How to use:

Key points:

  • Prevents Data-Races
  • Use RAII (lock_guard, unique_lock) - never manual lock/unlock
  • Lock multiple mutexes with std::scoped_lock (deadlock-free)
  • Minimize critical section size
  • Consider std::atomic for simple counters

Common mistakes:

  • Forgetting to unlock ‚Üí use RAII
  • Deadlocks ‚Üí lock in consistent order or use scoped_lock
  • Holding locks too long ‚Üí minimize critical sections

The rule: When multiple threads access shared data, protect it with std::mutex + RAII locks. Keep critical sections small. For simple atomics, use std::atomic instead.
