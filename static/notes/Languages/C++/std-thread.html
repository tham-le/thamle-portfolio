WTF is std::thread?

WTF is std::thread?

Nov 06, 20257 min read

  • cpp
  • threading
  • concurrency
  • cpp11

std::thread is C++‘s standard way to create and manage operating system threads. It’s been part of C++ since C++11.

Think of it as spawning a separate worker that runs code in parallel with your main program.

The Kitchen Analogy

The Restaurant Kitchen

Single Thread (One Chef)

One chef cooks everything sequentially:

  • Chop vegetables (2 minutes)
  • Grill meat (5 minutes)
  • Boil pasta (8 minutes)

Total time: 15 minutes (everything happens one after another)

Multiple Threads (Three Chefs)

Three chefs work in parallel:

  • Chef 1: Chop vegetables (2 minutes)
  • Chef 2: Grill meat (5 minutes) - starts immediately
  • Chef 3: Boil pasta (8 minutes) - starts immediately

Total time: 8 minutes (limited by slowest task)

That’s threading - do multiple things at the same time by using multiple workers (CPU cores).

Creating a Thread

Basic Example

What happens:

  1. std::thread t(worker_function) creates a new Operating-System thread
  2. The new thread runs worker_function() immediately
  3. t.join() blocks the main thread until t finishes
  4. Program exits when main thread finishes

Thread Lifecycle

join() - Wait for Completion

Output:

Waiting...
Thread done      (after 2 seconds)
Main done

join() blocks until the thread finishes.

detach() - Let It Run Free

Danger: If main exits before the detached thread finishes, undefined behavior!

Rule: Only detach if you’re certain the thread will finish before program exit (e.g., daemon threads in long-running services).

The Destructor Rule

Fatal error: Destroying a std::thread that hasn’t been joined or detached terminates the program.

Always call join() or detach() before the thread object is destroyed.

Thread Safety Issues

Data Races

Problem: counter++ is three operations:

  1. Read counter
  2. Add 1
  3. Write counter

Threads can interleave, causing lost updates.

Solution: Use std-mutex or std::atomic.

std::thread vs. Alternatives

ApproachUse CaseCostScalability
std::threadCPU-bound parallelismHigh (OS thread)Hundreds max
Boost-AsioI/O-bound asyncLow (event loop)Thousands
co_awaitAsync without blockingLow (coroutine)Thousands
std::asyncQuick parallel tasksHigh (thread pool)Hundreds

Rule of thumb:

  • CPU-bound work (math, compression) → std::thread
  • I/O-bound work (network, disk) → Boost-Asio or co_await

Performance Considerations

Thread Creation Cost

Creating an OS thread is expensive (~1-2ms overhead, ~2MB stack Memory).

Context Switching

Too many threads → excessive Context Switches → slower than single-threaded!

Optimal: Number of threads ≈ Number of CPU cores (for CPU-bound work).

Common Pitfalls

1. Forgetting to Join

2. Accessing Destroyed Variables

Fix: Capture by value or use std::shared_ptr.

3. Exception Safety

Fix: Use RAII wrapper or std::jthread (C++20).

C++20: std::jthread (Better std::thread)

std::jthread fixes the “forgot to join” problem. Prefer it in C++20+ code.

The Big Takeaway

std::thread in a Nutshell

std::thread is C++‘s standard way to create Operating-System threads for parallel execution.

Key points:

  • Creates real OS thread (expensive: ~1-2ms, ~2MB stack)
  • Must call join() (wait) or detach() (run free) before destruction
  • Use for CPU-bound parallel work (not I/O!)
  • Beware Data-Races - use std-mutex or std::atomic
  • Consider thread pools instead of creating threads per task

Alternatives:

  • std::jthread (C++20) - auto-joining
  • Boost-Asio + co_await - async I/O (lighter weight)
  • std::async - quick fire-and-forget tasks

The rule: std::thread gives you raw OS threads. Great for CPU parallelism, but heavy. For I/O, use Boost-Asio. For safety, use std::jthread in C++20+.
