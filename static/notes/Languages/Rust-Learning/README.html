This is a 30-day intensive program to reforge a C/C++ developer into a Rust Multimedia Engineer. Our textbook is the digital_video_introduction repository. Our goal is not just to learn Rust, but to learn how to build the very guts of video technology with it.

Week 1: Rust Fundamentals Through a Media Lens

The Goal: Master Rust‚Äôs core concepts, but apply them immediately to the domain of digital video. We won‚Äôt be making User structs; we‚Äôll be making Pixel and Frame structs. The fundamentals are the same, but the context is everything.

Day 1: The Pixel & The Toolchain

Objective: Set up your environment and handle the most basic unit of video: the pixel. Primary Resources:

  • Rust_book_link: Chapters 1, 2, 3.
  • digital_video_introduction: Section ‚ÄúBasic terminology‚Äù.

Daily Checklist: Definition of Done

  • Environment is perfect: rustup, cargo, rust-analyzer are installed.
  • Create a [[Rust - Structs]] note. Inside, define and implement a Pixel { r: u8, g: u8, b: u8 } struct.
  • Write a function that takes two Pixels and blends them.
  • Create a [[Rust - Enums and Pattern Matching]] note. Define an enum ColorModel { RGB, YCbCr }.
  • Write a basic program that creates a 10x10 ‚Äúimage‚Äù (a Vec<Pixel>) and fills it with a color.

Day 2: The Frame & The Borrow Checker

Objective: Understand Rust‚Äôs ownership and borrowing by managing a full frame of video data. Primary Resources:

  • The Rust Book: Chapter 4 (Ownership).
  • digital_video_introduction: Section ‚ÄúBasic terminology‚Äù.

Daily Checklist: Definition of Done

  • Create a struct Frame { width: u32, height: u32, pixels: Vec<Pixel> }.
  • Write a function that borrows a Frame (&Frame) and calculates its average brightness. This function must NOT take ownership.
  • Write a function that takes a mutable borrow of a Frame (&mut Frame) and inverts its colors.
  • In your [[Rust - Ownership]] note, explain why passing a raw pointer to a frame buffer in C is dangerous and how Rust‚Äôs borrow checker prevents that entire class of bugs.

Day 3: From RGB to YCbCr - Traits & Generics

Objective: Implement color model conversions using Rust‚Äôs core abstraction tools: traits. Primary Resources:

  • The Rust Book: Chapter 10 (Generics, Traits, Lifetimes).
  • digital_video_introduction: Section ‚ÄúConverting between YCbCr and RGB‚Äù.

Daily Checklist: Definition of Done

  • Define a trait ConvertibleColor with methods like to_ycbcr() and from_ycbcr().
  • Implement this trait for your Pixel struct.
  • Create new structs YCbCrPixel { y: u8, cb: u8, cr: u8 }.
  • Write a generic function that can operate on any Frame containing pixels that implement ConvertibleColor.
  • Write the conversion logic from the DVI repo as a pure Rust function.

Day 4: Chroma Subsampling - Slices & Lifetimes

Objective: Perform a real video compression technique while mastering slices and lifetimes. Primary Resources:

  • The Rust Book: Re-read Chapter 4.3 (Slices) and 10.3 (Lifetimes).
  • digital_video_introduction: Section ‚ÄúChroma subsampling‚Äù.

Daily Checklist: Definition of Done

  • Wrote a function subsample_420(y_plane: &[u8], u_plane: &[u8], v_plane: &[u8]) that correctly downsamples the chroma planes.
  • Wrote a function that takes slices of a frame‚Äôs pixel data to operate on a specific region, avoiding unnecessary copies.
  • Explained in your [[Rust - Lifetimes]] note how lifetimes would prevent you from returning a pointer to a subsampled chroma plane that outlives the original frame data.

Day 5-7: Consolidation Project I - A Simple Image Parser/Converter

Objective: Build a command-line tool that can read a simple, uncompressed image format and apply the concepts from this week. Primary Resources:

  • The Rust Cookbook: Recipes for File I/O and CLI arguments.
  • The PPM image format spec (it‚Äôs one of the simplest raw image formats).

Project Checklist: Definition of Done

  • The CLI tool accepts an input file path and an output file path.
  • It can parse a simple P6 .ppm file into your Frame struct.
  • It can convert the frame from RGB to YCbCr.
  • It can perform 4:2:0 chroma subsampling on the YCbCr data.
  • It can write the modified image planes to separate output files (e.g., output.y, output.u, output.v).
  • All I/O operations use Result for robust [[Rust - Error Handling (Result and Option)]].

Week 2: Deconstructing the Codec Pipeline

The Goal: Implement simplified versions of the core stages of a video codec. This week is a deep dive into the theory from digital_video_introduction, bringing it to life with Rust code.

Day 8: Frame Types & Inter Prediction

Objective: Model I, P, and B frames and implement a basic block-based motion search. Primary Resources:

  • digital_video_introduction: Sections ‚ÄúFrame types‚Äù and ‚ÄúTemporal redundancy‚Äù.

Daily Checklist: Definition of Done

  • Created an enum FrameType { I, P, B }.
  • Wrote a function find_best_match(block: &[Pixel], search_area: &Frame) -> (x, y) that performs a simple Sum of Absolute Differences (SAD) search.
  • The result of the function is a motion vector. Create a struct MotionVector { x: i16, y: i16 }.
  • Wrote a function to calculate the ‚Äúresidual‚Äù block by subtracting the matched block from the original.

Day 9: The Transform - Implementing a DCT

Objective: Understand and implement the Discrete Cosine Transform. Primary Resources:

  • digital_video_introduction: Section ‚Äú3rd step - transform‚Äù.
  • Any online resource explaining the 1D and 2D DCT-II formulas.

Daily Checklist: Definition of Done

  • Wrote a function that performs a 1D DCT on a Vec<f64>.
  • Used the 1D function to write a function that performs a 2D DCT on an 8x8 block of pixel data (as a [[f64; 8]; 8] array).
  • Wrote the inverse transform (IDCT) as well.
  • Verified that IDCT(DCT(block)) results in the original block (with some floating-point error).

Day 10: Quantization & Entropy Coding

Objective: Implement the lossy and lossless compression steps. Primary Resources:

  • digital_video_introduction: Sections ‚Äú4th step - quantization‚Äù and ‚Äú5th step - entropy coding‚Äù.

Daily Checklist: Definition of Done

  • Wrote a function that performs quantization and de-quantization on an 8x8 DCT coefficients block, using a quantization matrix.
  • Implemented a simple Run-Length Encoding (RLE) function for the quantized, zig-zag scanned coefficients.
  • Explored a Rust crate for Huffman or Arithmetic coding (like huffman-compress) and used it on the RLE output.

Day 11-14: Consolidation Project II - The ‚ÄúToy‚Äù Encoder

Objective: Combine all the pieces from this week into a single program that performs a rudimentary intra-frame or inter-frame encode. Primary Resources:

  • All your code from this week.

Project Checklist: Definition of Done

  • The program can take a single 8x8 block of pixel data.
  • Path A (Intra): It performs DCT ‚Üí Quantization ‚Üí RLE ‚Üí Entropy Coding and writes the compressed bytes to a file.
  • Path B (Inter): Given two blocks, it finds a motion vector, calculates the residual, and then performs the DCT‚ÜíQuant‚Üí‚Ä¶ pipeline on the residual.
  • You have a corresponding ‚ÄúToy Decoder‚Äù that can read the file, reverse the process, and reconstruct the block.
  • You can calculate the compression ratio you achieved.

Week 3: The Bitstream & Container Sprint

The Goal: Understand how compressed data is actually structured and packaged. This is where we move from abstract algorithms to parsing real-world data formats.

Day 15-17: Bit-Level Manipulation & Parsing with nom

Objective: Learn to read and write data at the bit level and use a real parsing library. Primary Resources:

  • The bitvec crate documentation.
  • The nom crate documentation and tutorials.
  • digital_video_introduction: Section ‚Äú6th step - bitstream format‚Äù.

Daily Checklist: Definition of Done

  • Used bitvec to write a sequence of variable-length codes to a Vec<u8>.
  • Wrote a nom parser to read a simple binary format (e.g., the header of your ‚ÄúToy Encoder‚Äù file).
  • Studied the structure of an H.264 NAL unit header.
  • Wrote a nom parser that can parse the NAL unit type from the first byte of a NAL unit.

Day 18-21: Consolidation Project III - MP4 Box Parser

Objective: Write a tool to inspect a real-world video container format. MP4 is built on ‚Äúboxes‚Äù (or ‚Äúatoms‚Äù), which is a perfect parsing exercise. Primary Resources:

  • The ISO Base Media File Format specification (ISO/IEC 14496-12). Look for a summary online.
  • ffmpeg or ffprobe to validate your tool‚Äôs output.

Project Checklist: Definition of Done

  • The tool takes the path to an .mp4 file.
  • It can identify and parse the top-level boxes (ftyp, moov, mdat).
  • For each box, it prints the 4-character code and its size.
  • It can recursively descend into the moov box and print the hierarchy of boxes within it (e.g., mvhd, trak).
  • Your tool‚Äôs output for a given file roughly matches the structure shown by ffprobe.

Week 4: Streaming, Performance, & Job Prep

The Goal: Bridge the gap from file-based processing to live streaming and prepare for interviews.

Day 22-26: Final Portfolio Project - A Streaming Tool

Objective: Build your capstone project that you will show to employers. Primary Resources:

  • tokio crate documentation.
  • digital_video_introduction: Section ‚ÄúOnline streaming‚Äù.

Project Ideas (Choose One):
