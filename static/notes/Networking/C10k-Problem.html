WTF is the C10k Problem?

The C10k problem refers to the challenge of handling 10,000 concurrent connections on a single server. In the late 1990s, this was considered impossible.

The name comes from ‚ÄúConcurrent 10k‚Äù connections.

The problem: Traditional ‚Äúone thread per connection‚Äù architecture collapses at high concurrency due to Context-Switch overhead and Memory exhaustion.

The solution: O with event loops (epoll, kqueue, IOCP).

The Restaurant Analogy

Serving 10,000 Customers

Traditional Approach (One Thread Per Connection)

Model: One waiter per customer

  • Customer 1 ‚Üí Waiter 1 (waits while customer decides)
  • Customer 2 ‚Üí Waiter 2 (waits while customer decides)
  • ‚Ä¶
  • Customer 10,000 ‚Üí Waiter 10,000 (waits‚Ä¶)

Problems:

  • Need 10,000 waiters (expensive!)
  • Waiters keep bumping into each other (Context-Switch)
  • Most waiters are idle (waiting)

Event-Driven Approach (epoll / Non-blocking-IO)

Model: One manager + event system

  • Manager checks: ‚ÄúWho‚Äôs ready to order?‚Äù
  • Event system: ‚ÄúTable 5, 23, 107 are ready!‚Äù
  • Manager serves only those tables
  • Move to next ready tables

Benefits:

  • One manager handles thousands
  • No idle waiting
  • Efficient CPU usage

That‚Äôs the C10k solution - event-driven architecture instead of thread-per-connection.

The Problem (Late 1990s)

Thread-Per-Connection Model

Why it fails at 10,000 connections:

Problem 1: Memory Exhaustion

Each thread needs:
- Stack: 2-8 MB
- Kernel structures: ~10 KB

10,000 threads √ó 2 MB = 20 GB RAM!

Most servers in 1999 had < 1 GB RAM.

Problem 2: Context Switch Overhead

10,000 threads competing for 1-4 [[CPU]] cores
‚Üí Constant [[Context-Switch]]ing
‚Üí ~10,000 context switches/second
‚Üí [[CPU]] spends more time switching than working!

Problem 3: Scalability Limit

OS limits:
- Max threads: ~1,000-2,000 (Linux 2.4)
- Max file descriptors: 1,024 (default ulimit)

You physically couldn‚Äôt create 10,000 threads on most systems.

The Solution: Event-Driven I/O

Key Insight

Most connections are idle!

Typical web server:

  • 10,000 connections open
  • ~100 actively sending/receiving data
  • 9,900 waiting for data

Don‚Äôt waste threads on idle connections.

Event Loop Pattern

Benefits:

  • One thread handles all connections
  • Only processes ready connections
  • No Context-Switch overhead
  • Constant Memory usage

Modern Descendants: C10M and C10G

C10M Problem (2013)

Challenge: 10 million concurrent connections on a single server.

Solution:

  • Kernel bypass (DPDK, io_uring)
  • User-space networking
  • Lock-free data structures
  • NUMA-aware design

Achieved by: Custom network stacks, specialized hardware.

C10G Problem (Future)

Challenge: 10 billion connections (hypothetical).

Solutions:

  • Distributed systems
  • Edge computing
  • Protocol optimizations (QUIC)

Common Mistakes

Mistake 1: Blocking in Event Loop

Fix: Offload CPU work to worker threads.

Mistake 2: Too Many Threads

Fix: Use event-driven I/O or thread pool.

Mistake 3: Forgetting to Configure Limits

Also configure: fs.file-max, net.core.somaxconn in /etc/sysctl.conf.

The Big Takeaway

C10k Problem in a Nutshell

The C10k problem was the challenge of handling 10,000 concurrent connections on a single server (late 1990s).

Traditional approach failed:

  • Thread-per-connection ‚Üí memory exhaustion (20 GB for 10k threads)
  • Excessive Context-Switching ‚Üí CPU overhead
  • OS limits (max threads, file descriptors)

Solution: Event-driven I/O

  • One thread + event loop (epoll, kqueue, IOCP)
  • Non-blocking-IO - only process ready connections
  • O(1) ready-event notification
  • Constant Memory usage

Modern implementations:

  • Nginx, Node.js (event loop)
  • Boost-Asio (C++20 co_await)
  • Tokio (Rust async/await)

Today: Solved problem. C10M (10 million) is the new frontier.

The rule: For high-concurrency servers, use event-driven I/O (epoll, Boost-Asio, async/await) instead of thread-per-connection. Modern frameworks have solved C10k - you just need to use Non-blocking-IO.

See also: epoll, Non-blocking-IO, Boost-Asio, co_await, Context-Switch
