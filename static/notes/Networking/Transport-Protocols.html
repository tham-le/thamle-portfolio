WTF are Transport Protocols?

It’s the menu of options for getting your data from point A to point B, with different tradeoffs

The Shipping Company Analogy

You need to send a package across the country. You have options:

Regular Mail (UDP): Cheap, fast, but sometimes stuff gets lost. No tracking. You just drop it in the mailbox and hope for the best.

Certified Mail (TCP): Every package is tracked, signed for, and guaranteed to arrive. Slower and more expensive, but rock solid reliable.

Express Courier (QUIC): Like certified mail, but the courier can take multiple routes simultaneously, switch vehicles mid-delivery, and doesn’t get stuck waiting if one road has traffic.

Carrier Pigeon (SCTP): Weird, specialized, rarely used, but perfect for specific situations like carrying messages between mobile cell towers.

These are your transport protocols. They all move data between applications, but with wildly different guarantees and performance characteristics.

Why Do We Need Different Transports?

The Network-Layer (IP) gets packets from computer to computer. But IP is unreliable, packets arrive out of order, things get lost. Applications need more.

The Transport Layer sits on top of IP and provides different services:

  • Reliability: Guaranteed delivery
  • Ordering: Data arrives in sequence
  • Flow control: Don’t overwhelm the receiver
  • Congestion control: Don’t overwhelm the network
  • Multiplexing: Multiple apps sharing one IP via ports

But here’s the thing: not every application needs all of these. A video stream doesn’t care if one frame gets lost. A database transaction absolutely cannot tolerate any loss.

So we have multiple transport protocols, each optimized for different use cases.

The Classic Duo: TCP and UDP

These are the original transport protocols from the 1980s, and they’re still dominant today.

TCP: Transmission Control Protocol

Philosophy: Reliability above all else. Every byte will arrive, in order, or the connection dies trying.TCP is a connection-oriented, end-to-end reliable protocol designed to fit into a layered hierarchy of protocols which support multi-network applications.

Key features:

  • Connection-oriented: Three-way handshake before data flows
  • Reliable: Every byte is acknowledged, lost packets are retransmitted
  • Ordered: Data arrives in sequence
  • Flow control: Receiver tells sender how much it can accept
  • Congestion control: Backs off when the network is overloaded

Header size: 20-60 bytes

Use cases:

  • Web browsing (HTTP/HTTPS)
  • File transfers (FTP, SFTP)
  • Email (SMTP, IMAP)
  • SSH, databases, anything where losing data would be catastrophic

The cost: Latency. The handshake, acknowledgments, and retransmissions add delay.

UDP: User Datagram Protocol

Philosophy: Speed over everything. Send the packet and pray.UDP is a connectionless protocol, meaning messages are sent without negotiating a connection and UDP does not keep track of what it has sent. It provides checksums for data integrity and port numbers for addressing, but has no handshaking dialogues and no guarantee of delivery, ordering, or duplicate protection.

Key features:

  • Connectionless: No handshake, just send
  • Unreliable: No ACKs, no retransmission, packets can get lost
  • Unordered: Packets might arrive out of sequence
  • No flow control: Fire hose mode, sender doesn’t care if receiver is overwhelmed
  • No congestion control: Application’s problem, not UDP’s

Header size: 8 bytes (tiny!)

Use cases:

  • DNS queries (one request, one response)
  • Video/audio streaming (lost frames are better than delayed frames)
  • Online gaming (fast updates matter more than perfect accuracy)
  • VoIP (real-time voice calls)
  • IoT sensors (small status updates)
  • DHCP (getting an IP address at boot)

The benefit: Speed and simplicity. No overhead, no delays, no baggage.

The Modern Contenders

TCP and UDP dominated for decades. But the internet evolved, and their limitations became painful. Enter the new generation.

SCTP: Stream Control Transmission Protocol

Philosophy: TCP, but better for specific use cases like telephony.SCTP is a reliable transport protocol that provides the message-oriented feature of UDP while ensuring reliable, in-sequence transport of messages with congestion control like TCP. Unlike UDP and TCP, the protocol supports multihoming and redundant paths to increase resilience and reliability.

Key features:

  • Multi-streaming: Send multiple independent data streams over one connection without head-of-line blocking
  • Multihoming: One endpoint can have multiple IP addresses; automatic failover if one path fails
  • Message-oriented: Like UDP, preserves message boundaries
  • Reliable: Like TCP, with ACKs and retransmission
  • Better security: Four-way handshake instead of three (harder to SYN flood)

Use cases:

  • Telephony signaling (SS7 over IP)
  • Mobile networks (4G/5G control plane)
  • WebRTC data channels
  • Anywhere you need TCP’s reliability but can’t afford head-of-line blocking

The problem: Almost nobody uses it. Middleboxes (firewalls, NATs) don’t understand SCTP and often block it. It’s technically superior but practically unusable on the public internet.

QUIC: Quick UDP Internet Connections

Philosophy: TCP’s reliability, but faster and built for the modern web.This is getting long. Let me create a complete, focused article that covers all the major transport protocols with clear comparisons. I’ll continue with the key protocols and wrap up with practical guidance.

QUIC was developed by Google in 2012 and standardized by the IETF in RFC 9000 in May 2021. It’s a UDP-based, stream-multiplexing, encrypted transport protocol that provides flow-controlled streams for structured communication, low-latency connection establishment, and network path migration.

Key features:

  • Built on UDP: Avoids TCP’s ossification problem (middleboxes can’t mess with it)
  • Always encrypted: TLS 1.3 baked in, everything encrypted by default
  • 0-RTT connection resumption: Clients can send data immediately on reconnection
  • 1-RTT initial connection: Faster than TCP+TLS (which takes 2-3 RTTs)
  • Stream multiplexing: Multiple independent streams, no head-of-line blocking
  • Connection migration: Change IP addresses (WiFi to cellular) without breaking the connection

The magic trick: QUIC combines what used to be three separate layers (TCP + TLS + HTTP/2) into one protocol on top of UDP.

Use cases:

  • HTTP/3 (the new web)
  • Video streaming (YouTube, Netflix experimenting)
  • Gaming
  • Any modern application that needs speed and security

Adoption: As of 2023, HTTP/3 represents about 25% of global internet traffic, primarily used by Google, Microsoft, and Facebook services.

HTTP/3: Not Really a Transport Protocol

Wait, HTTP/3? That’s an application protocol, but it’s worth mentioning because it’s built specifically for QUIC and is driving QUIC adoption.

HTTP/1.1 → runs on TCP, one request at a time per connection
HTTP/2 → runs on TCP, multiplexes requests but suffers from TCP’s head-of-line blocking
HTTP/3 → runs on QUIC, multiplexes requests with no head-of-line blocking

HTTP/3 is what’s making the web faster. When you load a website today, there’s a good chance it’s using HTTP/3 over QUIC.

Head-of-Line Blocking: The Problem QUIC Solves

This deserves special attention because it’s the killer feature of both SCTP and QUIC.

TCP’s problem: TCP is a single byte stream. If packet 5 gets lost, packets 6, 7, 8 sit in the kernel buffer waiting. Even if they arrived perfectly, the application can’t see them until packet 5 is retransmitted.

Imagine you’re loading a web page with 10 images. With HTTP/2 over TCP:

  • All 10 images share one TCP connection
  • Image 3’s packet gets lost
  • Images 4-10 are stuck waiting, even though their packets arrived fine
  • The whole page loads slower

QUIC’s solution: Each stream is independent. If image 3’s packet gets lost, only image 3 waits for retransmission. Images 4-10 keep flowing.

This is huge for performance on lossy networks (mobile, WiFi).

Connection Migration: Mobile’s Best Friend

Your phone switches from WiFi to cellular. What happens?

TCP: Your IP address changed. All TCP connections die. Every app reconnects from scratch.

QUIC: Uses Connection IDs instead of the IP address + port tuple to identify connections. When your IP changes, QUIC just keeps going. Seamless handoff.

This is why video calls and downloads don’t drop when you leave WiFi range.

The Ossification Problem

Why did we need QUIC when we could have just improved TCP?

Ossification: TCP’s header is unencrypted. For decades, middleboxes (firewalls, load balancers, NATs) have been inspecting and modifying TCP headers. Any change to TCP breaks these middleboxes.

Example: TCP Fast Open (a great feature from 2011) is still barely used because middleboxes drop packets with unknown options.

QUIC’s solution: Encrypt everything except the absolute minimum. Middleboxes can’t peek inside, so they can’t break future improvements. QUIC can evolve without permission.

How to Choose a Transport Protocol

Here’s a decision tree:

Do you need reliability?

  • No → UDP (DNS, status updates, sensors)
  • Yes → Keep reading

Do you need multiple independent streams?

  • Yes → QUIC (modern web apps, multiplayer games)
  • No → Keep reading

Do you need the connection to survive IP changes?

  • Yes → QUIC (mobile apps, long-lived connections)
  • No → Keep reading

Are you building something new with no legacy constraints?

  • Yes → QUIC (it’s better in almost every way)
  • No → TCP (universal compatibility, proven, works everywhere)

Are you in telecom/telephony?

  • Maybe → SCTP (if middleboxes support it)
  • Otherwise → QUIC

The Future

QUIC is winning. HTTP/3 adoption is growing fast. Major browsers, CDNs, and services all support it.

TCP isn’t going anywhere. Too much legacy, too universal, too embedded in kernels. It’ll be around for decades.

UDP remains essential. For truly latency-sensitive applications (VoIP, gaming, live streaming), nothing beats “just send it and hope.”

SCTP is niche. Technically excellent, practically blocked by middleboxes. Used in telecom where you control the network.

The Big Picture

Transport protocols are about tradeoffs:

TCP: Reliability at the cost of speed
UDP: Speed at the cost of reliability
SCTP: Flexibility at the cost of deployability
QUIC: Modern features at the cost of… actually, QUIC is just better, but requires recent infrastructure

Choosing Your Transport Building something new? Use QUIC if you can, TCP if you must.

Need raw speed? UDP, but handle reliability yourself.
Sending occasional small messages? UDP (like DNS does).
Transferring large files? TCP or QUIC.

The transport layer is where the rubber meets the road. Pick the wrong protocol and your application suffers. Pick the right one and everything just works.
