WTF is a Mesh Network?

The final boss. Everything we built clicks into place here.

You’ve made it through four articles. You know how TUN devices intercept packets, how NAT traversal punches direct paths between peers, how a control plane distributes keys and endpoints, and how WireGuard’s Noise handshake encrypts every byte in transit.

Now let’s zoom out and look at what all of that assembles into: a network topology. Because the technology stack we’ve built doesn’t mandate a specific shape. You can use it to build a centralized hub-and-spoke network, or a distributed full mesh. The choice has massive practical consequences for latency, throughput, resilience, and operational complexity.

This is the article where Tailscale, NetBird, and WireGuard’s raw mode finally get compared at the architecture level.

The Core Analogy

Hub-and-spoke is like a regional airport system. Every flight goes through a hub city. San Francisco to Denver? San Francisco to hub, hub to Denver. It doesn’t matter that a direct flight would take 2 hours; you’re flying through Chicago. The hub is operationally simple to manage, but every trip has extra distance built in.

Mesh networking is like having a car and a road atlas. You go directly from where you are to where you want to be, taking the optimal path. The coordination is harder (you need the atlas), but every trip is as short as it can possibly be.

The internet was designed as a mesh. The modern VPN industry mostly built hubs on top of it by accident, because hubs are easier to configure. WireGuard’s lightweight tunnels and tools like Tailscale exist to put the mesh back.

Hub-and-Spoke: How Traditional VPNs Work

In a hub-and-spoke VPN, every device connects to a central server. That server routes all traffic between clients.

      VPN Server (hub)
      203.0.113.1:1194
           |
    -------+---------+----------+
    |                |          |
 Laptop A         Laptop B    Server C
192.168.1.0/24   10.0.0.0/24  172.16.0.0/16

When Laptop A wants to reach Server C:

Laptop A -> [encrypted] -> VPN Server -> [encrypted] -> Server C

The packet goes to the hub first, gets decrypted, re-encrypted, and forwarded. Even if Laptop A and Server C are in the same building, the packet physically travels to the hub datacenter and back.

This model is operationally simple. You configure one server. Every client gets one config: “connect to the hub.” The hub handles all routing decisions.

The problems:

Latency. Every hop through the hub adds round-trip time. Laptop A in Paris talking to Server C also in Paris but through a hub in Virginia: 200ms+ instead of sub-1ms.

Single point of failure. The hub goes down, the whole network goes down.

Bandwidth bottleneck. All traffic is funneled through one machine. As your network grows, the hub becomes the limiting factor.

Scale. Hub-and-spoke with WireGuard means manually maintaining peer lists on the hub for every client, and every client needs a static route pointing at the hub. With 100 devices, you have 100 peer configs to manage on one machine.

Full Mesh: Every Peer Talks Directly

In a full mesh, every device has a direct encrypted tunnel to every other device it needs to reach. No intermediary.

Laptop A --- WireGuard tunnel --- Laptop B
   |                                  |
   +---- WireGuard tunnel ----+        |
   |                          |        |
Server C ---- WireGuard tunnel --- Server D

When Laptop A sends to Server C, the packet travels directly:

Laptop A -> [encrypted WireGuard] -> Server C

No hub. Optimal path. If both are on the same LAN, the traffic never leaves the LAN. If they’re in different datacenters, they take the direct internet path between them.

The scaling problem with raw WireGuard:

For N devices, a full mesh requires each device to maintain N-1 peer configurations. For 10 devices, that’s 9 peer entries per device, and 90 total configuration relationships to maintain. Add a device: update all 9 existing devices. Remove a device: update all 9. Rotate a key: update all 9.

At 10 devices this is annoying. At 50 devices it’s a full-time job. At 500 devices it’s impossible to maintain manually.

This is exactly the gap that Tailscale, NetBird, and similar tools fill. They give you full mesh topology with automatic configuration management.

The Hybrid Architecture: Centralized Control, Distributed Data

The insight that makes modern mesh VPNs work is separating the two concerns:

Control plane: centralized, low traffic, handles all the coordination. Key distribution, endpoint tracking, policy enforcement. This can be a hosted service or self-hosted. It carries bytes per second.

Data plane: fully distributed, peer-to-peer, handles all actual data. WireGuard UDP tunnels between every pair of communicating devices. This is where gigabits flow.

                  Coordination Server
                  (control plane only)
                        |
            +-----------+-----------+
            |           |           |
         Device A    Device B    Device C
            |                       |
            +-- Direct WireGuard ----+
                  (data plane)

The coordination server is the hub, but only for metadata: “Device C’s current endpoint is 203.0.113.55:41641, here’s its public key.” Once devices have that information, they establish direct WireGuard tunnels and data flows directly between them. The coordination server becomes irrelevant to data traffic.

The Coordination Server Is Not a VPN Concentrator

In a traditional hub-and-spoke VPN, the server handles data. In Tailscale/NetBird, the coordination server handles only control messages. Your data never passes through Tailscale’s or NetBird’s servers under normal operating conditions. The only exception is the DERP relay fallback when NAT traversal fails, and even then the relay only forwards already-encrypted WireGuard packets it cannot decrypt.

How the Full Stack Works Together

Let’s trace a complete packet journey through the entire stack we’ve built across this series. Alice’s laptop (100.64.0.1) wants to reach Bob’s server (100.64.0.2).

Phase 1: Bootstrap (Control Plane)

Before any packet flows, the control plane has already done its work:

  1. Alice installs the Tailscale/NetBird client. It generates a Curve25519 key pair. The public key is registered with the coordination server.
  2. Bob’s server does the same.
  3. The coordination server pushes a network map to Alice: “Peer 100.64.0.2 has public key BBBB..., current endpoint 203.0.113.99:41641.”
  4. The agent on Alice’s machine applies this to WireGuard: wg set wg0 peer BBBB... allowed-ips 100.64.0.2/32 endpoint 203.0.113.99:41641

Phase 2: First Packet (NAT Traversal + Handshake)

Alice’s application calls send() to 100.64.0.2:

  1. TUN device: The kernel routes the packet to wg0 per the routing table. The packet appears as raw bytes at WireGuard’s TUN file descriptor. (See TUN-TAP)

  2. Cryptokey lookup: WireGuard looks up 100.64.0.2 in its AllowedIPs table. Matches peer BBBB.... (See WireGuard-Protocol)

  3. No active session: WireGuard has no live session with BBBB.... It queues the packet and initiates a handshake.

  4. NAT traversal: The agent fires STUN probes to discover Alice’s current public endpoint and simultaneously sends hole-punch probes to Bob’s endpoint. (See NAT-Traversal)

  5. Noise handshake: WireGuard sends a Handshake Initiation message to 203.0.113.99:41641. The message contains Alice’s ephemeral public key and her static public key encrypted under Bob’s static public key. Bob responds with his own Handshake Response. Shared session keys are derived. (See WireGuard-Protocol)

  6. Key confirmation: Alice sends the queued packet, now encrypted with the session key, as the first transport data message. This serves as key confirmation.

Phase 4: Key Rotation (Perfect Forward Secrecy)

After 180 seconds (REKEY_AFTER_TIME) or after approximately 2^60 packets, WireGuard automatically re-runs the handshake. New ephemeral keys are generated. New session keys are derived. The old session keys are zeroed from memory. An attacker who records today’s traffic and compromises Alice’s static private key years later cannot decrypt that recorded traffic, because the ephemeral keys that generated those session keys no longer exist.

When Mesh Breaks: The Relay Fallback

A full mesh is the goal, but not always achievable. Two devices both behind symmetric NAT cannot hole-punch. A device on a corporate network with all outbound UDP blocked cannot reach a peer directly.

In these cases, Tailscale’s DERP relay and NetBird’s relay server step in. The topology degrades gracefully from full mesh to relay-assisted:

Best case:  Alice --- [direct WireGuard UDP] --- Bob

Fallback:   Alice --- [WireGuard via DERP relay] --- Bob
                           (still E2E encrypted)

The relay doesn’t introduce a security regression. The DERP server or NetBird relay only sees encrypted WireGuard ciphertext. But it does introduce latency. Tailscale periodically retries direct connectivity in the background and upgrades the connection when network conditions improve.

Checking Your Connection Type

On Tailscale: tailscale status shows each peer’s connection type. On NetBird: netbird status shows whether connections are direct or relayed. On raw WireGuard: wg show shows the endpoint and latest handshake time. If a handshake completed and data is flowing, the path is direct.

The Complete Picture

Across five articles, we’ve built the full mental model:

Layer 1: TUN/TAP (TUN-TAP) The kernel interface. A process opens /dev/net/tun, registers a virtual network interface, and intercepts all packets routed through it. WireGuard sits in a read/write loop on this file descriptor, receiving outgoing packets and injecting incoming ones.

Layer 2: NAT Traversal (NAT-Traversal) The connectivity layer. STUN discovers each peer’s public endpoint through their NAT. Simultaneous hole punching opens a direct UDP path between peers. DERP/relay provides fallback when punching fails.

Layer 3: Control Plane (Control-Plane) The coordination layer. The coordination server (Tailscale, Headscale, NetBird) distributes public keys, current endpoints, and ACL policies to every device. This is the difference between manually editing WireGuard configs on 50 machines and having them all update automatically in seconds.

Layer 4: WireGuard Protocol (WireGuard-Protocol) The cryptographic layer. The Noise_IK handshake derives shared session keys using Curve25519 ECDH. ChaCha20-Poly1305 encrypts and authenticates every packet. Session keys rotate every 180 seconds for perfect forward secrecy. The Cryptokey Routing Table ties destination IPs to peer identities.

Layer 5: Mesh Topology (this article) The architectural layer. Instead of routing all data through a central server (hub-and-spoke), peers communicate directly with each other. The control plane is the only centralized component and carries only metadata. All actual data flows peer-to-peer, encrypted end-to-end, over direct WireGuard tunnels when possible and relayed tunnels as fallback.

Common Misconceptions

❌ “Tailscale is just a WireGuard GUI.”

Tailscale is a full mesh VPN built on top of WireGuard as the data plane. It adds NAT traversal, a coordination server, identity-based access control, split DNS, exit nodes, subnet routing, and automatic key management. WireGuard is the tunnel primitive. Tailscale is the complete network product.

❌ “Mesh networking means no central server.”

A mesh data plane has no central server. A mesh control plane almost always does. The architectural insight is that the control plane carrying metadata and the data plane carrying actual traffic don’t need to be the same server. Making them separate is what allows the mesh to work without a data throughput bottleneck.

❌ “WireGuard tunnels are persistent connections like TCP.”

WireGuard is stateless at the protocol level. There is no “connect” or “disconnect” operation. You configure a peer, and packets flow when sent. The interface looks like a always-on connection, but WireGuard is actually silent until there’s data to send. Session state (the handshake) is created on demand and expires after REJECT_AFTER_TIME seconds of inactivity.
