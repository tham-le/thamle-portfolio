WTF is a TUN/TAP Device?

The kernel trick that lets your VPN pretend to be a real network card.

You install WireGuard and suddenly you have a new network interface called wg0. You can ping things through it. Routing tables respect it. Your OS treats it like it’s an actual ethernet card plugged into your machine.

There is no ethernet card. So what the hell is wg0?

It’s a TUN device. And once you understand what TUN/TAP actually is, the entire architecture of WireGuard, Tailscale, and NetBird will click into place. This article is the foundation for that whole series.

The Core Analogy

Think of the Linux networking stack as a postal sorting facility. Packets come in from physical hardware (the loading dock), the kernel sorts them, applies routing rules, and delivers them to the right destination (your application, or out another physical interface).

Now imagine you’re a new contractor who wants to receive mail at that facility, but you don’t have a physical loading dock. TUN/TAP lets you register a fake loading dock. The kernel routes packets to your fake dock just like any real one, and instead of the packets going to hardware, they land directly in your userspace application as bytes you can read from a file descriptor.

You read those bytes, do whatever you want with them (encrypt them, forward them, inspect them, throw them away), and when you write bytes back, the kernel receives them as if a real physical device just sent packets in.

You become the network card.

The Problem It Solves

Before TUN/TAP, if you wanted to implement a VPN in userspace, you had two bad options:

Option 1: Write a kernel driver. Full access, maximum performance, but you’re writing kernel code. One bug and the system panics. ASN.1 parsers in the kernel have historically been a CVE factory. Not great for security-sensitive VPN code.

Option 2: Operate purely at the socket layer. You intercept individual TCP connections, but you can’t intercept raw IP packets. You can’t easily handle UDP. The kernel’s routing table doesn’t know about your tunnel. This is basically what SOCKS proxies do, and it’s limited.

What you actually want is to sit between the routing table and the physical hardware. You want the kernel to think you’re a network interface, hand you arbitrary IP packets, and let you decide what to do with them. That’s exactly what TUN/TAP provides.

The Key Insight

TUN/TAP doesn’t bypass the kernel’s networking stack. It plugs into it as a first-class participant. The kernel routes packets to your interface the same way it routes packets to eth0. The difference is that instead of going to hardware, those packets go directly to your file descriptor.

TUN vs TAP: What’s the Difference?

The driver is called “TUN/TAP” because it provides two modes, and they operate at different layers of the network stack.

TUN (network TUNnel) operates at Layer 3. It deals in raw IP packets. When you read from a TUN file descriptor, you get a complete IP packet: IP header, TCP/UDP header, payload and all. No Ethernet header. WireGuard uses TUN because it only cares about routing IP traffic, not simulating an entire Ethernet segment.

TAP (network TAP) operates at Layer 2. It deals in Ethernet frames, MAC addresses included. When you read from a TAP file descriptor, you get a full Ethernet frame. TAP devices can participate in Ethernet bridging, ARP, and everything else that happens below IP. QEMU and VirtualBox use TAP to give virtual machines a realistic fake network card.

The rule of thumb:

TUNTAP
Layer3 (IP)2 (Ethernet)
Reads/WritesIP packetsEthernet frames
Use casesVPNs, tunnelsVMs, network bridges
ExamplesWireGuard, OpenVPNQEMU, VirtualBox

For our series on VPN-style networking, TUN is what matters. TAP is worth knowing about, but you won’t use it unless you’re building a VM hypervisor.

What is IFF_NO_PI?

Without IFF_NO_PI, the kernel prepends a 4-byte header to each packet containing flags and protocol type. With IFF_NO_PI, you get the raw IP packet with no preamble. Most VPN implementations set this flag because they want to deal with clean IP packets, not driver metadata.

Step 4: The Packet Loop

This is the heart of every VPN implementation. Your program sits in a loop, reading packets from the TUN fd, doing something with them, and writing packets back:

What’s happening here:

  • read(tun_fd, ...) blocks until a local application sends a packet destined for our TUN interface. The kernel strips the Ethernet layer (we’re TUN, not TAP) and hands us the raw IP packet.
  • write(tun_fd, ...) injects a packet back into the kernel’s networking stack. The kernel processes it as if it just arrived from the network. TCP/IP state machines, socket buffers, all of it kicks in normally.

This bidirectional file descriptor is the entire interface between your userspace VPN code and the kernel’s network stack.

The Full Packet Flow

Here’s what actually happens when you send a packet through a WireGuard-style tunnel, from the moment your application calls send() to the moment encrypted bytes leave your physical eth0:

Your application
    │
    │  send(sock, data, ...) -> TCP segment
    ▼
Kernel TCP/IP stack
    │
    │  Routing table lookup: "10.100.0.5 goes to wg0"
    ▼
TUN device (wg0)
    │
    │  Packet appears as bytes at read(tun_fd, ...)
    ▼
WireGuard / VPN userspace process
    │
    │  Encrypt packet with ChaCha20-Poly1305
    │  Wrap in UDP datagram
    │  send() to remote peer's real IP
    ▼
Kernel TCP/IP stack (again, but for the real socket)
    │
    │  Routing table lookup: "163.172.x.x goes to eth0"
    ▼
Physical ethernet interface (eth0)
    │
    ▼
Real network

The inverse path on the receiving side:

Physical ethernet interface (eth0)
    │
    ▼
WireGuard / VPN userspace process
    │
    │  Read encrypted UDP from real socket
    │  Decrypt and verify
    │  write(tun_fd, plaintext_ip_packet)
    ▼
TUN device (wg0)
    │
    ▼
Kernel TCP/IP stack
    │
    │  "This is a packet for 10.0.0.2, deliver to local socket"
    ▼
Your application's socket buffer

The kernel is fooled twice. Outgoing: it thinks the TUN device is a real NIC that will handle sending. Incoming: it thinks the TUN device received a packet from the network.

Common Misconceptions

❌ “TUN/TAP requires root.”

Creating a TUN device requires CAP_NET_ADMIN. But once you create a persistent TUN device and assign ownership to a user (ip tuntap add mode tun user alice), that user can open and use it without root. This is how WireGuard’s userspace tools work in controlled environments.

❌ “Userspace VPNs are slow because of TUN.”

The overhead comes from the user/kernel boundary crossing (the read/write calls), not TUN itself. Tailscale’s engineering team actually improved wireguard-go’s throughput past the kernel WireGuard module in certain conditions by using batched reads and enabling TSO/GRO offloads via ioctl. The kernel interface is a tool. How you use it determines your performance.

❌ “TAP is just a slower TUN.”

TAP operates at a fundamentally different layer. It’s not slower TUN, it’s a completely different abstraction. You use TAP when you need to simulate a real Ethernet network, complete with MAC addresses and ARP. VMs use TAP. VPNs use TUN. Wrong tool for the wrong job, not a speed trade-off.

Why This Matters for WireGuard, Tailscale, and NetBird

Here’s why TUN is the foundation for everything else in this series:

WireGuard’s kernel module registers itself as a network driver directly. When you create a wg0 interface, the kernel talks to WireGuard’s kernel code without any userspace in the path. Fast, minimal overhead.

wireguard-go and BoringTun (Rust) are userspace WireGuard implementations. They open /dev/net/tun, create a wg0 interface, and sit in exactly the packet loop described above. Cloudflare wrote BoringTun in Rust specifically because Go’s garbage collector introduced latency spikes in packet-heavy workloads. Same TUN mechanism, different process.

Tailscale uses wireguard-go under the hood. Its additional layers (NAT traversal, coordination server, access control) are all built on top of this same TUN foundation. The TUN device gives Tailscale a place to intercept packets before they leave your machine.

NetBird follows the same pattern. The architecture is: TUN device for packet interception, WireGuard protocol for encryption, a control plane for peer coordination.

All of them are, at their core, programs that sit in a read/write loop on a file descriptor, pretending to be a network card.

Quick Recap

TUN/TAP is a Linux kernel driver (/dev/net/tun) that lets a userspace process register a virtual network interface. The process opens the driver, issues an ioctl() to create the interface, configures IP addresses and routes via normal tools, then sits in a loop reading outgoing packets and writing incoming ones. The kernel routes packets to the virtual interface just like a real NIC. WireGuard, Tailscale, OpenVPN, and NetBird all use this mechanism.
