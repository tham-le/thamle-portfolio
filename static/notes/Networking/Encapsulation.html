Encapsulation

It’s just wrapping shit in more shit so the next layer doesn’t have to care

The Envelope Analogy

Imagine you’re sending a letter. You write your message on paper, put it in an envelope, write an address on that envelope, and drop it in a mailbox. The postal worker doesn’t open your envelope to read your letter. They just look at the address and send it along.

Then that envelope goes into a mail bag. The truck driver doesn’t care about individual envelopes. They just grab bags and drive.

That’s encapsulation. Each layer wraps the previous layer’s data and adds its own header. Each layer only cares about its own job.

Why Do We Need This?

Picture the nightmare alternative: every time you send data over the network, you’d need to know:

  • The exact path your data takes through routers
  • The MAC address of every device along the way
  • The physical cable topology
  • Whether it’s going over WiFi or Ethernet

Fuck that.

Encapsulation lets each Network-Layer do its job without micromanaging the others. The application doesn’t care about IP addresses. IP doesn’t care about MAC addresses. Everyone stays in their lane.

How It Actually Works

Let’s trace a simple HTTP request from your browser to a server. We’ll watch the data get wrapped like a Russian nesting doll.

Layer 7: Application Layer

Your browser creates an HTTP request:

That’s just text. Raw data. The application layer doesn’t care how this gets to the server.

Layer 4: Transport Layer (TCP)

TCP wraps your HTTP request and adds its own header:

[TCP Header: src port 54321, dst port 80, sequence numbers, etc.]
[Your HTTP request]

The TCP header contains:

  • Source port (your random port, like 54321)
  • Destination port (80 for HTTP)
  • Sequence numbers (so packets arrive in order)
  • Checksums (to detect corruption)

TCP calls this whole package a segment. It doesn’t know or care that the data inside is HTTP. Could be anything.

Layer 3: Network Layer (IP)

IP takes the TCP segment and wraps that:

[IP Header: src IP 192.168.1.100, dst IP 93.184.216.34, TTL, etc.]
[TCP Header + HTTP request]

The IP header contains:

  • Source IP address (your computer)
  • Destination IP address (example.com’s server)
  • TTL (time to live, prevents infinite loops)
  • Protocol field (tells the receiver “hey, this contains TCP”)

IP calls this a packet. It doesn’t peek inside at the TCP data. Not its job.

Layer 2: Data Link Layer (Ethernet)

Finally, Ethernet wraps the IP packet:

[Ethernet Header: src MAC, dst MAC]
[IP Header + TCP Header + HTTP request]
[Ethernet Trailer: checksum]

The Ethernet header contains:

  • Source MAC address (your network card)
  • Destination MAC address (your router’s MAC)
  • Type field (tells the receiver “this contains IP”)

This is called a frame. Ethernet doesn’t know what’s inside. Just delivers it to the next hop.

The Beautiful Part

When this frame arrives at your router, it strips off the Ethernet header, looks at the IP destination, and adds a new Ethernet header for the next hop.

The TCP and HTTP data? Untouched. The router doesn’t even look at them.

When it arrives at the destination server:

  1. Ethernet strips off the frame header → passes IP packet upward
  2. IP strips off the packet header → passes TCP segment upward
  3. TCP strips off the segment header → passes HTTP data upward
  4. Application reads the HTTP request

Each layer only handles its own wrapper.

Common Confusion: Headers vs. Data

Here’s what trips people up: at each layer, everything from the layer above is just data.

To TCP, your HTTP request is meaningless bytes. To IP, the TCP segment is meaningless bytes. Each layer treats the inner content as an opaque blob (called the payload).

The Protocol Field Each header has a "protocol" or "type" field that tells the next layer what to expect:

  • Ethernet header says “type 0x0800” → contains IP
  • IP header says “protocol 6” → contains TCP
  • TCP port 80 → probably contains HTTP

This is how decapsulation works going back up the stack.

The MTU Problem

Every network has a Maximum Transmission Unit (MTU). Usually 1500 bytes for Ethernet.

But what if your data is bigger? Let’s say you want to send 5000 bytes.

IP handles this through fragmentation. It breaks your data into chunks:

Original packet: [IP Header][4000 bytes of data]

After fragmentation:
Packet 1: [IP Header (fragment 1)][1480 bytes]
Packet 2: [IP Header (fragment 2)][1480 bytes]  
Packet 3: [IP Header (fragment 3)][1040 bytes]

Each IP header has a “fragment offset” field. The receiver uses these to reassemble the original packet.

TCP is smarter. It knows the MTU ahead of time and breaks data into segments that fit. This is called the Maximum Segment Size (MSS).

Why This Design Is Genius

Modularity: Want to add a new application protocol? Just use TCP or UDP. Done. No need to rewrite IP or Ethernet.

Flexibility: Your HTTP data can travel over Ethernet, WiFi, fiber, or a carrier pigeon. The application doesn’t change.

Debugging: When something breaks, you can isolate which layer is fucked. Is TCP retransmitting? Check Layer 4. Packets not arriving? Check Layer 3 routing.

Efficiency: Each layer only processes its own header. Routers don’t waste time parsing TCP or HTTP. They just read the IP destination and forward.

The Full Stack

Here’s what each layer adds:

LayerNameUnitHeader Info
7ApplicationMessageHTTP, DNS, etc.
4TransportSegmentPorts, sequence numbers
3NetworkPacketIP addresses, TTL
2Data LinkFrameMAC addresses
1PhysicalBitsVoltage/light signals

Going down (encapsulation): each layer adds a header.
Going up (decapsulation): each layer strips its header.
