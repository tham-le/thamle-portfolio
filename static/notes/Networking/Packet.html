WTF is a Packet?

A packet is a chunk of data sent over a network. It’s the fundamental unit of network communication.

Think of it like a letter in an envelope - it has addressing information (headers) and content (payload).

The Mail System Analogy

The Postal Letter

The Envelope (Headers)

  • From address: Source IP address
  • To address: Destination IP address
  • Stamp: Protocol type (TCP/UDP)
  • Routing marks: How to get there

The Letter Inside (Payload)

  • The actual message/data
  • Could be text, image data, video, etc.

The postal service doesn’t care what’s in the letter - they just deliver the envelope to the right address.

That’s a packet - headers tell where to go, payload is what you’re sending.

Packet Structure

Basic Anatomy

┌─────────────────────────────────────┐
│          HEADERS                    │  ← Routing & control info
├─────────────────────────────────────┤
│                                     │
│          PAYLOAD                    │  ← Actual data
│                                     │
└─────────────────────────────────────┘

Real Example: TCP/IP Packet

┌────────────────────────────────────┐
│  Ethernet Header (14 bytes)        │  Layer 2: MAC addresses
├────────────────────────────────────┤
│  IP Header (20+ bytes)             │  Layer 3: IP addresses, TTL
├────────────────────────────────────┤
│  TCP Header (20+ bytes)            │  Layer 4: Ports, seq numbers
├────────────────────────────────────┤
│  Application Data (0-1460 bytes)   │  Layer 7: Your actual data
└────────────────────────────────────┘

Total: ~1500 bytes maximum (MTU)

Headers stack like Russian dolls - each layer adds its own envelope.

The Layers Explained

Layer 2: Ethernet (Link Layer)

Ethernet Header:
┌─────────────┬─────────────┬──────┐
│ Dest MAC    │ Source MAC  │ Type │
│ (6 bytes)   │ (6 bytes)   │ (2)  │
└─────────────┴─────────────┴──────┘

What it does: Deliver packet to next hop on local network.

  • Destination MAC: Hardware address of next device
  • Source MAC: Hardware address of sender
  • Type: What’s inside (usually IP)

Layer 3: IP (Network Layer)

IP Header (simplified):
┌─────────┬──────┬────┬──────────┬───────────┐
│ Version │ TTL  │ Proto │ Source IP │ Dest IP │
│  (4)    │ (8)  │ (8) │  (32)     │  (32)    │
└─────────┴──────┴────┴──────────┴───────────┘

What it does: Route packet across the internet.

  • Source/Dest IP: e.g., 192.168.1.10 → 8.8.8.8
  • TTL (Time To Live): Hops remaining (prevents infinite loops)
  • Protocol: What’s inside (6 = TCP, 17 = UDP)

Layer 4: TCP/UDP (Transport Layer)

TCP Header:

┌────────┬────────┬────────┬─────────┬───────┐
│ Src    │ Dest   │ Seq    │ Ack     │ Flags │
│ Port   │ Port   │ Number │ Number  │       │
│ (16)   │ (16)   │ (32)   │ (32)    │ (8)   │
└────────┴────────┴────────┴─────────┴───────┘

What it does: Deliver to correct application, ensure reliability.

  • Source/Dest Port: Which program? (e.g., 80 = HTTP, 443 = HTTPS)
  • Sequence Number: Order of packets
  • Ack Number: “I received up to packet N”
  • Flags: SYN, ACK, FIN, RST

UDP Header (simpler):

┌────────┬────────┬────────┬──────────┐
│ Src    │ Dest   │ Length │ Checksum │
│ Port   │ Port   │        │          │
│ (16)   │ (16)   │ (16)   │ (16)     │
└────────┴────────┴────────┴──────────┘

What it does: Fast, unreliable delivery.

  • No sequence numbers, no acknowledgments
  • Used for video streaming, gaming (speed > reliability)

Why Packets?

1. Fair Sharing

Without packets:

User A: AAAAAAAAAAAAAA... (monopolizes network for 1 hour)
User B: (waits...)

With packets:

Network: A, B, A, B, A, B... (interleaved, everyone gets turns)

2. Error Recovery

If a single packet gets corrupted:

Sender: Here's packets 1, 2, 3, 4, 5
Receiver: Got 1, 2, 4, 5 (3 corrupted)
Receiver: "Resend packet 3 please"
Sender: Here's packet 3 again

Only resend what failed, not the entire file.

3. Routing Flexibility

Packet 1: Route through Chicago
Packet 2: Route through Dallas (Chicago congested)
Packet 3: Route through Chicago (congestion cleared)

Different packets can take different paths. The internet routes each packet independently.

4. Maximum Transmission Unit (MTU)

Physical networks have size limits:

  • Ethernet: 1500 bytes max
  • Internet: Usually 1500 bytes (standard MTU)
  • Jumbo frames: 9000 bytes (special networks)

Large data must be split into multiple packets:

5000 byte file:
┌──────────────┐
│ Packet 1     │  1500 bytes
├──────────────┤
│ Packet 2     │  1500 bytes
├──────────────┤
│ Packet 3     │  1500 bytes
├──────────────┤
│ Packet 4     │  500 bytes
└──────────────┘

Packet Journey Example

Sending “Hello” from your laptop to example.com:

Step 1: Application Layer

HTTP Request: GET / HTTP/1.1

Step 2: TCP Layer Adds Header

┌────────────────────┐
│ TCP Header         │  Src Port: 54321, Dest Port: 80
│ Payload: GET / ... │
└────────────────────┘

Step 3: IP Layer Adds Header

┌────────────────────┐
│ IP Header          │  Src: 192.168.1.10, Dest: 93.184.216.34
│ TCP Header         │
│ Payload: GET / ... │
└────────────────────┘

Step 4: Ethernet Layer Adds Header

┌────────────────────┐
│ Ethernet Header    │  Dest MAC: router's MAC
│ IP Header          │
│ TCP Header         │
│ Payload: GET / ... │
└────────────────────┘

Step 5: Physical Transmission

Packet travels as electrical signals / light pulses / radio waves:

Your Laptop → Router → ISP → Internet Backbone → example.com

At each hop:

  • Strip Ethernet header
  • Look at IP destination
  • Add new Ethernet header for next hop
  • Forward

Step 6: Destination Receives

example.com receives and unwraps the layers:

Strip Ethernet → Strip IP → Strip TCP → Read HTTP request

This is called de-encapsulation.

Packet Loss

Not all packets arrive:

  • Congestion: Router buffer full, drops packet
  • Corruption: Checksum fails, packet discarded
  • Route failure: No path to destination

TCP handles loss:

Sender: Packets 1, 2, 3, 4, 5
Receiver: Got 1, 2, 4, 5
Receiver: (Timeout waiting for 3)
Receiver: "Where's packet 3?"
Sender: Resends packet 3

UDP doesn’t care - if packet lost, too bad!

Packet Inspection

Using tcpdump (Linux/Mac)

Output:

15:23:45.123 IP 192.168.1.10.54321 > 93.184.216.34.80: Flags [S], seq 1234
15:23:45.150 IP 93.184.216.34.80 > 192.168.1.10.54321: Flags [S.], seq 5678, ack 1235

Shows: Source IP:Port → Dest IP:Port, TCP flags, sequence numbers

Using Wireshark (GUI)

Frame 1: 74 bytes on wire
Ethernet II, Src: aa:bb:cc:dd:ee:ff, Dst: 11:22:33:44:55:66
Internet Protocol Version 4, Src: 192.168.1.10, Dst: 93.184.216.34
Transmission Control Protocol, Src Port: 54321, Dst Port: 80

Wireshark decodes every layer - perfect for debugging.

Packet Overhead

Headers eat bandwidth:

Sending 1 byte of data via TCP/IP:

Ethernet: 14 bytes
IP:       20 bytes
TCP:      20 bytes
Payload:   1 byte
─────────────────
Total:    55 bytes

Efficiency: 1/55 = 1.8%! That’s why small packets are wasteful.

Solution: Send larger payloads (up to MTU size).

The Big Takeaway

Packet in a Nutshell

A packet is the fundamental unit of network data transmission. It consists of:

  • Headers - Addressing and control information (like an envelope)
  • Payload - Actual data being sent (like the letter inside)

Layered structure:

  • Ethernet (Layer 2): Local network delivery via MAC addresses
  • IP (Layer 3): Internet routing via IP addresses
  • TCP/UDP (Layer 4): Application delivery via ports
  • Application (Layer 7): Your actual data (HTTP, etc.)

Why packets?

  • Fair network sharing (interleaving)
  • Error recovery (resend only failed packets)
  • Routing flexibility (each packet can take different path)
  • MTU constraints (max ~1500 bytes per packet)

Each router unwraps and re-wraps packets as they traverse the network.

The rule: Everything on the internet is broken into packets. Headers tell routers where to send them. TCP ensures they arrive in order. UDP sends and forgets.
