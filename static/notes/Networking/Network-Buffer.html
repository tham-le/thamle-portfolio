WTF is a Network Buffer?

The plumbing between your code and the wire

You just learned how sockets work. You call write(), data appears on the other side, and life is good. But here’s the thing nobody tells you: your data didn’t go anywhere near the network cable when you called write().

Instead, it took a journey through a series of waiting rooms, queues, and holding areas before finally getting shoved onto the wire. This invisible plumbing is what we call network buffers, and understanding them is the difference between writing a toy program and writing production-grade network code.

Today, we’re going to follow a single byte on its journey from your write() call to the actual network hardware. We’ll see where it waits, why it waits, and what happens when these waiting rooms get too full.

The Core Analogy

Think about the plumbing in your house. When you turn on a faucet, water doesn’t teleport from the water main to your tap. It travels through pipes of different sizes, each with its own capacity.

A network buffer is exactly like the pipes in this system:

  • Your write() call is like pouring water into your sink drain
  • The send buffer is the pipe under your sink (limited capacity)
  • The receive buffer on the other end is the pipe leading to their faucet
  • The network is the main water line connecting both houses

Just like your sink can overflow if you pour water faster than it drains, your network buffers can overflow if you send data faster than the network can transmit it.

Beginner: The Journey of a Single Byte

Let’s trace what actually happens when you write this innocent line of code:

Step 1: Your Program’s write() Call

Your program says, “Hey kernel, please send these 5 bytes.”

What you think happens: The bytes go straight to the network card.

What actually happens: The kernel says, “Sure, I’ll get to it,” and copies your data into kernel space.

This copy is important. The kernel can’t trust your memory (you might free it or change it), so it makes its own copy in kernel-controlled memory.

Step 2: The Send Buffer (The First Waiting Room)

The kernel doesn’t immediately send your data. Instead, it dumps those bytes into the socket send buffer, a chunk of kernel memory dedicated to outgoing data for this specific socket.

Your Program's Memory:  [H][e][l][l][o]
                          ↓ copy (system call boundary)
Kernel Send Buffer:     [H][e][l][l][o][...empty space...]

This buffer is just a circular queue in kernel memory, managed by the operating system. The size is finite, typically 16KB to 64KB by default (varies by OS).

Why Buffer at All?

Batching efficiency. The kernel groups multiple small writes together into larger TCP packets. Sending one 1000-byte packet is way cheaper than sending 1000 one-byte packets. The buffer is where this batching magic happens.

Your write() call returns immediately after copying data to this buffer. It does NOT wait for the data to actually hit the network.

Step 3: The TCP Layer Wraps It Up

When the TCP stack decides it’s time to send (based on timers, buffer fullness, or other heuristics), it:

  1. Takes a chunk of data from the send buffer
  2. Wraps it in a TCP header (sequence numbers, checksums, window size, flags)
  3. Passes the packet down to the IP layer

The TCP layer also keeps a copy of sent data in case it needs to retransmit (if the receiver doesn’t acknowledge receipt). This is separate from the send buffer.

Step 4: The IP Layer and Below

The IP layer adds its own header (source/destination IP addresses) and passes the packet to the network driver. The driver places it in the network interface card’s (NIC) transmit queue, yet another buffer.

Application → Send Buffer → TCP Layer → IP Layer → NIC TX Queue → Wire

The NIC pulls packets from this queue and physically transmits them as electrical signals (or light pulses if fiber optic).

Step 5: The Receiver’s Side

The data travels across the network and arrives at the destination machine’s network card. Now the reverse journey begins:

  1. NIC Receive Queue: The network card’s hardware buffer stores incoming packets
  2. Interrupt & Driver: The NIC triggers an interrupt, and the driver pulls the packet into kernel memory
  3. IP Layer: Strips the IP header, verifies checksums
  4. TCP Layer: Strips the TCP header, checks sequence numbers, sends acknowledgment
  5. Kernel Receive Buffer: The TCP stack places the actual data into the socket receive buffer for the destination socket
  6. Application’s read() Call: When the receiving program calls read(), the kernel copies data from the receive buffer into the program’s memory
Wire → NIC RX Queue → Driver → IP → TCP → Receive Buffer → read()

At each step, data is copied from one buffer to another. Modern systems use techniques like zero-copy to reduce this overhead for large transfers.

Beginner: When Buffers Fill Up

Here’s where things get real. What happens when buffers can’t keep up?

Scenario 1: Send Buffer is Full

You’re writing faster than the network can transmit. The send buffer fills up.

If the send buffer doesn’t have room for all your data, write() will:

  • Block (default behavior): Wait until space frees up
  • Partial write: Accept what fits, return the number of bytes written
  • Return error (non-blocking mode): Return -1 with errno = EAGAIN

The kernel is saying, “Slow down! I can’t drain the pipe fast enough!”

Scenario 2: Receive Buffer is Full

The sender is transmitting faster than the receiver can read(). The receive buffer fills up.

What happens: TCP’s flow control mechanism kicks in. The receiver advertises a window size of 0 in its TCP acknowledgment packets, telling the sender “My buffer is full, stop sending!” The sender pauses transmission until the receiver drains its buffer and advertises a larger window.

This is automatic and transparent to your application. TCP handles it for you. But if the receiver never calls read(), the sender will eventually block on write() because it can’t send more data.

The Deadlock Scenario

If both sides fill their send buffers trying to write without reading, both will block forever. Always interleave reads and writes, or use separate threads/processes for reading and writing.

Intermediate: Inspecting and Tuning Buffer Sizes

You can see and change buffer sizes using socket options:

Platform differences:

  • Linux: Default ~87KB send, ~128KB receive (configurable via /proc/sys/net/core/wmem_default and rmem_default)
  • macOS: Default ~130KB for both
  • Windows: Default ~8KB send, ~8KB receive (much smaller!)

The OS might double your requested size internally for bookkeeping, so the value you get back from getsockopt() might be larger than what you set.

When to Tune Buffer Sizes

Increase buffers for:

  • High-bandwidth connections (gigabit networks)
  • Long-distance connections (satellite, intercontinental)
  • Bulk data transfers (file downloads, video streaming)
  • Applications where throughput matters more than latency

Keep buffers small for:

  • Low-latency applications (gaming, real-time audio)
  • Memory-constrained environments
  • Connections with many concurrent sockets (each socket has its own buffers)

The bandwidth-delay product is your guide:

Optimal Buffer Size = Bandwidth × Round-Trip Time

For a 1 Gbps connection with 100ms RTT:

1,000,000,000 bits/sec × 0.1 sec = 100,000,000 bits = 12.5 MB

You’d want buffers around 12-13MB to fully utilize the pipe. The defaults are nowhere near this.

Advanced: The TCP Perspective

From TCP’s point of view, buffers are critical for its reliability and flow control mechanisms.

TCP Window Size and Flow Control

Every TCP packet contains a window size field telling the sender how much buffer space the receiver has available:

TCP Header:
- Sequence Number: 1000
- Acknowledgment: 500
- Window Size: 32768  ← "I have 32KB of buffer space"

The sender won’t send more than this amount of unacknowledged data. As the receiver’s application calls read() and drains the receive buffer, the receiver advertises a larger window in subsequent ACK packets.

Zero Window: When the receive buffer is full, the receiver advertises window size 0. The sender stops sending data and periodically sends window probes (tiny packets) to check if space has freed up.

Nagle’s Algorithm and Buffer Aggregation

TCP has an optimization called Nagle’s algorithm: instead of immediately sending small writes, it waits a bit to batch them together into larger packets.

This improves efficiency but adds latency (up to 200ms in some cases). For interactive applications (SSH, games), you want to disable it:

Now each write() is sent immediately in its own packet.

The Retransmission Buffer

The send buffer is different from the retransmission buffer. After TCP sends data from the send buffer, it keeps a copy in the retransmission buffer until the receiver acknowledges receipt. If no ACK arrives within the timeout period, TCP retransmits from this buffer.

Send Buffer: [New data waiting to be sent]
     ↓
TCP sends packet
     ↓
Retransmission Buffer: [Sent but unacknowledged data]
     ↓ (ACK received)
Freed

This is why acknowledged data can be removed from memory, but unacknowledged data must be kept around.

Advanced: Buffer Memory Limits

Each socket consumes kernel memory for its buffers. On a server with 10,000 connections, even 64KB buffers add up:

10,000 sockets × (64KB send + 64KB receive) = 1.25 GB of kernel memory

Real-World Implications

1. Why Your Writes Succeed Even When the Network is Slow

Because write() only copies to the send buffer, it returns quickly even on a congested network. The actual transmission happens later in the background.

Implication: A successful write() doesn’t mean data reached the destination. It just means the kernel accepted it.

2. Why Application-Level Acknowledgments Matter

If you need confirmation that data was received and processed, implement application-level ACKs. TCP only tells you data was received by the receiver’s kernel, not that the application processed it.

3. Buffer Bloat

Large buffers can increase latency. If data sits in buffers for too long, interactive applications suffer. This is called bufferbloat.

Example: You’re playing an online game. Your roommate starts downloading a large file. The download fills the router’s buffers, and your game packets get stuck behind download packets. Your latency spikes.

Solutions:

  • Active Queue Management (AQM) algorithms like CoDel, FQ-CoDel
  • Quality of Service (QoS) rules prioritizing interactive traffic
  • Smaller buffer sizes on network equipment

4. Head-of-Line Blocking in HTTP/2

HTTP/2 multiplexes multiple requests over a single TCP connection. But if one packet is lost, TCP’s reliable delivery stalls ALL streams (they’re all waiting in the receive buffer for that missing packet).

HTTP3 solves this by using QUIC over UDP, which allows independent streams.
