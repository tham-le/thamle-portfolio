WTF is the WireGuard Protocol?

Three messages. Two peers. One shared secret. Here’s every step.

You’ve been taking WireGuard on faith for three articles. You know it uses a TUN device, you know the peers find each other via NAT traversal, you know the control plane distributes keys. But when WireGuard says “your packet is encrypted,” what does that actually mean at the byte level?

What algorithm? What key? How does a fresh peer connection bootstrap a shared secret when two parties have never communicated before? And what does “perfect forward secrecy” actually mean in practice, not just as a marketing phrase?

This article answers all of it. It’s the most technically demanding in the series, but you’ll come out of it understanding exactly why WireGuard is considered one of the best-designed security protocols written in the last decade.

The Core Analogy

Imagine you need to have a private conversation with a stranger in a public square. You both have a lockbox, and you’ve published the combination to your lockbox publicly (that’s your public key). Anyone can put a message in your box. Only you can open it (that’s your private key).

Here’s the trick: you both put sealed envelopes containing one-time combination codes into each other’s public lockboxes. Each of you opens your own box, reads the combination inside, and you both now have the same secret combination. Nobody watching from outside learned anything. That shared secret becomes the key for all your future conversation.

That’s a Diffie-Hellman key exchange. WireGuard’s handshake is a more sophisticated version of this, using Curve25519 elliptic curve math instead of combination locks, and it runs this exchange four times simultaneously with different key pairs to cover various failure scenarios.

The Cryptographic Primitives

Before the handshake, let’s name the tools WireGuard uses. These aren’t negotiable: WireGuard has no cipher suite negotiation. You get exactly this stack, no configuration options:

Curve25519 handles key exchange. It’s an elliptic curve designed by Daniel Bernstein. Each WireGuard peer has a static Curve25519 key pair that serves as their permanent identity. Curve25519 was chosen because it’s fast in software, resistant to timing side-channels by construction, and has a much smaller key size (32 bytes) than RSA while providing comparable or better security.

ChaCha20-Poly1305 is the AEAD (Authenticated Encryption with Associated Data) cipher used for all data. ChaCha20 is the stream cipher (it encrypts). Poly1305 is the MAC (it authenticates, detecting any tampering). They’re always used together. This combination was chosen over AES-GCM because it’s fast in pure software on hardware without AES acceleration, which matters enormously for mobile devices and embedded systems.

BLAKE2s is the hash function, used for key derivation and chaining keys through the handshake. It’s faster than SHA-256 at comparable security levels.

HKDF (HMAC-based Key Derivation Function, RFC 5869) derives symmetric session keys from the Diffie-Hellman outputs.

The Noise Protocol Framework is not a cipher; it’s a pattern language for building handshake protocols. WireGuard implements the specific pattern called Noise_IKpsk2_25519_ChaChaPoly_BLAKE2s. That name decodes as: Noise framework, IK handshake pattern (Initiator sends their key, responder’s Key is Known to the initiator beforehand), optional Pre-Shared Key in position 2, using Curve25519, ChaCha20-Poly1305, BLAKE2s.

The Cryptokey Routing Table

Before a single handshake packet flies, WireGuard needs to know one thing: which public key belongs to which destination IP range. This is the Cryptokey Routing Table, and it’s the heart of WireGuard’s identity model.

# A WireGuard interface configuration

[Interface]
PrivateKey = <base64-encoded Curve25519 private key>
ListenPort = 51820

[Peer]
PublicKey  = <peer A's Curve25519 public key>
AllowedIPs = 10.0.0.2/32

[Peer]
PublicKey  = <peer B's Curve25519 public key>
AllowedIPs = 10.0.0.3/32, 192.168.100.0/24

When WireGuard needs to send a packet to 10.0.0.2, it walks the AllowedIPs table, finds peer A’s public key, and uses that key to encrypt the packet. When it receives an encrypted packet, it decrypts it and checks that the decrypted source IP is in peer A’s AllowedIPs. If it doesn’t match, the packet is silently dropped.

This is powerful: authentication and routing are the same operation. If a packet decrypts and authenticates correctly, you know exactly who sent it. No spoofing possible. No need for separate firewall rules checking source IPs from untrusted sources.

AllowedIPs Is an Access Control List and a Routing Table Simultaneously

On the send side: AllowedIPs is a routing table. “Send packets destined for 10.0.0.2 to this peer.” On the receive side: AllowedIPs is an ACL. “Only accept packets claiming to be from 10.0.0.2 if they come from this peer’s key.” A packet that arrives encrypted under peer A’s key but claims to be from 10.0.0.4 (not in peer A’s AllowedIPs) is dropped immediately. The crypto proves the sender, the AllowedIPs prove the sender is allowed to claim that address.

The Handshake: Three Messages

The WireGuard handshake establishes a shared symmetric session key between two peers. It takes 1.5 round trips: initiator sends, responder sends, initiator sends first data packet (which serves as key confirmation). Here’s what each message contains.

Message 1: Initiator Hello

The initiator wants to talk to the responder. It knows the responder’s static public key (from the Cryptokey Routing Table). It sends:

Handshake Initiation message:
  u8   message_type          = 1
  u8   reserved[3]           = {0, 0, 0}
  u32  sender_index          (random session ID, used to route replies)
  u8   ephemeral_public[32]  (fresh Curve25519 public key, generated just for this session)
  u8   encrypted_static[48]  (initiator's static public key, encrypted)
  u8   encrypted_timestamp[28] (anti-replay timestamp, encrypted)
  u8   mac1[16]              (keyed MAC over the whole message, for DoS resistance)
  u8   mac2[16]              (optional, for cookie mechanism under load)

The cryptographic construction behind this message:

# Step 1: Start with a chaining key based on the protocol identifier
ck = HASH("Noise_IKpsk2_25519_ChaChaPoly_BLAKE2s")

# Step 2: Generate a fresh ephemeral key pair just for this session
e_priv, e_pub = DH_GENERATE()

# Mix the responder's static public key into the chaining key
ck = HKDF(ck, responder_static_pub)

# Step 3: Mix our ephemeral public key into the chain
ck, key = HKDF(ck, e_pub)

# Step 4: Perform DH between our ephemeral private key and responder's static public key
# This proves we know the responder's public key before they reply
dh_result = DH(e_priv, responder_static_pub)
ck, key = HKDF(ck, dh_result)

# Step 5: Encrypt our static public key using the derived key
# The responder will decrypt this to learn who is calling
encrypted_static = AEAD_ENCRYPT(key, 0, initiator_static_pub, hash_so_far)

# Step 6: Perform DH between our static private key and responder's static public key
dh_result2 = DH(initiator_static_priv, responder_static_pub)
ck, key = HKDF(ck, dh_result2)

# Step 7: Encrypt a timestamp (prevents replay attacks)
encrypted_timestamp = AEAD_ENCRYPT(key, 0, current_tai64n_timestamp, hash_so_far)

The key observation: the initiator’s static public key is encrypted in the message. An eavesdropper watching the network cannot identify who is initiating the connection. This is called identity hiding.

Message 2: Responder Hello

The responder receives message 1, decrypts the initiator’s static public key, and looks it up in its own Cryptokey Routing Table. If it’s a known peer, it responds:

Handshake Response message:
  u8   message_type          = 2
  u8   reserved[3]           = {0, 0, 0}
  u32  sender_index          (responder's random session ID)
  u32  receiver_index        (echoes the initiator's sender_index)
  u8   ephemeral_public[32]  (responder's fresh ephemeral key)
  u8   encrypted_empty[16]   (empty payload encrypted + authenticated, proves key knowledge)
  u8   mac1[16]
  u8   mac2[16]

The responder generates its own fresh ephemeral key pair, performs four Diffie-Hellman operations, and derives the same chaining key the initiator computed. The encrypted empty payload is the proof: it’s an AEAD-encrypted zero-length message. If the initiator can decrypt and verify it, both sides have confirmed they share the same session keys.

At this point, both sides have derived two symmetric keys from the handshake:

send_key, recv_key = HKDF(final_chaining_key, empty)

The initiator’s send_key equals the responder’s recv_key, and vice versa. These are the session keys for encrypted data transport.

The 1.5 RTT Nuance

After message 2, the responder cannot safely send data yet. The responder authenticated the initiator’s static key (step 5 in message 1 above), but entity authentication is incomplete until the initiator sends a data packet that decrypts correctly. The responder waits. When the initiator sends the first encrypted transport packet, that serves as implicit key confirmation. Only then does the responder begin sending data.

This is why WireGuard is described as 1.5-RTT: strictly speaking, the handshake + first data packet complete full mutual authentication before bidirectional data flows.

The Data Packet Format

Once the handshake is complete, data packets look like this:

Transport Data message:
  u8   message_type = 4
  u8   reserved[3]  = {0, 0, 0}
  u32  receiver_index   (identifies which session this belongs to)
  u64  counter          (monotonically increasing, serves as the nonce)
  u8   encrypted_payload[variable length]  (your actual IP packet, encrypted)
  u8   auth_tag[16]     (Poly1305 authentication tag, appended by AEAD)

The counter serves double duty as the nonce and as replay protection. The receiver maintains a sliding window of accepted counters. Any packet with a counter already seen or too far in the past is dropped. This prevents replay attacks where an attacker records and re-sends packets.

Perfect Forward Secrecy: What It Actually Means

This phrase gets thrown around constantly. Here’s what it concretely means in WireGuard.

Without PFS: If an attacker records all your encrypted traffic today and later compromises your long-term private key, they can decrypt all of that recorded traffic retroactively.

With PFS: Each session uses ephemeral key pairs that exist only in RAM for the duration of the session. When the session ends, those ephemeral keys are explicitly zeroed in memory. An attacker who compromises your long-term static key later cannot decrypt past sessions, because the ephemeral keys that generated those session keys no longer exist anywhere.

WireGuard achieves PFS by:

  1. Using fresh ephemeral Curve25519 key pairs in every handshake (the e_priv, e_pub generated in message 1 above).
  2. Rekeying automatically every REKEY_AFTER_TIME seconds (180 seconds by default) or after REKEY_AFTER_MESSAGES packets (roughly 2^60), whichever comes first.
  3. Explicitly zeroing ephemeral keys and symmetric session keys from memory after use using memzero_explicit() (the kernel-provided function that won’t be optimized out by the compiler).

This is the difference between a protocol with PFS and one without. It’s not magic. It’s a discipline of destroying key material the moment it’s no longer needed.

Why These Algorithms, Not Others?

This is worth understanding because it’s a deliberate rejection of complexity.

ChaCha20-Poly1305 over AES-GCM: AES-GCM is faster on CPUs with hardware AES instructions (AES-NI). ChaCha20-Poly1305 is faster everywhere else. Mobile devices, embedded systems, and older hardware often lack AES-NI. ChaCha20 also has no known timing side-channels in software implementations, while AES implementations have historically been vulnerable to cache-timing attacks on hardware without AES-NI. By choosing ChaCha20, WireGuard gets consistent security properties everywhere.

Curve25519 over RSA or NIST curves: RSA keys at comparable security levels are enormous (2048 to 4096 bits). NIST curves (P-256, P-384) have a history of questionable parameter choices and potential NSA influence. Curve25519 was designed with transparent parameters that are verifiably not backdoored, is extremely fast, and has a 32-byte key size.

No cipher negotiation at all: SSL/TLS allows clients and servers to negotiate which cipher suite to use. This flexibility has historically led to downgrade attacks where an attacker forces both parties to agree on a weaker algorithm. WireGuard simply doesn’t have this. There is one algorithm stack. You cannot be downgraded.

WireGuard Is Not Quantum-Resistant by Default

Curve25519 and the DH construction would be broken by a sufficiently capable quantum computer using Shor’s algorithm. WireGuard supports an optional pre-shared symmetric key (PSK) that can be added to the handshake. If a quantum attacker breaks the DH, the PSK still protects the session. This provides post-quantum resistance at the cost of having to distribute and manage an extra shared secret per peer pair.

Silent Drop Policy: Why WireGuard Is Hard to Fingerprint

One more design choice worth noting: WireGuard responds to packets from unknown sources with absolute silence. No error messages. No ICMP port unreachable. No RST packets. Nothing.

If you scan a WireGuard port on the internet with a random packet, you get no response. The port appears closed. A legitimate peer that sends a correctly formatted handshake initiation, encrypted under a key that matches a configured peer, will get a response. Everyone else gets silence.

This makes WireGuard interfaces significantly harder to fingerprint and enumerate than traditional VPN endpoints, which typically respond to probes with recognizable error packets.
