WTF is Certificate Verification?

Why your browser trusts amazon.com but not sketchy-bank-login.ru

You connect to https://github.com. The server sends you a certificate claiming “I am github.com.” How does your computer know this isn’t a lie?

This is the certificate verification problem, and it’s more interesting than you think. Different systems solve it in completely different ways.

Prerequisites

This article assumes you understand:

  • What TLS is and why it exists
  • Basic certificate structure

If not, read WTF is TLS? first. For protocol details, see TLS: Under the Hood.

The Core Problem

When a server sends you a certificate, anyone could have created it:

Certificate claiming:
  "I am github.com"
  Public Key: [random key]
  
Who created this?
  ✓ GitHub (legitimate)
  ✗ Attacker trying to steal your password
  ✗ Government doing surveillance
  ✗ Your ISP injecting ads

You need a way to distinguish legitimate certificates from fraudulent ones.

The Trust Anchor Analogy

Think of certificate verification like passport verification at a border crossing.

Without root certificates:

  • You show up with a passport saying “I’m John Smith”
  • The border agent has no way to verify this
  • They have to either trust everyone (insecure) or reject everyone (useless)

With root certificates:

  • Your passport is issued by a government (Certificate Authority)
  • The border agent has a book of trusted government seals (root certificate store)
  • They check: “Is this passport signed by a government we recognize?”
  • If yes, they verify the signature matches
  • Only then do they trust your passport

Root certificates are that book of trusted seals. They’re the starting point for all trust decisions.

What Are Root Certificates?

A root certificate is a self-signed certificate from a trusted Certificate Authority (CA):

Root Certificate:
┌────────────────────────────────────┐
│ Subject: DigiCert Global Root CA   │
│ Issuer: DigiCert Global Root CA    │  <- Self-signed
│ Public Key: [CA's public key]      │
│ Valid: 2006-2031                   │
│ Signature: [signed by itself]      │
└────────────────────────────────────┘

Key properties:

  1. Self-signed: Subject and Issuer are the same
  2. Pre-installed: Shipped with your OS/browser
  3. Long-lived: Valid for 10-25 years
  4. Highly protected: If the private key is compromised, the entire trust model breaks

Your operating system/browser comes with ~100-150 trusted root certificates from organizations like:

  • DigiCert
  • Let’s Encrypt
  • GlobalSign
  • Sectigo
  • Government CAs (various countries)

The Verification Process

When you connect to a website, here’s what happens:

1. Server Sends Certificate Chain

github.com sends:
┌─────────────────────────────────────┐
│ End-Entity Certificate              │
│ Subject: github.com                 │
│ Issuer: DigiCert TLS RSA SHA256 CA  │ <- Intermediate CA
│ Public Key: [github's key]          │
└─────────────────────────────────────┘
         ↓ signed by
┌─────────────────────────────────────┐
│ Intermediate Certificate            │
│ Subject: DigiCert TLS RSA SHA256 CA │
│ Issuer: DigiCert Global Root CA     │ <- Root CA
│ Public Key: [intermediate's key]    │
└─────────────────────────────────────┘

The root certificate is not sent (you already have it).

2. Build the Chain

Your client assembles the chain from end-entity to root:

github.com 
  → DigiCert TLS RSA SHA256 CA (intermediate)
    → DigiCert Global Root CA (root, in your trust store)

If any certificate is missing, verification fails.

3. Validate Each Link

For each certificate in the chain, verify:

a. Signature Validation

If the signature doesn’t match, someone tampered with the certificate.

b. Validity Period

Expired certificates are rejected (even if signature is valid).

c. Name Constraints

Prevents CAs from issuing certificates outside their authorized scope.

d. Key Usage

Ensures certificates are only used for their designated purpose.

e. Revocation Status (Optional)

Most browsers check OCSP, but it’s often skipped for performance (soft-fail).

4. Verify Hostname

This prevents attackers from using a valid certificate for a different domain.

5. Final Decision

Only if all checks pass does your client trust the connection.

Where Do Root Certificates Live?

This is where it gets interesting. There are three fundamentally different approaches:

Approach 1: Hardcoded in the Application

Architecture:

Your Application
├── Binary contains embedded root certificates
└── No interaction with OS certificate store

Example: Mozilla Firefox, webpki-roots (Rust)

How it works:

The certificates are compiled into your binary. They’re literal byte arrays in your executable.

Pros:

  • ✅ Consistent across all platforms
  • ✅ No OS dependencies
  • ✅ Application fully controls trust decisions
  • ✅ Works in sandboxed environments

Cons:

  • ❌ Must ship updates to add new root CAs
  • ❌ Can’t use user-installed certificates (corporate proxies, development)
  • ❌ Binary size increase (small, ~100KB)
  • ❌ Security updates require application redeployment

When to use:

  • Embedded systems with no OS certificate store
  • Applications that need consistent behavior across platforms
  • When you want full control over trusted CAs

Approach 2: OS Certificate Store (Read-Only)

Architecture:

Your Application
  ↓ reads certificates from
OS Certificate Store
  /usr/share/ca-certificates/      (Linux)
  Keychain                          (macOS)
  Certificate Manager               (Windows)

Example: curl, Go’s crypto/tls, rustls-native-certs (Rust)

How it works:

The application reads certificates from the OS but does validation itself.

Pros:

  • ✅ Uses certificates installed by users/admins
  • ✅ Works with corporate proxies that install custom root CAs
  • ✅ OS handles root CA updates
  • ✅ Respects system security policies

Cons:

  • ❌ Different behavior on different OSes
  • ❌ Must parse various certificate formats (PEM, DER, PKCS#7)
  • ❌ Doesn’t work on iOS/Android (sandboxed, no file access)
  • ❌ Application still responsible for validation logic

When to use:

  • Desktop applications
  • Server-side applications
  • When users need to install custom root CAs
  • Corporate environments with internal CAs

Approach 3: OS Validation APIs (Full Delegation)

Architecture:

Your Application
  ↓ asks OS to validate
OS Certificate Validation API
  ├── SecTrustEvaluate      (macOS/iOS)
  ├── CertVerifyCertificateChainPolicy (Windows)
  └── Custom implementation (Android)

Example: Chrome, Safari, rustls-platform-verifier (Rust)

How it works:

The application delegates everything to the OS. You hand it a certificate, it tells you valid or not.

Pros:

  • ✅ Uses OS certificate store
  • ✅ Uses OS validation logic (handles edge cases)
  • ✅ Proper revocation checking (OS maintains OCSP cache, CRLs)
  • ✅ Works on mobile (iOS, Android)
  • ✅ Benefits from OS security updates automatically
  • ✅ Handles platform-specific policies (certificate pinning, etc.)

Cons:

  • ❌ Different behavior on different platforms
  • ❌ Less control over validation process
  • ❌ Debugging is harder (OS is a black box)
  • ❌ Depends on OS quality (old Android versions have issues)

When to use:

  • Mobile applications (iOS, Android)
  • Applications that need robust revocation checking
  • When you want the OS to handle all edge cases
  • Production web browsers

Case Study: The Rust Ecosystem

Let’s see how real Rust networking crates handle this. This is enlightening because you can see all three approaches in one ecosystem.

The Foundation: rustls

rustls is the pure-Rust TLS implementation. It doesn’t handle root certificates itself. You must provide them:

This design is intentional: rustls provides the TLS protocol, but certificate sourcing is application policy.

Three Certificate Crates

1. webpki-roots (Approach 1: Hardcoded)

What it is: Mozilla’s root certificate list embedded in the binary.

Update process:

  • Mozilla updates their root store
  • webpki-roots maintainers pull changes
  • You bump your dependency and rebuild
  • Users get updated roots when you deploy

2. rustls-native-certs (Approach 2: OS Store)

What it does:

  • Linux: Reads /etc/ssl/certs, /usr/share/ca-certificates
  • macOS: Queries Keychain API
  • Windows: Uses CertOpenSystemStore API

Limitations:

  • ❌ Doesn’t work on iOS (no file system access)
  • ❌ Doesn’t work on Android (different certificate storage)
  • ⚠️ Only loads certificates, validation is still rustls’s responsibility

3. rustls-platform-verifier (Approach 3: OS APIs)

What it does:

  • macOS/iOS: Uses SecTrustEvaluate
  • Windows: Uses CertVerifyCertificateChainPolicy
  • Android: Uses Java X509TrustManager via JNI
  • Linux: Falls back to rustls-native-certs + rustls validation

Why it’s better for mobile:

iOS App using webpki-roots:
  ✗ Can't trust user-installed enterprise certificates
  ✗ Can't access iOS's OCSP cache
  ✗ Ignores iOS certificate policies

iOS App using rustls-platform-verifier:
  ✓ Uses iOS's certificate validation
  ✓ Respects iOS security policies
  ✓ Handles revocation properly

How a Typical Rust Project Handles It

Let’s trace through common crates:

reqwest (HTTP Client)

Certificate source:

  • If rustls-tls-webpki-roots feature: Uses webpki-roots (Approach 1)
  • If rustls-tls-native-roots feature: Uses rustls-native-certs (Approach 2)
  • No option for rustls-platform-verifier (Approach 3)

Problem: Pull request #2286 tried to add platform verifier support but is blocked by rustls-platform-verifier #58.

Workaround: None currently. You’re stuck with Approach 1 or 2.

async-tungstenite (WebSocket Client)

Certificate source:

  • Default: Feature-dependent (webpki-roots or rustls-native-certs)
  • Custom: Can use any ClientConfig you build (including platform verifier!)

Flexibility: High. You can use Approach 3:

quinn (QUIC Implementation)

Certificate source:

  • Explicit API for platform verifier (Approach 3)
  • Explicit API for custom root store (Approach 1 or 2)

Flexibility: Excellent. quinn directly exposes both options.

Trade-offs: When to Use Each Approach

Use Approach 1 (Hardcoded) When:

✅ Embedded systems with no OS certificate infrastructure
✅ Consistency is critical across all platforms
✅ Sandboxed environments (some WASM scenarios)
✅ Testing/development with controlled certificate sets

❌ Don’t use when:

  • Users need to install custom root CAs (corporate proxies)
  • You’re deploying on mobile (iOS/Android)
  • You need robust revocation checking

Use Approach 2 (OS Store) When:

✅ Desktop applications (Linux/macOS/Windows)
✅ Server-side applications
✅ Corporate environments with custom internal CAs
✅ You need explicit control over validation logic

❌ Don’t use when:

  • Deploying on mobile (iOS/Android)
  • You want OS-level revocation checking
  • Certificate parsing is a concern (format variations)

Use Approach 3 (OS APIs) When:

✅ Mobile applications (iOS, Android)
✅ Production web browsers
✅ Robust revocation checking is required
✅ You want the OS to handle edge cases

❌ Don’t use when:

  • You need consistent behavior across platforms (testing)
  • You want full control over validation process
  • You’re debugging certificate issues (OS is opaque)

Platform Differences

Different operating systems have different certificate infrastructures:

Linux

Store location:

Format: Usually PEM (Base64 encoded)

Management:

APIs: No standard OS API. Applications read files directly.

macOS

Store location: Keychain (database, not files)

Access:

APIs: Security.framework, SecTrustEvaluate()

Windows

Store location: Certificate Store (registry-based)

Access:

APIs: CryptoAPI, CertVerifyCertificateChainPolicy()

iOS

Store location: Keychain (sandboxed, per-app)

Access: No direct file access. Use SecTrustEvaluate() API.

User certificates: Settings → General → About → Certificate Trust Settings

Key limitation: Apps can’t access the certificate files directly. Must use OS validation APIs.

Android

Store location:

/system/etc/security/cacerts/     # System CAs
/data/misc/user/*/cacerts-added/  # User CAs

Access: Apps use Java X509TrustManager API, not direct file access.

Key limitation: Since Android 7 (API 24), apps don’t trust user-installed CAs by default (must opt-in via manifest).

The Big Picture

Certificate verification isn’t a single algorithm, it’s a policy decision about trust architecture:

┌─────────────────────────────────────────────────────┐
│              Certificate Verification               │
└─────────────────────────────────────────────────────┘
                        │
        ┌───────────────┼───────────────┐
        │               │               │
        ▼               ▼               ▼
  Approach 1       Approach 2      Approach 3
  (Hardcoded)      (OS Store)      (OS API)
        │               │               │
        ├──────────┬────┴────┬──────────┤
        │          │         │          │
     Embedded   Desktop   Server    Mobile
     Systems              Side

The trend: Industry is moving toward Approach 3 (OS validation APIs):

  • Chrome switched from NSS to OS validators
  • Safari has always used OS validation
  • Mobile apps require it for proper security

Why?

  1. Revocation checking is hard (OCSP, CRL, certificate transparency)
  2. Edge cases are endless (name constraints, policy constraints, cross-certification)
  3. Security updates are critical (OS vendors can push fixes faster than app updates)
