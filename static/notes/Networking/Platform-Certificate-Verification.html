WTF is Platform Certificate Verification?

Your TLS code works on Linux. It crashes on Android. Welcome to platform verification.

You write a TLS client in Rust. It connects fine on your Linux dev machine. You cross-compile for Android, deploy — and it rejects perfectly valid certificates. Same code, same server, same cert. WTF?

Certificate verification is not a universal operation. Every operating system has its own trust store, its own verification engine, and its own opinions about what “valid” means. Linux is lenient. Android is strict. iOS is somewhere in between. If you’re shipping a cross-platform app, you need to understand the differences or your users will stare at connection errors.

The Core Analogy

Three countries, three passport offices:

  • Linux is the laid-back country. “Passport looks legit? Come on in. We couldn’t check the revocation list, but whatever — you’re probably fine.”
  • Android is the strict country. “Passport looks legit, but we can’t verify it wasn’t revoked? Denied. Come back when we can confirm.”
  • iOS is the efficient country. “We’ll check everything behind the scenes using our own systems. You don’t need to configure anything.”

All three accept the same passports (certificates). But their verification process — what they check and how they handle failures — is completely different.

What Verification Actually Involves

When a TLS client connects to a server, it receives a certificate chain. Verifying that chain involves:

  1. Chain building — constructing a path from the server cert to a trusted root CA
  2. Signature verification — checking each cert is signed by the next in the chain
  3. Expiration check — is the cert still valid?
  4. Revocation check — has the cert been revoked? (via OCSP or CRL)
  5. Name verification — does the cert match the hostname?

Steps 1-3 and 5 are straightforward. Step 4 is where everything falls apart.

The Revocation Problem

Certificate revocation is how CAs say “this cert was compromised, don’t trust it.” Two mechanisms exist:

OCSP (Online Certificate Status Protocol) — ask the CA’s responder “is this cert revoked?” in real-time. Fast but adds latency and leaks which sites you visit to the OCSP server.

CRL (Certificate Revocation List) — download a list of all revoked certs from the CA. No per-request privacy leak, but the lists are large.

OCSP Stapling — the server pre-fetches its own OCSP response and includes it in the TLS handshake. Best of both worlds: no extra round trip, no privacy leak.

Here’s where it gets messy: Let’s Encrypt deprecated their OCSP responders in 2025. Certs now include CRL URLs instead. Those CRL URLs use plain HTTP (http://c.lencr.org/). And Android blocks cleartext HTTP by default.

How Each Platform Handles It

Linux (webpki / OpenSSL)

Most Rust TLS libraries on Linux use either webpki (via rustls) or OpenSSL for verification.

Trust store: /etc/ssl/certs/ or the Mozilla root CA bundle.

Revocation: Fail-open. If the OCSP responder is unreachable or the CRL can’t be fetched, the certificate is accepted anyway. The philosophy: a revocation check failure shouldn’t break connectivity.

Certificate valid? -> Accept
OCSP check fails?  -> Accept anyway (fail-open)
CRL unreachable?   -> Accept anyway (fail-open)

This is why your app works on Linux. It’s forgiving.

Android (TrustManager)

Android uses the Java TrustManager for certificate verification, which delegates to the system’s certificate store.

Trust store: System CA certificates in /system/etc/security/cacerts/ plus user-installed CAs.

Revocation: Fail-closed. If Android can’t verify revocation status, the certificate is rejected. The philosophy: if we can’t confirm it’s not revoked, we don’t trust it.

Certificate valid?  -> Continue to revocation check
OCSP check fails?   -> REJECT
CRL unreachable?    -> REJECT

The Let’s Encrypt CRL is served over http://c.lencr.org/ — and Android’s default network security config blocks cleartext HTTP. So the CRL fetch fails, and Android rejects the cert.

Setting cleartextTrafficPermitted="true" globally disables HTTP protection for your entire app. Only allow it for specific CRL domains. CRL URLs are intentionally HTTP (the data isn’t sensitive) — this is by design, not a bug.

iOS (Security.framework)

iOS handles verification through Apple’s Security.framework.

Trust store: Apple’s curated CA root store, updated via OS updates.

Revocation: Handled automatically. Security.framework manages OCSP and CRL checks internally, including caching and fallback. No configuration needed.

Certificate valid?  -> Accept
Revocation check?   -> Handled transparently by the OS
CRL over HTTP?      -> No problem (iOS handles it internally)

This is why iOS “just works.” Apple’s framework does the heavy lifting.

Summary

Certificate verification is platform-specific. Linux is fail-open (accepts certs when revocation checks fail). Android is fail-closed (rejects them). iOS handles everything transparently via Security.framework. When shipping cross-platform Rust, use rustls-platform-verifier to delegate to each OS’s native stack. On Android, prepare for JNI initialization, companion AAR dependencies, and network security config exceptions for CRL endpoints. On iOS and desktop, it just works.
