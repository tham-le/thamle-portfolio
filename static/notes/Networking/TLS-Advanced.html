TLS: Under the Hood (Advanced)

From handshake packets to certificate chains: how TLS actually works

You’ve read WTF is TLS? and understand that TLS encrypts your data, authenticates servers, and prevents tampering. Now let’s get into the details: how does the handshake actually work? What’s inside those certificates? Why is QUIC’s approach different from TCP+TLS?

This is the deep dive.

Prerequisites

This article assumes you understand:

  • Basic TLS concepts (encryption, authentication, integrity)
  • TCP and UDP fundamentals
  • What HTTP and HTTPS are

If not, start with WTF is TLS? first.

TLS Protocol Architecture

TLS is actually two protocols in one:

┌─────────────────────────────────────┐
│        TLS Handshake Protocol       │  <- Negotiate security
│   (Key exchange, authentication)    │
├─────────────────────────────────────┤
│        TLS Record Protocol          │  <- Encrypt data
│   (Fragmentation, compression,      │
│    encryption, authentication)      │
└─────────────────────────────────────┘

TLS Handshake Protocol:

  • Negotiates cipher suites
  • Exchanges keys
  • Authenticates peers
  • Runs once per connection

TLS Record Protocol:

  • Fragments application data into records
  • Encrypts each record
  • Adds authentication tags
  • Runs continuously during the session

This separation is important: the handshake sets everything up, then the record layer handles the heavy lifting of encrypting your actual data.

TLS 1.3 Handshake (Technical Details)

TLS 1.3 (RFC 8446, August 2018) was a major overhaul. Let’s trace exactly what happens.

Full Handshake (1-RTT)

Client                                               Server

Key  ^ ClientHello
Exch | + key_share*
     | + signature_algorithms*
     | + psk_key_exchange_modes*
     v + pre_shared_key*       -------->
                                                  ServerHello  ^ Key
                                                 + key_share*  | Exch
                                            + pre_shared_key*  v
                                        {EncryptedExtensions}  ^  Server
                                        {CertificateRequest*}  v  Params
                                               {Certificate*}  ^
                                         {CertificateVerify*}  | Auth
                                                   {Finished}  v
                               <--------  [Application Data*]
     ^ {Certificate*}
Auth | {CertificateVerify*}
     v {Finished}              -------->
       [Application Data]      <------->  [Application Data]

Breaking it down:

1. ClientHello (Client → Server)

Key innovation: Client sends its key_share immediately. TLS 1.2 required an extra round trip to negotiate the key exchange algorithm first.

2. ServerHello + More (Server → Client)

The server responds with multiple messages (encrypted after ServerHello):

Critical security feature: Everything after ServerHello is encrypted. TLS 1.2 sent certificates in plaintext, leaking which sites you visited.

3. Client Authentication (Client → Server)

Both sides have now verified:

  • The handshake wasn’t tampered with
  • They derived the same keys
  • (Optionally) client provided a certificate

Total round trips: 1 (down from 2 in TLS 1.2)

0-RTT Mode (Zero Round Trip Time)

For resumed connections, TLS 1.3 allows sending data immediately:

Client                                               Server

ClientHello
+ early_data
+ key_share*
+ psk_key_exchange_modes
+ pre_shared_key
(Application Data*)        -------->
                                                  ServerHello
                                             + pre_shared_key
                                                 + key_share*
                                        {EncryptedExtensions}
                                                   {Finished}
                           <--------  [Application Data*]
(EndOfEarlyData)
{Finished}                 -------->
[Application Data]         <------->  [Application Data]

How it works:

  1. Client and server shared a PSK (Pre-Shared Key) from a previous session
  2. Client encrypts data with derived key from PSK
  3. Sends encrypted data in first flight
  4. Server processes early data while handshake completes

0-RTT Replay Attacks

Early data is vulnerable to replay attacks. An attacker can capture the first ClientHello + early data and replay it later. The server sees what looks like a valid request.

Mitigation: Only use 0-RTT for idempotent operations (GET requests, not POST). Many implementations disable it by default.

What TLS 1.3 Removed

TLS 1.3 aggressively pruned dangerous features from TLS 1.2:

Removed FeatureWhy It Was Dangerous
RSA key exchangeNo forward secrecy (compromise private key → decrypt all past sessions)
Static DHSame problem as RSA
CBC mode ciphersBEAST, Lucky13, POODLE attacks
RC4 stream cipherBiased output, practical attacks
SHA-1, MD5Collision attacks demonstrated
CompressionCRIME attack (compression side-channel)
RenegotiationComplex attack surface, removed entirely
Custom DHE parametersLogjam attack, weak parameter choices

What’s left: Only AEAD (Authenticated Encryption with Associated Data) ciphers:

  • AES-GCM
  • ChaCha20-Poly1305
  • AES-CCM

All require forward secrecy (ephemeral key exchange).

Certificate Deep Dive

Certificates are X.509 documents that bind a public key to an identity. Let’s decode one.

X.509 Certificate Structure

Certificate (TBSCertificate):
┌────────────────────────────────────────────┐
│ Version: 3                                 │
│ Serial Number: 0x04:7f:3a:23:...(20 bytes)│
├────────────────────────────────────────────┤

├────────────────────────────────────────────┤
│ Issuer:                                    │
│   CN = DigiCert TLS RSA SHA256 2020 CA1   │

│   C = US                                   │
├────────────────────────────────────────────┤
│ Validity:                                  │
│   Not Before: 2024-11-01 00:00:00 UTC     │

├────────────────────────────────────────────┤

└────────────────────────────────────────────┘
Signature Algorithm: sha256WithRSAEncryption
Signature: 4a:2b:c9:8f:...

Key fields explained:

Serial Number: Unique identifier for this certificate. Used in revocation lists.

Issuer: The CA that signed this certificate. This creates the chain of trust.

Subject: Who this certificate belongs to. For TLS, this is usually a domain name.

Subject Alternative Names (SANs): Additional domains this certificate is valid for. Modern certificates use SANs instead of CN.

Key Usage: What the key can be used for (signing, encryption, etc.)

Authority Information Access (AIA): Where to check certificate status:

  • OCSP: Online Certificate Status Protocol (real-time check)
  • CRL: Certificate Revocation List (batch check)

Certificate Revocation

How do you invalidate a certificate before it expires?

CRL (Certificate Revocation List)

CRL Structure:
┌────────────────────────────────────┐
│ Issuer: DigiCert CA                │
│ This Update: 2025-12-30 00:00      │
│ Next Update: 2025-12-31 00:00      │
├────────────────────────────────────┤
│ Revoked Certificates:              │
│   Serial: 0x1234...                │
│   Revocation Date: 2025-12-15      │
│   Reason: Key Compromise           │
│                                    │
│   Serial: 0x5678...                │
│   Revocation Date: 2025-12-20      │
│   Reason: Certificate Hold         │
└────────────────────────────────────┘

Problem: CRLs can be huge (megabytes). Browsers don’t want to download them.

OCSP (Online Certificate Status Protocol)

Better approach: Ask the CA in real-time:

OCSP Request:
  Certificate Serial: 0x1234...
  Hash Algorithm: SHA-256

OCSP Response:
  Certificate Status: good
  This Update: 2025-12-30 12:00
  Next Update: 2025-12-31 12:00
  Signature: [signed by CA]

Problem: Privacy leak. The CA learns which sites you visit.

OCSP Stapling

Best approach: Server gets OCSP response and staples it to the certificate during handshake:

Server -> Client:
  Certificate
  + OCSP Response (signed by CA, fresh timestamp)

Benefits:

  • No privacy leak (client doesn’t contact CA)
  • Faster (no extra round trip)
  • Reduces CA load

Certificate Transparency

Modern solution: All certificates are logged in public append-only logs (Certificate Transparency logs):

  1. CA issues certificate
  2. CA submits certificate to CT logs
  3. CT logs return Signed Certificate Timestamps (SCTs)
  4. Server presents SCTs during handshake
  5. Browser verifies SCTs prove certificate was logged

Why this matters: Prevents CAs from issuing fraudulent certificates secretly. If a bad certificate is issued, it appears in public logs where anyone can detect it.

Certificate Validity Periods

The industry is moving toward shorter certificate lifespans:

YearMaximum Validity
201539 months (3.25 years)
2018825 days (27 months)
2020398 days (13 months)
202947 days (approved)

Why shorter?

  • Limits damage from compromised keys
  • Forces automation (Let’s Encrypt model)
  • Reduces reliance on revocation checking
  • Faster crypto algorithm migration

Protocol Stack Integration

How does TLS fit with different transport protocols?

Traditional: TCP + TLS

┌──────────────────┐
│   Application    │  <- Your code: HTTP, SMTP, etc.
├──────────────────┤
│   TLS Layer      │  <- Encryption, authentication
│  (TLS 1.3)       │
├──────────────────┤
│   TCP Layer      │  <- Reliable transport
│  (Stream)        │     - In-order delivery
├──────────────────┤     - Retransmission
│   IP Layer       │     - Flow control
└──────────────────┘

Connection establishment:

Timeline of TCP + TLS handshake:

0ms    Client -> Server: TCP SYN
20ms   Server -> Client: TCP SYN-ACK
40ms   Client -> Server: TCP ACK
       ✓ TCP connection established (1 RTT)

40ms   Client -> Server: TLS ClientHello
60ms   Server -> Client: TLS ServerHello + Certificate + Finished
80ms   Client -> Server: TLS Finished
       ✓ TLS connection established (1 RTT)

Total: 2 RTT before application data

Characteristics:

  • 2 RTT minimum (TCP handshake + TLS handshake)
  • Head-of-line blocking: One lost packet blocks entire stream
  • Separate layers: TLS doesn’t know about TCP, TCP doesn’t know about TLS

Modern: QUIC (Integrated TLS 1.3)

QUIC (RFC 9000, May 2021) embeds TLS 1.3 directly:

┌──────────────────┐
│   Application    │  <- HTTP/3, DNS-over-QUIC
├──────────────────┤
│   QUIC Layer     │  <- Transport + TLS integrated
│  - Streams       │     - TLS 1.3 handshake
│  - Reliability   │     - Built-in encryption
│  - Encryption    │     - Per-stream ordering
├──────────────────┤
│   UDP Layer      │  <- Unreliable datagrams
└──────────────────┘

Connection establishment:

Timeline of QUIC handshake:

0ms    Client -> Server: Initial Packet
                         + ClientHello
                         + key_share
20ms   Server -> Client: Initial + Handshake Packets
                         + ServerHello
                         + Certificate
                         + Finished
40ms   Client -> Server: Handshake Packet
                         + Finished
       ✓ Connection established (1 RTT)

With 0-RTT:
0ms    Client -> Server: Initial Packet
                         + Early Application Data
       ✓ Data sent immediately (0 RTT)

Key differences:

FeatureTCP + TLSQUIC
Handshake2 RTT1 RTT (or 0 RTT)
EncryptionOptionalMandatory
Head-of-line blockingYes (single stream)No (multiple streams)
Connection migrationNo (tied to 4-tuple)Yes (connection ID)
Ossification resistancePoor (middleboxes inspect)Good (encrypted headers)

QUIC’s TLS Integration (RFC 9001)

QUIC doesn’t use TLS as a separate layer. Instead:

  1. QUIC acts as the TLS record layer:

    • QUIC packets replace TLS records
    • QUIC provides encryption, not TLS

  2. TLS provides only the handshake:

    • Key exchange (Diffie-Hellman)
    • Authentication (certificates)
    • Key derivation

  3. Cryptographic handshake messages are in CRYPTO frames:

QUIC Initial Packet:
┌────────────────────────────────────┐
│ Header (unencrypted):              │
│   Type: Initial                    │
│   Version: 1                       │
│   Destination Connection ID        │
│   Source Connection ID             │
│   Packet Number                    │
├────────────────────────────────────┤
│ Payload (encrypted with initial    │
│          keys derived from conn    │
│          IDs):                     │
│   CRYPTO Frame:                    │
│     ClientHello (TLS 1.3)          │
│   PADDING Frame                    │
└────────────────────────────────────┘

Why this is brilliant:

  • No double encryption: TLS normally encrypts, then QUIC would encrypt again (wasteful)
  • Faster: Handshake and transport setup combined
  • Mandatory security: Can’t run QUIC without encryption
  • Ossification resistance: Nearly everything is encrypted, preventing middlebox interference

Computational Cost

Modern CPUs have hardware acceleration for crypto:

AES-GCM (common TLS cipher):

  • Software: ~1-2 GB/s per core
  • AES-NI (Intel/AMD): 10-20 GB/s per core
  • ARMv8 Crypto: 5-10 GB/s per core

ChaCha20-Poly1305 (mobile-friendly):

  • Software: 2-4 GB/s per core
  • NEON (ARM): 5-8 GB/s per core

Bottom line: Encryption is essentially free on modern hardware. TLS overhead is dominated by handshake latency, not CPU cost.

Bandwidth Overhead

Per-record overhead (TLS 1.3):

TLS Record Header:
  Content Type: 1 byte
  Legacy Version: 2 bytes
  Length: 2 bytes
  = 5 bytes header

AEAD Authentication Tag:
  = 16 bytes (GCM/Poly1305)

Total overhead per record: 21 bytes

For a 1400-byte Ethernet frame:

  • Useful payload: ~1379 bytes
  • Overhead: ~1.5%

Negligible for most applications.

TLS False Start

Optimization in TLS 1.2 (obsoleted by TLS 1.3’s 1-RTT):

Client sends application data before receiving server’s Finished message:

Client                         Server
ClientHello         -------->
                    <--------  ServerHello
                               Certificate
                               ServerKeyExchange
                               ServerHelloDone
ClientKeyExchange
ChangeCipherSpec
Finished
Application Data    -------->  <- False Start!
                    <--------  ChangeCipherSpec
                               Finished
Application Data    <------->  Application Data

Risk: Client sends data before server is fully authenticated. Requires additional security properties from cipher suite.

TLS 1.3 makes this unnecessary by completing handshake in 1-RTT.

Security Considerations

Forward Secrecy

Definition: Compromise of long-term keys (server’s private key) doesn’t allow decryption of past sessions.

How TLS 1.3 achieves this:

Every connection uses ephemeral key exchange:

Connection 1:
  Client generates ephemeral DH private key
  Server generates ephemeral DH private key
  Both derive session key
  ✓ Session key destroyed after connection

Connection 2:
  NEW ephemeral keys generated
  Completely unrelated to Connection 1
  ✓ Independent session key

If server's long-term private key is compromised:
  → Attacker can impersonate server in future
  ✗ But cannot decrypt past recorded sessions

TLS 1.2’s RSA key exchange doesn’t have forward secrecy:

RSA key exchange (removed in TLS 1.3):
  Client encrypts session key with server's public key
  Attacker records encrypted session

Later, if server's private key is compromised:
  ✓ Attacker can decrypt session key
  ✓ Attacker can decrypt entire recorded session

This is why TLS 1.3 removed RSA key exchange entirely.

Additional protection (downgrade sentinels):

If server is forced to negotiate TLS 1.2 or earlier, it includes a special random value in ServerHello:

TLS 1.2 downgrade:
  server_random = [...24 bytes...] + 0x44 0x4F 0x57 0x4E 0x47 0x52 0x44 0x01
                                     ^^^^^^^^^^^^ "DOWNGRD\x01"

TLS 1.1 or earlier:
  server_random = [...24 bytes...] + 0x44 0x4F 0x57 0x4E 0x47 0x52 0x44 0x00
                                     ^^^^^^^^^^^^ "DOWNGRD\x00"

TLS 1.3 clients detect this sentinel and abort the connection, preventing downgrade.
