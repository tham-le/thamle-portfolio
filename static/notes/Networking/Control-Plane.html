WTF is a Control Plane?

WireGuard is a sports car with no steering wheel. The control plane is the steering wheel.

You’ve gone through TUN-TAP and NAT-Traversal. You know how packets flow through a TUN device and how two peers behind NAT punch through to a direct connection. At this point you could technically wire up a WireGuard tunnel manually.

Now do this for every peer. On every device. Every time you add a machine. Every time a key rotates. Every time a device changes IP. Every time someone leaves the company and you need to revoke their access across every other device in the network.

That’s the problem the control plane solves.

The Core Analogy

Think of raw WireGuard like a collection of phones where everyone can call anyone, but only if they already have each other’s private number memorized. No directory. No contacts app. You want to add a new person to the group? You have to personally call every existing member and give them the new number, then call the new person and give them everyone else’s number.

A control plane is the contacts app. It maintains the directory, distributes numbers automatically when someone joins, revokes them when someone leaves, and handles the fact that people’s phone numbers change constantly.

What WireGuard Deliberately Left Out

WireGuard’s author Jason Donenfeld made an explicit design decision: WireGuard has no control plane. This wasn’t an oversight. The whitepaper states it directly. WireGuard is a cryptographic tunnel primitive. It does one thing: encrypt and route packets between a set of manually configured peers. Key exchange, peer discovery, dynamic routing updates, access control policies, all of that is intentionally out of scope.

This is a beautiful choice for a low-level tool. It keeps the codebase to around 4,000 lines in the kernel, which makes it auditable by a single person in a few days. It has no attack surface for certificate management bugs, no TLS stack, no user database. It is exactly as complex as the tunnel layer needs to be.

But for any real deployment beyond two machines, you need something that manages the configuration. That something is the control plane.

What a Control Plane Actually Does

A control plane for WireGuard needs to solve four problems:

1. Key distribution. Every peer needs every other peer’s public key. In a network of N devices, that’s N×(N-1) key relationships to maintain. The control plane holds the authoritative list and pushes updates when it changes.

2. Endpoint tracking. Devices move. Laptops switch networks. Home IP addresses change. The control plane tracks the current public endpoint (IP:port) of each device and distributes updates so peers can find each other.

3. IP address assignment. Tunnel IPs (the 10.x.x.x addresses inside the VPN) need to be unique and consistently assigned. The control plane acts as the IPAM (IP Address Management) layer.

4. Access control. Not every device should talk to every other device. The control plane enforces policies: who can reach what, on which ports, under what conditions.

The Control Plane Carries Almost No Traffic

Here’s the elegant part of the architecture: the control plane only distributes metadata (keys, endpoints, policies). It never touches your actual data. Once two peers know each other’s keys and endpoints, all data flows directly between them over WireGuard UDP. The control plane can be slow, centralized, and even briefly unavailable without interrupting established tunnels. The data plane is fully distributed and peer-to-peer.

The Three Implementations

Tailscale: Centralized, Managed

Tailscale’s coordination server at login.tailscale.com is a centralized key-value store for your tailnet. When a device joins:

  1. It authenticates via an identity provider (Google, GitHub, Okta, any OIDC provider).
  2. It registers its WireGuard public key and current endpoint.
  3. The coordination server distributes a network map to all other devices: “Here is the updated list of peers with their keys and endpoints.”
  4. Each device applies the map to its local WireGuard configuration.

The coordination server also handles ACLs (Access Control Lists) defined in a HuJSON policy file. Those policies are pushed down to each device as part of the network map. The device enforces them locally using packet filtering.

Tailscale architecture:

    Coordination Server (login.tailscale.com)
           |            |            |
    [key store]   [endpoint map]  [ACL policy]
           |            |            |
    -------+------------+------------+-------
           |                         |
      Device A                   Device B
      [wg config]                [wg config]
           |                         |
           +------- Direct WireGuard UDP ----+
                    (data plane, P2P)

The coordination server is the hub, but only for control messages. Data is always peer-to-peer when NAT traversal succeeds.

Headscale: Self-Hosted Open Source Control Plane

Headscale is a community-built reimplementation of the Tailscale coordination server protocol. It’s compatible with the official Tailscale clients, meaning you install the standard Tailscale app on your devices but point it at your own Headscale server instead of login.tailscale.com.

This gives you:

  • Full data sovereignty: no metadata leaves your infrastructure
  • No device limits (Tailscale free tier caps at 100 devices)
  • Full control over user management and ACL policies

The trade-off is operational burden. You run the server, maintain it, back it up, and handle availability.

NetBird: Self-Hostable, Peer-Centric

NetBird is a fully open-source mesh VPN with its own control plane implementation called the Management Server. Unlike Tailscale’s proprietary coordination server (Headscale reimplements the protocol but not the server), NetBird ships both the client and the server as open source under a permissive license.

NetBird’s architecture adds one concept Tailscale doesn’t have: Signal Server for peer discovery using WebRTC-style ICE candidate exchange, and a Relay Server (their TURN equivalent) as fallback when direct connections fail.

NetBird architecture:

  Management Server          Signal Server
  (key store, ACLs)    (ICE/STUN coordination)
         |                       |
    -----+-------+---------------+------
         |       |               |
     Device A    +--- Relay ----+   Device B
     [wg config]    (fallback)       [wg config]
         |                               |
         +------- Direct WireGuard UDP --+

The management server handles the same job as Tailscale’s coordination server: key distribution, policy enforcement, and network map delivery. But the signal server handles the real-time peer discovery (exchanging ICE candidates for hole punching), which Tailscale handles through DERP.

Key Rotation and Revocation

One of the most practically important things the control plane handles is key lifecycle.

Without a control plane, revoking access for a departed employee means logging into every single device in your network and removing that person’s public key from the WireGuard [Peer] sections. With 50 devices, that’s 50 SSH sessions.

With a control plane, you remove the user from the coordination server. It pushes an updated network map to all devices simultaneously. Within seconds, every device drops the peer’s public key from their WireGuard configuration and the revoked device can no longer establish tunnels to anyone. One action, instant effect everywhere.

The same applies to key rotation. When a device generates a new keypair (for security hygiene or because the old key was potentially compromised), it registers the new key with the coordination server, which propagates the update. All peers update their config. Old tunnels using the previous key expire as WireGuard’s session timer runs out.

The Privacy Trade-off

There’s an honest trade-off here worth naming.

When you use Tailscale’s hosted coordination server, Tailscale’s infrastructure knows:

  • Which devices are in your network
  • Each device’s current public IP address and endpoint
  • Your network topology and routing configuration
  • Which ACL policies you’ve defined

What they cannot see: your actual data. WireGuard keys are generated on the device and never leave it. The coordination server only ever sees public keys, never private keys. All data is end-to-end encrypted.

If your threat model requires that nobody outside your organization knows your network topology, Headscale or NetBird self-hosted are the right choices. If you just need secure encrypted connectivity and trust Tailscale as a service provider, the hosted option is operationally simpler.

The "Zero Trust" Label

Both Tailscale and NetBird market themselves as “zero trust networking.” What this means in practice: each device authenticates with its WireGuard key, and ACL policies are enforced per-device at the packet level rather than relying on network perimeter. A compromised device inside the VPN can only reach what it’s explicitly allowed to reach, not everything. This is the meaningful part of “zero trust” in this context.

What’s Next

You now understand the full three-layer architecture of a modern mesh VPN: the TUN device that intercepts packets, the NAT traversal mechanism that punches through to direct P2P connections, and the control plane that manages the configuration layer above all of it.
