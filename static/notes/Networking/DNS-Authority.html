WTF is DNS Authority?

WTF is DNS Authority?

Nov 14, 202510 min read

  • networking
  • dns
  • infrastructure

Or: How does the entire internet agree on what google.com means?

You just learned how DNS works. Your computer asks “where’s github.com?” and gets back an IP address. But here’s the real question: who decided that github.com points to that specific IP address? And how does every DNS server on Earth know the same answer?

Welcome to the world of DNS authority, where we’ll discover who actually controls the internet’s naming system.

The Core Analogy

Think of DNS authority like a giant library system with a strict hierarchy.

At the top, you have the Library of Congress (the root). They don’t have every book, but they have a master catalog telling you which regional library handles which topics.

Then you have regional libraries (.com, .org, .uk). They don’t store every book either, but they know which local library has what you need.

Finally, you have local libraries (github.com, google.com). These actually have the books (IP addresses) you’re looking for.

Each level has authority over its section. The Library of Congress can’t tell your local library which books to stock, but they can tell everyone where to find your local library.

Beginner: The Trust Hierarchy

Who’s in Charge?

DNS has a clear chain of command:

Root Zone (.)
    ↓
ICANN (Internet Corporation for Assigned Names and Numbers)
    ↓
TLD Operators (.com, .org, .uk, etc.)
    ↓
Domain Registrars (GoDaddy, Namecheap, Google Domains, etc.)
    ↓
Domain Owners (You!)
    ↓
Authoritative Nameservers (Your DNS provider)

Let’s break this down from the top.

The Root Zone: The Master List

At the very top is the root zone. It’s literally a text file that lists every TLD (.com, .org, .uk, .pizza) and which servers are authoritative for them.

Here’s what part of it looks like:

com.                 172800  IN  NS  a.gtld-servers.net.
com.                 172800  IN  NS  b.gtld-servers.net.
org.                 172800  IN  NS  a0.org.aone.net.
uk.                  172800  IN  NS  ns1.nic.uk.

This file is managed by ICANN and IANA (Internet Assigned Numbers Authority). They’re nonprofit organizations that coordinate the global internet infrastructure.

The 13 Root Servers (Sort Of)

There are “13” root DNS servers named A through M:

a.root-servers.net.
b.root-servers.net.
c.root-servers.net.
...
m.root-servers.net.

But here’s the trick: there aren’t actually 13 physical servers. There are hundreds of servers worldwide, all using anycast routing. When you query a.root-servers.net, your request goes to the nearest server in that cluster.

Why 13 names? Historical reasons. Early DNS packets could only fit 13 server addresses in a UDP response without fragmenting.

TLD Operators: Managing .com, .org, etc.

Each TLD has an operator:

  • .com and .net: Verisign
  • .org: Public Interest Registry (PIR)
  • .uk: Nominet
  • .io: Internet Computer Bureau (ICB)
  • .pizza: Yes, this exists. Someone manages it.

These operators maintain the authoritative DNS servers for their TLD. When you register mysite.com, you’re asking Verisign (through a registrar) to add an entry to the .com zone.

Domain Registrars: Your Gateway

You don’t talk to Verisign directly. You use a registrar like:

  • GoDaddy
  • Namecheap
  • Google Domains (now Squarespace)
  • Cloudflare
  • Gandi

Registrars are accredited by ICANN to sell domain names. They handle the paperwork and money, then tell the TLD operator to update the zone file.

What Happens When You Register a Domain

Let’s say you register coolproject.com:

  1. You pay the registrar (10−15/year for .com)
  2. Registrar contacts Verisign: “Add coolproject.com to the .com zone”
  3. You provide nameserver addresses: Where should DNS queries for coolproject.com go? 
    ns1.cloudflare.com
ns2.cloudflare.com
  4. Verisign updates the .com zone with your nameservers
  5. Your nameserver serves the actual records (A, AAAA, MX, etc.)

Now, when someone looks up coolproject.com:

Root → "Ask .com servers"
.com → "Ask ns1.cloudflare.com"
ns1.cloudflare.com → "coolproject.com is at 104.21.48.23"

Intermediate: How Synchronization Works

The Zone File System

DNS isn’t a single database. It’s millions of independent databases (zone files) that reference each other.

A zone file is a text file listing all DNS records for a domain:

; Zone file for example.com
$TTL 3600
@       IN  SOA   ns1.example.com. admin.example.com. (
                  2025111401 ; Serial number (YYYYMMDDnn)
                  7200       ; Refresh (2 hours)
                  3600       ; Retry (1 hour)
                  1209600    ; Expire (2 weeks)
                  86400 )    ; Minimum TTL (1 day)

@       IN  NS    ns1.example.com.
@       IN  NS    ns2.example.com.
@       IN  A     93.184.216.34
www     IN  A     93.184.216.34
mail    IN  A     93.184.216.35
@       IN  MX    10 mail.example.com.

Key fields:

  • SOA (Start of Authority): Metadata about this zone
  • Serial number: Incremented every time the zone changes
  • NS records: Which servers are authoritative for this zone
  • A/AAAA records: The actual IP addresses

DNS Propagation: Why Changes Take Time

When you update a DNS record, it doesn’t instantly affect everyone. Why?

TTL (Time To Live) tells caches how long to keep a result:

example.com.  3600  IN  A  93.184.216.34
              ^^^^
              This is the TTL (1 hour)

If you change example.com to point to a new IP, here’s what happens:

  1. You update your zone file on your authoritative nameserver
  2. Recursive resolvers (8.8.8.8, etc.) still have the old answer cached
  3. After 3600 seconds (1 hour), the cache expires
  4. Next query fetches the new answer

This is why “DNS propagation takes 24-48 hours” is a thing. It’s not that DNS is slow, it’s that:

  • Different resolvers cache at different times
  • Some ignore TTL recommendations (bad behavior, but it happens)
  • Your ISP might have multi-level caching

Pro tip: Before making a critical DNS change, lower your TTL to 300 seconds (5 minutes) a day in advance. Then make your change. Then raise TTL back to normal.

Zone Transfers: How Nameservers Stay in Sync

If you have multiple nameservers (you should for redundancy), they need to stay synchronized.

Primary nameserver (master): Holds the authoritative zone file Secondary nameservers (slaves): Copy data from the primary

Zone transfers happen over TCP (not UDP like normal queries):

Secondary: "What's your serial number?"
Primary: "2025111401"
Secondary: "Mine is 2025111305. Send me updates."
Primary: [Sends entire zone or just changes]

This uses two methods:

  1. AXFR (Full zone transfer): Sends the entire zone file
  2. IXFR (Incremental zone transfer): Sends only changes since last serial

The secondary checks periodically (based on the Refresh value in SOA).

How Everyone Stays Consistent

The genius of DNS is that consistency isn’t instantaneous. It’s eventual consistency:

  1. Root zone updates rarely: Adding new TLDs is a big deal
  2. TLD zones update when domains are registered/renewed
  3. Domain zones update when owners make changes
  4. Caching at every level smooths out the load

The root zone is distributed via HTTPS to all root servers every few hours. It’s digitally signed (more on this in Advanced).

Advanced: DNSSEC and the Root Key Ceremony

The Security Problem

Traditional DNS has zero security. Anyone can:

  • Spoof responses: Fake DNS server sends wrong IP
  • Cache poisoning: Trick a resolver into caching malicious answers
  • Man-in-the-middle: Intercept and modify DNS queries

DNSSEC (DNS Security Extensions) solves this with cryptographic signatures.

How DNSSEC Works

Every DNS response is signed with a private key. Clients verify the signature with a public key.

example.com.  3600  IN  A      93.184.216.34
example.com.  3600  IN  RRSIG  A 8 2 3600 (
              20251201000000 20251101000000 12345 example.com.
              [cryptographic signature data] )

The chain of trust:

Root Zone (.) 
  ↓ [signed by root key]
.com Zone
  ↓ [signed by .com key]
example.com
  ↓ [signed by example.com key]
www.example.com → 93.184.216.34

Each level signs the public key of the level below it. If any signature is invalid, the resolver rejects the answer.

The Root Key Signing Ceremony

The root zone’s private key is the most important key on the internet. If it’s compromised, someone could redirect all DNS traffic.

So ICANN does something amazing: The Root Key Signing Ceremony.

Four times a year, people from around the world gather in a secure facility:

  1. 21 people (Trusted Community Representatives) hold physical key cards
  2. 7 must be present to unlock the Hardware Security Module (HSM)
  3. They’re in a Faraday cage (no signals in or out)
  4. Video cameras record everything
  5. They generate or use the Key Signing Key (KSK) to sign the root zone

This is not a joke. This actually happens. The ceremony is livestreamed and audited.

Why this matters: If the root zone is compromised, the entire internet’s DNS becomes untrustworthy. This process ensures no single entity can control it.

Anycast Routing for Root Servers

Remember those “13” root servers? Here’s how anycast makes them actually hundreds:

Traditional routing: IP address → specific server

Anycast routing: Same IP address announced from multiple locations. Routers send traffic to the “nearest” one based on BGP routing.

Example: a.root-servers.net (198.41.0.4) is announced from:

  • Los Angeles
  • London
  • Tokyo
  • Sydney
  • …150+ other locations

When you query it, your ISP’s router sends your packet to the closest instance. This provides:

  • Low latency: Queries go to the nearest server
  • High availability: If one server fails, traffic routes to the next nearest
  • DDoS resistance: Attack traffic is distributed across many servers

BGP Hijacking: The Dark Side

DNS relies on BGP (Border Gateway Protocol) to route traffic. But BGP has no authentication.

What can go wrong:

In 2008, Pakistan Telecom tried to block YouTube domestically. They announced a more specific BGP route for YouTube’s IP space. Routers worldwide saw this as “the best path” and sent ALL YouTube traffic to Pakistan.

Result: YouTube was down globally for hours.

This has happened multiple times:

  • 2018: Google’s 8.8.8.8 hijacked by China Telecom
  • 2019: Cloudflare’s 1.1.1.1 briefly redirected

Solutions:

  • RPKI (Resource Public Key Infrastructure): Cryptographically signs BGP announcements
  • ROV (Route Origin Validation): Routers verify BGP signatures

But adoption is slow because BGP is ancient (1989) and everyone needs to upgrade.

The DNS Root Trust Anchor

How do you trust the root zone’s public key in the first place? This is the trust anchor problem.

Your operating system ships with the root zone’s public key hardcoded:

; Root trust anchor (built into your OS)
. 172800 IN DNSKEY 257 3 8 (
    AwEAAaz/tAm8yTn4Mfeh5eyI96WSVexTBAvkMgJzkKTOiW1vkIbzxeF3
    [more key data...]
    ) ; Key ID = 20326

This key is called the trust anchor. Your OS vendor (Microsoft, Apple, Linux distros) includes it.

When ICANN does the Key Signing Ceremony, they’re updating this key. Then OS vendors push updates.

The 2018 Root Key Rollover was a big deal. ICANN changed the root key for the first time since DNSSEC was deployed. Some old devices that didn’t update lost DNSSEC validation.

Real-World Implications

Censorship and DNS

Governments censor the internet by:

  1. Blocking at the ISP DNS level: ISP’s resolver returns fake IPs
  2. Poisoning DNS caches: Inject false answers into resolvers
  3. Blocking authoritative nameservers: Firewall traffic to specific DNS servers

Workarounds:

  • Use DNS over HTTPS (DoH) to bypass ISP DNS
  • Use public resolvers (8.8.8.8, 1.1.1.1)
  • Use VPNs to route around geographic blocks

The Fragility of Centralization

DNS has choke points:

  • ICANN controls the root zone: US-based nonprofit
  • Verisign has a monopoly on .com: They can raise prices (and do)
  • 13 root server organizations: If they collude, they could manipulate DNS

Alternatives being explored:

  • Blockchain-based DNS (Namecoin, Handshake): Decentralized, no ICANN
  • ENS (Ethereum Name Service): Decentralized naming for crypto addresses

But these lack adoption because traditional DNS just works.
