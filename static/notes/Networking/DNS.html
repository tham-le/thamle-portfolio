WTF is DNS?

The internet’s phone book, but way more interesting than that boring analogy suggests

You type google.com into your browser and instantly see the search page. But here’s the thing: computers don’t speak “google.com”. They speak IP addresses like 142.250.185.46. So WTF just happened? How did your browser magically know where to go?

That’s DNS. And it’s doing way more work than you think.

The Core Analogy

Think of DNS like a restaurant reservation system.

You call and say “I want a table at Chez François.” The host doesn’t just write down “Chez François” in their book. They write down the actual address: “1234 Main Street”. Later, when you show up, they know exactly where to send you.

DNS works the same way. You ask for github.com, and DNS translates that into 140.82.121.4. Your computer uses that IP address to actually connect.

But here’s where it gets interesting: just like a good reservation system keeps a cache of popular restaurants, DNS caches answers so it doesn’t have to look up google.com every single time you visit.

The Problem

In the early internet (we’re talking 1970s ARPANET), there were like 50 computers total. They literally maintained a text file called HOSTS.TXT that listed every computer’s name and IP address. You’d download this file, and boom, you could connect to any machine by name.

This was a beautiful dumpster fire for obvious reasons:

  1. It didn’t scale: As the internet grew, this file became massive
  2. Updates were slow: Someone had to manually update it and everyone had to re-download
  3. No organization: Everything was in one flat list
  4. Collisions: What if two organizations wanted the same name?

By 1983, Paul Mockapetris said “enough of this nonsense” and invented DNS. Now we have billions of devices, and the system still works.

How It Actually Works

DNS is a distributed, hierarchical database. Let’s break that down.

The Hierarchy

Domain names work from right to left:

api.github.com
^^^ ^^^^^^ ^^^
│      │   └─── Top-Level Domain (TLD)
│      └─────── Second-Level Domain (your actual site)
└────────────── Subdomain

Each level is managed by different servers:

  1. Root servers (.): Know where to find .com, .org, .uk, etc.
  2. TLD servers (.com): Know where to find github.com, google.com, etc.
  3. Authoritative servers (github.com): Know the actual IP addresses

The Lookup Process

When you ask for www.github.com, here’s what happens:

Your Computer
    ↓ "Where's www.github.com?"
    ↓
Recursive Resolver (your ISP or 8.8.8.8)
    ↓ "Where's .com?"
    ↓
Root Server
    ↓ "Ask the .com TLD server at 192.5.6.30"
    ↓
TLD Server (.com)
    ↓ "Ask github.com's nameserver at 185.199.108.153"
    ↓
Authoritative Nameserver (github.com)
    ↓ "www.github.com is at 140.82.121.4"
    ↓
Your Computer
    → Connects to 140.82.121.4

This looks slow, but it’s not. Why?

Caching Everywhere

Every level caches results:

  1. Your browser: Keeps DNS results in memory
  2. Your OS: Maintains a DNS cache
  3. Your router: Often caches locally
  4. Recursive resolver: Caches heavily (Google’s 8.8.8.8 serves billions of requests)

When you visit github.com the second time, your computer already knows the answer. No network requests needed.

DNS Records: More Than Just IP Addresses

DNS isn’t just “name to IP”. It stores different types of records:

A Record (Address)

Maps a domain to an IPv4 address:

github.com → 140.82.121.4

AAAA Record

Same, but for IPv6:

github.com → 2606:50c0:8000::154

CNAME (Canonical Name)

Points one name to another:

www.github.com → github.com

This is useful for aliasing. When github.com’s IP changes, you only update one record.

MX Record (Mail Exchange)

Tells email where to go:

github.com → mail.google.com (priority 10)

TXT Record

Arbitrary text data. Used for:

  • Email verification (SPF, DKIM)
  • Domain ownership verification
  • Configuration hints

DNS Uses UDP (Mostly)

DNS queries typically use UDP on port 53 because:

  1. Speed: No connection setup (no three-way handshake like TCP)
  2. Simplicity: Single request, single response
  3. Stateless: Each query is independent

The packet structure is simple:

DNS Query Packet:
+------------------+
| Transaction ID   | (16 bits, random, for matching responses)
| Flags            | (query type, recursion, etc.)
| Questions        | (what you're asking for)
| Answers          | (empty in query)
+------------------+

But: If the response is too big (over 512 bytes in old DNS, 4096 bytes with EDNS0), DNS falls back to TCP. This happens with:

  • Large TXT records
  • DNSSEC signatures
  • Zone transfers between nameservers

Common Pitfalls

❌ DNS is NOT instant

Even with caching, DNS adds latency. First lookup can take 50-200ms.

Solution: Warm up connections. Many HTTP clients support connection pooling and DNS caching.

❌ DNS can lie (or be outdated)

TTL (Time To Live) tells caches how long to keep a result. If github.com has a TTL of 3600 seconds (1 hour), your cache won’t check for updates for an hour.

When you change a DNS record, it takes time to propagate. This is called DNS propagation.

❌ DNS is unencrypted by default

Traditional DNS sends queries in plaintext. Your ISP can see every domain you look up.

Modern solutions:

  • DNS over HTTPS (DoH): Encrypts DNS queries inside HTTPS
  • DNS over TLS (DoT): Encrypts DNS queries with TLS

Both prevent ISP snooping but require client support.

Performance: Why Google’s 8.8.8.8 is Fast

Google’s public DNS (8.8.8.8) is fast because:

  1. Massive cache: They see billions of queries, so popular domains are always cached
  2. Anycast routing: 8.8.8.8 isn’t one server. It’s thousands of servers worldwide. Your query goes to the nearest one
  3. Optimized resolver: Custom software, not off-the-shelf BIND

Cloudflare’s 1.1.1.1 works similarly and claims to be even faster (with privacy focus).

The /etc/hosts Hack

Before checking DNS, your OS checks /etc/hosts (or C:\Windows\System32\drivers\etc\hosts on Windows):

# /etc/hosts
127.0.0.1   localhost
192.168.1.100   myserver.local

This is why 127.0.0.1 is called localhost. It’s hardcoded.

Use cases:

  • Block ads: 0.0.0.0 ads.example.com
  • Local development: 127.0.0.1 myapp.local
  • Testing without real DNS
