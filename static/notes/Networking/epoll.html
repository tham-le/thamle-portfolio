WTF is epoll?

When building a high-performance server, you need to monitor thousands of sockets efficiently. Checking each one in a loop would waste CPU time. You need a way to ask the Operating System: “Wake me up when ANY of these sockets has data ready.”

On Linux, that mechanism is epoll (event poll).

It’s the secret sauce behind Nginx, Redis, and every high-performance Event Loop on Linux.

The Security Guard Analogy

The Building Security System

The Dumb Way (Polling)

You’re a security guard watching 10,000 doors. You walk to each door, check if anyone is knocking, then move to the next door. By the time you finish checking all 10,000 doors, the first person gave up and left!

Problem: Wastes time checking doors with no activity.

The select() Way (Better, But Still Wasteful)

You install a basic alarm system. You give it a list of doors to watch. It rings when ANY door is knocked on. But it doesn’t tell you WHICH door - you still have to check all 10,000 doors to find the right one!

Problem: Still O(n) - you check every door after each alarm.

The epoll Way (Genius)

You install a modern alarm system. It tells you EXACTLY which doors were knocked on:

  • “Door 42, Door 127, and Door 8,933 have visitors!”

You only check those specific doors. Instant response, no wasted effort.

Result: O(1) per event. Scales to millions of doors!

That’s epoll - it gives you a list of ready file descriptors, not just a notification.

The Technical Story

Before epoll: select() and poll()

select() - The Original (1983)

Problems:

  • Limited to 1024 file descriptors (FD_SETSIZE)
  • O(n) - must check every socket after wake-up
  • Copies entire fd_set to/from kernel on each call

poll() - Improvement (1997)

Better: No 1024 limit. Still bad: O(n) complexity.

Enter epoll (2002) - The Game Changer

The magic: epoll_wait() returns only the ready file descriptors, not all of them!

Why epoll is Fast

1. O(1) Wake-up Complexity

  • select/poll: O(n) - iterate through all file descriptors
  • epoll: O(1) - kernel maintains ready list internally

2. No Repeated Setup

  • select/poll: Rebuild fd list every call
  • epoll: Register once with epoll_ctl(), reuse forever

3. Only Returns Ready FDs

  • select/poll: Returns ALL fds, you check which are ready
  • epoll: Returns ONLY ready fds

4. Edge-Triggered Option

epoll can work in two modes:

Level-Triggered (default):

Socket has data → epoll notifies
Socket still has data → epoll notifies again
Socket still has data → epoll notifies again...

Good for simple code, but can spam notifications.

Edge-Triggered (EPOLLET):

Socket has data → epoll notifies ONCE
You must drain ALL data before epoll notifies again

More efficient, but requires careful programming.

epoll vs kqueue vs IOCP

Different operating systems have different mechanisms:

OSMechanismNotes
LinuxepollBest for high connection counts
macOS/BSDkqueueSimilar to epoll, slightly different API
WindowsIOCPCompletion-based, not readiness-based
PortablelibuvCross-platform library that uses the best mechanism per OS

Libraries like Tokio (Rust) and Boost.Asio (C++) abstract over these differences.

The C10k Problem Solution

The C10k Problem asked: “Can a server handle 10,000 concurrent connections?”

Before epoll:

  • 10,000 sockets × select() O(n) = Terrible performance
  • OR 10,000 threads × 1MB stack = 10GB RAM + context switching hell

After epoll:

  • 10,000 sockets × epoll O(1) = Blazing fast
  • 1 thread handling all connections = Minimal memory

epoll solved the C10k problem and enabled:

  • Nginx - 100,000+ concurrent connections
  • Redis - Millions of operations per second
  • Modern web infrastructure

The Big Takeaway

epoll in a Nutshell

epoll is Linux’s efficient way to monitor thousands of file descriptors and get notified when they’re ready for I/O. Instead of checking every socket (O(n)), epoll returns only the ready ones (O(1)).

It’s the foundation of:

  • High-performance event loops
  • await implementations on Linux
  • Modern servers handling 100,000+ connections

The trade-off: More complex API, but the performance gains are worth it for high-scale applications.

If you’re building anything that needs to handle thousands of concurrent connections efficiently on Linux, you’re using epoll - either directly or through a library that wraps it.
