WTF is the Transport Layer?

It’s the difference between “I don’t care if you got my message” and “I need to know you got every single byte”

The Delivery Service Analogy

You need to send something to a friend. You have two options:

Option 1: Regular Mail (UDP) You drop your letter in the mailbox. Maybe it arrives, maybe it doesn’t. Maybe it takes 3 days, maybe 3 weeks. You have no idea. You don’t get a tracking number. You don’t get confirmation. You just… hope.

Option 2: Certified Mail with Tracking (TCP) You hand your letter to a postal worker who gives you a receipt. They track every step. “Arrived at sorting facility.” “Out for delivery.” “Delivered and signed.” If something goes wrong, they tell you. If it gets lost, they resend it.

That’s the difference between UDP and TCP. Both are Layer 4 protocols (the Transport Layer), but they have completely different philosophies.

Why Do We Need a Transport Layer?

The Network-Layer (IP) gets packets from one computer to another. That’s it. IP doesn’t care:

  • If packets arrive out of order
  • If packets get lost
  • If packets get duplicated
  • Which application on your computer should receive the data

The Transport Layer solves these problems. It sits between the Network-Layer and your applications, providing:

  1. Port numbers: Multiple applications can use the network simultaneously
  2. Reliability (optional): Guaranteed delivery and ordering
  3. Flow control (optional): Don’t overwhelm the receiver
  4. Error detection: Checksums to catch corruption

There are two main Transport Layer protocols:

  • TCP (Transmission Control Protocol): Reliable, ordered, connection-oriented
  • UDP (User Datagram Protocol): Unreliable, connectionless, fast

Let me explain both.

Port Numbers: The Apartment Number

Your computer has one IP address, but dozens of applications might be using the network at the same time. Your browser, your email client, Spotify, a game, whatever.

How does incoming data know which application it’s for?

Port numbers.

Think of your IP address as a building address, and port numbers as apartment numbers. The IP gets the packet to your building (your computer). The port number gets it to the right apartment (the right application).

Port numbers are 16 bit integers: 0 to 65535.

Some ports are well known (reserved for specific services):

  • Port 80: HTTP (web traffic)
  • Port 443: HTTPS (encrypted web traffic)
  • Port 22: SSH (secure shell)
  • Port 25: SMTP (email)
  • Port 53: DNS (domain name lookups)

Your applications use ephemeral ports (random high numbers like 54321) for outgoing connections.

When you visit a website:

  • Your browser picks a random port (say, 54321)
  • It connects to the server’s port 80 (or 443 for HTTPS)
  • The server sends responses back to your port 54321
  • Your OS knows: “Port 54321 belongs to the browser, route data there”

TCP: The Reliable Workhorse

TCP is what you use when you cannot afford to lose data. Web browsing, file downloads, email, SSH, anything where missing data would be a disaster.

The minimum TCP header is 20 bytes, and can be up to 60 bytes with options.

Let me break down the critical fields:

Source Port & Destination Port (4 bytes total)

Where the data is coming from and going to. Your browser might use port 54321 as the source, connecting to the server’s port 80.

Sequence Number (4 bytes)

This is the magic of TCP. Every byte in the TCP stream gets a number.

If you send “Hello” (5 bytes), and the sequence number is 1000, then:

  • Byte ‘H’ = sequence 1000
  • Byte ‘e’ = sequence 1001
  • Byte ‘l’ = sequence 1002
  • Byte ‘l’ = sequence 1003
  • Byte ‘o’ = sequence 1004

The next segment you send starts at sequence 1005. This lets the receiver put packets back in order even if they arrive scrambled.

Acknowledgment Number (4 bytes)

When you receive data, you send back an ACK saying “I got everything up to byte X, send me byte X next.”

If the receiver gets bytes 1000 to 1004, they reply with ACK 1005, meaning “I have everything up to 1004, send 1005 next.”

Flags (6 bits)

Control bits that manage the connection:

  • SYN: “Let’s start a connection”
  • ACK: “I acknowledge your data”
  • FIN: “I’m done sending data”
  • RST: “Something’s wrong, kill the connection”
  • PSH: “Push this data to the application immediately”
  • URG: “This data is urgent”

Window Size (2 bytes)

Flow control. The receiver tells the sender: “I can accept this many bytes right now.” If the receiver’s buffer is full, it advertises a window of 0, and the sender stops.

Checksum (2 bytes)

Detects corruption. If the checksum doesn’t match, the segment is silently dropped, and TCP will retransmit it.

TCP in Action: The Three-Way Handshake

Before [TCP]] can send any data, it must establish a connection. This is the famous three-way handshake.

Let me search for the exact handshake process to make sure I get the details right.Perfect. Here’s how the three-way handshake works:

Step 1: SYN (Client to Server)

The client sends a SYN (Synchronize Sequence Number) segment to the server to establish a connection, with an initial sequence number.

Client → Server
Flags: SYN
Seq: 1000 (random initial sequence number)
Ack: 0

The client is saying: “Hey, I want to talk. I’m starting my sequence numbers at 1000.”

Step 2: SYN + ACK (Server to Client)

The server responds with both SYN and ACK flags set, acknowledging the client’s SYN and providing its own initial sequence number.

Server → Client
Flags: SYN + ACK
Seq: 5000 (server's random initial sequence number)
Ack: 1001 (acknowledging client's SYN)

The server is saying: “Got it! I’ll start my sequence numbers at 5000. I’m ready for your byte 1001.”

Step 3: ACK (Client to Server)

Client → Server
Flags: ACK
Seq: 1001
Ack: 5001 (acknowledging server's SYN)

The client is saying: “Acknowledged! I’m ready for your byte 5001. Let’s go!”

After this three-way handshake, the connection status on both sides changes to ESTABLISHED and both are ready to start the actual data transfer.

Why Three Steps?

According to RFC 793, the main reason for the three-way handshake is to prevent old duplicate connection initiations from causing confusion, and to allow both parties to synchronize their segment sequence numbers.

Both sides need to agree on starting sequence numbers. If you only had two steps, you couldn’t be sure the other side got your acknowledgment.

TCP Data Transfer: Reliable Delivery

Once the connection is established, TCP can send data. Here’s how reliability works:

Sending Data

Client → Server
Seq: 1001
Payload: "GET /index.html HTTP/1.1" (24 bytes)

The client sends 24 bytes starting at sequence 1001.

Acknowledging Data

Server → Client
Ack: 1025 (1001 + 24)

The server says: “I got everything up to byte 1024. Send me byte 1025 next.”

Retransmission on Loss

If the server doesn’t ACK within a timeout period, the client retransmits:

Client → Server (again)
Seq: 1001
Payload: "GET /index.html HTTP/1.1"

The server might have received the original but the ACK got lost. That’s fine. TCP handles duplicate data by checking sequence numbers.

TCP Closing: The Four-Way Handshake

When you’re done, you close the connection with a four-way handshake:

1. Client → Server: FIN (I'm done sending)
2. Server → Client: ACK (Got it)
3. Server → Client: FIN (I'm done too)
4. Client → Server: ACK (Got it)

Why four steps instead of three? Because TCP is full duplex. Each direction needs to close independently. The server might still have data to send even after the client says “I’m done.”

Flow Control: Don’t Overwhelm the Receiver

The Window Size field in the TCP header implements flow control.

The receiver advertises how much buffer space it has:

Server → Client
Window: 4096

This means: “I can accept 4096 bytes before my buffer is full. Don’t send more than that without waiting for an ACK.”

As the receiver processes data, it sends more ACKs with updated window sizes:

Server → Client
Ack: 5000
Window: 8192 (buffer space freed up)

If the window hits zero, the sender stops. When buffer space opens up, the receiver sends a “window update” and the sender resumes.

Congestion Control: Don’t Overwhelm the Network

Flow control protects the receiver. Congestion control protects the network.

TCP uses algorithms like slow start and congestion avoidance to figure out how fast it can send without causing packet loss on the network.

It starts slow, sending a few segments. If those are acknowledged, it doubles the rate. If packets start dropping (detected by missing ACKs), it backs off.

This is why TCP connections start slow but get faster. It’s probing the network to find the optimal rate.

UDP: The Unreliable Alternative

Now let’s talk about UDP (User Datagram Protocol).

UDP is the complete opposite of TCP. It’s a thin wrapper around IP that adds port numbers and… that’s basically it.

The UDP Header

That’s it. 8 bytes total. No sequence numbers, no ACKs, no retransmission, no flow control, no congestion control, no connection setup, no nothing.

UDP in Action

You just send data:

Client → Server
UDP packet:
  src_port: 54321
  dst_port: 53 (DNS)
  length: 40
  checksum: 0xAB12
  payload: [DNS query for "example.com"]

Maybe it arrives, maybe it doesn’t. UDP doesn’t care.

When to Use UDP

UDP is perfect when:

  • Speed matters more than reliability: Real-time video, VoIP, online games
  • The application handles retransmission: DNS queries (if no response, the app retries)
  • Broadcasting or multicasting: Sending to multiple receivers simultaneously
  • Small, self-contained messages: Each packet is independent

Examples:

  • DNS: Queries are small. If you don’t get a response, just ask again.
  • Streaming video: If you lose a frame, who cares? Keep going. Don’t slow down to retransmit old data.
  • Online games: Player position updates every 50ms. If you lose one update, the next one is coming soon anyway.
  • DHCP: Getting an IP address when you boot up. Small request/response, retries built into the protocol.

The Big Picture

The Transport Layer sits between your application and the network. It provides two fundamental services:

  1. Multiplexing (via port numbers): Multiple applications share one IP address
  2. Reliability (via TCP): Guaranteed, ordered delivery when you need it

Think of it this way:

  • Network-Layer (IP): Get packets from computer A to computer B
  • Transport Layer (TCP/UDP): Get data from application A to application B

Without the Transport Layer, every application would need to implement its own retransmission, ordering, and flow control. TCP does this once, properly, so applications don’t have to.

TCP vs UDP Decision Use TCP when you need reliability: web browsing, file transfers, email, SSH, databases.

Use UDP when you need speed and can tolerate loss: video streaming, VoIP, gaming, DNS.

How do applications actually use TCP/UDP? That’s where sockets come in, the programming interface to the Transport Layer.
