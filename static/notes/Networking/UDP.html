WTF is UDP?

TCP’s faster, simpler, and slightly reckless cousin

Alright, we’ve spent the last few articles diving deep into TCP. We’ve seen its buffers, its flow control, its congestion control, its sliding windows, its acknowledgments, its retransmissions… TCP is a beautiful, sophisticated piece of engineering.

But here’s the thing: sometimes TCP is overkill.

Imagine you’re broadcasting a live sports game. A packet gets lost. Do you want TCP to pause the stream, retransmit that lost packet, and make sure everything arrives in perfect order while the game continues without you? Hell no. You’d rather skip that frame and keep the stream moving. The moment has already passed.

Or imagine you’re playing an online shooter. Your position update from 100ms ago finally arrives after being retransmitted three times. Congratulations, that information is now completely useless. You’ve already been shot.

For these scenarios, TCP’s reliability guarantees aren’t just unnecessary, they’re actively harmful.

Enter UDP (User Datagram Protocol), the protocol that says, “I’ll send your data. Whether it arrives? Not my problem.”

Today, we’re going to understand when you’d willingly throw away TCP’s reliability, what you get in return, and how to build on top of UDP when you need some reliability but not all of TCP’s overhead.

The Postcard vs. Registered Mail Analogy

Remember our postal service analogy for IP? Let’s revisit it with UDP.

TCP is registered mail:

  • You fill out forms
  • You get a tracking number
  • The postal service confirms delivery
  • If something goes wrong, they retry
  • Everything arrives in order
  • Slow but reliable

UDP is a postcard:

  • You write your message
  • You slap on an address
  • You toss it in the mailbox
  • You have no idea if it arrives
  • You have no idea what order multiple postcards arrive in
  • Fast and simple, but zero guarantees

UDP's Philosophy "I'll get your message to the mailbox. After that? Good luck. Maybe it arrives, maybe it doesn't. Maybe it arrives three times. Maybe out of order. Not my circus, not my monkeys."

This sounds terrible, right? Why would anyone use this?

Because UDP is blazingly fast and has almost zero overhead.

What UDP Doesn’t Do (And Why That’s The Point)

Let’s compare UDP to TCP by listing everything UDP deliberately doesn’t do:

FeatureTCPUDP
Connection SetupYes (three-way handshake)No (just send)
Guaranteed DeliveryYes (retransmits lost packets)No
Ordered DeliveryYes (reassembles out-of-order packets)No
Flow ControlYes (sliding window)No
Congestion ControlYes (CWND, slow start, etc.)No
Error CheckingYes (checksum, but optional in IPv4)Yes (minimal checksum)
Overhead per packet20+ bytes8 bytes

UDP’s header is comically simple:

UDP Header (8 bytes total):
┌────────────────┬────────────────┐
│  Source Port   │   Dest Port    │  (4 bytes)
├────────────────┴────────────────┤
│        Length of packet          │  (2 bytes)
├────────────────┬────────────────┤
│    Checksum    │                │  (2 bytes)
└────────────────┴────────────────┘

That’s it. Source port, destination port, length, checksum. Compare this to TCP’s 20-byte header plus all the state management, timers, and algorithms we discussed in previous articles.

UDP is Stateless UDP doesn't maintain a "connection." Every packet is independent. The sender doesn't know or care about the receiver's state. Just fire and forget.

When You Actually Want UDP

So when is this chaos useful? More often than you’d think:

1. Real-Time Streaming (Video/Audio)

Live video streaming is UDP’s killer app. If a frame gets lost at time T, retransmitting it at time T+200ms is worthless. The viewer has already moved on. Better to just skip that frame and keep the stream flowing.

Examples: Zoom, Discord voice chat, live sports broadcasts, WebRTC

2. Online Gaming

In a fast-paced game, your position update from 100ms ago is ancient history. Games send constant position updates. If one is lost, the next one is coming in 16ms anyway (60 FPS). Retransmitting old data just adds lag.

Examples: First-person shooters, racing games, real-time strategy games

3. DNS Queries

When you look up google.com, you send a small UDP packet to a DNS server asking for the IP Address. The DNS server sends back a small UDP packet with the answer. If it gets lost? Just send another query. No need for TCP’s connection overhead for a single request-response.

Examples: Every DNS lookup on the internet

4. IoT and Sensor Networks

A temperature sensor sending readings every 10 seconds doesn’t need reliability. If one reading is lost, another is coming soon. UDP’s low overhead is perfect for battery-powered devices.

Examples: Smart home sensors, industrial monitoring

5. Broadcasting and Multicasting

UDP supports sending one packet to multiple recipients simultaneously (multicast). TCP can’t do this because it’s connection-oriented. You can’t have a connection with multiple receivers.

Examples: IPTV, stock ticker updates, distributed systems discovery

The Dark Side: What Can Go Wrong

UDP’s simplicity comes with real risks:

1. Packet Loss is Your Problem

TCP retransmits. UDP doesn’t. If a packet is lost in transit, you’ll never know unless you build your own detection mechanism.

Client sends: "Hello!"
[ packet gets lost in a router somewhere ]
Client waits forever...

Solution: Application-level acknowledgments. The application layer must implement its own “did you get that?” logic if needed.

2. Packets Can Arrive Out of Order

UDP packets are independent. The network can deliver them in any order.

Client sends:  Packet 1, Packet 2, Packet 3
Server receives: Packet 3, Packet 1, Packet 2

Solution: Add sequence numbers to your application protocol.

3. Packets Can Arrive Multiple Times

Network glitches can cause duplication. UDP doesn’t deduplicate.

Client sends: "Fire missile!"
Network hiccups and duplicates the packet
Server receives: "Fire missile!" twice
[ fires two missiles ]

Solution: Application-level deduplication using unique IDs.

4. No Congestion Control = Network Abuse

UDP doesn’t slow down when the network is congested. If you blast UDP packets as fast as you can, you’ll congest the network for everyone.

This is why UDP applications need to implement application-level rate limiting. Don’t be that person who kills the network.

With Great Power Comes Great Responsibility UDP gives you raw speed by removing safety rails. You're now responsible for not being a jerk to the network. Implement your own rate limiting, or use protocols that do it for you (like QUIC).

Building Reliability on Top of UDP

Many modern protocols use UDP as a foundation and add selective reliability on top:

QUIC: UDP With Smarts

QUIC (used by 3) is built on UDP but adds:

  • Selective reliability (some streams are reliable, others aren’t)
  • Congestion control
  • Encryption (built-in, not optional like TLS)
  • Faster connection setup than TCP
  • Better handling of packet loss

QUIC is essentially “what if we took the good parts of TCP and rebuilt them on UDP without TCP’s historical baggage?”

Custom Game Protocols

Online games often build custom protocols on UDP:

  • Critical data (chat messages, scores) gets application-level acknowledgments
  • Non-critical data (position updates) is sent without acknowledgment
  • Sequence numbers prevent applying old position updates
  • Prediction and interpolation smooth over packet loss

When to Choose What

Here’s your decision tree:

Use TCP when:

  • Correctness matters more than speed
  • You’re transferring files, web pages, emails, or any data where loss is unacceptable
  • You want the protocol to handle reliability for you
  • You’re okay with occasional latency spikes when retransmission happens

Use UDP when:

  • Real-time delivery matters more than perfect delivery
  • Old data becomes worthless quickly
  • You’re okay implementing your own reliability logic (if needed)
  • You need multicast/broadcast
  • You want minimal latency and overhead

Use a protocol built on UDP (like QUIC) when:

  • You want UDP’s speed but some of TCP’s reliability
  • You’re building something modern and can avoid TCP’s legacy issues

The Big Picture

UDP: Fast, Simple, Reckless UDP is the bare minimum transport protocol. It adds addressing (ports) on top of IP, a checksum, and that's it. No connections, no reliability, no flow control, no congestion control.

Advantages:

  • Minimal overhead (8-byte header)
  • No connection setup delay
  • Perfect for real-time, time-sensitive data
  • Supports multicast/broadcast

Disadvantages:

  • No delivery guarantees
  • No ordering guarantees
  • No congestion control
  • Application must implement any needed reliability

The Tradeoff: UDP trades TCP’s guarantees for raw speed. Whether that’s a good trade depends entirely on your use case.

UDP is proof that sometimes the best solution is the simplest one. Strip away all the complexity, accept the risks, and move fast. For real-time applications, that’s often the right call.
