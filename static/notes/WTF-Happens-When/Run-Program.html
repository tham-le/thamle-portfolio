WTF happens when you run ./program

You type ./program in your terminal and hit Enter. What the hell just happened? Let’s trace through every single step from keystroke to running process.

This is the complete systems programming story - no skipping anything.

1. You press Enter in the shell

Keyboard interrupt

When you press Enter:

  1. Hardware keyboard sends scancode to the keyboard controller
  2. Keyboard controller generates a hardware interrupt (IRQ 1 on x86)
  3. CPU receives interrupt, looks up handler in Interrupt Descriptor Table (IDT)
  4. Kernel interrupt handler reads scancode from I/O port 0x60
  5. Keyboard driver translates scancode to character (\n for Enter)
  6. TTY layer receives character and wakes up the shell process

The shell was blocked in a read() system call waiting for input.

See: Operating-System, System-Call

2. Shell parses the command

Command line parsing

The shell (bash, zsh, etc.) now has the string: ./program

Parsing steps:

  1. Tokenization: Split by whitespace → ["./program"]
  2. Expansion: Check for variables ($VAR), wildcards (*.txt), etc.
  3. Path resolution:
    • ./ means current directory
    • Shell calls stat("./program") to check if file exists
    • Checks if file is executable (x permission bit)

Permission check:

If the file doesn’t exist or isn’t executable, shell prints error and stops.

See: Filesystem

3. Shell calls fork()

To run your program, the shell needs to create a new Process.

The fork() system call

What fork() does:

  1. System call: Shell makes syscall(__NR_fork)
  2. Kernel enters: CPU switches to kernel mode (ring 0)
  3. Process duplication:
    • Kernel creates new task_struct (process control block)
    • Copies parent’s memory mappings (Copy-on-Write)
    • Copies file descriptor table
    • Copies signal handlers
    • Assigns new PID
  4. Return twice:
    • Returns PID of child to parent
    • Returns 0 to child

Why Copy-on-Write?

Copying all memory would be slow. Instead, kernel marks all pages as read-only and shares them. Only when either process writes to a page does the kernel copy it.

Before fork():
Parent: [Memory Pages]

After fork():
Parent: [Memory Pages] ─┐
                        ├─ Shared (read-only)
Child:  [Memory Pages] ─┘

After write by child:
Parent: [Memory Pages]     (original)
Child:  [New Copy]         (separate)

See: Process, Virtual-Memory, Copy-on-Write

4. Child process calls execve()

Now the child process needs to replace itself with your program.

The execve() system call

System call enters kernel:

User mode:  execve() library function
            ↓ (syscall instruction)
Kernel mode: sys_execve() in kernel

See: System-Call

5. Kernel loads the ELF binary

The kernel needs to load your program from disk into memory.

Step 1: Read ELF header

Kernel checks:

  • Magic number is 0x7f 'E' 'L' 'F'
  • Architecture matches (x86-64)
  • File format is correct

See: ELF

Step 2: Parse program headers

Program headers describe memory segments:

$ readelf -l program

Program Headers:
  Type           Offset   VirtAddr           PhysAddr           FileSiz  MemSiz   Flg Align
  LOAD           0x000000 0x0000000000400000 0x0000000000400000 0x001234 0x001234 R E 0x1000
  LOAD           0x001000 0x0000000000601000 0x0000000000601000 0x000345 0x000678 RW  0x1000

Segments:

  • Code segment: Read + Execute (.text section)
  • Data segment: Read + Write (.data, .bss sections)

See: ELF

Step 3: Clear old address space

The child process’s old memory (copied from shell) is thrown away.

Step 4: Map ELF segments into memory

For each LOAD segment:

Example mapping:

Virtual Address Space:
0x00400000: [Text Segment] (R-X)  ← Code
0x00601000: [Data Segment] (RW-)  ← Initialized data
0x00602000: [BSS Segment]  (RW-)  ← Uninitialized data (zero-filled)

Key point: Memory is demand-paged. The kernel doesn’t read the entire file into RAM yet - it just sets up the mappings. When the program accesses a page, a page fault loads it from disk.

See: Virtual-Memory, Page-Fault, mmap

Step 5: Set up stack

Stack layout:

High addresses
    ↓
[Environment variables]
[Command line arguments]
[argc]
    ↓
Stack grows downward
    ↓
Low addresses

See: Stack

Step 6: Find interpreter

If the ELF has a PT_INTERP segment, it specifies a dynamic linker:

$ readelf -l program | grep interpreter
[Requesting program interpreter: /lib64/ld-linux-x86-64.so.2]

The kernel loads the dynamic linker instead of jumping directly to your program.

See: Dynamic-Linking

6. Virtual memory is set up

At this point, the process has a brand new Virtual-Memory layout:

0xFFFFFFFFFFFFFFFF  ┌──────────────┐
                    │   Kernel     │  (Ring 0, not accessible)
0xFFFF800000000000  ├──────────────┤
                    │              │
                    │   (Unmapped) │
                    │              │
0x00007FFFFFFFFFFF  ├──────────────┤
                    │    Stack     │  ← Growing downward
                    ├──────────────┤
                    │              │
                    │   (Unmapped) │
                    │              │
                    ├──────────────┤
                    │  Heap (brk)  │  ← Growing upward (malloc)
                    ├──────────────┤
                    │     BSS      │  (Uninitialized data)
                    ├──────────────┤
                    │     Data     │  (Initialized data)
                    ├──────────────┤
                    │     Text     │  (Code)
0x0000000000400000  └──────────────┘

Page Table:

The kernel creates a page table mapping virtual addresses to physical addresses:

Virtual Addr    →   Physical Addr
0x00400000      →   0x12345000  (Code page)
0x00401000      →   0x23456000  (Code page)
0x00601000      →   0x34567000  (Data page)
...

TLB (Translation Lookaside Buffer):

  • CPU caches recent virtual→physical translations
  • On Context-Switch, TLB is flushed (expensive!)

See: Virtual-Memory, Page-Table, TLB

7. Dynamic linker runs

If your program uses shared libraries (.so files), the dynamic linker runs before your main().

Dynamic linker: /lib64/ld-linux-x86-64.so.2

Steps:

  1. Reads ELF headers of the main program
  2. Finds dependencies: 
    $ ldd program
    linux-vdso.so.1 (0x00007fff...)
    libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
    /lib64/ld-linux-x86-64.so.2
  3. Loads each shared library:
  4. Relocations: Fixes up addresses
    • PLT (Procedure Linkage Table): For function calls
    • GOT (Global Offset Table): For global variables
  5. Runs constructors: Functions marked __attribute__((constructor))

Lazy binding (PLT/GOT)

First call to printf():

1. Call printf@PLT
2. PLT jumps to GOT[printf] (initially points back to PLT)
3. PLT calls dynamic linker
4. Dynamic linker resolves printf address
5. Dynamic linker updates GOT[printf]
6. Future calls go directly to printf

First call: ~1000 cycles
Later calls: ~5 cycles

See: Dynamic-Linking, PLT-GOT

8. Program starts executing

Finally, control transfers to your program!

Entry point

Actual entry point is NOT main()!

The real entry is _start in crt1.o:

__libc_start_main() does:

  1. Set up environment
  2. Register atexit() handlers
  3. Call main(argc, argv, envp)
  4. Call exit(return_value)

See: C-Runtime

Your main() runs

Stack frame for main:

High addresses
    ↓
[Return address to __libc_start_main]
[Saved frame pointer]
[Local variables]
[Stack grows down...]
    ↓
Low addresses

Calling printf:

  1. Arguments pushed to stack (or passed in registers)
  2. call printf pushes return address
  3. printf executes (from libc.so)
  4. printf calls write() system call
  5. Kernel writes to terminal
  6. Return to main

See: Stack, Calling-Convention, System-Call

9. Program exits

When main() returns:

Exit sequence

  1. __libc_start_main receives return value
  2. Calls exit(0)
  3. exit() does cleanup:
    • Flushes stdio buffers
    • Calls functions registered with atexit()
    • Calls destructors for global objects (C++)
  4. Calls _exit(0) system call

_exit() system call

In the kernel:

  1. Cleanup process resources:
    • Close all file descriptors
    • Unmap all memory (free page tables)
    • Release locks
    • Detach from TTY
  2. Notify parent: Send SIGCHLD signal to parent
  3. Become zombie: Process enters EXIT_ZOMBIE state
  4. Wait for parent: Parent must call wait() to reap exit status

Parent (shell) reaps child

Kernel:

  1. Returns exit code to parent
  2. Frees task_struct of child
  3. Process is fully dead

Shell:

See: Process, Zombie-Process

10. Behind the scenes

CPU perspective

Throughout execution:

  1. Fetch-Decode-Execute loop:

    while (true) {
    instruction = fetch(PC);
    decode(instruction);
    execute(instruction);
    PC = next_instruction;
}

  2. Privilege levels:

    • User mode (Ring 3): Your program
    • Kernel mode (Ring 0): System calls

  3. System call transition:

    User mode:   printf() → write() → syscall instruction
                                  ↓
Kernel mode:                   sys_write()
                                  ↓
User mode:                     return to printf()

See: CPU, System-Call

Memory management details

Page faults:

When your program accesses a page not yet loaded:

  1. CPU generates page fault exception
  2. Kernel page fault handler:
    • Check if address is valid (in VMA)
    • Read page from disk (ELF file)
    • Update page table
    • Return to user mode
  3. Instruction re-executed (now succeeds)

Demand paging means:

  • Faster startup (don’t load entire program)
  • Less memory used (only load needed pages)

See: Page-Fault, Virtual-Memory

Scheduler

Your process doesn’t run continuously. The kernel scheduler switches between processes:

Timer interrupt (every ~1-10ms):

1. Hardware timer fires interrupt
2. Kernel scheduler runs
3. Picks next process to run
4. Context switch:
   - Save current registers
   - Save stack pointer
   - Switch page tables
   - Restore next process's registers
5. Return to (possibly different) process

Cost of context switch:

  • Save/restore registers: ~100 cycles
  • Switch page tables: ~1000 cycles
  • TLB flush: All cached translations lost
  • Cache misses: New process has different working set

See: Context-Switch, Scheduler

File descriptor inheritance

The process inherits file descriptors from the shell:

These survive across fork() and execve().

When printf() calls write(1, "Hello\n", 6), it writes to the terminal.

See: File-Descriptor

Complete timeline

Let’s put it all together:

0.000000s: Press Enter
0.000001s: Keyboard interrupt
0.000002s: Shell wakes up, parses "./program"
0.000010s: Shell calls fork()
0.000050s: Kernel creates child process (CoW)
0.000051s: Child calls execve()
0.000100s: Kernel loads ELF header
0.000150s: Kernel sets up virtual memory
0.000200s: Dynamic linker loads libc.so
0.000500s: Dynamic linker resolves symbols
0.000600s: _start runs
0.000610s: __libc_start_main() calls main()
0.000620s: main() calls printf()
0.000630s: printf() calls write() syscall
0.000640s: Kernel writes to terminal
0.000650s: main() returns
0.000660s: exit(0) cleanup
0.000670s: _exit(0) syscall
0.000680s: Kernel sends SIGCHLD to shell
0.000690s: Shell calls wait(), reaps child
0.000700s: Shell prints new prompt

Total time: ~0.7ms (heavily depends on system load, I/O, etc.)

Key takeaways

Process creation:

  1. fork() creates new process (CoW memory)
  2. execve() replaces process with new program

ELF loading:

  1. Kernel parses ELF headers
  2. Maps segments into virtual memory (demand-paged)
  3. Sets up stack
  4. Runs dynamic linker

Program execution:

  1. Dynamic linker loads libraries
  2. _start calls __libc_start_main()
  3. main() runs
  4. exit() cleans up
  5. _exit() terminates process
  6. Parent reaps zombie

Performance implications

Why processes are expensive

Creating a process involves:

  • System calls (fork, execve): ~1-10μs each
  • Memory setup: TLB flush, page table creation
  • ELF loading: Disk I/O (if not cached)
  • Dynamic linking: Symbol resolution

Alternatives:

  • Threads: Share address space, much faster std-thread
  • vfork(): Don’t copy memory (deprecated)
  • posix_spawn(): Optimized for fork+exec

Why system calls are slow

Each system call:

  1. Mode switch: User mode → Kernel mode (~50-100 cycles)
  2. Kernel work: Actual operation
  3. Mode switch back: Kernel mode → User mode (~50-100 cycles)

Total: 200-1000 cycles (compare to ~1 cycle for register operation)

Optimization:

  • Batch operations (e.g., writev() instead of multiple write())
  • Use mmap() instead of read()/write() for large files
  • Buffer I/O (stdio does this)

See: System-Call, mmap

The Big Picture

Every time you run a program:

Userspace:  Shell → fork() → execve() → Dynamic Linker → main()
               ↕        ↕         ↕             ↕          ↕
Kernel:     Scheduler  Copy   ELF Loader   Page Faults  Syscalls
               ↕        ↕         ↕             ↕          ↕
Hardware:   Timer    Memory    Disk         MMU         CPU

This is systems programming. Every layer matters. Every optimization counts. Every design decision has tradeoffs.
