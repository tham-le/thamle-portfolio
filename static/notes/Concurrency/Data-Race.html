WTF is a Data Race?

WTF is a Data Race?

Nov 06, 20255 min read

  • concurrency
  • threading
  • race-conditions
  • safety

Imagine two people trying to update the same shared spreadsheet at exactly the same time, without any coordination. Chaos, right?

That’s a data race - one of the most insidious bugs in concurrent programming.

The Bank Account Analogy

The Joint Bank Account

You and your partner share a bank account with $1000.

Thread 1 (You):

  1. Read balance: $1000
  2. Add $100
  3. Write balance: $1100

Thread 2 (Partner):

  1. Read balance: $1000
  2. Add $50
  3. Write balance: $1050

If both happen at the same time, here’s what can go wrong:

Time | Thread 1      | Thread 2      | Actual Balance
-----|---------------|---------------|---------------
1    | Read: $1000   |               | $1000
2    |               | Read: $1000   | $1000
3    | Add $100      |               | $1000
4    |               | Add $50       | $1000
5    | Write: $1100  |               | $1100
6    |               | Write: $1050  | $1050  ❌

Expected: 1000+100 + 50=1150 Actual: 1050(thelastwritewins,losing100!)

This is a data race.

The Technical Definition

A data race occurs when:

  1. Two or more threads access the same memory location
  2. At least one access is a write
  3. The accesses are not synchronized

The result is Undefined Behavior - your program might crash, produce wrong results, or appear to work fine (until it doesn’t).

Data Race vs Race Condition

These are often confused:

Data Race

  • Technical definition: Unsynchronized concurrent access to shared memory
  • Always a bug: Leads to Undefined Behavior
  • Example: Two threads writing to the same variable

Race Condition

  • Broader concept: Program behavior depends on timing
  • Sometimes intentional: Can be by design
  • Example: Two threads racing to acquire a Lock

All data races are bugs. Not all race conditions are bugs.

Why Data Races are Evil

  1. Non-deterministic: The bug appears randomly, making it almost impossible to reproduce
  2. Platform-dependent: Might work on your machine, fail in production
  3. Optimization-dependent: Compiler optimizations can make data races worse
  4. Silent corruption: Data gets corrupted without crashes or errors

How Languages Handle Them

C++

  • Default: Nothing stops you from creating data races
  • Solution: Use [[std::mutex]], [[std::atomic]], or avoid shared mutable state
  • Tools: Thread sanitizers can detect data races at runtime

Rust

  • Compile-time prevention: The Borrow Checker makes data races impossible*
  • Rule: Either:
    • Multiple immutable references (&T) OR
    • One mutable reference (&mut T)
  • Never both at the same time

*Unless you use unsafe, which is required for low-level code.

Go

  • Philosophy: “Share memory by communicating” (channels)
  • Race detector: Built-in tool to catch data races

Java

  • synchronized keyword: Built-in locks
  • volatile: Ensures visibility across threads
  • java.util.concurrent: Thread-safe collections

The Big Takeaway

Data Race in a Nutshell

A data race happens when multiple threads access the same memory without synchronization, with at least one writing. It causes Undefined Behavior - crashes, corruption, or worse.

Rust prevents data races at compile time through the Borrow Checker. C++ requires manual synchronization with mutexes or atomics.

The golden rule: If you’re sharing mutable data between threads, you need synchronization. Period.
