WTF is X11?

The display system built for a world that no longer exists

You boot a Linux machine from the 1990s, 2000s, or even early 2010s. Your desktop appears. Windows open, move, resize. You type, you click, pixels change. For decades, there was one system making this happen on Unix machines: X11.

But here’s the weird part: the X server runs on your machine, the client runs on your machine, and they talk to each other over a network protocol… even though they’re on the same computer. WTF is that about?

Let’s rewind to 1984 and understand why the X Window System was designed the way it was, and why it dominated Unix graphics for 40 years.

The Multi-Terminal University Problem

Think of MIT in 1984. You have expensive mainframes and minicomputers in machine rooms. You have dozens of terminals at desks scattered across campus. Students and researchers need to run programs on the powerful machines while sitting at their terminals.

The problem: Everyone wants graphical interfaces, not just text. But your terminal doesn’t have the computing power to run the application AND draw the graphics. The powerful computer in the machine room doesn’t have a screen attached to it.

The solution: Split the job in two.

  • The application runs on the powerful machine (the “client”)
  • The display server runs on your desk terminal (the “server”)
  • They talk to each other over the campus network

This is where X11’s seemingly backwards terminology comes from. The server is at your desk because it serves your screen, keyboard, and mouse to the remote applications. The client is the application that connects to your display server, wherever that application happens to be running.

In the restaurant analogy from the Wayland article, imagine a kitchen (application server) in one building, and dining rooms (display servers) scattered across campus. The kitchen sends plated dishes over a delivery network to whichever dining room ordered them.

This made perfect sense in 1984.

What X11 Actually Is

X11 is three things at once:

  1. A protocol: The X11 protocol defines how clients and servers talk to each other
  2. A display server: The X server manages your screen, keyboard, and mouse
  3. An architecture: The entire X Window System including libraries, window managers, and more

The “11” comes from the protocol version. X went through rapid evolution from X1 (1984) to X10 (1986), then X11 was released in September 1987. The protocol designers said “we don’t think it will ever change again, okay. There’s not gonna be a version 12.”

They were right. X11 has been at version 11 since 1987, nearly 40 years.

The Core Architecture: Network Transparent by Design

X uses a client-server model where the X server program runs on a computer with a graphical display and communicates with various client programs.

Here’s the key insight: the client and server can communicate securely over the Internet by tunneling the connection over an encrypted connection. They can run on the same machine or on different ones, possibly with different architectures and operating systems.

The Communication Model

The X server acts as a go-between for the user and the client programs, accepting requests on TCP port 6000 (plus the display number) for graphical output from the client programs and displaying them to the user, and receiving user input and transmitting it to the client programs.

Four types of packets flow over the network:

  1. Request: Client asks server to do something (“create a window”, “draw a rectangle”)
  2. Reply: Server sends information back to client
  3. Event: Server notifies client about something (“mouse moved”, “key pressed”, “window exposed”)
  4. Error: Server tells client something went wrong

This all happens asynchronously. Clients send requests without waiting for replies (when they don’t need one). The server sends events whenever something interesting happens.

The Display Variable

You’ve probably seen DISPLAY=:0 or DISPLAY=localhost:10.0 in error messages. This tells clients where to find the X server.

The format is: hostname:displaynumber.screennumber

  • :0 means local connection, display 0, screen 0
  • 192.168.1.100:0 means connect to that IP address, display 0
  • localhost:10.0 typically means SSH X11 forwarding

On the same host, X uses Unix domain sockets for efficient communication instead of TCP. But the protocol is identical.

What The X Server Actually Does

The X server is the program that sits closest to your hardware. It has three main jobs:

1. Output Management

The X server can perform rendering itself. X provides basic drawing primitives:

  • Fill rectangles
  • Draw lines
  • Display text
  • Copy rectangular regions
  • Composite images

Applications send commands like “draw a red rectangle from (10,10) to (100,100)” and the X server translates these into actual GPU commands.

The X server maintains a framebuffer (the actual pixels on screen) and coordinates which pixels belong to which window.

2. Input Handling

The X server captures all input from keyboard, mouse, and other devices. It routes input events to the appropriate client based on:

  • Which window has focus
  • Where the mouse pointer is
  • Which client asked to receive which events

The X server receives input from the kernel (via evdev or similar) and packages it into X11 protocol events.

3. Resource Management

The X server manages shared resources:

  • Windows: Rectangular areas that clients draw into
  • Pixmaps: Off-screen images stored in server memory
  • Graphics Contexts (GCs): Drawing state (colors, line widths, fonts)
  • Colormaps: Color management (important before true color displays)
  • Cursors: Mouse pointer shapes

All these objects live in the server. Clients reference them by ID numbers.

The Separation of Concerns

Here’s where X11 gets interesting (and complicated): X deliberately does not specify the user interface.

The X11 protocol provides drawing primitives, but it does NOT provide:

  • Buttons
  • Menus
  • Scroll bars
  • Window title bars
  • Window decorations
  • Desktop icons

All of that is someone else’s job.

The Three-Layer Cake

Layer 1: X Server

  • Manages hardware
  • Provides drawing primitives
  • Routes input

Layer 2: Window Manager

  • A special X client that controls window appearance
  • Draws title bars and borders
  • Handles window positioning, resizing, minimizing
  • Defines how windows interact

Layer 3: Widget Toolkit

  • Client-side libraries (GTK, Qt, Motif, etc.)
  • Implements buttons, menus, scroll bars
  • Uses X drawing primitives to render widgets

This separation means you can mix and match. Run KDE apps on GNOME. Use a tiling window manager with Qt applications. Everything talks the same X11 protocol underneath.

But it also means massive complexity. Three separate programs (server, window manager, compositor) have to coordinate perfectly.

Windows and Drawing: How It Actually Works

Let’s trace what happens when you open a terminal window:

Step 1: Create the Window

The terminal application (X client) sends a CreateWindow request to the X server:

  • Position: (100, 100)
  • Size: 800x600
  • Attributes: Want keyboard input, want mouse events

The server allocates window resources and assigns it a Window ID (like 0x2400001).

Step 2: Map the Window

Windows start unmapped (invisible). The client sends a MapWindow request to make it visible.

The window manager intercepts this. It decides where the window actually goes, adds decorations (title bar, borders), and then allows it to be shown.

Step 3: Expose Events

When a window becomes visible, the server sends Expose events to tell the client “these pixels need to be drawn.”

Why? Because X traditionally did not keep window contents in off-screen buffers. When a window was covered and then uncovered, those pixels were lost. The application had to redraw them.

Step 4: Drawing

The client uses X drawing primitives to render content:

  • XDrawString() to draw text
  • XFillRectangle() to draw filled rectangles
  • XCopyArea() to copy pixmap data

Each drawing call is a protocol request sent to the server. The server executes these drawing commands and updates the framebuffer.

Step 5: Input

User presses a key. The server generates a KeyPress event and sends it to the focused window’s client. The client application receives the event and responds.

The Extension System

The X11 protocol was designed to be extensible. The core protocol is minimal, and functionality is added through extensions.

At the protocol level, every extension can be assigned new request/event/error packet types.

Important extensions:

  • MIT-SHM: Shared memory for faster image transfer (avoids copying pixels over the socket)
  • XRender: Modern 2D rendering with alpha blending, gradients
  • Composite: Redirect window contents to off-screen buffers (needed for compositing)
  • XFixes: Various fixes and additions (cursor hiding, regions)
  • GLX: OpenGL rendering in X windows
  • Xinerama/RandR: Multi-monitor support

Applications query which extensions are available and adapt accordingly.

The Compositing Revolution

Traditional X had no compositing. Windows drew directly to the screen. This was fast but limited:

  • No translucency effects
  • No live window previews
  • Screen tearing during updates
  • No smooth animations

The Composite extension changed this. It allows window contents to be redirected to off-screen buffers.

A compositing manager (separate program) reads these buffers, applies effects, and composites them into a final screen image.

But this is NOT built into X. It’s a separate program that uses X extensions. Common compositing managers include:

  • Compiz: Famous for its 3D desktop cube
  • Picom: Lightweight standalone compositor
  • Mutter (GNOME) and KWin (KDE): Integrated compositors

This layered approach created more complexity and more places for things to go wrong.

The Beautiful Complications

X11’s design philosophy: “Provide mechanism, not policy.”

Translation: X gives you the tools to build a GUI, but doesn’t tell you how to use them. This created incredible flexibility:

  • Hundreds of window managers to choose from
  • Complete customization of desktop appearance
  • Network transparency as a core feature
  • True multi-user capabilities

But it also created problems:

Complexity: Three or more programs (X server, window manager, compositor, applications) have to coordinate Performance: Drawing commands sent over a protocol have overhead Security: Any X client can spy on any other (read keystrokes, screenshots, inject input) Latency: Network round trips add delay, even on localhost

Network Transparency: The Killer Feature (That Killed Performance)

X11 was designed when network transparency mattered. You could run an application on a powerful server and display it on your terminal.

Or more securely with SSH:

Every X drawing command goes over the network. For simple apps, this is fine. For complex apps (web browsers, video players), this is terrible.

Modern X implementations use Unix domain sockets for local connections and shared memory extensions to avoid copying pixels, but the protocol structure remains.

The tradeoff: Network transparency added complexity that penalized the common case (local applications) to support the rare case (remote applications).

The Security Disaster

X’s design lacks sandboxing, so any application can access and manipulate keyboard, mouse, display and clipboard content.

Practically, this means:

  • Any X client can read your keystrokes (yes, including passwords)
  • Any X client can take screenshots of other windows
  • Any X client can inject fake keyboard/mouse events into other windows
  • Any X client can move/resize other windows

This is a fundamental design flaw. X was built in a more trusting era where all programs on a machine belonged to the same user and were assumed to be cooperative.

Modern systems try to mitigate this:

  • Run X server without root privileges
  • Use SSH tunneling for remote connections
  • Sandboxing extensions (XACE) that few apps use
  • Just accept the risk

Why X11 Lasted So Long

Despite its problems, X11 ruled Unix graphics from 1987 to approximately 2015 (and still runs on many systems today). Why?

1. It worked well enough For typical desktop use, X11 was fine. The problems were more theoretical than practical for most users.

2. Massive ecosystem Thousands of applications written for X11. Decades of window managers, toolkits, and extensions.

3. No viable alternative Attempts to replace X failed repeatedly. The network was too entrenched.

4. Continuous improvement X11 adapted. Compositing extensions, GPU acceleration, better font rendering. The protocol stayed the same, but implementations improved.

5. “Good enough” is hard to beat Replacing working software that everyone depends on is extraordinarily difficult.

The Big Picture

When you run X11:

  1. The X server starts (Xorg, XFree86, etc.)
  2. It talks to the Kernel via DRM/KMS to control display hardware
  3. It listens on a Unix socket (:0) and optionally TCP port 6000
  4. A window manager connects as a special client
  5. The compositor (if using one) connects via Composite extension
  6. Applications connect as clients
  7. Applications send drawing commands
  8. The server executes them and updates the screen
  9. Input events flow back to applications

Everything goes through the server. It’s the central hub, but unlike Wayland’s compositor, it’s separate from the window manager and has no built-in compositing.

This is X11: a network-transparent, endlessly flexible, beautifully complicated display system designed for a world of terminals and timesharing that somehow survived into the age of personal computers.
