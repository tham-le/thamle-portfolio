WTF is CUSE?

FUSE’s weird cousin that lets you implement device drivers in userspace

You’ve heard of FUSE. Maybe you’ve even used SSHFS to mount a remote directory, or NTFS-3G to read Windows drives on Linux. FUSE lets you implement filesystems in userspace without touching kernel code. Great.

But what if you want to implement /dev/dsp? Or create a custom serial port? Or build a virtual sound card? These aren’t filesystems. They’re character devices. And this is where CUSE (Character device in Userspace) comes in.

CUSE is FUSE’s lesser-known sibling. While FUSE lets you mount directories, CUSE lets you create /dev entries. Same philosophy, different target. Let’s figure out what that actually means.

The Character Device Problem

Think about what lives in /dev:

These are character devices (the c in the permissions). They represent hardware or virtual devices that applications can interact with through standard file operations: open(), read(), write(), ioctl().

The traditional way to create these: Write a kernel module. Load it with insmod. Debug kernel panics. Reboot constantly. Suffer.

The CUSE way: Write a regular userspace program using libfuse. Run it without root privileges. Debug with gdb and printf. No kernel crashes.

This is the same revolution FUSE brought to filesystems, now applied to device drivers.

The Core Analogy: Device Impersonation

Think of /dev/dsp (the old sound device) like a restaurant’s kitchen window.

Traditional kernel driver: The kitchen is built into the restaurant’s foundation. If you want to change how food is prepared, you need to do construction work. One mistake brings down the whole building.

CUSE driver: The kitchen is actually a food truck parked outside. The restaurant has a window (the CUSE interface) that passes orders to the truck and returns plates to customers. From the customer’s perspective, it’s identical. But now you can:

  • Replace the food truck without demolishing the restaurant
  • Debug recipes without risking the building’s structural integrity
  • Run multiple food trucks simultaneously
  • Let anyone operate a food truck without contractor licenses

The /dev/dsp device node is the window. Your userspace program is the food truck. CUSE is the system that connects them.

How CUSE Actually Works

CUSE builds directly on top of FUSE’s infrastructure. In fact, the kernel implementation is only about 1,000 lines of code because it reuses almost everything from FUSE.

The Architecture

Three key components:

  1. The CUSE kernel module (cuse.ko)

    • Provides /dev/cuse for communication
    • Creates actual character device nodes in /dev
    • Routes operations to userspace handlers

  2. The libfuse library (same one FUSE uses)

    • Provides the protocol implementation
    • Handles kernel communication
    • Marshals requests and responses

  3. Your userspace program

    • Implements the device behavior
    • Responds to open(), read(), write(), ioctl() calls
    • Runs with normal user privileges (if part of fuse group)

The Communication Flow

When an application opens your CUSE device:

Application              Kernel              Your Program
    |                       |                      |
    |---open("/dev/mydevice")-->|                  |
    |                       |                      |
    |                   [CUSE module]              |
    |                       |                      |
    |                       |---FUSE_OPEN--------->|
    |                       |                      |
    |                       |<--Reply: fd=42-------|
    |<-----fd=5-------------|                      |
    |                       |                      |
    |---write(5, "data")------>|                   |
    |                       |---FUSE_WRITE-------->|
    |                       |    (your handler)    |
    |                       |<--Reply: written-----|
    |<----bytes written-----|                      |

Every operation on the character device gets translated into a FUSE protocol message and sent to your userspace program. Your program processes it and sends back a response. From the application’s perspective, it’s just a normal device.

The /dev/cuse Control Device

CUSE exposes a special control device at /dev/cuse. This is where your userspace program connects to announce what device it wants to implement.

The handshake:

  1. Your program opens /dev/cuse

  2. CUSE sends a CUSE_INIT request

  3. Your program responds with device information:

    • Device name (e.g., "mydevice")
    • Major and minor numbers (optional, kernel can assign)
    • Permission mode

  4. The kernel creates /dev/mydevice as a character device

  5. Your program now receives all operations on that device

Critical difference from FUSE: There’s no mount command and no mount helper. Your program talks directly to /dev/cuse. This is both simpler (no suid helpers) and more restrictive (requires membership in the fuse group).

CUSE vs UIO: Two Paths to Userspace Drivers

Linux offers another mechanism for userspace drivers: UIO (Userspace I/O). How does CUSE compare?

Key Differences

UIO is for real hardware with interrupts. You write kernel code to handle IRQs, then expose memory-mapped regions to userspace. Best for: device drivers that need fast hardware access.

CUSE is for virtual devices or software-only functionality. No kernel code required. No interrupt handling. Best for: emulating legacy devices, protocol translation, testing.

Example use cases:

Use CaseBest ChoiceWhy
PCIe network cardUIONeeds interrupt handling
Virtual serial portCUSESoftware only, no hardware
Sound card emulationCUSEOSS compatibility layer
FPGA control interfaceUIOMemory-mapped registers
Custom /dev node for IPCCUSEJust needs char device semantics

Think of UIO as “I have hardware but want userspace control.” Think of CUSE as “I want a /dev node with no hardware involved.”

Real-World CUSE Applications

CUSE was originally created to solve a specific problem: OSS emulation.

The OSS Proxy Story

The Open Sound System (OSS) was deprecated years ago in favor of ALSA, but many legacy applications still expected OSS device files like /dev/dsp and /dev/mixer. The in-kernel OSS emulation was becoming painful to maintain.

Solution: A CUSE-based OSS proxy that implements these devices in userspace, translating OSS calls to ALSA on the backend. Way smaller than the kernel emulation, easier to debug, and users can kill it if it misbehaves.

Other Use Cases

Virtual serial ports:

  • Serial-over-network drivers that forward serial port operations to remote devices
  • Testing serial applications without hardware
  • Protocol converters (e.g., named pipes to serial device)

Hardware emulators:

  • Virtual sound cards for testing
  • Fake video capture devices (useful for videoconferencing testing)
  • Mock sensors for embedded development

Legacy compatibility:

  • Keeping old software working without maintaining kernel drivers
  • Gradual migration from deprecated interfaces

The Limitations

CUSE is powerful but not magic. Several constraints exist:

Performance

Every operation crosses the kernel/userspace boundary twice (application → kernel → userspace handler → kernel → application). This adds overhead.

Benchmark comparison (approximate):

  • Kernel character driver: 100,000 ops/sec
  • CUSE device: 30,000 ops/sec
  • Network socket: 50,000 ops/sec

For many use cases, this is fine. For high-performance hardware access, stick to kernel drivers or UIO.

No Interrupt Handling

CUSE drivers cannot handle hardware interrupts directly. Those must be handled by something else, typically a downstream kernel driver.

This is by design. CUSE is for software devices or devices where a kernel component handles the interrupt-driven part.

Limited ioctl() Support

While CUSE supports ioctl(), complex ioctls that pass kernel pointers or expect specific kernel behaviors may not work correctly. The CUSE interface has to marshal all data across the kernel boundary.

Security Considerations

A CUSE device running as your user can only do what your user can do. This is actually a feature. A buggy CUSE device can’t corrupt kernel memory or crash the system. Worst case, you kill the process.

But this also means CUSE devices can’t directly access hardware or perform privileged operations.

CUSE and FUSE: The Shared Foundation

CUSE is not a separate project. It’s built on top of FUSE’s infrastructure, reusing the same protocol handling, just redirecting operations to a character device instead of a filesystem mount .

What they share:

  • Same libfuse library
  • Same kernel communication protocol
  • Same request/reply message format
  • Same event loop model

What differs:

  • FUSE: mount creates a filesystem hierarchy

  • CUSE: /dev/cuse creates a single device node

  • FUSE: Operations like readdir(), mkdir(), unlink()

  • CUSE: Operations like open(), read(), write(), ioctl()

This shared foundation means the actual CUSE kernel implementation is only around 1,000 lines of code because it leverages all of FUSE’s existing machinery.

When Should You Use CUSE?

Use CUSE when:

✅ You need a character device without hardware
✅ You’re emulating a legacy device interface
✅ You’re prototyping a driver before committing to kernel code
✅ You want to debug device logic without kernel crashes
✅ You need dynamic device creation/destruction
✅ Security isolation matters (userspace crashes don’t take down the kernel)

Don’t use CUSE when:

❌ You need maximum performance (use kernel drivers)
❌ You need to handle hardware interrupts (use UIO or kernel drivers)
❌ You need complex memory-mapped I/O (use UIO)
❌ The device semantics don’t fit character device model (maybe use FUSE for filesystem-like access)

The Big Picture

CUSE extends the FUSE philosophy from filesystems to devices. Both say: “Most driver logic doesn’t need kernel privileges. Let’s run it in userspace where it’s safer, easier to debug, and simpler to develop.”

The stack looks like this:

┌─────────────────────────────┐
│   Application (cat, echo)    │
│    Uses: open(), read()      │
└──────────┬──────────────────┘
           │ System calls
┌──────────▼──────────────────┐
│   Kernel VFS Layer           │
│   Character device node      │
└──────────┬──────────────────┘
           │ CUSE interface
┌──────────▼──────────────────┐
│   FUSE/CUSE Kernel Module    │
│   Protocol handling          │
└──────────┬──────────────────┘
           │ /dev/cuse communication
┌──────────▼──────────────────┐
│   Your Userspace Program     │
│   Device logic               │
└─────────────────────────────┘

From the application’s perspective, /dev/echo_device is just another character device. It doesn’t know or care that the implementation runs in userspace.

This is the power of abstraction. CUSE maintains the character device interface contract while completely changing the implementation strategy.
