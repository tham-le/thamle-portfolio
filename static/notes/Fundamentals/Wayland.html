WTF is Wayland?

The display protocol that merged the middleman into the manager

You click your mouse. A window moves. Pixels change color. You type on your keyboard, and characters appear on screen. WTF is actually making this happen?

The answer is simpler than you think, at least on Wayland. Your applications don’t talk directly to your graphics card. They don’t draw pixels straight to your screen. Instead, they talk to something called a display server, and Wayland is the language they use to have that conversation.

Let’s break down what’s actually happening when you stare at your screen.

The Restaurant Kitchen Analogy

Think of your desktop like a busy restaurant kitchen.

In the Wayland model:

  • Your applications are the line cooks. Each one prepares their dish (renders their window content) completely on their own station.
  • The compositor is the head chef AND the expeditor combined. It takes all the finished dishes (rendered window buffers), arranges them beautifully on the pass (your screen), and decides what gets served when.
  • The protocol is the kitchen’s communication system. Cooks call out “window ready!”, “need focus!”, “closing down station 3!“.

Here’s the key insight: each cook is responsible for making their own dish look perfect. They don’t ask the head chef to add garnish or plate it for them. They render everything themselves, hand over a complete buffer of pixels, and the compositor just arranges these finished products on screen.

This is fundamentally different from older systems where the display server did a lot of the cooking itself.

What Actually Is a Display Server?

A display server is the piece of software that sits between your applications and your graphics hardware. It has three jobs:

  1. Output: Get pixels from applications onto your screen
  2. Input: Capture your keyboard, mouse, and touch events and route them to the right application
  3. Coordination: Make sure multiple windows can coexist without stepping on each other’s toes

Without a display server, every application would need to know how to talk directly to your specific graphics card, handle multi-monitor setups, manage window stacking order, and coordinate with every other running application. That’s a nightmare.

The display server abstracts all of that away. Applications just say “here’s my window content, please display it” and the display server figures out the hardware-specific details.

The Wayland Way: Compositor as King

Wayland is the language (protocol) that applications use to talk to a display server, but here’s where it gets interesting: in Wayland, the server and compositor are one and the same.

Let’s unpack that.

The Compositor Model

In Wayland’s architecture, there’s a single program called the compositor that does everything:

  • Acts as the display server (coordinates applications)
  • Manages window stacking and positioning
  • Handles all the visual effects (transparency, shadows, animations)
  • Controls input routing (who gets the keyboard focus?)
  • Talks directly to the Kernel to render the final image

There is no separation of concerns here. The compositor is an all-in-one package.

Popular Wayland compositors include:

  • Mutter (used by GNOME)
  • KWin (used by KDE Plasma)
  • Sway (tiling window manager)
  • Weston (the reference implementation)

Each desktop environment essentially writes its own Wayland compositor. This gives them total control over the display stack.

Client Responsibility

Here’s what makes Wayland different: Wayland does not expose any API for rendering, but delegates to clients such tasks (including the rendering of fonts, widgets, etc.).

Your application is responsible for:

  • Rendering its entire window content into a buffer
  • Drawing window decorations (title bar, borders) via a graphics toolkit
  • Handling all the text rendering, widget drawing, everything visual

Then it hands this complete, pre-rendered buffer to the compositor and says “put this on screen.”

The compositor’s job is dead simple: composite these buffers together, add effects, and send the final image to the display hardware.

How Wayland Communication Works

The Wayland protocol is sent over a UNIX domain stream socket, usually named wayland-0 in your $XDG_RUNTIME_DIR (typically /run/user/1000/).

The Protocol

The Wayland protocol is an asynchronous object-oriented protocol. Here’s what that means in practice:

Object-oriented: Everything in Wayland is an object with methods you can call.

  • wl_display represents the connection itself
  • wl_compositor creates surfaces
  • wl_surface is a window (or part of one)
  • wl_keyboard, wl_pointer handle input

Asynchronous: Clients send requests without waiting for immediate responses. The compositor sends events back when something interesting happens.

Messages: All communication happens via structured messages:

  • Requests: Client to compositor (“create a surface”, “attach this buffer”, “give me focus”)
  • Events: Compositor to client (“you got focus”, “mouse moved”, “window was resized”)

The Connection Dance

When an application starts:

  1. Connect to the compositor: Open a Socket connection to wayland-0
  2. Get the registry: Ask “what interfaces do you support?”
  3. Bind to interfaces: Say “I want to use wl_compositor and wl_shm”
  4. Create a surface: Request a wl_surface object (this is your window)
  5. Render content: Draw pixels into shared memory
  6. Attach and commit: Tell the compositor “here’s my buffer, please display it”

The compositor then decides where on screen to put that surface, adds any effects, and composites it with all the other surfaces.

Shared Memory: The Buffer Exchange

Here’s a critical detail: if the client uses a software (CPU) renderer and the result is stored in system memory, then client and compositor can use shared memory to implement the buffer communication without extra copies.

The buffer is where your application renders its pixels. This could be:

  • Shared memory via wl_shm (software rendering)
  • DMA-BUF (GPU-accelerated, zero-copy directly from VRAM)

The compositor and client share access to the same memory region. The client writes pixels, the compositor reads them. No copying needed (in the ideal case).

Think of it like a double-sided whiteboard in our restaurant kitchen. The cook writes on one side, the expeditor reads from the other side. They’re looking at the same physical object.

This is way more efficient than older systems that copied pixels multiple times through the display pipeline.

Security by Design

One huge advantage of Wayland: Wayland isolates the input and output of every window, achieving confidentiality, integrity and availability for both.

What does this mean practically?

Each application is sandboxed. An application cannot:

  • See what other windows are displaying (screen scraping is hard)
  • Inject keyboard events into other applications (no keylogging)
  • Manipulate other windows’ positions or content
  • Read your clipboard without permission

The compositor mediates everything. If Application A wants to read the clipboard, the compositor decides if that’s allowed. If Application B wants to screenshot the screen, the compositor handles the permission flow.

This is a massive security improvement over older display systems where any application could spy on any other application.

The Extension Model

The core Wayland protocol is intentionally minimal. A Wayland compositor can define and export its own additional interfaces, which has become the standard way to implement Wayland protocol extensions.

This means:

  • Core protocol: Basic window management, input, buffers
  • Extensions: Screenshot APIs, screen recording, special compositor features

Popular extension protocols:

  • xdg-shell: Standard window management (maximize, minimize, fullscreen)
  • xdg-decoration: Server-side window decorations
  • wlr-layer-shell: Desktop shell components (panels, notifications)
  • linux-dmabuf: Zero-copy buffer sharing with GPU

Compositors can choose which extensions to implement. This keeps the core protocol clean while allowing innovation.

The Tradeoff: Compositor Power

Here’s the thing: Wayland gives all the power to the compositor.

Compositor decides:

  • Where windows appear
  • How input is routed
  • What extensions are available
  • Security policies

This is great for integration. Your desktop environment controls the entire display stack. No external window manager. No separate compositor. Everything works together seamlessly.

But it also means Wayland behavior varies by compositor. The experience on GNOME (using Mutter) might differ from KDE (using KWin) in subtle ways. Features available in one compositor might not exist in another.

This is by design. Wayland gives compositors freedom to innovate and optimize for their specific use case.

Legacy Application Support: XWayland

What about older applications that speak the old display protocol?

For legacy applications that have not yet been ported to run natively on Wayland, a compatibility layer known as XWayland provides the necessary support.

XWayland is an X11 server that runs as a Wayland client. Let that sink in.

It’s like having a translator in our restaurant kitchen who speaks the old kitchen slang. Legacy apps think they’re talking to an old-style display server, but XWayland translates their requests into Wayland protocol messages.

The compositor sees XWayland as just another client. XWayland sees old apps as X11 clients. Everyone’s happy.

This means you can run a mix of native Wayland apps and legacy X11 apps on the same desktop without issues.

Why Wayland Exists

Modern graphics systems need:

  • Low latency: Every millisecond counts for smooth animations
  • Security: Applications shouldn’t spy on each other
  • Efficiency: No unnecessary pixel copies
  • Simplicity: Clean protocol, easy to implement

Wayland is the next-generation display server for Unix-like systems, designed and built by the alumni of the venerable Xorg server. The people who maintained the old system for decades looked at its accumulated complexity and said “we can do better.”

They designed Wayland from scratch with modern hardware in mind:

  • GPUs that can composite efficiently
  • Security requirements for multi-user systems
  • Touch input and high-DPI displays
  • Clean separation between system and applications

The Big Picture

When you boot a Linux system with Wayland:

  1. The compositor starts (e.g., Mutter, KWin)
  2. It talks to the Kernel via DRM/KMS to control the display hardware
  3. It creates a Socket at wayland-0
  4. Applications connect to this socket
  5. Applications render their windows into buffers
  6. The compositor arranges these buffers on screen
  7. Input events flow back through the compositor to applications

Everything flows through the compositor. It’s the central hub, the single source of truth for what’s on your display.

This is Wayland: a lean protocol, a compositor-centric architecture, and a clean break from decades of accumulated complexity.
