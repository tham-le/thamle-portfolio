WTF is a Windows Package Installer?

Why does Windows have two completely different ways to install software, and why should you care?

You download software on Windows and get hit with a choice: setup.exe or setup.msi. Sometimes you get both. Sometimes the EXE just extracts an MSI. WTF is going on here?

This confusion exists because these two formats solve installation in fundamentally different ways. One is a wild west of “do whatever you want,” and the other is a tightly controlled database that Windows itself understands.

The Core Analogy

Think of installing software like getting furniture delivered to your apartment:

EXE installers are like hiring a random person off Craigslist. They show up, assemble your furniture however they see fit, maybe drill some holes in unexpected places, and leave. If something goes wrong halfway through, you’re stuck with a half assembled IKEA nightmare. Want to undo what they did? Good luck figuring out what they changed.

MSI installers are like hiring a licensed contractor who files detailed blueprints with the city. The building inspector (Windows Installer service) reviews the plans, tracks every change, and can reverse the work if something goes wrong. Need to fix a broken piece later? The inspector knows exactly what was installed and where.

This difference matters enormously for system administrators, security, and your own sanity.

Part 1: WTF is an EXE?

The PE Format Foundation

Before we talk about installers, you need to understand what an EXE actually is.

EXE files use the Portable Executable (PE) format, Windows’ native binary format since Windows NT 3.1. The PE format is used for native executables, dynamic link libraries (.dll), kernel modules, control panel applications, and many other file types.

Every EXE starts with a DOS header (the famous “MZ” bytes, named after Mark Zbikowski), followed by headers that tell Windows how to load the program into memory. Each PE file includes a DOS executable header, which generally displays the message “This program cannot be run in DOS mode.”

Here’s the thing: an EXE is just an executable program. It can do literally anything code can do. When that program happens to be an installer, it’s just a program that:

  1. Extracts files from itself
  2. Copies them to your disk
  3. Modifies the registry
  4. Creates shortcuts
  5. Whatever else the developer coded

There’s no standard, no blueprint, no rules.

EXE Installer Frameworks

Since building an installer from scratch is tedious, developers use frameworks:

NSIS (Nullsoft Scriptable Install System): Created by the folks who made Winamp. NSIS is a powerful open source installer creation tool for Windows. It provides a high degree of flexibility through a scripting language, allowing developers to create custom, professional installation setups. NSIS is famous for producing tiny installers and being infinitely customizable.

Inno Setup: Inno Setup is a free installer for Windows programs that supports a wide range of features including disk spanning, password protection, uninstall capabilities, and silent installs. It uses a Pascal style scripting language and is popular for its ease of use.

InstallShield: The commercial heavyweight, used by enterprises for decades.

Each framework produces a self contained EXE that bundles your files, compression logic, UI, and installation scripts into one binary. The end result looks similar to users, but the internal implementation varies wildly.

The Problem with EXE Installers

User: "I want to uninstall this program"
EXE Installer: "I left an uninstaller somewhere... maybe in Program Files? 
               Or was it AppData? Anyway, it might not clean up everything.
               Good luck!"

EXE installers have several pain points:

  1. No standard uninstall mechanism: Each installer implements its own uninstall logic (or doesn’t)
  2. No repair capability: If files get corrupted, you typically reinstall from scratch
  3. No rollback on failure: If installation fails halfway, you’re left with a mess
  4. Enterprise deployment nightmare: System admins can’t easily push EXE installers via Group Policy
  5. Security concerns: EXE files could hide malware, which is why MSI files are preferred over EXE file installers, especially in enterprise environments where security is crucial.

Part 2: WTF is an MSI?

The Database Approach

MSI files are fundamentally different. They’re not programs; they’re databases.

The installation information, and optionally the files themselves, are packaged in installation packages, loosely relational databases structured as COM Structured Storages and commonly known as “MSI files”.

When you double click an MSI, you’re not running the MSI itself. You’re handing it to msiexec.exe, the Windows Installer service, which reads the database and performs the installation according to its contents.

User double-clicks setup.msi
         │
         ▼
    msiexec.exe (Windows Installer Service)
         │
         ├── Reads database tables
         ├── Validates consistency
         ├── Plans installation
         ├── Executes with rollback support
         └── Logs everything

Inside the MSI Database

The MSI file uses separate streams for each of the tables in the database. Streams are used to store things like UI graphics, CustomAction DLLs, and even the binaries to be installed.

The key tables include:

TablePurpose
FileWhat files to install
ComponentLogical groupings of files
FeatureUser visible installation options
RegistryRegistry entries to create
DirectoryFolder structure
CustomActionCustom installation logic

You can actually open an MSI file and browse its tables using Microsoft’s Orca tool or even 7 Zip for a quick peek at the structure.

The Hierarchy: Products, Features, Components

MSI organizes installations hierarchically:

Product: A single, installed, working program (or set of programs) is a product. A product is identified by a unique GUID (the ProductCode property) providing an authoritative identity throughout the world.

Feature: What users see in “custom installation” dialogs. A feature is a hierarchical group of components. A feature may contain any number of components and other sub features. Smaller packages can consist of a single feature.

Component: Components are collections of resources that are always installed or uninstalled as a whole on the computer. Resources could be files, registries, shortcuts, or basically anything else that can be installed.

This hierarchy enables powerful features like installing only spell check but not thesaurus, or removing a feature without touching others.

The Self Healing Magic

Here’s something cool: MSI packages can repair themselves.

A key path is a specific file, registry key, or ODBC data source that the package author specifies as critical for a given component. When an MSI based program is launched, Windows Installer checks the existence of key paths. If there is a mismatch between the current system state and the value specified in the MSI package (e.g., a key file is missing), the related feature is re installed. This process is known as self healing or self repair.

Delete a critical DLL? Windows Installer notices and reinstalls it. This is why you sometimes see “Configuring Microsoft Office…” when launching Word after updates.

Transactional Installation

The vision driving development was that setup needed to be a transacted set of changes to a target machine that could be aborted and cleaned up if an error occurred or the user cancelled setup.

If an MSI installation fails at step 47 of 50, Windows Installer rolls back all 46 previous steps, leaving your system exactly as it was before. EXE installers? You get whatever partial state existed when it crashed.

When to Use Each

Use MSI when:

  • Deploying to enterprise environments
  • You need reliable uninstall/repair
  • System admins need to customize installations (via MST transform files)
  • Security auditing is required

Use EXE when:

  • You need maximum UI customization
  • Installation requires complex logic MSI can’t express
  • Targeting consumer markets where “just works” matters more than enterprise features
  • You’re wrapping an MSI anyway (many EXE installers extract and run an MSI internally)

The Plot Twist: EXE Often Contains MSI

Here’s a dirty secret of Windows installers: many “EXE” installers are just self extracting archives that contain an MSI.

setup.exe
    │
    ├── Extracts to temp folder
    │       │
    │       ├── actual_installer.msi
    │       ├── prerequisites.msi
    │       └── runtime.msi
    │
    └── Calls msiexec.exe on each MSI

Why do this? The EXE wrapper can:

  • Check prerequisites (like .NET Framework version)
  • Show a fancier UI than MSI’s built in dialogs
  • Chain multiple MSI installations together
  • Download additional content during install

So when someone says “use MSI for enterprise,” they often mean “make sure there’s an MSI available,” even if the primary installer is an EXE.

The Modern Era: MSIX

Microsoft noticed that MSI, despite its power, has limitations for modern app deployment. In 2018, they introduced MSIX, which combines the best of MSI (declarative, transactional) with modern features:

  • Containerized installation: Apps get their own isolated view of the filesystem and registry
  • Clean uninstall guaranteed: When you remove an MSIX app, everything goes with it
  • Differential updates: Only changed bits get downloaded
  • Microsoft Store integration: Though MSIX works outside the Store too

MSIX is becoming the recommended format for new Windows applications, but MSI isn’t going anywhere. Over 90% of organizations still actively use MSI files either via vendors or created in house. The format’s maturity and extensive Windows engineering investment keeps it cemented as a key enterprise tool.
