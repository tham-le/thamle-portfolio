WTF is FUSE?

WTF is FUSE?

Nov 26, 202515 min read

  • fuse

  • userspace

  • vfs

The system that lets you implement filesystems as regular programs

You run ls on a directory. Files appear. You cat a file. Content shows up. Normal filesystem stuff.

But what if I told you that directory isn’t backed by a disk? What if it’s actually pulling files over SSH from a server halfway around the world? Or reading from a zip archive? Or fetching from Google Drive? And all of this without a single line of kernel code?

That’s FUSE. Filesystem in Userspace. The system that turned filesystem development from “write kernel modules, reboot constantly, debug kernel panics” into “write a regular program, use gdb, iterate quickly.”

Let’s understand how this black magic works.

The Kernel Filesystem Problem

Traditionally, filesystems lived in the Kernel. Every filesystem (ext4, NTFS, XFS, etc.) required kernel code. This made sense because filesystems need deep integration with the kernel’s Virtual Filesystem (VFS) layer.

But this created brutal problems:

  1. Kernel crashes: One bug in your filesystem code takes down the entire system
  2. Root privileges required: You can’t load kernel modules without root
  3. Slow iteration: Change code, recompile kernel module, reboot, test, crash, repeat
  4. Language restrictions: Must use C, can’t use Python/Go/Rust easily
  5. Deployment headaches: Distributing kernel modules is a nightmare

Meanwhile, interesting filesystem ideas kept emerging:

  • “I want to mount my SSH server as a local directory"
  • "I want to access Windows NTFS drives on Linux"
  • "I want to encrypt files transparently"
  • "I want to access cloud storage as a filesystem”

All of these required writing kernel code. Until FUSE.

The Core Analogy: The Restaurant Intermediary

Think of a filesystem like a restaurant.

Traditional kernel filesystem: The restaurant (kernel) has a kitchen built into its foundation. If you want to change the menu, you need to hire contractors, do construction work, and potentially damage the building’s structure. One kitchen fire brings down the whole restaurant.

FUSE filesystem: The restaurant has a service window. Behind that window, you can plug in any food truck you want. The restaurant still handles customers (applications), takes orders (system calls), and serves food (data). But the actual cooking happens in the food truck (userspace program).

If the food truck catches fire, you just disconnect it and the restaurant keeps running. You can swap food trucks without reconstruction. Anyone can operate a food truck without a contractor’s license.

The VFS layer is the restaurant’s front of house (takes orders from customers).
The FUSE kernel module is the service window (passes orders to the back).
Your program is the food truck (does the actual work).

Applications don’t know or care where their data comes from. They just see a normal filesystem.

How FUSE Actually Works

FUSE consists of two components: the fuse kernel module (maintained in the regular kernel repositories) and the libfuse userspace library. Together, they create a bridge between kernel space and userspace.

The Three-Layer Architecture

Layer 1: The Kernel Module

  • Lives in the kernel as a VFS filesystem implementation
  • Registers with VFS as type “fuse”
  • Provides /dev/fuse, a character device for communication
  • Routes filesystem operations to userspace

Layer 2: The libfuse Library

  • Provides functions to mount the filesystem, unmount it, read requests from the kernel, and send responses back
  • Implements the FUSE protocol
  • Handles all the messy kernel communication
  • Offers high-level and low-level APIs

Layer 3: Your Filesystem Program

  • Regular userspace application
  • Implements filesystem operations (read, write, stat, etc.)
  • Can be written in any language with FUSE bindings
  • Runs with normal user privileges

The Communication Flow

When you run ls /mnt/sshfs:

Application              Kernel VFS           FUSE Module         Your Program
    |                        |                     |                   |
    |--readdir("/mnt/sshfs")-->|                   |                   |
    |                        |                     |                   |
    |                    [VFS routing]             |                   |
    |                        |                     |                   |
    |                        |---FUSE_READDIR----->|                   |
    |                        |                     |                   |
    |                        |                  [write to             |
    |                        |                   /dev/fuse]            |
    |                        |                     |                   |
    |                        |                     |--read /dev/fuse-->|
    |                        |                     |                   |
    |                        |                     |             [your readdir()
    |                        |                     |              implementation]
    |                        |                     |                   |
    |                        |                     |<---file list------|
    |                        |                     |                   |
    |                        |<--directory entries-|                   |
    |                        |                     |                   |
    |<---file list-----------|                     |                   |

The file descriptor to use for communication between the userspace filesystem and the kernel must be obtained by opening the FUSE device (‘/dev/fuse’). Every filesystem operation becomes a message sent through this device.

The Protocol: How Messages Work

The FUSE protocol defines how the kernel and your program talk. Every message from the kernel to userspace starts with the fuse_in_header struct, with the most interesting fields being the opcode (the operation to perform) and nodeid (the file or directory to operate on).

Key Concepts

Inodes and NodeIDs:

  • The root directory has a fixed nodeid of 1. The nodeid is an arbitrary value chosen by the userspace filesystem.
  • Your filesystem must remember which files/directories correspond to which nodeids
  • The kernel uses nodeids to reference files instead of paths

Request Types (opcodes):

  • FUSE_LOOKUP: Find a file by name in a directory
  • FUSE_GETATTR: Get file attributes (size, permissions, timestamps)
  • FUSE_READ: Read data from a file
  • FUSE_WRITE: Write data to a file
  • FUSE_READDIR: List directory contents
  • FUSE_OPEN: Open a file
  • FUSE_RELEASE: Close a file
  • Plus dozens more for mkdir, rmdir, rename, symlink, etc.

The Mount Process

Unlike regular filesystems, FUSE mounts don’t require root privileges for the actual filesystem code. Here’s how it works:

Using fusermount3

The fusermount3 program is installed setuid root, done to allow normal users to mount their own filesystem implementations.

The steps:

  1. Your program calls fuse_mount() from libfuse
  2. libfuse runs fusermount3 (setuid root helper)
  3. fusermount3 opens /dev/fuse and passes the fd back to your program
  4. fusermount3 performs the actual mount() system call
  5. Your program now receives requests through the /dev/fuse fd

Safety restrictions:

To limit the harm that malicious users can do, fusermount3 enforces limitations: The user can only mount on a mountpoint for which they have write permission.

This means I can mount a FUSE filesystem on ~/my_mount but not on /etc/evil_mount.

Real-World FUSE Filesystems

FUSE isn’t just for toy examples. Production systems rely on it daily.

SSHFS: Network Filesystem Over SSH

SSHFS enables secure remote access to files over a network using the SSH protocol, allowing users to mount a remote directory from a server onto their local machine.

Behind the scenes, SSHFS translates FUSE operations into SFTP protocol commands. The remote server doesn’t need any special software beyond sshd.

NTFS-3G: Windows Filesystem on Linux

NTFS-3G is a stable, open-source, third-party NTFS driver for Linux that allows read-write access to NTFS partitions.

Windows NTFS is complex and proprietary. Writing a kernel driver would be difficult and risky. Using FUSE, ntfs-3g implements NTFS in userspace where bugs don’t crash the kernel.

EncFS: Transparent Encryption

EncFS intercepts read/write operations and transparently encrypts/decrypts data.

Performance: The Trade-Off

FUSE adds overhead. FUSE incurs performance overhead primarily due to frequent context switches between the kernel and the userspace daemon, as well as the serialization and deserialization of requests over the FUSE protocol via the /dev/fuse device.

Typical overhead:

FUSE filesystems typically exhibit a 2-3× slowdown compared to native kernel filesystems like Ext4 in worst-case scenarios, such as random small reads or multi-threaded metadata operations, though performance degradation can be as low as imperceptible (within 5%) for sequential large-block I/O on optimized setups.

Where it hurts most:

  • Metadata operations (stat, readdir on many files)
  • Small, random reads/writes
  • Multi-threaded workloads

Where it’s fine:

  • Large sequential I/O
  • Network-bound operations (SSHFS already has network latency)
  • Applications with their own caching

Why accept the cost?

  1. Safety: Userspace bugs don’t crash the kernel
  2. Development speed: Iterate without rebooting
  3. Portability: Same code works across kernel versions
  4. Flexibility: Use any programming language
  5. Unique capabilities: Some filesystems only make sense in userspace (SSHFS, cloud storage)

For many use cases, the convenience massively outweighs the performance cost.

The Two APIs: High-Level vs Low-Level

libfuse offers two APIs: a “high-level”, synchronous API, and a “low-level” asynchronous API.

High-Level API

Works with paths:

  • Easier to use
  • Path-based operations
  • Synchronous (callback returns when done)
  • Good for simple filesystems

Low-Level API

Works with inodes:

  • More complex but more powerful
  • Inode-based operations (like the kernel)
  • Asynchronous (can reply later)
  • Better performance, more control
  • Used by serious implementations

Most simple filesystems use the high-level API. Production systems often use low-level for performance.

Security and Limitations

FUSE adds safety but also constraints.

Security Measures

Isolation: A FUSE filesystem runs with the mounting user’s privileges. If my FUSE program has a bug, it can only damage what I can already damage. No kernel corruption.

Mount restrictions:

The mount owner should not be able to use this capability to compromise the system: mount owner should not be able to get elevated privileges with the help of the mounted filesystem, mount owner should not get illegitimate access to information from other users’ and the super user’s processes.

Allow_other:

By default, access is restricted to the mounting user. The allow_other option overrides this security measure, restricting file access to the user mounting the filesystem, but this option is by default only allowed to root.

Known Issues

Permission caching bug:

FUSE has an unresolved security bug: if the default_permissions mount option is not used, the results of the first permission check performed by the filesystem for a directory entry will be re-used for subsequent accesses as long as the inode is present in the kernel cache, even if the permissions have since changed.

This is a kernel issue that’s been known since 2006 but remains unfixed.

Possible Deadlocks

There are a couple of ways to deadlock a FUSE filesystem. If a process issuing a FUSE filesystem request is interrupted by a signal, the kernel handles this with INTERRUPT requests.

The solution: Connections can be monitored and aborted via the FUSE control filesystem at /sys/fs/fuse/connections, which allows administrative intervention if the daemon becomes unresponsive.

The Big Picture

FUSE changed filesystem development from specialized kernel work to regular application programming.

Before FUSE:

  • Write kernel module in C
  • Debug kernel panics
  • Reboot constantly
  • Need root access
  • Risk system crashes

With FUSE:

  • Write regular program in any language
  • Use standard debugging tools
  • Run as normal user
  • Iterate quickly
  • Bugs are isolated

The stack looks like this:

┌─────────────────────────────────────┐
│    Applications (ls, cat, vim)      │
│    Use standard system calls         │
└──────────────┬──────────────────────┘
               │ open(), read(), write()
┌──────────────▼──────────────────────┐
│       Kernel VFS Layer               │
│    (Virtual Filesystem abstraction)  │
└──────────────┬──────────────────────┘
               │
        ┌──────┴──────┐
        │             │
    [ext4]      [FUSE kernel]
    [NTFS]            │
    [XFS]             │ /dev/fuse
                      │
┌─────────────────────▼─────────────────┐
│         libfuse                        │
│    (Protocol implementation)           │
└─────────────────────┬──────────────────┘
                      │ Callbacks
┌─────────────────────▼─────────────────┐
│    Your Filesystem Program             │
│  (SSHFS, NTFS-3G, EncFS, etc.)        │
│    Runs in userspace                   │
└────────────────────────────────────────┘

Applications see a normal filesystem. The kernel sees a normal VFS implementation. Your program just responds to callbacks. Everyone’s happy.
