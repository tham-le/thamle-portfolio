ASCII: The OG Character Set

ASCII (American Standard Code for Information Interchange) was created in 1963. It‚Äôs a 7 bit encoding, which means it can represent exactly 128 characters (2^7 = 128).

ASCII was originally designed as a 7 bit code, using values from 0 to 127. Why 7 bits? At the time, computers had limited memory and processing power. Using 7 bits was a compromise between practicality and functionality, and it allowed for compatibility with existing 6 bit encodings.

Here‚Äôs what ASCII includes:

Unicode: One Encoding to Rule Them All

Unicode was created in 1991 with a radical idea: assign a unique number to every character in every human language, ever. Not just current languages. Dead languages. Emoji. Mathematical symbols. Musical notation. Everything.

UTF-8: The Genius Variable Width Encoding

UTF-8 (Unicode Transformation Format, 8 bit) is the most popular way to encode Unicode characters into bytes. As of July 2025, almost every webpage uses UTF-8. Why? Because it‚Äôs brilliant.

How UTF-8 Works

UTF-8 is a variable width encoding that uses 1 to 4 bytes per character, depending on the code point. Characters that are used more frequently take fewer bytes.

1 byte (for code points U+0000 to U+007F):

  • This is exactly ASCII! The first 128 characters are identical.
  • Binary format: 0xxxxxxx
  • Example: ‚ÄòA‚Äô = 01000001

UTF-8 is backward compatible with ASCII. A file containing only ASCII characters is identical whether encoded as ASCII or UTF-8.

2 bytes (for code points U+0080 to U+07FF):

  • Example: ‚Äò√©‚Äô (U+00E9) = 11000011 10101001

3 bytes (for code points U+0800 to U+FFFF):

  • Example: ‚Äò‰∏≠‚Äô (U+4E2D) = 11100100 10111000 10101101

4 bytes (for code points U+10000 to U+10FFFF):

  • Covers emoji, rare scripts, historical characters
  • Example: ‚Äôüí©‚Äô (U+1F4A9) = 11110000 10011111 10010010 10101001

Notice the pattern? The leading bits tell you how many bytes the character uses:

  • Starts with 0: One byte character
  • Starts with 110: Two byte character (first byte)
  • Starts with 1110: Three byte character (first byte)
  • Starts with 11110: Four byte character (first byte)
  • Starts with 10: Continuation byte (not the first byte)

Why UTF-8 Won

  1. Backward compatible with ASCII: Old programs expecting ASCII just work with UTF-8.
  2. Space efficient: English text uses 1 byte per character, same as ASCII.
  3. No byte order issues: Unlike UTF-16 and UTF-32, there‚Äôs no ambiguity about which byte comes first.
  4. Self synchronizing: If you jump into the middle of a UTF-8 string, you can figure out where characters start by looking at the bit patterns.

UTF-16 and UTF-32: The Alternatives

UTF-16: Uses 2 bytes for most common characters (BMP), 4 bytes for rare ones. UTF-16 encodes characters using either one 16 bit code unit or a pair of 16 bit surrogate pairs. Used internally by Windows, Java, and JavaScript.

Problem: Not backward compatible with ASCII, and you still need variable width for emoji and rare characters.

UTF-32: Uses exactly 4 bytes for every character, period. Simple, but wastes space. A file containing only ASCII characters would be four times larger in UTF-32 than in UTF-8. Rarely used.
