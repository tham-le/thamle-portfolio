WTF is LD_PRELOAD?

The environment variable that lets you hijack ANY function in ANY program

You run a program. It calls malloc(). But instead of getting memory from the system allocator, your custom function executes. It logs the allocation, checks for leaks, and then calls the real malloc(). The program has no idea anything changed.

WTF just happened?

You used LD_PRELOAD, Linux’s most powerful debugging tool and simultaneously one of its biggest security headaches. It lets you intercept and replace ANY library function in ANY dynamically linked program without recompiling, without modifying binaries, without even having source code.

It’s like having a universal remote control for every function call in every program on your system.

Let’s understand how this works, why it’s both brilliant and dangerous, and how to actually use it.

The Restaurant Switcheroo

Think of a program as a restaurant kitchen making dishes. When the chef needs salt, they don’t keep salt in the kitchen; they call a supplier (the C Standard Library).

Normal operation:

  • Chef calls “give me salt”
  • Order goes to the regular supplier (libc)
  • Salt arrives from the expected source

With LD_PRELOAD:

  • You intercept the phone line
  • When chef calls “give me salt”, YOUR supplier answers first
  • You can: give them your salt, log the request, modify the order, or pass it to the real supplier
  • The chef never knows you’re in the middle

That’s LD_PRELOAD. It hijacks the connection between programs and their libraries, letting you inject your own code at the function call level.

The Problem This Solves

Programs on Linux use Shared Libraries (.so files) for most functionality. When you call printf(), malloc(), or open(), you’re calling functions from these shared libraries.

The traditional problem:

  • You want to debug how a program uses malloc()
  • But you don’t have source code
  • And you can’t modify the binary
  • And recompiling the entire system is insane

The LD_PRELOAD solution: You can specify a shared library to load before any other library, and symbols in that library will override symbols in other libraries.

Your custom malloc() gets called instead of the system’s malloc(). You can log it, modify it, or pass through to the real one. The program never knows.

This enables:

  • Debugging: Log every function call without modifying code
  • Testing: Replace functions with mocks
  • Performance profiling: Inject timing measurements
  • Bug workarounds: Fix broken library behavior without patches
  • Security research: Analyze program behavior
  • Malware (unfortunately): Inject rootkits and spyware

How It Actually Works

When you run a program, the Dynamic Linker (ld.so) loads all required shared libraries before main() executes. The LD_PRELOAD environment variable specifies additional shared objects to be loaded before all others.

The Loading Order

Libraries specified in LD_PRELOAD are loaded and added to the link map in left-to-right order:

  1. LD_PRELOAD libraries (your custom code)
  2. Libraries specified in the binary’s DT_NEEDED section
  3. Libraries from LD_LIBRARY_PATH
  4. System default paths (/lib, /usr/lib)

Symbol Resolution

When a program calls a function:

  1. Dynamic linker searches loaded libraries in order
  2. First match wins
  3. Your preloaded library is searched FIRST
  4. If you define malloc(), yours is called instead of libc’s

The key insight: You can override symbols in standard libraries by creating a library with the same symbols and specifying it in LD_PRELOAD.

The Security Problem

Here’s the terrifying part: LD_PRELOAD can be used for malicious purposes, such as injecting malware into processes.

The Security Mitigations

For security reasons, the dynamic linker determines if a binary should run in secure-execution mode. In secure execution mode:

  1. Preload pathnames containing slashes are ignored
  2. Shared objects are preloaded only from standard directories and only if they have set-user-ID mode bit enabled
  3. LD_PRELOAD is completely stripped from the environment

Secure execution mode activates when:

  • The binary is setuid or setgid
  • Real UID != Effective UID
  • Real GID != Effective GID
  • The process has file capabilities

If LD_PRELOAD wasn’t ignored for setuid processes, a user could run /bin/passwd with a malicious library that spawns a root shell.

Why this matters: The loader will ignore LD_PRELOAD if ruid != euid, preventing privilege escalation attacks.

Limitations and Gotchas

1. Only Works on Dynamically Linked Programs

LD_PRELOAD is specific to processes that use ld.so to dynamically load libraries. It won’t work with statically compiled executables.

Check if a program is dynamically linked:

2. Symbol Must Be Dynamic

If a program has malloc() inlined or statically linked, your override won’t work. The function must be dynamically resolved.

3. Constructor Execution Order

If your library has a __attribute__((constructor)) function, it runs BEFORE main(), but possibly AFTER constructors in other preloaded libraries.

4. Thread Safety

Your replacement functions MUST be thread-safe if the original is. Logging to files without locking can corrupt output in multithreaded programs.

The Big Picture

When you use LD_PRELOAD:

  1. You write a Shared Library with replacement functions
  2. You compile it as a .so file with -shared -fPIC
  3. You set LD_PRELOAD=/path/to/your.so
  4. You run the target program
  5. The Dynamic Linker loads your library FIRST
  6. Symbol resolution finds YOUR functions before system functions
  7. Your code executes on every call
  8. You can log, modify, or pass through to the real function

This is LD_PRELOAD: a frighteningly powerful mechanism that lets you hijack function calls in running programs without touching their code, perfect for debugging and testing, and equally perfect for malware.

Alternatives and Related Techniques

/etc/ld.so.preload

System-wide preloading file. The /etc/ld.so.preload file is an alternative to the LD_PRELOAD environment variable, working similarly but applying system-wide. Requires root to modify.

LD_LIBRARY_PATH

Changes where the linker SEARCHES for libraries, but doesn’t force specific libraries to load first. LD_PRELOAD dictates which libraries are loaded first to potentially override symbols, while LD_LIBRARY_PATH specifies additional directories where the dynamic linker should search for libraries.

LD_AUDIT

Hooks into the dynamic linking process itself. More powerful than LD_PRELOAD but much more complex.

ptrace()

The System Call that debuggers use. Can intercept and modify ANY system call, not just library functions. Even more powerful, but requires the traced process to cooperate.
