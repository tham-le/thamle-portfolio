WTF are PDAs and FSMs?

The simpler, dumber cousins of Turing Machines that actually run most of your software

The Core Analogy

Imagine three workers in our warehouse:

  1. FSM Worker: Has terrible memory. Can only remember their current mood (state). Reads the conveyor belt but can’t write anything. Makes decisions based solely on current mood and what they see. This worker handles your regex matching, network protocols, and traffic lights.

  2. PDA Worker: Same as FSM worker, but gets a stack of Post-it notes. Can put new notes on top or remove the top note. Still can’t write on the conveyor belt. This worker handles your programming language parsers, XML validators, and matching parentheses.

  3. Turing Machine Worker: Can write on the belt, move anywhere, infinite memory. We met this genius last time.

Here’s the beautiful hierarchy: FSM ⊂ PDA ⊂ Turing Machine. Each one is strictly more powerful than the previous.

The Power Hierarchy

  • FSM: Can recognize regular languages (regex)
  • PDA: Can recognize context-free languages (most programming languages)
  • Turing Machine: Can recognize anything computable

FSM: The Amnesia Machine

A Finite State Machine (FSM) is like a Turing Machine that got lobotomized. It can only:

  • Be in one of a finite number of states
  • Read input symbols one at a time
  • Transition to a new state based on current state + input symbol
  • Accept or reject the input

That’s it. No writing. No going backwards. No memory except the current state.

Why FSMs Matter

Despite being “dumb,” FSMs are EVERYWHERE:

  • Regex engines: Every regex is compiled to an FSM
  • Network protocols: TCP state machine (LISTEN, SYN_SENT, ESTABLISHED, etc.)
  • Lexical analysis: Tokenizing your source code
  • Game AI: Enemy behavior patterns
  • UI components: Button states (normal, hover, pressed, disabled)

Pseudo-Code: A URL Validator FSM

STATES = {START, SCHEME, COLON, SLASH1, SLASH2, DOMAIN, PATH, ACCEPT, REJECT}

FUNCTION validate_url(input_string):
    current_state = START
    
    FOR EACH character IN input_string:
        current_state = transition(current_state, character)
        IF current_state == REJECT:
            RETURN False
    
    RETURN current_state IN {DOMAIN, PATH, ACCEPT}

FUNCTION transition(state, char):
    MATCH (state, char):
        // Start state
        CASE (START, 'h'):
            RETURN SCHEME
        CASE (START, _):
            RETURN REJECT
            
        // Building "http" or "https"
        CASE (SCHEME, 't'):
            RETURN SCHEME
        CASE (SCHEME, 'p'):
            RETURN SCHEME
        CASE (SCHEME, 's'):
            RETURN SCHEME  // optional 's'
        CASE (SCHEME, ':'):
            RETURN COLON
            
        // Expecting "://"
        CASE (COLON, '/'):
            RETURN SLASH1
        CASE (SLASH1, '/'):
            RETURN SLASH2
            
        // Domain name
        CASE (SLASH2, letter_or_digit):
            RETURN DOMAIN
        CASE (DOMAIN, letter_or_digit_or_dot):
            RETURN DOMAIN
        CASE (DOMAIN, '/'):
            RETURN PATH
            
        // Path (everything after domain)
        CASE (PATH, any_valid_char):
            RETURN PATH
            
        DEFAULT:
            RETURN REJECT

This FSM can tell if something looks like a URL, but it can’t count (can’t ensure equal numbers of opening and closing tags) or remember what it saw before (can’t check if a variable was declared before use).

The Assembly Connection

FSMs map perfectly to jump tables in Assembly:

; Pseudo-assembly FSM for detecting binary numbers divisible by 3
; States: ZERO (remainder 0), ONE (remainder 1), TWO (remainder 2)

fsm_div3:
    xor rax, rax          ; State ZERO initially
    
process_bit:
    movzx rbx, byte [rsi] ; Read next bit
    inc rsi
    
    ; Jump table: (state * 2 + bit) * 8
    lea rcx, [rax + rax]  ; state * 2
    add rcx, rbx          ; + bit
    jmp [jump_table + rcx * 8]
    
state_zero_bit_0:
    mov rax, 0    ; Stay in ZERO
    jmp check_end
    
state_zero_bit_1:
    mov rax, 1    ; Go to ONE
    jmp check_end
    
state_one_bit_0:
    mov rax, 2    ; Go to TWO
    jmp check_end
    
state_one_bit_1:
    mov rax, 0    ; Go to ZERO
    jmp check_end
    
; ... and so on

The OCaml Connection

In OCaml, FSMs are naturally expressed as recursive functions with pattern matching:

(* Pseudo-OCaml FSM *)
type state = Start | Building | Accept | Reject

let rec run_fsm state input =
    match input with
    | [] -> state = Accept  (* End of input *)
    | head :: tail ->
        let next_state = 
            match (state, head) with
            | (Start, 'a') -> Building
            | (Building, 'b') -> Building
            | (Building, 'c') -> Accept
            | _ -> Reject
        in
        if next_state = Reject then false
        else run_fsm next_state tail

PDA: The Stack Machine

A Pushdown Automaton (PDA) is an FSM with a stack. This one addition makes it exponentially more powerful. It can:

  • Everything an FSM can do
  • Push symbols onto a stack
  • Pop symbols from the stack
  • Make decisions based on state + input + top of stack

Why PDAs Matter

PDAs are the theoretical foundation for:

  • Programming language parsers: Matching braces, parsing nested structures
  • XML/HTML validators: Ensuring tags are properly nested
  • Context-free grammar recognizers: Most programming language syntax
  • Calculator evaluation: Handling nested parentheses in expressions

The Killer Feature: Counting and Matching

FSMs can’t count, but PDAs can. They can recognize patterns like:

  • a^n b^n (same number of a’s followed by same number of b’s)
  • Balanced parentheses
  • Palindromes (with a center marker)
  • Nested structures

Pseudo-Code: Balanced Parentheses Checker

FUNCTION check_balanced_parens(input_string):
    stack = Empty Stack
    
    FOR EACH char IN input_string:
        IF char == '(':
            PUSH '(' onto stack
        ELSE IF char == ')':
            IF stack is empty:
                RETURN False  // More closing than opening
            POP from stack
        // Ignore other characters
    
    RETURN stack is empty  // True if all opened were closed

// More complex PDA for multiple bracket types
FUNCTION check_brackets(input_string):
    stack = Empty Stack
    opening = {'(', '[', '{'}
    closing = {')', ']', '}'}
    pairs = {('(', ')'), ('[', ']'), ('{', '}')}
    
    FOR EACH char IN input_string:
        IF char IN opening:
            PUSH char onto stack
        ELSE IF char IN closing:
            IF stack is empty:
                RETURN False
            top = POP from stack
            IF (top, char) NOT IN pairs:
                RETURN False  // Mismatched brackets
    
    RETURN stack is empty

PDA for Context-Free Grammar Parsing

Here’s where PDAs shine. They can parse most programming languages:

// PDA for simple arithmetic expressions
// Grammar: E -> E + T | T
//          T -> T * F | F  
//          F -> (E) | number

STATES = {START, EXPR, TERM, FACTOR, ACCEPT}
STACK_SYMBOLS = {E, T, F, +, *, (, ), NUM, $}

FUNCTION parse_expression(input):
    state = START
    stack = ['$']  // Bottom marker
    
    WHILE input not empty OR stack not just ['$']:
        current_input = peek(input)
        stack_top = peek(stack)
        
        // Predict which production to use
        action = parse_table[stack_top][current_input]
        
        MATCH action:
            CASE PUSH(symbols):
                // Push right-hand side of production
                pop(stack)
                FOR symbol IN reverse(symbols):
                    push(stack, symbol)
                    
            CASE MATCH:
                // Terminal symbol matches input
                pop(stack)
                consume(input)
                
            CASE ERROR:
                RETURN False
    
    RETURN True

The Assembly Implementation

PDAs in Assembly use the CPU’s built-in stack:

; Pseudo-assembly PDA for palindrome with center marker '#'
; Example: "abc#cba" is accepted

check_palindrome:
    ; First phase: push everything before '#'
push_phase:
    lodsb                    ; Load byte from [rsi] to al
    cmp al, '#'
    je match_phase          ; Found center marker
    cmp al, 0
    je reject              ; End without center marker
    push rax               ; Push onto stack
    jmp push_phase
    
    ; Second phase: pop and match everything after '#'
match_phase:
    lodsb                   ; Get next character
    cmp al, 0
    je check_stack_empty   ; End of input
    pop rbx                ; Pop from stack
    cmp al, bl            ; Compare with stack
    jne reject            ; Mismatch
    jmp match_phase
    
check_stack_empty:
    ; Stack should be empty for valid palindrome
    cmp rsp, initial_rsp
    jne reject
    
accept:
    mov rax, 1
    ret
    
reject:
    mov rax, 0
    ret

The OCaml Implementation

OCaml’s lists make perfect stacks for PDAs:

(* Pseudo-OCaml PDA *)
type symbol = A | B | Open | Close | EOF

(* Check if input has form a^n b^n *)
let rec pda_anbn state stack input =
    match (state, stack, input) with
    (* Push A's onto stack *)
    | (Start, stack, A :: rest) -> 
        pda_anbn PushingAs (A :: stack) rest
    | (PushingAs, stack, A :: rest) -> 
        pda_anbn PushingAs (A :: stack) rest
        
    (* Start popping when we see B's *)
    | (PushingAs, stack, B :: rest) -> 
        pda_anbn PoppingAs stack (B :: rest)
    | (PoppingAs, A :: stack_rest, B :: input_rest) -> 
        pda_anbn PoppingAs stack_rest input_rest
        
    (* Accept if stack empty when input empty *)
    | (PoppingAs, [], []) -> true
    | _ -> false

(* More complex: parsing nested structures *)
let rec parse_nested tokens stack =
    match (tokens, stack) with
    | (Open :: rest, stack) -> 
        parse_nested rest (Open :: stack)
    | (Close :: rest, Open :: stack_rest) -> 
        parse_nested rest stack_rest
    | (Close :: _, []) -> 
        false  (* Unmatched closing *)
    | ([], []) -> 
        true   (* Balanced *)
    | ([], _) -> 
        false  (* Unclosed opening *)
    | _ -> 
        parse_nested (List.tl tokens) stack

The Power Hierarchy in Practice

Here’s how to choose which automaton you need:

DECISION TREE:

Do you need to remember counting/nesting?
├── NO: Use FSM
│   ├── Examples: Regex, tokenization, protocols
│   └── Implementation: State variables, switch/match statements
│
└── YES: Do you need unlimited memory or random access?
    ├── NO: Use PDA
    │   ├── Examples: Parsing, bracket matching, simple languages
    │   └── Implementation: Explicit stack + states
    │
    └── YES: Use Turing Machine (real computer)
        ├── Examples: Compilers, interpreters, general computation
        └── Implementation: Just write normal code

Real-World Example: JSON Validator

JSON parsing needs a PDA because of nested objects/arrays:

FUNCTION validate_json(text):
    state = EXPECT_VALUE
    stack = []
    
    FOR token IN tokenize(text):
        MATCH (state, token.type):
            CASE (EXPECT_VALUE, OBJECT_START):
                push(stack, OBJECT)
                state = EXPECT_KEY_OR_END
                
            CASE (EXPECT_VALUE, ARRAY_START):
                push(stack, ARRAY)
                state = EXPECT_VALUE_OR_END
                
            CASE (EXPECT_VALUE, STRING | NUMBER | BOOL | NULL):
                state = check_container_continuation(stack)
                
            CASE (EXPECT_KEY_OR_END, OBJECT_END):
                IF top(stack) != OBJECT: RETURN false
                pop(stack)
                state = check_container_continuation(stack)
                
            // ... many more cases
    
    RETURN stack.is_empty() AND state == DONE

The Computational Hierarchy

The Automata Power Levels

  • FSM: Regular languages. Can’t count. O(1) memory. Implements regex, lexers, protocols.
  • PDA: Context-free languages. Can count with stack. O(n) memory. Implements parsers, validators.
  • Linear Bounded Automaton: Context-sensitive languages. Tape limited to input size. Rarely used.
  • Turing Machine: Recursively enumerable languages. Unlimited memory. Can compute anything computable.

The fascinating part? Most practical computing happens at the FSM and PDA levels. You only need Turing completeness for general-purpose programming. Your regex engine doesn’t need it. Your JSON parser doesn’t need it. They’re purposely limited to be more analyzable and efficient.
