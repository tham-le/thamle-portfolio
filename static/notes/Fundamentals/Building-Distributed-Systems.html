WTF: How Do You Actually Build a Distributed System?

You understand the theory. Now let’s build something.

The previous article explained why distributed systems are hard — partial failure, no shared clock, unreliable networks. This article is about how you actually build one. Not in theory. In code.

We’ll start with a single server, then progressively distribute it: add communication between services, replicate state, handle failures, and coordinate work. Each step introduces a real pattern you’ll use in production.

Step 0: The Single Server

Everything starts here. One process, one machine, one database.

This is your baseline. Everything is in-memory, single-threaded-safe via the mutex, and lives in one process. No network, no replication, no distributed problems.

Now let’s break it apart.

Step 1: Put It on the Network (RPC)

The first step to “distributed” is making your service callable over the network. The standard approach: RPC (Remote Procedure Call).

RPC makes a network call look like a function call:

// What you write:
let result = remote_store.get("user:123");

// What actually happens:
1. Serialize "get" + "user:123" into bytes
2. Send bytes over TCP to the remote server
3. Remote server deserializes, calls local get()
4. Remote server serializes the result
5. Send result bytes back
6. Deserialize into your local type

RPC Is Not a Function Call

The network can fail. The remote server can crash. The call can timeout. Every RPC can return three outcomes: success, failure (server rejected it), or unknown (timeout — did it execute or not?). The “unknown” case is the hardest. If you sent “transfer $100” and got a timeout, did the money move? You need idempotency — making the same call twice produces the same result — to handle this safely.

Step 2: Service Discovery

You have multiple services. Service A needs to call Service B. But where is Service B running? What IP? What port? What if it moved?

Hardcoded addresses don’t scale:

Service discovery solves this. Services register themselves, and clients look them up dynamically.

DNS-Based Discovery

The simplest approach — use DNS with short TTLs:

kvstore.internal.example.com → 10.0.1.5, 10.0.1.6, 10.0.1.7

The client resolves the DNS name and gets multiple IPs. Round-robin or random selection provides basic load balancing. When a server dies, update DNS.

Limitation: DNS caching means clients may not see changes for seconds or minutes.

Service Registry (etcd, Consul, ZooKeeper)

A dedicated coordination service where services register and discover each other:

// Service B starts up and registers itself
etcd.put("/services/kvstore/node1", "10.0.1.5:8080")
etcd.put("/services/kvstore/node2", "10.0.1.6:8080")

// Service A discovers Service B
nodes = etcd.get_prefix("/services/kvstore/")
// → ["10.0.1.5:8080", "10.0.1.6:8080"]

Services attach a lease (TTL) to their registration. If a service crashes without deregistering, the lease expires and it’s automatically removed. Clients can watch for changes and update their connection pools in real time.

Health Checks

Discovery alone isn’t enough. A service might be registered but unhealthy (out of memory, deadlocked, overwhelmed). Health checks verify that registered services are actually working:

Load balancers and service meshes poll this endpoint and route traffic away from unhealthy instances.

Step 3: Replication (Don’t Lose Data)

Your KV store has one copy of the data. The server crashes. The data is gone.

Replication maintains copies on multiple machines. The fundamental question: who accepts writes?

Single-Leader Replication

One node (the leader) accepts all writes. It forwards changes to followers:

Client writes "X=5"
       |
    Leader (accepts write, logs it)
       |
   ┌───┼───┐
   ↓   ↓   ↓
 F1   F2   F3   (followers apply the write)

Reads can go to any replica (for throughput) or only to the leader (for consistency).

When the leader dies: followers detect the failure (no heartbeat) and elect a new leader. This is where consensus algorithms come in.

The Replication Log

The leader doesn’t send raw state — it sends a log of operations. Each follower replays the log to reconstruct the state:

Log:  [SET x=1] [SET y=2] [DEL x] [SET y=5]
                                            ↓
State after replaying log:  { y: 5 }

This is the same idea behind database WAL (Write-Ahead Log), Kafka topics, and event sourcing. The log is the source of truth. The state is derived from the log.

Log-Based Architecture

Many distributed systems are built entirely around a replicated log. Kafka is literally a distributed log. Database replication uses a log. Event sourcing stores every change as a log entry. If you understand the replicated log pattern, you understand the backbone of modern distributed data systems.

Step 4: Consensus (Agree on a Leader)

When the leader dies, the remaining nodes must agree on a new leader. This is the consensus problem.

Raft is the standard algorithm for this. It’s designed to be understandable (unlike Paxos). Here’s the core:

Raft in Practice

Every node is in one of three states: follower, candidate, or leader.

Normal operation:
  Leader sends heartbeats every 150ms
  Followers reset their election timer on each heartbeat

Leader dies:
  A follower's election timer expires (random 150-300ms)
  It becomes a candidate, increments its term, votes for itself
  It asks other nodes for votes
  If it gets a majority → becomes leader
  If someone else wins → becomes follower

The randomized election timeout is the key insight — it prevents two nodes from campaigning simultaneously (most of the time). If they do collide, both elections fail and they retry with new random timeouts.

Why Majority Matters

Raft requires a majority (more than half) for both elections and commits. With 5 nodes, a majority is 3. This means:

  • Up to 2 nodes can crash and the system keeps working
  • There’s always overlap between any two majorities — preventing split-brain (two leaders)
5 nodes, 2 crash:
  Remaining 3 = majority → can elect leader, can commit writes

5 nodes, 3 crash:
  Remaining 2 < majority → system stops (safe: no incorrect results)

Step 5: Sharding (Scale Beyond One Machine)

One machine’s memory and disk are finite. Sharding splits data across multiple machines by key:

Shard 0: keys where hash(key) % 3 == 0
Shard 1: keys where hash(key) % 3 == 1
Shard 2: keys where hash(key) % 3 == 2

Consistent Hashing

Simple modulo sharding breaks when you add or remove a shard (every key’s assignment changes). Consistent hashing maps both keys and nodes onto a ring:

        Node A
       /      \
      /   Ring  \
    Node C --- Node B

Key "user:1" → position 73° → closest node clockwise = Node B
Key "user:2" → position 210° → closest node clockwise = Node C

When Node B dies, only its keys move to the next node on the ring. The other shards are unaffected. When you add a new node, it only takes keys from its neighbors.

The Router Layer

Clients need to know which shard to talk to. Three approaches:

Client-side routing: the client has the shard map and routes directly. Fast, but every client needs the map.

Proxy routing: a stateless proxy sits in front of all shards and routes requests. Clients talk to one endpoint.

Gossip-based: nodes share the shard map with each other and redirect clients. Redis Cluster uses this — any node can redirect you to the right shard with MOVED.

Step 6: Handle Failure Gracefully

In a distributed system, failure is not exceptional — it’s normal. Design for it.

Circuit Breaker

If a downstream service is dead, stop sending it requests. A circuit breaker tracks failures and “opens” (short-circuits) after a threshold:

Closed (normal):     requests flow through
                     failure count tracks errors
                         ↓ (threshold exceeded)
Open (tripped):      all requests fail immediately (don't even try)
                     timer starts
                         ↓ (timer expires)
Half-Open (testing): allow one request through
                     success → Closed / failure → Open

Idempotency

The timeout problem: you sent a request, got no response. Did it execute? You want to retry, but what if it already executed and the response was lost?

Make operations idempotent — executing them twice produces the same result:

The client generates a unique request_id before the first attempt and reuses it on retries. The server deduplicates by ID.

Step 7: Observability (See What’s Happening)

A single-server system is debuggable with println. A distributed system with 10 services and 50 instances? You need structured observability.

The Three Pillars

Logs — what happened, in text:

Metrics — numeric measurements over time:

kvstore_requests_total{method="GET", status="200"} 14523
kvstore_request_duration_seconds{method="GET", quantile="0.99"} 0.045
kvstore_replication_lag_seconds{follower="node2"} 0.003

Distributed tracing — follow a request across services:

Trace ID: abc123
├── [Gateway]      GET /user/123          2ms
│   ├── [AuthService]  validate_token     5ms
│   └── [KVStore]      get("user:123")    3ms
│       └── [Replica]  forward_read       1ms

A trace ID is generated at the entry point and propagated through every service call (via HTTP headers or gRPC metadata). Tools like Jaeger or Zipkin collect and visualize these traces.

Without distributed tracing, debugging “why was this request slow?” across 10 services is nearly impossible.

The Architecture Patterns

These building blocks combine into standard architectures:

Request-Reply (Synchronous)

Client → API Gateway → Service A → Service B → Database
                                  ↘ Service C → Cache

Simple, familiar, easy to reason about. But: every hop adds latency, and a slow downstream service blocks the entire chain.

Event-Driven (Asynchronous)

Service A → [Message Queue] → Service B
                             → Service C
                             → Service D

Services communicate through events on a message queue (Kafka, RabbitMQ, NATS). Service A publishes “user.created” and doesn’t wait. Services B, C, D consume the event independently.

Advantages: services are decoupled, failures are isolated (the queue buffers events), easy to add new consumers.

Trade-off: eventual consistency. Service B might not see the event for seconds. Debugging is harder — there’s no single request to trace.

CQRS (Command Query Responsibility Segregation)

Separate the write path from the read path:

Writes → Command Service → Primary DB → Replication → Read Replicas
Reads  → Query Service → Read Replicas (optimized for queries)

The write model and read model can have different schemas, different databases, different scaling strategies. Write-heavy and read-heavy workloads don’t compete for the same resources.

Summary

Building a distributed system is a progression: put your service on the network (RPC/gRPC), add service discovery (DNS or etcd), replicate state for fault tolerance (leader-follower with a write-ahead log), implement consensus for leader election (Raft), shard data for scale (consistent hashing), handle failures gracefully (timeouts, retries, circuit breakers, idempotency), and add observability (logs, metrics, distributed tracing). Start by building a replicated key-value store — the simplest system that exercises all these patterns. Then read “Designing Data-Intensive Applications” and implement Raft from the paper.
