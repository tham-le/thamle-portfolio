WTF is Zero Copy?

Stop shuffling data like a bureaucrat passing forms around

The Filing Cabinet Analogy

Imagine you need to move a document from a filing cabinet in room A to a desk in room B. The normal way? You walk to room A, photocopy the document, carry the copy to your office, photocopy it again, then carry that to room B. Four trips, two photocopies, and your legs are tired.

Zero copy says: just slide the damn filing cabinet from room A to room B. Or better yet, give room B a key to the filing cabinet.

That’s what happens when your computer moves data around normally. It copies data from disk to kernel memory, from kernel memory to your program’s memory, from your program’s memory back to kernel memory, and finally from kernel memory to the network card. Every copy burns CPU time and memory bandwidth.

Zero copy techniques let the kernel move data directly from source to destination, skipping all those pointless photocopies.

The Problem: Death By a Thousand Copies

Let’s say you’re building a web server that needs to serve a file. Here’s what happens the traditional way with read() and write():

Looks simple, right? But here’s what’s actually happening under the hood:

  1. DMA copies file from disk to kernel buffer (kernel page cache)
  2. CPU copies from kernel buffer to your userspace buffer (the buffer variable)
  3. CPU copies from userspace buffer back to kernel socket buffer (for the write() call)
  4. DMA copies from kernel socket buffer to network card

That’s four data transfers and four context switches (two for read(), two for write()). Your CPU is doing busy work, copying data it doesn’t even care about. All you wanted was to shove the file through the socket, but the kernel made you an unwilling middleman.

For small files? Who cares. For a busy web server pushing gigabytes per second? This becomes your bottleneck.

Enter sendfile(): The OG Zero Copy

Linux kernel 2.1 introduced sendfile() specifically to solve this problem. Instead of forcing data through userspace, it tells the kernel: “Move this file to this socket. I don’t need to touch it.”

Here’s the magic that happens now:

  1. DMA copies file from disk to kernel buffer
  2. Kernel copies file descriptor info to socket buffer (just metadata, not the actual data)
  3. DMA gathers data from kernel buffer and sends to network card

We cut it down to two context switches (enter kernel, leave kernel) and eliminated the CPU copies entirely. The CPU just sets up the transfer and walks away.

With modern network cards that support scatter/gather DMA, step 2 becomes even cheaper. The network card can pull data directly from multiple non-contiguous memory locations, so the kernel doesn’t even need to assemble a contiguous buffer.

sendfile() Limitations

sendfile() only works for file to socket transfers. The source must support mmap() operations (so, actual files, not pipes or sockets) and the destination must be a socket. Need something more flexible? Keep reading.

splice(): The Swiss Army Knife

sendfile() is great, but what if you need more flexibility? What if you want to move data between two sockets? Or from a socket to a file? Enter splice(), the primitive that actually powers sendfile() under the hood.

splice() moves data through a pipe, which acts as a kernel buffer. You splice data into the pipe, then splice it out to the destination. No userspace copies.

The flags deserve explanation:

  • SPLICE_F_MOVE: Hint to the kernel to move pages instead of copying (though the kernel may still copy if it needs to)
  • SPLICE_F_MORE: Tell the kernel more data is coming, so it can batch operations

The splice() Gotcha

The pipe has a limited buffer size (typically 64KB by default). If you try to splice more than the pipe can hold in one shot, you’ll get partial transfers. You need to loop and handle this correctly, checking both splice calls.

mmap(): Mapping Your Way to Zero Copy

mmap() takes a completely different approach. Instead of moving data, it maps a file directly into your process’s address space. The file becomes “memory” that you can read and write with normal pointer operations.

The brilliance here is demand paging. The file isn’t actually read into RAM when you call mmap(). The kernel just sets up the mapping. When you first access data[100], a page fault occurs, the kernel loads that 4KB page from disk into memory, and your access continues. Future accesses to that page are just memory reads—no system calls.

Changes you make to a MAP_SHARED mapping eventually get written back to the file, but asynchronously. You can force it with msync():

mmap() for Inter-Process Communication

Two processes can mmap() the same file with MAP_SHARED, giving them both direct access to the same physical memory pages. No pipes, no sockets, no copying. Process A writes, process B reads, all through shared memory.

You still need synchronization (like semaphores or lock files) to avoid race conditions, but you’ve eliminated all data copying.

mmap() Trade-offs

mmap() isn’t free. Setting up the mapping is expensive, involving page table manipulations and TLB flushes. For small files or single accesses, traditional read() can be faster. Use mmap() when you have large files with repeated or random access patterns.

MSG_ZEROCOPY: Zero Copy for Send Operations

All the techniques above are about moving existing data. But what if you’re generating data in your program that needs to go out on the network? You’re stuck copying it to the kernel, right?

Not anymore. Linux 4.14 added MSG_ZEROCOPY for send(), allowing the kernel to directly DMA your userspace buffers to the network card.

Here’s the catch: you can’t modify the buffer until the kernel is done with it. The kernel pins your memory pages, sends them, and then notifies you via the error queue when it’s safe to reuse the buffer.

The error queue can coalesce multiple notifications. If you send three buffers, you might get one notification covering all three.

When MSG_ZEROCOPY Hurts Performance

The setup overhead (pinning pages, managing notifications) is significant. MSG_ZEROCOPY only wins for sends larger than ~10KB. For small messages, regular send() is faster. The kernel may also fall back to copying if your buffer isn’t DMA-able or if you hit resource limits (locked page limits, socket buffer limits).

When to Actually Use Zero Copy

Zero copy is not a magic performance bullet. Here’s when each technique makes sense:

Use sendfile() when:

  • You’re serving static files over the network (web servers, file servers)
  • The file is already in the page cache (hot data)
  • You don’t need to modify the data in transit

Use splice() when:

  • You need to move data between arbitrary file descriptors
  • You’re proxying data (socket to socket, socket to file)
  • You want the most flexible zero copy primitive

Use mmap() when:

  • You have large files with random or repeated access patterns
  • Multiple processes need to share data
  • You want the file to “feel like” memory

Use MSG_ZEROCOPY when:

  • You’re sending large buffers (>10KB) you generate yourself
  • You can tolerate the complexity of completion notifications
  • You’re hitting CPU bottlenecks on send

Don’t use zero copy when:

  • You’re dealing with small transfers (<4KB)
  • Your data needs processing or transformation in userspace
  • The setup overhead exceeds the copy cost

The Reality Check

Here’s what nobody tells you: on modern systems with gigantic caches, a simple read() + write() loop with an 8KB buffer often keeps everything in L1 cache. The “copy” is just moving cache lines around, which is blazingly fast.

Zero copy shines when:

  • You’re moving massive amounts of data
  • Your CPU is the bottleneck, not your I/O
  • You’re serving hot data from the page cache (so no disk seeks)

For a typical web app serving small API responses? Regular I/O is fine. For a CDN pushing 40 Gbps of video? Zero copy is the difference between 100% CPU usage and 30% CPU usage.

What We Skipped

There are more zero copy techniques we didn’t cover:

  • copy_file_range(): Copies between files entirely in kernel space
  • vmsplice(): Splices userspace memory into a pipe (tricky, lots of gotchas)
  • tee(): Duplicates pipe data without copying
  • AF_XDP: True zero copy for raw packet processing

Each has its niche, but they’re more specialized.

The Big Picture

Zero copy is about reducing unnecessary work. Every copy burns CPU cycles and memory bandwidth. Every context switch flushes caches and TLBs. For high-throughput systems, these costs add up fast.

But zero copy comes with complexity. You lose the simplicity of “read into buffer, do stuff, write buffer.” You gain performance but sacrifice flexibility and sometimes correctness (hello, MSG_ZEROCOPY buffer lifecycle management).

The lesson? Profile first, optimize second. If you’re not CPU-bound or memory-bandwidth-bound, traditional I/O is simpler and often fast enough. When you do hit those limits, zero copy is your escape hatch.
