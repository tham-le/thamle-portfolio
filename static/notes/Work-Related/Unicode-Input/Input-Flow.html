Issue 1: IME/VK Character Flow on Android

Your diagram shows: “IME/VK sent one Character”

Reality: The flow is more complex:

User taps '中' on virtual keyboard
         ↓
Android InputMethodService
         ↓
InputConnection.commitText("中")
         ↓
Your Client App receives the character
         ↓
JNI bridge to Rust
         ↓
Create UnicodeInput packet

Why: The IME (Input Method Editor) doesn’t directly send to your app. It goes through Android’s InputConnection API.

Issue 2: Missing Packet Creation Layer

Between the GUI and kymux, you need:

Character received from OS
         ↓
Client Input Handler (Rust)
         ↓
Create InputPacket {
    target: Host,
    payload: Unicode(UnicodeInput {
        character: '中',
        modifiers: 0x00
    })
}
         ↓
Serialize to bytes [10, 1, 0x2D, 0x4E, 0x00, 0x00, 0x00]
         ↓
Send via kymux

Issue 3: Missing Deserialization on Host Side

After kymux receives raw bytes, before InputPacket:

Raw bytes from network
         ↓
Network receiver thread
         ↓
Deserialize bytes → InputPacket
         ↓
Route to Host Handler
         ↓
Extract UnicodeInput from payload
         ↓
Platform Injector

Issue 4: Platform Injector Details Missing

The “Unicode Input” box should show the platform-specific conversion:

UnicodeInput { character: '中', modifiers: 0x00 }
         ↓
Windows Injector
         ↓
Convert UTF-32 (0x4E2D) → UTF-16 (0x4E2D)
         ↓
SendInput with KEYEVENTF_UNICODE flag
         ↓
Windows Kernel receives INPUT structure
         ↓
Character appears in focused application

Detailed Explanation: How the JNI Code Works

This is a JNI (Java Native Interface) bridge that connects Android Java/Kotlin code with the native Rust/C libkynput library. Let me break down how it works.

High-Level Architecture

┌─────────────────────────────────────────────────────────────┐
│              Android Application (Java/Kotlin)               │
│  - UI components (buttons, virtual keyboard, touch events)  │
│  - Activity/View lifecycle                                   │
└──────────────────────┬──────────────────────────────────────┘
                       │ JNI calls
                       │ (nativeOnMousePosition, nativeOnKeyboardKey, etc.)
                       ▼
┌─────────────────────────────────────────────────────────────┐
│              JNI Bridge (libclientjni-jni.c)                │
│  - Converts Java types to C types                           │
│  - Manages memory across Java/C boundary                    │
│  - Handles thread attachment                                │
└──────────────────────┬──────────────────────────────────────┘
                       │ C API calls
                       │ (libkynput_keyboard_key_to_packet, etc.)
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                 libkynput (Rust/C Library)                   │
│  - Creates InputPacket structures                           │
│  - Serializes packets                                        │
│  - Sends via network (kymux)                                 │
└─────────────────────────────────────────────────────────────┘

Key Data Structures

The Core Client Structure

This structure is the central state that connects Java and native code. It’s allocated when the Java app starts and stored as a long field in the Java object.

The Input Handler

This handles all input from Android (keyboard, mouse, gamepad) and forwards it to libkynput.

Initialization Flow

1. JNI_OnLoad: Library Initialization

What happens:

  1. Android loads the shared library (.so file)
  2. JNI_OnLoad is called automatically
  3. It caches Java class and method references for fast callbacks
  4. It stores the JavaVM pointer for thread attachment later

Why cache class/method IDs?

  • Looking them up every time is expensive
  • We’ll use these IDs to call Java methods from native code (callbacks)

2. nativeNew: Client Creation

What happens:

  1. Java calls LibVLC.nativeNew() from constructor
  2. Native code allocates kyber_client structure
  3. Stores a pointer to this structure in Java object’s mInstance field
  4. This pointer is used in all subsequent JNI calls

3. nativeStart: Connecting to Host

What happens:

  1. Java calls this with connection parameters (IP, port, certificates)
  2. Native code converts Java types to C types
  3. Creates libclient connection
  4. Registers callback for events from the host
  5. Initiates connection

Important: The actual libkynput pipeline isn’t created yet! It’s created later when the control plane connects (in libclient_event_handler).

Input Flow: How Keyboard Input Works

Let’s trace what happens when a user types a key on Android:

Step 1: Android Detects Key Press

Step 2: JNI Receives the Call

Key transformations:

Android KeyCode (int) → Scancode Set 1 (uint16_t) → InputPacket

Example:

AKEYCODE_A (29) → Scancode 0x1E → KeyboardKey { scancode: 0x1E, pressed: true }

Step 3: Send Packet to Pipeline

This hands the packet off to libkynput (Rust), which:

  1. Serializes it to bytes
  2. Sends it over the network via kymux
  3. Delivers it to the Windows host

Event Callback Flow (Host → Android)

The flow also works in reverse. When the host sends something (like cursor data), it comes back through JNI.

The Event Handler

Critical concept: Thread Attachment

This callback runs on a native thread created by libclient (not the main Android thread). To call Java methods from this thread, we must:

  1. Attach the thread to the JVM:
  1. Get a JNIEnv for this thread
  2. Call Java methods using the env
  3. Detach when done (or let it detach automatically)

Without attachment, calling Java methods would crash.

Cursor Consumer: Receiving Cursor Data

What’s happening:

  1. Host sends cursor image over network
  2. libkynput receives it, deserializes to Cursor packet
  3. Cursor consumer is notified
  4. Native code:
    • Extracts pixel data
    • Converts color format (BGRA → RGBA)
    • Creates Java ByteBuffer pointing to native memory (zero-copy!)
    • Calls Java callback method onCursorReceived()
  5. Java code displays the cursor image

Mouse Position: Coordinate Transformation

This is interesting because it shows coordinate space conversion:

Why transformation is needed:

Android Screen: 1080x2400 (portrait, touch)
  ┌──────────┐
  │          │
  │ [Video]  │  ← Video is 1920x1080 (landscape), letterboxed
  │          │
  └──────────┘

Windows Desktop: 1920x1080
  ┌─────────────────┐
  │                 │
  └─────────────────┘

User taps at Android (540, 800):

  1. Without transformation: Would send (540, 800) to Windows → wrong!
  2. With transformation: Converts to Windows (960, 540) → correct!

The videoLayout object tracks:

  • Host resolution (1920x1080)

Complete Input Flow Example

Let’s trace a complete keypress from Android to Windows:

1. User presses 'A' key on Android virtual keyboard
   ↓
2. Android OS: onKeyDown(AKEYCODE_A, event)
   ↓
3. Java: libVLC.nativeOnKeyboardKey(0, AKEYCODE_A, true)
   ↓
4. JNI: Java_com_kyber_libclient_LibVLC_nativeOnKeyboardKey()
   - Get client instance from Java object
   - Convert AKEYCODE_A → scancode 0x1E
   ↓
5. JNI: libkynput_keyboard_key_to_packet(HOST, 0x1E, true)
   - Creates KeyboardKey { scancode: 0x1E, pressed: true }
   - Wraps in InputPacket { target: Host, payload: Keyboard(...) }
   ↓
6. JNI: handler_send_pkt()
   - Calls libkynput_consumer_consume()
   ↓
7. [Enters Rust land - libkynput]
   - Router processes packet
   - Applies filters
   - Serializes: [0, 1, 0x1E, 0x00, 0x01]
   ↓
8. [Network layer - kymux/QUIC]
   - Encrypts packet
   - Sends over internet
   ↓
9. [Windows Host receives]
   - Deserializes packet
   - Calls Windows injector
   - SendInput() with scancode 0x1E
   ↓
10. Windows: 'A' character appears in Notepad

Summary

This JNI bridge:

  1. Connects Java and Native: Allows Android app to use Rust/C library
  2. Manages Lifecycle: Creates/destroys client, starts/stops streaming
  3. Forwards Input: Converts Android events → libkynput packets
  4. Receives Callbacks: Gets cursor data from host, converts to Android format
  5. Handles Threads: Attaches native threads to JVM for callbacks
  6. Transforms Coordinates: Maps Android touch → Windows desktop coordinates
  7. Manages Memory: Carefully tracks Java references and native allocations

The code you saw is essentially a translation layer that makes a Rust library usable from Android Java/Kotlin, handling all the messy details of crossing the language boundary.
