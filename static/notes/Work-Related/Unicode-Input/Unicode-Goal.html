Goals

  • Virtual keyboards (mobile, VR, Android TV) can send text input to the host
  • IME (Input Method Editor) composition for CJK languages works correctly
  • Emoji and special characters can be typed from any client platform
  • Characters beyond the Basic Multilingual Plane (U+10000+) are supported
  • Optional modifier keys (Ctrl, Alt, Shift, Meta) can be combined with Unicode input

Problem Statement

On some platforms, keyboards do not generate traditional key press/release events:

  • Mobile: iOS and Android virtual keyboards emit characters directly
  • VR Headsets: Quest, Vision Pro use spatial input methods
  • Android TV: Remote-controlled on-screen keyboards
  • IME Systems: Chinese, Japanese, Korean input methods produce composed characters

Traditional scancode-based input (KeyboardKey) cannot represent:

  • Emoji (ğŸ˜€, ğŸ‰, â¤ï¸)
  • CJK characters (ä¸­, æ—¥, í•œ)
  • Mathematical symbols (âˆ‘, âˆ«, âˆš)
  • Characters requiring multiple key events to compose

Implementation Status

Protocol (Core)

  • New UnicodeInput type in libkynput/src/types/unicode.rs
  • Add InputType::Unicode enum variant

Host (Input Injection)

    • Requires UTF-16 encoding with surrogate pair handling
  • Linux: FFI using X11 Ctrl+Shift+U input method
    • Converts UTF-32 to hexadecimal sequence
  • macOS: FFI using CGEventKeyboardSetUnicodeString
    • Requires UTF-16 encoding with surrogate pair handling

Client (Input Capture)

  • Android: JNI bridge for InputConnection.commitText()
    • Requires UTF-16 to UTF-32 conversion (surrogate pair decoding)
  • iOS: Objective-C bridge for UITextInput protocol
    • Requires UTF-16 to UTF-32 conversion
  • Web: JavaScript bridge for input event
    • Requires UTF-16 to UTF-32 conversion via codePointAt()

Platform Testing

  • Windows: ASCII, emoji, CJK, surrogate pairs
  • Linux: Compatibility with GTK/Qt applications
  • macOS: Native text input validation
  • Android: Virtual keyboard integration
  • iOS: QuickPath and dictation support
  • Web: Cross-browser compatibility

Edge Cases

Surrogate Pairs

Characters: Emoji (U+1F600+), rare CJK, mathematical symbols Handling: Automatic in UTF-32, explicit encoding required for UTF-16

Invalid Codepoints

Range: 0xD800-0xDFFF (surrogates), > 0x10FFFF Action: Reject during deserialization with error

Configuration Options

Security

  • Allow/deny clipboard to Unicode conversion
  • Whitelist/blacklist specific codepoint ranges
  • Rate limiting for Unicode packets

Compatibility

  • Fallback to scancode for ASCII on platforms without Unicode support
  • Optional IME composition event support
  • Configurable encoding for X11 (Ctrl+Shift+U vs other methods)

Performance Considerations

Bandwidth:

  • 7 bytes per character (2 header + 4 UTF-32 + 1 modifiers)
  • Typical session: ~1000 characters/hour = ~7 KB/hour
  • Impact: Negligible compared to video stream

Latency:

  • Windows: ~1ms (native SendInput)
  • macOS: ~1ms (native Core Graphics)
  • Linux: ~50-100ms (Ctrl+Shift+U method)
  • Network: Existing packet latency (same as keyboard)
