Analysis of Current libkynput Architecture

What This Code Tells Us

Looking at this InputPacket structure, I can see several important architectural decisions:

1. Client-Host Model (Not Client-Server)

This is a remote desktop/game streaming architecture, not a traditional client-server model. Think Parsec, Steam Remote Play, or Moonlight:

  • Host: The computer being controlled (has the mouse/keyboard physically)
  • Client: The device doing the controlling (iPad, phone, another PC)

Input flow:

Client (iPad) → Captures touch/keyboard → Sends packets → Host (PC) → Injects input

2. Custom Serialization Protocol

The protocol uses:

  • Type byte: Identifies what kind of input (keyboard, mouse, etc.)
  • Target: Which side should process it
  • Payload bytes: Custom serialized data for each input type

This means you’re NOT using serde/bincode automatically. Each input type implements its own serialize()/deserialize().

3. Existing Input Types

Key observation: You already have Clipboard(ClipboardEvent) for transferring text. This suggests the architecture can handle arbitrary text, but clipboard is different from keyboard input.

4. Type System with Enum Mapping

There’s an InputType enum (not shown in this code) that maps to u8 values for the wire protocol. You’ll need to add a new type.

Wire Protocol Format

Understanding the network format:

┌─────────────────────────────────────────────────────────┐
│                    InputPacket Wire Format               │
├─────────────────────────────────────────────────────────┤
│ Byte 0: Type (u8)         │ 10 = Unicode                │
│ Byte 1: Target (u8)       │ 0 = Client, 1 = Host        │
│ Bytes 2-5: Codepoint      │ UTF-32, little-endian       │
│ Byte 6: Modifiers         │ Bit flags                   │
└─────────────────────────────────────────────────────────┘

Example: Sending ’😀’ to Host:

[10, 1, 0x00, 0xF6, 0x01, 0x00, 0x00]
 │   │   └──────┬──────────┘   └──┘
 │   │          │                │
 │   │    0x1F600 (emoji)    No modifiers
 │   │
 │   InputTarget::Host
 InputType::Unicode

Key Architectural Insights

  1. Separation of Concerns: Protocol (wire format) is separate from execution (platform injection)

  2. Target-Based Routing: The InputTarget field determines which side processes the packet, allowing bidirectional communication

  3. Custom Serialization: Each payload type owns its serialization logic, giving fine control over wire format

  4. Type Safety: The InputType enum ensures packets are deserialized with the correct parser

  5. Platform Abstraction: The InputInjector trait hides platform differences from the packet handler
