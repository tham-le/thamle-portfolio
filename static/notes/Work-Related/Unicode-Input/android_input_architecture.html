WTF is Android Input Architecture?

Your app doesn’t actually touch that keyboard

The Restaurant Kitchen Analogy

Imagine you’re running a restaurant kitchen. When customers want food, they don’t burst through the door and tell you directly. Instead, they tell a waiter, who writes down the order and hands it to you through a service window. You never deal with customers yourself; you just get these formatted order slips.

That’s exactly how Android keyboard input works. Your app never directly sees someone tapping keys on the virtual keyboard. The Input Method Editor (IME), which is the keyboard app, talks to the Android system. The system then delivers formatted text events to your app through specific interfaces.

You’re not catching keystrokes. You’re catching the result of someone typing.

Why This Crazy Indirection?

You might think: “Why can’t my app just detect touches on the keyboard?”

Three excellent reasons:

  1. Security: If any app could see raw keystrokes, password managers would be useless. Your banking password would be visible to every app on your phone.
  2. Flexibility: Gboard, SwiftKey, voice input, handwriting, emoji pickers—all need to work seamlessly with every app. Your app doesn’t need to know about 50 different keyboard implementations.
  3. Intelligence: The keyboard does smart stuff like autocorrect, swipe typing, and predictive text before sending the final result to your app. You get clean, processed text, not raw gesture data.

The Two Paths: Simple vs. Custom

When capturing keyboard input, you have two approaches. Let’s start with the simple one.

Path 1: The Easy Way (EditText)

EditText is Android’s built-in text input view that handles all the complexity for you. When you drop an EditText into your app, you get a complete text editing system out of the box.

Here’s what happens when the user types:

User taps "H" on keyboard
    ↓
Keyboard (IME) processes the tap
    ↓
Android system receives "H" from keyboard  
    ↓
System calls InputConnection methods
    ↓
EditText receives text via InputConnection
    ↓
EditText updates its display
    ↓
Your TextWatcher callbacks fire

The timing matters:

  • beforeTextChanged(): Text is “abc”, user types “d”, you see “abc”
  • onTextChanged(): Text is now “abcd”, you see “abcd”
  • afterTextChanged(): Text is “abcd”, and you can modify it

The Recursive Trap

If you modify text in afterTextChanged(), it triggers the entire cycle again. This can create infinite loops if you’re not careful. Always check if modification is needed before making changes.

Path 2: The Hard Way (Custom Input)

Sometimes you need a custom view that receives text input. This requires implementing InputConnection, the interface between the IME and your editor.

Think of InputConnection as the waiter’s notepad. The IME (keyboard) writes orders on it, and your view reads those orders.

Here’s the architecture:

┌─────────────────┐
│   Keyboard App  │ (Gboard, SwiftKey, etc.)
│      (IME)      │
└────────┬────────┘
         │ commitText("H")
         │ deleteSurroundingText(1, 0)
         │ setComposingText("typ")
         ↓
┌─────────────────┐
│ InputConnection │ Interface between IME and app
└────────┬────────┘
         │ Methods like:
         │ - commitText()
         │ - deleteSurroundingText()  
         │ - getTextBeforeCursor()
         ↓
┌─────────────────┐
│   Your View     │ (EditText or custom)
│  (Editor)       │
└─────────────────┘

What Happens During Typing?

Let’s trace a single letter through the system. User taps “H”:

Step 1: User’s finger touches the keyboard at coordinates matching the “H” key.

Step 2: The keyboard app (IME) processes this touch event and decides it’s the letter H.

Step 3: The IME calls inputConnection.commitText("H", 1) on the active InputConnection.

Step 4: The InputConnection implementation (either from EditText or your custom code) receives this call and updates the text.

Step 5: If using EditText, any registered TextWatchers get notified via their three callbacks.

Step 6: The UI updates to show the new character.

All of this happens in milliseconds, giving the illusion of direct keyboard input.

The Composing Text Mystery

Ever notice how text has an underline while you’re swiping on the keyboard? That’s composing text—temporary text that the IME might change.

The IME uses setComposingText("typ", 1) instead of commitText(). This tells your app: “Show this, but I might change my mind.”

User swipes across "typing"
↓
IME: setComposingText("t", 1)    → Shows "t" with underline
IME: setComposingText("ty", 1)   → Shows "ty" with underline  
IME: setComposingText("typ", 1)  → Shows "typ" with underline
User lifts finger
↓
IME: commitText("typing", 1)     → Finalizes as "typing"

Composing spans are marked with the SPAN_COMPOSING flag to help users track what part of the text the IME is currently focusing on.

This is why you can swipe “helo” and it suggests “hello.” The IME keeps the text marked as composing until you accept or reject the suggestion.

What About Raw KeyEvents?

You can send raw KeyEvents through InputConnection using sendKeyEvent(), but this is discouraged for normal text input. It’s mainly for special keys like arrow keys or hardware keyboard events.

The IME typically uses high-level methods like commitText() instead of simulating individual key presses. This is more efficient and handles complex input like emoji and autocorrect properly.

The Big Picture

Here’s the complete flow from keyboard tap to your app:

┌──────────────┐
│  User taps   │
│  "H" key     │
└──────┬───────┘
       │
       ↓
┌──────────────────┐
│ Keyboard App     │ Touch event → determine "H"
│ (IME)            │ 
└──────┬───────────┘
       │ Calls InputConnection methods
       │ commitText("H", 1)
       ↓
┌──────────────────┐
│ Android System   │ Routes call to focused view
└──────┬───────────┘
       │
       ↓
┌──────────────────┐
│ InputConnection  │ Interface implementation
│ (from EditText   │ Updates internal text buffer
│  or custom view) │
└──────┬───────────┘
       │
       ↓
┌──────────────────┐
│ EditText or      │ Displays new text
│ Custom View      │ 
└──────┬───────────┘
       │
       ↓
┌──────────────────┐
│ TextWatcher      │ Your app's callbacks fire
│ Callbacks        │ beforeTextChanged()
│                  │ onTextChanged()  
│                  │ afterTextChanged()
└──────────────────┘

The system is designed so your app doesn’t care which keyboard the user has installed. Whether they’re swiping on Gboard, using voice input, or typing on a hardware keyboard, your app sees the same InputConnection interface.
