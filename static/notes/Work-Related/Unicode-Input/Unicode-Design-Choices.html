Architecture Analysis: Unicode Input Design Choices

Core Architectural Decisions

1. Why UnicodeInput is a Separate Type (Not Part of KeyboardKey)

Reasoning:

Semantic Clarity: These represent fundamentally different input models:

  • KeyboardKey: ‚ÄúA physical key was pressed/released‚Äù (hardware event)
  • UnicodeInput: ‚ÄúInsert this character‚Äù (logical event)

Type Safety: Prevents invalid states at compile time:

Wire Protocol Efficiency: Different serialization needs:

  • KeyboardKey: 3 bytes (2 bytes scancode + 1 byte pressed)
  • UnicodeInput: 5 bytes (4 bytes UTF-32 + 1 byte modifiers)

Mixing them would waste bandwidth:

2. Why Modifiers are u8 Bit Flags (Not a Struct)

Reasoning:

Minimal Wire Format: 1 byte vs 4 bytes (or more with alignment):

Struct approach: [ctrl: 1 byte][alt: 1 byte][shift: 1 byte][meta: 1 byte] = 4 bytes
Bitflag approach: [0x05] = 1 byte (Ctrl + Shift)

Follows Existing Pattern: Look at the modifiers.rs file you provided:

This is for tracking state, not for wire protocol. For network transmission, bit flags are standard.

Platform Compatibility: OS APIs use bit flags:

3. Why UTF-32 on the Wire (Not UTF-8 or UTF-16)

Reasoning:

Rust‚Äôs Native Representation: char is UTF-32 (Unicode scalar value):

Fixed Size = Simpler Parsing: Always 4 bytes, no length calculation needed:

No Ambiguity with Surrogates: UTF-16 has surrogate pairs that complicate validation:

Space Trade-off is Acceptable:

  • Each character: 4 bytes (UTF-32) vs 1-4 bytes (UTF-8)
  • For ASCII: 3 bytes wasted per character
  • But Unicode packets are rare compared to mouse/keyboard events
  • Total bandwidth impact is negligible in real usage

Reasoning:

Different Semantic Models:

  • Characters are instantaneous: insert and done

6. Why Separate from Clipboard

You noted that Clipboard already exists. Why add Unicode?

Reasoning:

Different User Intent:

  • Clipboard: ‚ÄúTransfer this text to/from clipboard‚Äù (persistent storage)
  • Unicode: ‚ÄúType this character now‚Äù (immediate input)

Different Platform Behavior:

Different Timing:

  • Clipboard: Asynchronous, survives after app closes
  • Unicode: Synchronous, affects focused window immediately

Different Size Constraints:

  • Clipboard: Can hold megabytes of text
  • Unicode: Single character (4 bytes max)

Overall Architecture Philosophy

The design follows libkynput‚Äôs principles:

1. One Type = One Concept

Each payload type represents exactly one kind of input event. No polymorphism, no optional fields.

2. Explicit is Better Than Implicit

3. Wire Protocol First

Types are designed around efficient serialization. The struct layout maps directly to bytes on the wire.

4. Platform Abstraction at the Edges

The core protocol is platform-agnostic (UTF-32). Platform-specific encoding (UTF-16, UTF-8) happens in the injector layer.

5. Zero Runtime Overhead

No allocations, no vtables, no boxing. Just plain data.

6. Composition Over Inheritance

Instead of a base InputEvent class with subclasses, we have:

  • Trait for serialization (InputEvent)
  • Enum for variants (Payload)
  • Separate structs for each concept

This is Rust‚Äôs idiomatic approach and matches libkynput‚Äôs existing design.
