Wire Protocol

Unicode packets use 7 bytes on the wire:

[Type: 1 byte][Target: 1 byte][Codepoint: 4 bytes LE][Modifiers: 1 byte]

Example for ‘中’ with Ctrl:

[0x0A][0x01][0x2D][0x4E][0x00][0x00][0x01]
  │     │      └────┬────┘       └──┘ └──┘
  │     │           │             │    Modifiers (Ctrl)
  │     │      UTF-32 0x4E2D     │
  │     Target: Host              │
  Type: Unicode                   │

# Changelog

## [Unreleased]

### Added
- Unicode input support for virtual keyboards and IMEs
- New `UnicodeInput` packet type with UTF-32 encoding
- Platform injectors for Windows, Linux, and macOS
- JNI bridge function `nativeOnUnicodeInput` for Android
- Support for emoji and supplementary plane characters (U+10000 to U+10FFFF)
- Modifier key support for Unicode input (Ctrl, Alt, Shift, Meta)

### Technical Details
- Wire protocol uses UTF-32 (4 bytes per character)
- Windows: Uses SendInput with KEYEVENTF_UNICODE flag
- Linux: Uses Ctrl+Shift+U input method for GTK/Qt compatibility
- macOS: Uses CGEventKeyboardSetUnicodeString API
- Proper surrogate pair handling for emoji and rare characters

Priority Order for Implementation

If you want to implement incrementally, follow this order:

Phase 1: Core Protocol (Required)

  1. src/types/unicode.rs
  2. src/types/mod.rs
  3. src/packet.rs
  4. src/lib.rs
  5. Tests: tests/unicode_integration.rs

Phase 2: C API (For JNI)

  1. src/capi/unicode.rs
  2. include/libkynput.h
  3. Tests: tests/capi_unicode.rs

Phase 3: Host Platform (Pick one first)

Windows:

  1. src/platform/windows/ffi.c
  2. src/platform/windows/injector.rs
  3. build.rs (Windows section)

OR Linux:

  1. src/platform/linux/ffi.c
  2. src/platform/linux/injector.rs
  3. build.rs (Linux section)

OR macOS:

  1. src/platform/macos/ffi.c
  2. src/platform/macos/injector.rs
  3. build.rs (macOS section)

Phase 4: Host Handler

  1. src/platform/mod.rs (InputInjector trait)
  2. src/host/handler.rs

Phase 5: Android Client

  1. libclient-jni/libclientjni-jni.c (nativeOnUnicodeInput)
  2. com/kyber/libclient/LibVLC.java
  3. com/kyber/libclient/InputView.java
  4. MainActivity.java
  5. res/layout/activity_main.xml

Phase 6: Documentation & Polish

  1. README.md
  2. CHANGELOG.md
  3. Additional tests
  4. Platform-specific edge cases

This is the complete list! Each file/function is necessary for the full Unicode support feature. Start with Phase 1 (core protocol) and work your way down. Each phase builds on the previous one, so you’ll have a working implementation at each step.
