Designing a Flexible Protocol

Now, for your real question: “The host can be customized too, what is the best way to do it?”

You need a protocol that can handle different remote machine configurations:

  • Windows (expects UTF-16 or scan codes)
  • Linux (expects UTF-8 or scan codes)
  • Different keyboard layouts
  • Different input methods

The Bottom Line

For your remote desktop app:

  1. Android side: Always UTF-8 for files, logs, configs
  2. Internal: Don’t worry about UTF-16, Kotlin handles it
  3. Protocol: Negotiate with remote host at connection
  4. Flexible: Support UTF-8 Unicode, UTF-16 Unicode, and scan codes
  5. Configurable: Let users choose per-host settings

The UTF-16 thing is a red herring. Just use UTF-8 everywhere in your code, and let Kotlin’s String handle the internal representation. When you need to send to a Windows host that wants UTF-16, convert explicitly at that moment.

UTF-8 everywhere still applies. UTF-16 is just Android’s internal memory format, not something you expose externally.
