Understanding Unicode: A Complete Explanation

What is Unicode?

Unicode is a universal character encoding standard that assigns a unique number (called a ‚Äúcode point‚Äù) to every character in every writing system in the world, plus symbols, emoji, and control characters.

The Problem Unicode Solves

Before Unicode, different computer systems used different encoding schemes:

ASCII (American):
'A' = 65

Latin-1 (Western Europe):
'A' = 65
'√©' = 233

Shift-JIS (Japanese):
'A' = 65
'„ÅÇ' = 0x82A0 (two bytes)

Big5 (Chinese):
'A' = 65
'‰∏≠' = 0xA4A4 (two bytes)

The chaos: The same byte sequence meant different characters on different systems. A document created in Japan would display as gibberish in Europe.

Unicode‚Äôs solution: One universal numbering system where every character has exactly one number, period.

Code Point Range

Unicode defines code points from U+0000 to U+10FFFF:

  • That‚Äôs 1,114,112 possible code points
  • Currently, about 149,000 are assigned to characters
  • The rest are reserved for future use

Unicode Planes

Unicode divides code points into 17 ‚Äúplanes‚Äù of 65,536 code points each:

Plane 0: Basic Multilingual Plane (BMP)
  U+0000 to U+FFFF
  Contains most common characters (Latin, Greek, Cyrillic, CJK, etc.)
  
Plane 1: Supplementary Multilingual Plane (SMP)
  U+10000 to U+1FFFF
  Contains historic scripts, musical symbols, emoji
  
Plane 2: Supplementary Ideographic Plane (SIP)
  U+20000 to U+2FFFF
  Contains additional CJK characters
  
Planes 3-13: Currently unused
  Reserved for future use
  
Plane 14: Supplementary Special-purpose Plane (SSP)
  U+E0000 to U+EFFFF
  Contains tags and variation selectors
  
Planes 15-16: Private Use Areas
  U+F0000 to U+10FFFF
  For custom/proprietary characters

Why this matters for libkynput: Characters in Plane 1+ (like emoji ‚ÄôüòÄ‚Äô at U+1F600) require special handling in UTF-16, which we‚Äôll explain below.

Encodings: How to Store Code Points

A code point is an abstract number. An encoding is how you represent that number in bytes for storage or transmission.

UTF-32: The Simple One

UTF-32 stores each code point as exactly 4 bytes (32 bits).

Character: 'A'
Code Point: U+0041
UTF-32LE (little-endian): [0x41, 0x00, 0x00, 0x00]
UTF-32BE (big-endian):    [0x00, 0x00, 0x00, 0x41]

Character: '‰∏≠'
Code Point: U+4E2D
UTF-32LE: [0x2D, 0x4E, 0x00, 0x00]
UTF-32BE: [0x00, 0x00, 0x4E, 0x2D]

Character: 'üòÄ'
Code Point: U+1F600
UTF-32LE: [0x00, 0xF6, 0x01, 0x00]
UTF-32BE: [0x00, 0x01, 0xF6, 0x00]

Advantages:

  • Simple: Code point value = byte value
  • Fixed width: Every character is always 4 bytes
  • Random access: Easy to jump to the Nth character

Disadvantages:

  • Wasteful: ASCII text uses 4√ó the space it needs
  • Not widely used: Most systems use UTF-8 or UTF-16

Why libkynput uses UTF-32 internally: Rust‚Äôs char type is UTF-32, making it the natural representation.

UTF-16: The Windows Standard

UTF-16 uses 2 bytes (16 bits) for most characters, but 4 bytes for characters above U+FFFF using ‚Äúsurrogate pairs.‚Äù

Simple Case: BMP Characters (U+0000 to U+FFFF)

These fit in one 16-bit unit:

Character: 'A'
Code Point: U+0041
UTF-16: [0x0041]  (2 bytes)

Character: '‰∏≠'
Code Point: U+4E2D
UTF-16: [0x4E2D]  (2 bytes)

Complex Case: Supplementary Characters (U+10000 to U+10FFFF)

These require surrogate pairs (two 16-bit units):

Character: 'üòÄ'
Code Point: U+1F600

Step 1: Subtract 0x10000
  0x1F600 - 0x10000 = 0x0F600

Step 2: Split into 10-bit high and low parts
  High 10 bits: 0x0F600 >> 10 = 0x003D
  Low 10 bits:  0x0F600 & 0x3FF = 0x0200

Step 3: Add to surrogate base values
  High surrogate: 0xD800 + 0x003D = 0xD83D
  Low surrogate:  0xDC00 + 0x0200 = 0xDE00

UTF-16: [0xD83D, 0xDE00]  (4 bytes)

The Surrogate Range:

  • High surrogates: 0xD800 to 0xDBFF (1,024 values)
  • Low surrogates: 0xDC00 to 0xDFFF (1,024 values)
  • These 2,048 code points are reserved and never represent characters by themselves
  • They only have meaning in pairs

Why this is tricky:

Why Windows uses UTF-16:

  • Historical: Windows NT was designed in the early 1990s when Unicode only had 65,536 code points (all fit in 16 bits)
  • When Unicode expanded beyond U+FFFF, Microsoft added surrogate pair support to maintain backward compatibility
  • Now Windows is stuck with UTF-16 forever

Summary: Key Takeaways

For libkynput Implementation

  1. Code points are abstract numbers (U+0000 to U+10FFFF)
  2. Encodings are concrete bytes (UTF-8, UTF-16, UTF-32)
  3. Rust‚Äôs char is UTF-32 (makes it our natural wire format)
  4. Windows needs UTF-16 (requires surrogate pair handling for emoji)
  5. Linux needs hex strings (Ctrl+Shift+U method)
  6. macOS needs UTF-16 (Core Graphics API)

Mental Model

Think of Unicode like this:

  • Code point = Street address (universal, unique identifier)
  • Encoding = GPS coordinates format (different ways to write the same location)
  • Character = The actual house (what you see on screen)
  • Grapheme = The property (might include garage, house, fence = multiple code points)

libkynput transmits ‚Äústreet addresses‚Äù (code points in UTF-32 format), and each platform‚Äôs injector converts them to the ‚ÄúGPS format‚Äù (encoding) that the local OS expects.
