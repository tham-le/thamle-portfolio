Big-O notation describes how an algorithm‚Äôs runtime or space requirements grow as the input size increases.

The Library Analogy

Finding a Book in a Library

O(1) - Direct Access

Scenario: You know the exact shelf number

  • Time: Constant, regardless of library size
  • Example: Array access by index

O(log n) - Binary Search

Scenario: Books are sorted, you keep halving the search space

  • 1000 books ‚Üí ~10 steps
  • 1,000,000 books ‚Üí ~20 steps
  • Example: Binary search

O(n) - Linear Scan

Scenario: Check every book one by one

  • 1000 books ‚Üí 1000 checks
  • 1,000,000 books ‚Üí 1,000,000 checks
  • Example: Linear search

O(n¬≤) - Compare Everything

Scenario: Compare every book with every other book

  • 1000 books ‚Üí 1,000,000 comparisons
  • Example: Bubble sort

That‚Äôs Big-O - expressing how work scales with input size.

O(1) - Constant Time

Performance: Independent of input size.

Examples:

  • Array access by index: arr[5]
  • Hash table lookup (average case): dict[key]
  • Push/pop from stack

Key insight: May have multiple operations, but still O(1) if count doesn‚Äôt depend on n.

O(log n) - Logarithmic Time

Performance: Doubles input size ‚Üí adds just 1 operation.

Examples:

  • Binary search
  • Balanced binary search tree operations (BST, AVL, Red-Black)
  • Heap insert/delete

Why so fast? You eliminate half the remaining data each step.

O(n) - Linear Time

Performance: Double input ‚Üí double operations.

Examples:

  • Linear search
  • Finding min/max in unsorted array
  • Traversing linked list
  • Printing all elements

Key insight: Single loop over n elements = O(n).

O(n log n) - Linearithmic Time

Performance: Efficient sorting algorithms.

Examples:

  • Merge sort
  • Quick sort (average case)
  • Heap sort
  • Most efficient comparison-based sorts

Why this complexity? Divide-and-conquer with merging.

O(n¬≤) - Quadratic Time

Performance: Nested loops over n elements.

Examples:

  • Bubble sort, selection sort, insertion sort
  • Checking all pairs in array
  • Naive string matching

Warning: Slow for large inputs. Avoid if possible.

O(2‚Åø) - Exponential Time

Performance: Each added element doubles the work.

Examples:

  • Recursive Fibonacci (naive)
  • Generating all subsets (2‚Åø subsets of n elements)
  • Brute-force password cracking

Practical limit: n < 30 for reasonable runtime.

Fix for Fibonacci: Use dynamic programming ‚Üí O(n).

Rules for Calculating Big-O

1. Drop Constants

2. Drop Non-Dominant Terms

3. Different Inputs = Different Variables

4. Best/Average/Worst Case

Usually discuss worst case unless specified.

The Big Takeaway

Big-O describes scalability:

  • How does runtime grow with input size?
  • Ignore constants and non-dominant terms
  • Focus on the worst case (usually)

Essential for:

  • Choosing the right algorithm
  • Interview questions
  • Optimizing code
  • Understanding trade-offs

Golden rule: Aim for O(n log n) or better for large datasets.
