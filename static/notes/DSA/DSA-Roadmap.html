WTF are Data Structures & Algorithms?

Choosing the right structure is half the solution. The other half is knowing which algorithm fits.

You have a million records and need to find one by ID. An unsorted array means scanning all million. A hash table means one lookup. That’s not a clever trick — it’s a 50,000x speedup from choosing the right data structure.

Data structures and algorithms are inseparable: the structure determines which algorithms are efficient, and the algorithm determines which structure you need. This article maps the landscape — what exists, what it’s good for, and how the pieces connect.

Data Structures: The Storage Side

Every data structure makes a trade-off. Fast reads or fast writes. Memory efficiency or access speed. Sorted order or insertion speed. There is no universal best — only the best fit for your access pattern.

Arrays

Contiguous memory, O(1) random access by index.

OperationTime
Access by indexO(1)
Search (unsorted)O(n)
Insert/Delete (middle)O(n)
Append (amortized)O(1)

Use when: you need fast indexed access and mostly append. The CPU cache loves arrays — sequential access is the fastest memory pattern there is.

Dynamic arrays (Vec in Rust, std::vector in C++) resize automatically. Amortized O(1) append — most pushes are constant time, occasional resizing is O(n), but spread across all operations it averages to O(1). See Algorithm-Theory for amortized analysis.

Linked Lists

Nodes with pointers. O(1) insert/delete at known positions, O(n) access by index.

OperationSinglyDoubly
Access by indexO(n)O(n)
Insert at headO(1)O(1)
Insert at tailO(n) / O(1)*O(1)
Delete at known nodeO(n)O(1)

*O(1) with tail pointer.

Use when: you need frequent insertions/deletions at arbitrary positions and don’t need random access. Kernel data structures use intrusive linked lists extensively. Cache performance is poor compared to arrays due to pointer chasing.

Stacks and Queues

Stack (LIFO): push/pop from one end. Function call stacks, undo/redo, expression parsing, DFS traversal.

Queue (FIFO): enqueue at back, dequeue from front. BFS traversal, task scheduling, print queues, Operating-System process scheduling.

Deque: push/pop from both ends. Sliding window problems, work-stealing schedulers.

All operations O(1).

Hash Tables

Map keys to values via a hash function. O(1) average for insert, delete, and lookup.

OperationAverageWorst
InsertO(1)O(n)
LookupO(1)O(n)
DeleteO(1)O(n)

Worst case happens when all keys collide into the same bucket. Good hash functions and load factor management keep this rare. The birthday paradox (see Mathematics-for-CS) determines collision probability.

Use when: you need fast key-based lookups. DNS resolution, caches, symbol tables in Compilers, database indexes.

Trade-off: no ordering. If you need sorted iteration, use a tree instead.

Trees

Hierarchical structures with parent-child relationships.

Binary Search Tree (BST): left < root < right. O(log n) search/insert/delete when balanced. Degenerates to O(n) when skewed.

Balanced BSTs (AVL, Red-Black): self-balancing to guarantee O(log n). Red-Black trees power std::map in C++ and BTreeMap in Rust’s standard library.

B-Trees: wide, shallow trees optimized for disk I/O. Each node holds many keys, minimizing the number of disk reads. Every filesystem and database index you’ve ever used is built on B-trees or their variants.

Tries (prefix trees): store strings character-by-character. O(m) lookup where m is key length, independent of the number of stored strings. Autocomplete, spell checkers, IP routing tables.

Heaps (priority queues): min-heap or max-heap. O(log n) insert, O(1) peek at min/max, O(log n) extract. Dijkstra’s algorithm, task scheduling, merge of k sorted lists.

Graphs

The most general structure: vertices and edges modeling arbitrary relationships.

G = (V, E)
Directed / undirected
Weighted / unweighted
Cyclic / acyclic (DAG)

Representations:

  • Adjacency list: O(V + E) space, efficient for sparse graphs
  • Adjacency matrix: O(V²) space, O(1) edge lookup, efficient for dense graphs

Use when: modeling networks, dependencies, state machines, social connections, routing. See Graph-Algorithms for traversals, shortest paths, and MSTs.

Algorithms: The Processing Side

Sorting

The most studied problem in CS. The lower bound for comparison-based sorting is Ω(n log n) — you can’t do better by comparing elements.

AlgorithmAverageWorstSpaceStableBest for
Quick sortO(n log n)O(n²)O(log n)NoGeneral purpose, cache-friendly
Merge sortO(n log n)O(n log n)O(n)YesNeed stability, linked lists
Heap sortO(n log n)O(n log n)O(1)NoGuaranteed O(n log n), in-place
Insertion sortO(n²)O(n²)O(1)YesNearly sorted, small arrays
Counting sortO(n + k)O(n + k)O(k)YesSmall integer key range

In practice, standard library sorts (like Rust’s sort) use hybrid approaches — typically introsort (quicksort + heapsort fallback) or timsort (merge sort + insertion sort for small runs).

See Sorting-Algorithms for detailed implementations.

Searching

Binary search: O(log n) on sorted data. Simple but the details matter — off-by-one errors are the classic bug. Useful far beyond arrays: git bisect uses binary search to find the commit that introduced a bug.

Hash-based: O(1) average via hash tables. When you need exact-match lookups.

Graph traversal: BFS for shortest path in unweighted graphs, DFS for exploring all reachable nodes, topological sort for dependency ordering.

Choosing the Right Structure

The decision usually comes down to your access pattern:

NeedStructure
Fast indexed accessArray
Fast key-value lookupHash table
Sorted order + fast lookupBalanced BST (Red-Black, AVL)
Frequent insert/delete at endsDeque
Priority access (min/max)Heap
Hierarchical dataTree
Relationships between entitiesGraph
Prefix matchingTrie
Disk-friendly sorted storageB-tree

Where DSA Meets Systems

Data structures aren’t academic abstractions. They’re the building blocks of every system:

  • Compilers use hash tables for symbol lookup, trees for ASTs, graphs for control flow analysis. See Compiler.
  • Operating systems use queues for scheduling, trees for filesystems (inodes), hash tables for page table lookups. See Operating-System.
  • Databases use B-trees for indexes, hash tables for joins, heaps for priority scheduling.
  • Networking uses tries for IP routing, graphs for topology, queues for packet buffering. See Router.
  • Memory allocators use free lists (linked lists), buddy systems (binary trees), and segregated fits.

The performance difference between choosing a hash table vs. a sorted array for a hot path can be the difference between a system that handles 10k requests/second and one that handles 100k.

Summary

Data structures define how data is stored; algorithms define how it’s processed. Arrays for indexed access, hash tables for key lookups, trees for sorted data, graphs for relationships. The right choice depends on your access pattern — there’s no universal winner. For algorithm design strategies (divide & conquer, DP, greedy), see Algorithm-Theory.
