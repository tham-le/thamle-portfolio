1. Binary Search: The Guessing Game Strategy

The Algorithm

Repeatedly divide search space in half until you find the answer.

Real-World Applications

üéØ Guessing Number Games

Think of a number between 1 and 100

Bad strategy (Linear Search):

Good strategy (Binary Search):
"Is it 50?" ‚Üí "Lower"
"Is it 25?" ‚Üí "Higher"
"Is it 37?" ‚Üí "Lower"
"Is it 31?" ‚Üí "Higher"
...
‚Üí Maximum 7 guesses (log‚ÇÇ 100 ‚âà 6.6)

üìñ Dictionary Lookup

When you look up a word in a physical dictionary:

  1. Open middle ‚Üí Too far? Go left half. Not far enough? Go right half.
  2. Repeat until you find the word
  3. Exactly binary search!

üîß Troubleshooting: Finding Bad Commit

üè† Finding Perfect Temperature

üí∞ Price Negotiation

...converge to fair price faster

üìö Library Book Search

  • Check if your number is higher or lower
  • Navigate to that section
  • Repeat

Why it works: Data is sorted ‚Üí logarithmic search time

2. Greedy Algorithms: The Vending Machine

The Algorithm

Make the locally optimal choice at each step, hoping for global optimum.

Real-World Applications

üíµ Making Change

Cashier giving you $0.87 in change

Greedy approach (what cashiers do):
1. Largest coin ‚â§ remaining? Use it.
2. Repeat until done.

$0.87:
- 3 quarters ($0.75) ‚Üí $0.12 left
- 1 dime ($0.10) ‚Üí $0.02 left
- 2 pennies ‚Üí Done!

Result: 6 coins (optimal!)

üìÖ Activity Selection / Meeting Scheduler

Schedule maximum meetings in conference room

Meetings:
A: 9-10am
B: 9:30-11am
C: 10:30-12pm
D: 11am-12pm

Greedy strategy: Always pick meeting that ends earliest
1. Pick A (ends 10am)
2. Pick C (ends 12pm, starts after A)
Result: A + C (2 meetings)

Why not B? It ends later, blocks more options.

üéí Packing a Suitcase

Limited weight, multiple items

Greedy by value-to-weight ratio:
- Item 1: $50, 2kg ‚Üí ratio: 25
- Item 2: $40, 1kg ‚Üí ratio: 40  ‚Üê Pick first
- Item 3: $30, 1kg ‚Üí ratio: 30  ‚Üê Pick second

Pack highest ratio items first

üöó Gas Station Stops

Road trip: 500 miles, 300-mile tank

Gas stations at: 100, 200, 250, 350, 450 miles

Greedy: Drive as far as possible before refueling
- Fill at start
- Pass 100, 200, 250 (can reach next)
- Stop at 350 (can't reach 450 without refueling)
- Stop at 450
Result: 2 stops (optimal)

üõí Checkout Line Selection

Supermarket: Multiple checkout lines

Greedy: Pick shortest line right now

Limitation: May not be optimal (short line might have slow customer)
Better: Consider items per person, but greedy is fast decision

Greedy limitation: Doesn‚Äôt always give optimal solution, but it‚Äôs fast and often ‚Äúgood enough.‚Äù

3. Graph Algorithms: Navigation & Networks

The Algorithm

Model relationships as nodes and edges, find paths or patterns.

Real-World Applications

üó∫Ô∏è GPS Navigation (Dijkstra‚Äôs Algorithm)

Finding shortest path from home to work

Map as graph:
- Intersections = nodes
- Roads = edges (with travel time as weight)

Dijkstra's algorithm:
1. Start at home (distance = 0)
2. Check all neighbors, track shortest distance to each
3. Visit closest unvisited node
4. Repeat until destination reached

Google Maps uses enhanced version considering:
- Real-time traffic
- Road types
- Turn restrictions

Simplified example:

Home --5min--> Mall --10min--> Work
Home --20min--> Park --8min--> Work

Dijkstra finds: Home ‚Üí Mall ‚Üí Work (15 min)
Better than: Home ‚Üí Park ‚Üí Work (28 min)

‚úàÔ∏è Flight Route Planning

Find cheapest flight from NYC to Tokyo

Graph:
- Cities = nodes
- Flights = edges (price as weight)

Algorithm explores connections:
NYC ‚Üí London ‚Üí Tokyo ($500 + $400 = $900)
NYC ‚Üí LA ‚Üí Tokyo ($300 + $500 = $800) ‚Üê Cheaper!

Airlines use this for:
- Finding routes with layovers
- Optimizing fuel costs
- Crew scheduling

üë• Social Network Friend Suggestions (BFS)

"People you may know"

Graph:
- People = nodes
- Friendships = edges

Breadth-First Search (BFS):
1. Your friends = distance 1
2. Friends of friends = distance 2  ‚Üê Suggest these!
3. Distance 3, 4, ... (too distant)

Why BFS? Explores by "degrees of separation"

Example:

You ‚Üí Alice ‚Üí Bob
You ‚Üí Carol ‚Üí Bob

Bob appears at distance 2 via 2 paths ‚Üí Strong suggestion!

üåê Web Crawling (DFS/BFS)

Google indexing the web

Graph:
- Web pages = nodes
- Hyperlinks = edges

Crawler uses BFS:
1. Start at seed URLs
2. Visit page, extract all links
3. Add links to queue
4. Repeat

Result: Discovers connected pages systematically

üîå Network Routing

Internet packet delivery

Router graph:
- Routers = nodes
- Connections = edges (bandwidth, latency as weight)

Algorithm finds:
- Fastest path considering current load
- Backup routes if connection fails
- Load balancing across multiple paths

üöá Subway System Planning

Find route between stations with fewest transfers

Graph:
- Stations = nodes
- Train lines = edges

BFS finds shortest path by number of stops
Weighted version considers:
- Transfer time
- Wait time
- Walking distance

4. Sorting: Organizing Information

The Algorithm

Arrange elements in order (ascending/descending).

Real-World Applications

üìä Leaderboards & Rankings

Video game leaderboard: Sort players by score

Quick Sort approach (like game servers do):
1. Pick pivot (middle score)
2. Partition: Higher scores left, lower right
3. Recursively sort each partition

Real-time updates: Insert new score in sorted position (Binary Search + Insert)

üìß Email Inbox

Gmail sorting by date/importance

Merge Sort approach:
- Divide emails into chunks
- Sort each chunk
- Merge sorted chunks

Why Merge Sort?
- Stable (maintains order of equal elements)
- Efficient for large datasets
- Good for external sorting (can't fit all in memory)

üéµ Spotify Playlist Organization

Sort by: Recently played, artist name, song length

Counting Sort for play counts:
- Count frequency of each play count
- Reconstruct sorted list

Efficient when range is small (0-1000 plays)
O(n + k) instead of O(n log n)

üì¶ Warehouse Inventory

Amazon warehouse: Sort packages by delivery time

Heap Sort approach:
- Min heap: Package with earliest deadline at top
- Extract min ‚Üí Ship first
- Continue until all shipped

Priority queue naturally handles:
- New packages arriving
- Deadline changes

üè¶ Bank Transaction History

Sort by date and amount

Multi-key sort:
1. Primary: Date (newest first)
2. Secondary: Amount (largest first within same date)

Implementation:
- Sort by amount
- Stable sort by date (preserves amount order within each date)

üé∞ Shuffling (Reverse of Sorting)

Spotify shuffle, card games

Fisher-Yates Shuffle:
1. Start from last element
2. Pick random element before it, swap
3. Move to previous element
4. Repeat

Result: Uniformly random permutation

5. Hash Tables: Instant Lookups

The Algorithm

Map keys to array indices for O(1) access.

Real-World Applications

üì± Phone Contacts

Find contact by name instantly

Hash table:
- Name ‚Üí Hash function ‚Üí Index
- "Alice" ‚Üí hash("Alice") ‚Üí Index 42
- contacts[42] = Alice's info

Search "Alice": Instant lookup, no need to scan all contacts

üîê Password Storage

Website login: Check password

Don't store plain password!
Store hash:
- User creates password: "secret123"
- Hash it: hash("secret123") ‚Üí "a4b3c2d1..."
- Store hash, not password

Login attempt:
- User enters "secret123"
- Hash it: "a4b3c2d1..."
- Compare hashes (match = correct password)

Why? Can't reverse hash to get password (one-way function)

üåê DNS (Domain Name System)

Type "google.com" ‚Üí Get IP address

DNS as distributed hash table:
- Domain name = key
- IP address = value

Cache:
- Your computer stores recent lookups
- No need to query DNS server again
- Hash table enables O(1) lookup

üõí Shopping Cart

E-commerce cart: Track items and quantities

Hash table:
- Product ID = key
- Quantity = value

Operations:
- Add item: cart[product_id] = quantity
- Update quantity: cart[product_id] += 1
- Remove: delete cart[product_id]

All O(1) operations!

üéÆ Game State Tracking

Video game: Track player inventory

Hash table:
- Item name = key
- Count = value

inventory = {
    "sword": 1,
    "potion": 5,
    "gold": 150
}

Check if player has item: O(1)
Use item: inventory["potion"] -= 1

üìö Word Frequency Counter

Analyze document: Count word occurrences

Hash table approach:
word_count = {}
for word in document:
    word_count[word] = word_count.get(word, 0) + 1

Result: {"the": 45, "algorithm": 12, ...}

Applications:
- Search engine indexing
- Spam detection
- Autocomplete suggestions

üßÆ Expression Evaluation (Stack)

Using stack:
1. Scan left to right
2. Push operands
3. Pop for operations (respect precedence)

Postfix: 3 5 2 * +

Evaluate postfix with stack:
1. Push 3
2. Push 5
3. Push 2
4. Pop 2, pop 5, multiply ‚Üí Push 10
5. Pop 10, pop 3, add ‚Üí Result: 13

üì¶ Browser History (Stack)

Visit pages: A ‚Üí B ‚Üí C

Forward stack: []
Back stack: [A, B, C]

Back button:
- Pop C from back stack ‚Üí Move to forward stack
- Display B

Forward button:
- Pop C from forward stack ‚Üí Move to back stack
- Display C

8. Dynamic Programming: Smart Optimization

The Algorithm

Solve complex problems by breaking into overlapping subproblems, cache results.

Real-World Applications

üí∞ Financial Planning

Maximize profit from stock trading

Problem: Buy and sell stocks to maximize profit

DP approach:
- State: Day i, holding stock or not
- Transition: Buy, sell, or hold
- Memoize: Best profit at each state

Example:
Prices: [7, 1, 5, 3, 6, 4]
Best: Buy at 1, sell at 6 ‚Üí Profit = 5

üì¶ Knapsack: Resource Allocation

Limited budget, maximize value

Business: Allocate $1M across projects
- Project A: $300k, ROI 40%
- Project B: $500k, ROI 50%
- Project C: $400k, ROI 35%

DP finds optimal combination within budget

üß¨ DNA Sequence Alignment

- Sequence 1: AGGTAB
- Sequence 2: GXTXAYB
- LCS: GTAB

Used for:
- Identifying mutations
- Evolutionary relationships
- Medical diagnostics

DP builds table of solutions to subproblems

üéØ Edit Distance: Spell Check

"recieve" ‚Üí "receive" (1 edit)

Calculate minimum edits (insert, delete, replace):

DP table compares character by character:
- Match: No cost
- Mismatch: Try all operations, pick cheapest

üöö Delivery Route Optimization

Visit N locations, minimize distance

Traveling Salesman Problem (TSP):
- Visit each city once
- Return to start
- Minimize total distance

- O(n¬≤ 2‚Åø) - better than brute force O(n!)

- UPS route planning

- Circuit board drilling

üéÆ Game Scoring: Coin Change

- Try each coin

- Cache results
