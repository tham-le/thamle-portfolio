Algorithms in the Wild

Every app you use, every service you depend on — there’s an algorithm behind it. Here’s a field guide.

Algorithms aren’t abstract puzzles for interview prep. They’re engineering tools solving real problems at massive scale. Google Maps runs Dijkstra variants on billions of edges. Netflix uses recommendation algorithms to drive 80% of what people watch. Your phone’s autocomplete is a trie with probabilistic ranking.

This article connects the algorithms you study to the systems where they run.

Binary Search: Halving the Search Space

The idea: if data is sorted, check the middle element. Eliminate half the remaining space. Repeat.

Where it shows up:

Dictionary/phonebook lookup — open to the middle, go left or right. This is binary search, and it’s how humans naturally search sorted data.

git bisect — find which commit introduced a bug across 1,000 commits in ~10 tests:

Database indexes — B-tree lookups are generalized binary search. Every WHERE id = X query on an indexed column uses this.

Temperature/parameter tuning — binary search between “too low” and “too high” converges faster than linear scanning.

Why it works: logarithmic search — O(log n) — because each step halves the problem. 1,000,000 items? 20 comparisons max.

Greedy Algorithms: Local Decisions, Global Results

The idea: at each step, make the locally best choice. Don’t reconsider.

Making change — cashiers do this intuitively:

Meeting room scheduling — maximize non-overlapping meetings by always picking the one that ends earliest. Provably optimal.

Huffman coding — compress data by assigning shorter codes to frequent symbols. Every ZIP file and JPEG uses this.

Gas station stops — drive as far as possible before refueling. Minimizes stops.

The trap: greedy doesn’t always work. For US coins it’s optimal, but for coin systems like {1, 3, 4}, greedy gives 4+1+1 for 6 cents when 3+3 is better. Always verify the greedy-choice property or use dynamic programming instead.

Graph Algorithms: Modeling Relationships

The idea: model entities as nodes, relationships as edges. Then traverse, search, or optimize over the structure.

GPS Navigation (Dijkstra’s Algorithm)

Intersections = nodes
Roads = edges (weighted by travel time)

Google Maps runs a modified Dijkstra’s algorithm considering real-time traffic, road types, and turn restrictions. The core insight: always expand the closest unvisited node.

Home --5min--> Mall --10min--> Work     (15 min total)
Home --20min--> Park --8min--> Work     (28 min total)

Dijkstra finds: Home → Mall → Work

Social Networks (BFS)

“People you may know” = friends-of-friends = nodes at distance 2 in the friendship graph. BFS naturally explores by distance from the source.

You → Alice → Bob     (distance 2 via Alice)
You → Carol → Bob     (distance 2 via Carol)

Bob appears via 2 paths → strong recommendation

Network Routing

Every packet you send traverses a graph of routers. OSPF (Open Shortest Path First) uses Dijkstra. BGP (Border Gateway Protocol) uses path-vector algorithms. See Router and Networking.

Dependency Resolution

Package managers (cargo, npm, apt) model dependencies as a directed acyclic graph and use topological sort to determine install order. Circular dependencies = cycle in the graph = error.

Sorting: The Foundation

Sorting enables binary search, makes duplicates adjacent, and structures data for merging. The choice of algorithm matters.

Quick sort — general purpose, cache-friendly (in-place, sequential access). O(n log n) average, O(n²) worst case. Most standard library sorts use this as a base.

Merge sort — stable (preserves relative order of equal elements), guaranteed O(n log n). Used when stability matters (database result ordering) or for external sorting (data too large for RAM).

Counting/radix sort — O(n) when key range is bounded. Used for sorting integers, IP addresses, or fixed-width strings.

Insertion sort — O(n) on nearly-sorted data. Hybrid algorithms (like timsort) use it for small subarrays where the overhead of divide-and-conquer isn’t worth it.

See Sorting-Algorithms for implementations.

Hash Tables: O(1) Everything

DNS resolution — your machine caches domain → IP mappings in a hash table. google.com → 142.250.80.46 in one lookup.

Password storage — store hash(password), not the password. On login, hash the input and compare. One-way function: can’t reverse the hash to recover the password. See Mathematics-for-CS for the underlying math.

Caches — every cache (browser, CDN, CPU TLB) is fundamentally a hash table mapping keys to stored values. LRU caches combine a hash table with a doubly-linked list for O(1) lookup and O(1) eviction.

De-duplication — seen this element before? Hash set membership check in O(1). Log processing, network packet dedup, database joins.

Stacks and Queues: Order Matters

Undo/redo — every text editor maintains a stack of actions. Undo pops the last action; redo pushes it back.

Browser history — back button pops from the history stack, forward pushes it back. Two stacks working together.

Expression evaluation — the shunting-yard algorithm uses a stack to convert infix (3 + 5 * 2) to postfix (3 5 2 * +), then evaluates the postfix with another stack. Every calculator and Compiler does this.

Print queues / task scheduling — FIFO ordering ensures fairness. First job submitted, first job processed.

Video buffering — a queue of frames: download ahead (enqueue), play in order (dequeue). Smooth playback depends on the queue staying non-empty.

Rate limiting — sliding window log: a queue of timestamps. Remove entries older than the window. If the queue length exceeds the limit, reject the request. O(1) amortized.

Trees: Hierarchy and Search

File systems — directories and files form a tree. ls -R is a DFS traversal. du -s computes subtree sizes bottom-up.

Autocomplete — a trie stores all known words. Type “ca” → traverse to the c→a node → all descendants are suggestions (“cat”, “car”, “can”). O(prefix length) lookup, independent of dictionary size.

Decision trees — loan approval, medical diagnosis, spam classification. Each internal node is a yes/no question; leaves are decisions. Random forests and gradient boosting (XGBoost, LightGBM) are ensembles of decision trees.

Game AI — minimax trees explore future game states. Alpha-beta pruning eliminates branches that can’t affect the outcome. Chess engines evaluate millions of positions per second this way.

DOM rendering — HTML is a tree. The browser traverses it (DFS) to compute layout and paint pixels.

Dynamic Programming: Trading Memory for Time

Spell check (edit distance) — how many insertions, deletions, and substitutions to transform “recieve” into “receive”? A 2D DP table compares character by character. Every autocorrect system uses this.

Route optimization — the Traveling Salesman Problem (visit N cities, minimize total distance). The Held-Karp algorithm uses DP to solve it in O(n² · 2ⁿ) — still exponential, but vastly better than the O(n!) brute force.

DNA sequence alignment — find the longest common subsequence between two genetic sequences. DP builds a table of subproblem solutions. The Smith-Waterman algorithm (used in BLAST) is a DP variant.

Stock trading — maximize profit from a sequence of prices. DP tracks the best profit at each day for each state (holding/not holding), making optimal buy/sell decisions.

Resource allocation — the knapsack problem: given a budget and a set of projects with costs and returns, which subset maximizes ROI? DP finds the optimal combination.

See Dynamic-Programming for implementations and Algorithm-Theory for when DP vs greedy applies.

Sliding Window: Streaming Data

Moving averages — track the 7-day average stock price. Instead of recomputing the sum every day, subtract the oldest value and add the newest. Constant-time update.

Rate limiting — “max 100 requests per minute.” Maintain a window of timestamps, slide it forward as time passes. O(1) amortized per request.

Network monitoring — detect anomalies by tracking metrics over a sliding time window. Alert when the current window deviates from historical baselines.

The Pattern

Every system you use combines these primitives:

You do this…Algorithm behind it
Search on GoogleRanking algorithms + hash table indexes
Open Google MapsDijkstra’s shortest path
Stream videoQueue (buffer) + compression (Huffman/arithmetic coding)
Swipe a credit cardFraud detection (decision trees, graph analysis)
Get a packageRoute optimization (TSP variants)
Type a messageTrie (autocomplete) + edit distance (autocorrect)

The algorithms are invisible, but they’re running billions of times per second across every layer of the stack.

Summary

Algorithms aren’t interview trivia — they’re the engines inside every system. Binary search powers database indexes. Dijkstra powers GPS routing. Hash tables power caches and DNS. DP powers spell check and sequence alignment. Understanding where algorithms show up in real systems makes you better at choosing the right tool when building your own.
