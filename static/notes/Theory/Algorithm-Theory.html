WTF is Algorithm Design?

You know the data structures. You know Big-O. The question is: given a new problem, which strategy do you reach for?

Algorithm design isn’t about memorizing sorting algorithms or graph traversals — those are tools you look up. Design is about paradigms: recurring strategies that apply across hundreds of different problems. Once you internalize the five or six major paradigms, new problems start looking like variations on patterns you’ve already solved.

This article focuses on the paradigm selection problem. For specific algorithms (sorting, graph traversal, string matching), see DSA-Roadmap. For complexity analysis, see Big-O-Notation. For the theoretical limits of what’s tractable, see Theory-of-Computation.

The Core Analogy

You’re in a maze. Here are your options:

  • Divide and conquer: Split the maze into quadrants, solve each quadrant independently, combine the paths.
  • Dynamic programming: As you explore, write the shortest path to each intersection on the wall. Never re-explore a path you’ve already measured.
  • Greedy: At every fork, take the corridor that points most directly toward the exit. Hope for the best.
  • Backtracking: Try every path. When you hit a dead end, retrace to the last fork and try another.
  • Branch and bound: Like backtracking, but you carry a measuring tape. If your current path is already longer than the best one you’ve found, stop and backtrack immediately.

Each strategy has problems where it’s optimal and problems where it fails. The art is matching the paradigm to the problem structure.

Divide and Conquer

Strategy: Break the problem into independent subproblems, solve them recursively, combine results.

When it works: Subproblems don’t overlap, and combining results is cheap.

The recurrence T(n) = 2T(n/2) + O(n) solves to O(n log n) via the Master Theorem.

Classic applications:

  • Merge sort, quick sort
  • Binary search
  • Karatsuba multiplication (multiply two n-digit numbers in O(n^1.585) instead of O(n^2))
  • Strassen’s matrix multiplication
  • Closest pair of points

The signal: if you can split the input in half, solve both halves independently, and combine in linear time, you probably want divide and conquer.

Dynamic Programming

Strategy: Break into overlapping subproblems, store solutions to avoid recomputation.

When it works: The problem has optimal substructure (optimal solution contains optimal solutions to subproblems) and overlapping subproblems (the same subproblem appears multiple times).

The canonical example — Fibonacci without and with DP:

The DP recipe:

  1. Define subproblems (what’s the state?)
  2. Find the recurrence relation
  3. Compute bottom-up (iterative table) or top-down (memoized recursion)
  4. Reconstruct the solution if needed

Classic applications:

  • Longest common subsequence (LCS)
  • Edit distance (Levenshtein)
  • Knapsack problem
  • Matrix chain multiplication
  • Shortest paths (Floyd-Warshall, Bellman-Ford)
  • Compiler instruction scheduling

The signal: if you find yourself computing the same function with the same arguments repeatedly, or if you’re considering all subsets/subsequences but the optimal solution has recursive structure, reach for DP.

Greedy vs DP: Greedy makes one choice per step and never looks back. DP considers all choices and picks the best. If greedy gives the correct answer, it’s faster. But greedy fails on problems like 0/1 knapsack where local optimality doesn’t imply global optimality.

Greedy Algorithms

Strategy: At each step, make the locally optimal choice. Never reconsider.

When it works: The problem has the greedy-choice property (a locally optimal choice leads to a globally optimal solution) and optimal substructure.

The greedy choice — always pick the activity that finishes earliest — provably gives the optimal solution. The proof uses an exchange argument: swapping any other choice for the greedy one never makes the solution worse.

Classic applications:

  • Activity/interval scheduling
  • Huffman coding (optimal prefix-free codes)
  • Dijkstra’s shortest path
  • Prim’s and Kruskal’s minimum spanning tree
  • Fractional knapsack

The signal: if the problem asks for an optimal schedule, optimal ordering, or optimal selection, and you can prove that a local rule never hurts the global solution, greedy is your fastest option.

The trap: greedy feels right on many problems where it’s wrong. Always prove correctness (greedy-choice property + optimal substructure) or find a counterexample. The 0/1 knapsack is the classic greedy trap — taking the highest value-per-weight item first can miss the optimal solution.

Backtracking

Strategy: Explore all possibilities, pruning invalid branches as early as possible.

When it works: The problem is a search over a combinatorial space, and you can detect invalid partial solutions before fully constructing them.

Classic applications:

  • N-Queens (place N queens on an N×N board with no attacks)
  • Sudoku solver
  • Subset sum
  • Graph coloring
  • Generating all permutations/combinations

Complexity: Worst case is exponential (O(2^n) or O(n!)), but effective pruning makes many instances tractable in practice.

The signal: if the problem says “find all solutions” or “does a valid configuration exist” over a combinatorial space, and you can check validity of partial solutions, backtracking is the natural approach.

Branch and Bound

Strategy: Backtracking + bounding. Track the best solution found so far and prune any branch whose lower bound exceeds it.

The quality of the bound function determines performance. A tight lower bound prunes aggressively; a loose one degrades to plain backtracking.

Classic applications:

  • Traveling salesman problem
  • Integer programming
  • Job scheduling
  • Any NP-hard optimization where you need exact solutions for small instances

See Theory-of-Computation for why these problems are NP-hard.

Amortized Analysis

Not a design paradigm — a way of thinking about cost that reveals hidden efficiency.

Some data structures have operations that are usually cheap but occasionally expensive. Amortized analysis shows the average cost per operation over a sequence.

Dynamic array (Vec in Rust, std::vector in C++):

  • Most push operations: O(1)
  • Occasionally: O(n) when the array doubles in size
  • Amortized cost per push: O(1)

Three techniques:

  • Aggregate: total cost of n operations / n
  • Accounting: assign “credits” to cheap operations, spend them on expensive ones
  • Potential method: define a potential function that captures stored-up work

This explains why hash tables with resizing are O(1) amortized despite occasional O(n) rehashes, and why splay trees are O(log n) amortized despite occasional O(n) operations.

Online Algorithms

Setting: Decisions must be made as input arrives, without knowledge of future inputs.

Competitive analysis compares the online algorithm’s cost to the optimal offline algorithm (which sees all input upfront).

Paging (cache replacement):

  • LRU (Least Recently Used): k-competitive (at most k times worse than optimal, where k is cache size)
  • Optimal offline (Bélády’s MIN): evict the page used furthest in the future — requires seeing the future

The ski rental problem: You’re skiing for an unknown number of days. Buying skis costs 100.Rentingcosts10/day. Strategy: rent for 10 days, then buy. Competitive ratio: 2 (you spend at most 2x the optimal cost, regardless of how many days you end up skiing).

Applications in systems:

  • OS page replacement (Operating-System)
  • Cache eviction strategies
  • Real-time task scheduling
  • Load balancing

Cache-Oblivious Algorithms

Goal: Optimize for the memory hierarchy without knowing cache parameters (line size, cache capacity).

Standard algorithms are tuned for specific cache sizes. Cache-oblivious algorithms use recursive decomposition so that at some level of recursion, the working set fits in cache — regardless of the actual cache size.

This achieves optimal cache performance across all levels of the memory hierarchy simultaneously — L1, L2, L3, and main memory.

See CPU, Memory.

Approximation Algorithms

When a problem is NP-hard and you need a guaranteed-quality solution, not just a heuristic.

Approximation ratio: for minimization, the algorithm guarantees A ≤ α · OPT.

Key results:

  • Vertex cover: 2-approximation
  • Metric TSP: 1.5-approximation (Christofides)
  • Set cover: O(log n)-approximation (greedy)

Applications in systems:

  • Compiler optimization (register allocation, instruction scheduling)
  • Network design and routing
  • Resource scheduling

Choosing the Right Paradigm

Signal in the problemParadigm
Independent subproblems, combine resultsDivide and conquer
Overlapping subproblems, optimal substructureDynamic programming
Local choice provably leads to global optimumGreedy
Search all configurations, can prune earlyBacktracking
Optimization with computable boundsBranch and bound
NP-hard, need guaranteed qualityApproximation
Input arrives incrementallyOnline algorithms

The paradigm is the starting point, not the answer. Real problems often combine paradigms — a branch-and-bound algorithm might use DP to compute bounds, or a divide-and-conquer algorithm might use greedy at the base case.

Summary

Algorithm design is about recognizing which strategy fits your problem’s structure. Divide and conquer splits into independent subproblems. DP handles overlapping subproblems by memoizing. Greedy commits to local optima — fast when it works, wrong when it doesn’t. Backtracking searches exhaustively with pruning. Branch and bound adds cost bounds to prune faster. When the problem is NP-hard, approximation algorithms give guaranteed-quality solutions in polynomial time. The paradigm tells you how to attack the problem; Theory-of-Computation tells you whether a polynomial solution can exist at all.
