WTF is an Operating System?

When you run a program, play a game, or browse the web, something is orchestrating everything behind the scenes - managing memory, scheduling tasks, handling file access, and coordinating hardware. That something is the Operating System (OS).

It’s the middleman between your programs and the hardware.

The Building Manager Analogy

The Apartment Building Manager

Think of your computer as an apartment building:

  • Tenants (Programs): Word, Chrome, Spotify, your game
  • Building Manager (OS): Manages everything - assigns apartments (memory), schedules maintenance (CPU time), handles utilities (I/O devices)
  • Building Resources: Electricity (CPU), water (memory), mailboxes (storage), security (permissions)

Without a manager:

  • Tenants would fight over apartments
  • No one would schedule maintenance
  • Chaos would reign

With a manager (OS):

  • Each tenant gets their own space
  • Fair scheduling of resources
  • Security and isolation
  • Shared amenities work smoothly

That’s what an OS does - it brings order to chaos.

What the OS Actually Does

1. Process Management

The OS runs multiple programs simultaneously by rapidly switching between them:

Time | Program Running
-----|----------------
0ms  | Chrome
10ms | Spotify
20ms | Your Code Editor
30ms | Chrome
40ms | Spotify
...

This is context switching - the OS saves one program’s state, loads another’s, and switches so fast you don’t notice.

2. Memory Management

Every program needs RAM. The OS:

  • Allocates memory to programs
  • Prevents programs from accessing each other’s memory (Memory Protection)
  • Uses Virtual Memory to give each program its own address space
  • Manages the Stack and Heap
Program A's View:  [0x00000000 -------- 0xFFFFFFFF]
Program B's View:  [0x00000000 -------- 0xFFFFFFFF]
                    ↑ Both think they own all memory!

The OS translates these “virtual” addresses to actual physical RAM addresses.

3. File System Management

The OS organizes storage into files and directories:

  • Reads and writes files
  • Manages permissions (who can access what)
  • Handles disk caching for performance
  • Provides abstraction over raw disk blocks

Without the OS, you’d be manually managing disk sectors!

4. I/O Device Management

The OS provides drivers and interfaces for:

  • Keyboard and mouse input
  • Display output
  • Network cards
  • USB devices
  • Printers, cameras, etc.

Programs don’t talk directly to hardware - they ask the OS via system calls.

5. Scheduling

The OS Scheduler decides which program gets CPU time. Strategies include:

  • Round-robin: Each program gets a time slice in turn
  • Priority-based: Important tasks get more CPU time
  • Multi-level feedback: Dynamic priority adjustment

This is crucial for understanding concurrency.

The Kernel vs User Space

The OS has two main parts:

Kernel Space (Privileged)

  • The core of the OS
  • Direct hardware access
  • Memory protection enforcement
  • Can execute any instruction

User Space (Restricted)

  • Where your programs run
  • No direct hardware access
  • Protected from crashing the system
  • Must ask kernel for privileged operations

When your program needs to read a file, it makes a System Call - a request to the kernel:

Popular Operating Systems

Unix-like (Linux, macOS, BSD)

  • Philosophy: Everything is a file
  • Scheduler: Completely Fair Scheduler (Linux)
  • Popular for: Servers, development, embedded systems
  • Examples: Ubuntu, Debian, macOS, Android

Windows

  • Philosophy: Object-oriented kernel
  • Scheduler: Multilevel feedback queue
  • Popular for: Desktop, gaming, enterprise
  • Examples: Windows 10, Windows 11, Windows Server

Real-Time OS (RTOS)

  • Philosophy: Guaranteed response times
  • Scheduler: Deterministic, priority-based
  • Popular for: Embedded systems, robotics, aerospace
  • Examples: FreeRTOS, VxWorks

Why This Matters

For Threading

When you create a Thread, you’re asking the OS to manage multiple execution paths. The OS schedules threads across CPU cores and handles context switching.

For Process vs Thread

  • Process: Full isolation, separate memory space (heavy)
  • Thread: Shared memory space within a process (lighter)

The OS provides both because they solve different problems.

For await and Event Loop

Modern async programming tries to avoid OS thread overhead by doing cooperative scheduling in user space. But ultimately, O operations still require OS support via syscalls like epoll.

For Performance

Understanding the OS helps you write faster code:

  • Avoid unnecessary syscalls (they’re expensive)
  • Use buffering to batch I/O operations
  • Understand CPU cache effects
  • Know when the OS will Context Switch

The Big Takeaway

Operating System in a Nutshell

The OS is the software layer that sits between your programs and the hardware. It manages resources (CPU, memory, I/O), provides isolation and security, and gives programs a consistent interface to work with.

Without an OS, every program would have to:

  • Manage hardware directly
  • Implement its own memory management
  • Handle all I/O devices
  • Coordinate with other programs

The OS does all this so your programs don’t have to.

Understanding the OS is fundamental to systems programming. Every performance optimization, every concurrency pattern, every I/O operation ultimately comes down to: “How efficiently can I work with the OS?”
