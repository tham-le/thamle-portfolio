WTF is a Context Switch?

A context switch is when the Operating-System stops running one process/thread and starts running another. The OS must:

  1. Save the current process’s state (registers, program counter, stack pointer)
  2. Load the next process’s state
  3. Jump to where the new process left off

It’s how modern computers multitask - rapidly switching between processes so fast it feels like they’re running simultaneously.

The Juggling Analogy

The Juggler

Imagine a juggler with many balls (processes) but can only throw one ball at a time:

Without Context Switching (Single-Tasking)

  • Juggle one ball until done
  • Then juggle the next ball
  • One task at a time, others wait

With Context Switching (Multitasking)

  • Throw ball A (run process A for 10ms)
  • Catch and save ball A’s position (save state)
  • Throw ball B (run process B for 10ms)
  • Catch and save ball B’s position (save state)
  • Throw ball C (run process C for 10ms)
  • Catch and save ball C’s position (save state)
  • Back to ball A…

Result: All balls stay in the air (all processes make progress)

Cost: Time spent catching and throwing (context switch overhead)

That’s context switching - rapidly switching between tasks to create the illusion of parallelism.

What Gets Saved During a Context Switch?

CPU Registers

Current Process (Before Switch):
├── Program Counter (PC): 0x4005a8  ← Where we are in the code
├── Stack Pointer (SP): 0x7fff1000  ← Top of the stack
├── General Registers:
│   ├── RAX: 42
│   ├── RBX: 0x400000
│   ├── RCX: 100
│   └── ... (all CPU registers)
├── Instruction Pointer (IP): 0x400600
└── Flags: 0x0246

→ OS saves all of this to memory

Next Process (After Switch):
├── Program Counter: 0x401234  ← Restore from saved state
├── Stack Pointer: 0x7ffe2000
├── General Registers:
│   ├── RAX: 99
│   ├── RBX: 0x500000
│   └── ...
└── ... (restore everything)

Everything in CPU registers must be saved/restored.

Memory Context

Process A's View:
Virtual Address 0x1000 → Physical Address 0x5000

Context Switch

Process B's View:
Virtual Address 0x1000 → Physical Address 0x8000  (different!)

The OS updates the page table so each process sees its own Memory.

When Context Switches Happen

1. Time Slice Expires (Preemption)

Process A running...
Timer interrupt! (10ms elapsed)
→ OS: "Your turn is over, saving your state..."
→ Context switch to Process B
Process B running...

Linux default: 100-1000 times per second per CPU core.

2. Process Blocks (Waiting for I/O)

Process A: read(file, buffer);  // ⏸️ Waits for disk
→ OS: "You're waiting, let's run someone else"
→ Context switch to Process B
Process B running...

Disk read completes!
→ OS: "Process A can run again"
→ Context switch back to Process A

Voluntary switch - process gives up CPU while waiting.

3. Higher Priority Process Arrives

Low-priority Process A running...
High-priority Process B wakes up!
→ OS: "Sorry A, B is more important"
→ Context switch to Process B (preempt A)

Priority-based scheduling.

4. Explicit Yield

Process voluntarily gives up CPU.

The Cost of Context Switching

Direct Costs

Typical Context Switch Time: 1-10 microseconds (μs)

What happens in those microseconds:
1. Save current registers      → ~500 ns
2. Update page tables          → ~1-2 μs
3. TLB flush                   → ~1-3 μs
4. Load new registers          → ~500 ns
5. Jump to new code            → ~100 ns

Pure overhead: The CPU does zero useful work during this time.

Indirect Costs (Cache Misses)

Process A fills L1/L2/L3 cache with its data
→ Context switch to Process B
Process B fills cache with its data (evicting A's data)
→ Context switch back to Process A
Process A's cache is COLD → Cache misses!

Cache thrashing: Processes keep evicting each other’s data from cache.

Real cost: Can be 10-100x the direct cost due to cache misses.

Context Switch vs. Other Operations

OperationTimeCost
Function call~1 ns✅ Negligible
System call~100 ns✅ Cheap
Thread context switch~1-3 μs⚠️ Moderate
Process context switch~3-10 μs⚠️ Expensive
Cache miss (L3)~40 ns⚠️ Expensive if frequent

Context switches are 1000x more expensive than system calls.

Thread vs. Process Context Switch

Process Context Switch (Expensive)

Process A → Process B
- Save all registers
- Switch page tables (entire virtual memory mapping)
- Flush TLB (translation lookaside buffer)
- Flush some caches
- Security context change

Cost: ~3-10 μs

Thread Context Switch (Cheaper)

Thread A → Thread B (same process)
- Save registers
- ✅ Same page tables (shared memory space)
- ✅ No TLB flush needed
- ✅ Shared cache data

Cost: ~1-3 μs

Threads are cheaper to switch because they share memory.

Reducing Context Switch Overhead

1. Use Fewer Threads

Rule: Threads ≈ CPU cores for CPU-bound work.

2. Use Async I/O

Boost-Asio with co_await avoids context switches for I/O.

3. CPU Affinity

Benefit: Better cache locality, fewer context switches between cores.

Context Switching in Different Scenarios

Scenario 1: Web Server (I/O-Bound)

1000 connections, each thread blocks on I/O
→ Lots of context switches (threads waiting)
→ Solution: Async I/O event loop (Nginx, Node.js)

Scenario 2: Video Game (CPU-Bound)

60 FPS = 16ms per frame
→ Minimize context switches!
→ Solution: Dedicate threads to specific tasks

Scenario 3: Database (Mixed)

Some threads handle queries (CPU-bound)
Some threads wait for disk (I/O-bound)
→ Tune thread pool size carefully

The Big Takeaway

Context Switch in a Nutshell

A context switch is when the Operating-System stops one process/thread and runs another by:

  1. Saving current CPU state (registers, program counter)
  2. Loading next process’s state
  3. Updating Memory mappings (for processes)

When it happens:

  • Time slice expires (every ~10ms)
  • Process blocks (waiting for I/O)
  • Higher priority process arrives

Cost:

  • Direct: 1-10 microseconds (pure overhead)
  • Indirect: Cache misses (can be 10-100x more)

Thread switch < Process switch (threads share memory)

Reduce overhead:

  • Use fewer threads (thread pool)
  • Use async I/O instead of blocking
  • Pin threads to CPU cores

The rule: Context switches enable multitasking but cost performance. For CPU-bound work, use threads ≈ number of cores. For I/O-bound work, use async I/O (Boost-Asio, co_await) to avoid context switches entirely.
