WTF is a Computer?

Let’s build a real, working computer from scratch.

Starting with Logic-Gates, we’ll build:

  1. Memory (registers, RAM)
  2. ALU (arithmetic)
  3. Control unit
  4. Complete CPU
  5. Run actual programs

No magic. No hand-waving. Every piece explained.

This is the answer to “But How Do It Know?”

1. Review: What we have so far

Building blocks

We can build from Transistors:

Basic gates:

  • NOT (inverter)
  • AND, OR, NAND, NOR
  • XOR, XNOR

Arithmetic:

  • Half adder (A + B)
  • Full adder (A + B + Carry)
  • 8-bit adder (chain full adders)

Selection:

  • Multiplexer (choose input)
  • Decoder (activate output)

Memory:

  • SR latch
  • D latch
  • D flip-flop

See: Logic-Gate

2. Building memory

Register (stores one number)

8-bit register = 8 D flip-flops:

         Clock
            │
D[7:0] ─┬───┼───[FF]─── Q[7]
        ├───┼───[FF]─── Q[6]
        ├───┼───[FF]─── Q[5]
        ├───┼───[FF]─── Q[4]
        ├───┼───[FF]─── Q[3]
        ├───┼───[FF]─── Q[2]
        ├───┼───[FF]─── Q[1]
        └───┴───[FF]─── Q[0]

Operations:

  • Clock pulse: Store D into register
  • Always outputs: Current value on Q

Add enable signal:

Enable ──┐
         ├─[AND]── Clock'
Clock ───┘         │
                [FF...]
  • Enable=1, Clock=pulse → Store
  • Enable=0 → Hold (ignore clock)

This is a CPU register!

Register file (multiple registers)

CPU needs many registers:

Register File (8 registers × 8 bits)

Write Data[7:0] ────────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
                        │     │     │     │     │     │     │     │
Write Enable ───[DEC]───┤     │     │     │     │     │     │     │
Write Addr[2:0] ─┘      │     │     │     │     │     │     │     │
                        ▼     ▼     ▼     ▼     ▼     ▼     ▼     ▼
                      [R0]  [R1]  [R2]  [R3]  [R4]  [R5]  [R6]  [R7]
                        │     │     │     │     │     │     │     │
Read Addr A[2:0] ───[MUX]────┴─────┴─────┴─────┴─────┴─────┴─────┘
                     │
                     └──── Read Data A[7:0]

Read Addr B[2:0] ───[MUX]──(same)──── Read Data B[7:0]

Operations:

  • Write: Decoder enables one register, stores Write Data
  • Read A: Multiplexer selects one register → Read Data A
  • Read B: Second multiplexer → Read Data B (two reads at once!)

Example: 8 registers × 8 bits = 64 bits total storage

x86-64 has 16 registers × 64 bits = 1024 bits

See: Register

RAM (Random Access Memory)

Need lots of memory (KB, MB, GB)!

One memory cell (1 bit):

Write Data ───┬─[AND]─┬─[SR Latch]─── Read Data
              │       │
Write Enable ─┴       │
                      │
Read Enable ──────────┴─[Buffer]

Memory array (8 bits × 4 addresses):

Address[1:0] ──[Decoder]─┬─ Enable[0] ─[Cell 0]
                         ├─ Enable[1] ─[Cell 1]
                         ├─ Enable[2] ─[Cell 2]
                         └─ Enable[3] ─[Cell 3]

Write Data[7:0] ─────────────┬──────────────────────
                             │(to all cells)
Read Data[7:0] ──────────────┴──────────────────────
                             (from selected cell)

Scale up:

  • 2-bit address → 4 locations
  • 16-bit address → 65,536 locations (64KB)
  • 32-bit address → 4GB

Modern DRAM: Uses capacitors (smaller), needs refresh.

See: Memory, DRAM

3. Building the ALU

Arithmetic Logic Unit - the brain!

Adder (8-bit)

Built from 8 full adders:

A[7:0] ─┬─[FA]─[FA]─[FA]─[FA]─[FA]─[FA]─[FA]─[FA]─┬─ Sum[7:0]
B[7:0] ─┘   │    │    │    │    │    │    │    │  │
Cin ────────┴────┴────┴────┴────┴────┴────┴────┴──┴─ Cout
            (Carry ripples left)

Problem: Slow (carry ripple delay)

Solution (for now): Accept it. Real CPUs use carry-lookahead.

Subtractor

A - B = A + (-B) = A + (NOT B) + 1

Two’s complement:

NOT B ─── Flip all bits
   +1 ─── Add 1
  = -B ─── Negative of B

Circuit:

A[7:0] ────────────┬─[Adder]─── Result[7:0]
                   │
B[7:0] ──[NOT]─────┤  (invert B if subtracting)
                   │
Sub flag ──────────┴─ Cin (1 for subtract, 0 for add)

If Sub=1:

  • NOT B (flips all bits)
  • Cin=1 (adds 1)
  • Result = A + NOT B + 1 = A - B ✓

Logic operations

AND, OR, XOR: Bit-by-bit operations

A[7:0] ──┬─[AND]─┬─ A & B
B[7:0] ──┤       │
         │       │
         ├─[OR]──┼─ A | B
         │       │
         └─[XOR]─┴─ A ^ B

Example:

A = 10110010
B = 11001100

A AND B = 10000000  (both 1)
A OR  B = 11111110  (either 1)
A XOR B = 01111110  (different)

Shifter

Shift left/right:

Shift Left by 1:
  Input:  10110010
  Output: 01100100  (multiply by 2)

Shift Right by 1:
  Input:  10110010
  Output: 01011001  (divide by 2)

Circuit (barrel shifter):

A[7:0] ──[MUX]── B[7:0]
         │ │
      0  │ └─ A[6:0], 0    (shift left 1)
      1  └─── A[7:1], 0    (shift right 1)

Shift amount selects MUX

Multi-bit shifts: Chain multiple 1-bit shifters.

Complete ALU

Combines all operations:

A[7:0] ─────┬────────────┐
            │            │
B[7:0] ─────┼────────────┤
            │            │
            ├─[Adder]────┤
            ├─[AND]──────┤
            ├─[OR]───────┤
            ├─[XOR]──────┤
            ├─[Shifter]──┤
            │            │
Op[2:0] ────┴─[MUX]──────┴─── Result[7:0]
                │
                ├──── Zero flag (Result == 0)
                ├──── Carry flag
                └──── Negative flag (bit 7)

Operation codes:

Op  │ Operation
────┼────────────
000 │ A + B
001 │ A - B
010 │ A AND B
011 │ A OR B
100 │ A XOR B
101 │ NOT A
110 │ A << 1
111 │ A >> 1

Flags (status outputs):

  • Zero (Z): Result is all zeros
  • Carry (C): Addition overflowed
  • Negative (N): Bit 7 is 1
  • Overflow (V): Signed overflow

See: ALU

4. Building control logic

Instruction format

Simple 16-bit instruction:

┌────────┬────────┬────────┬────────┐
│ Opcode │  Reg1  │  Reg2  │  Reg3  │
│ 4 bits │ 3 bits │ 3 bits │ 6 bits │
└────────┴────────┴────────┴────────┘

Example:
ADD R1, R2, R3  →  R1 = R2 + R3

Binary: 0001 001 010 011 000000
        │    │   │   │
        │    │   │   └─ R3 (src2)
        │    │   └─ R2 (src1)
        │    └─ R1 (dest)
        └─ ADD (opcode 0001)

Instruction set:

Opcode │ Instruction    │ Meaning
───────┼────────────────┼───────────────
0000   │ NOP            │ No operation
0001   │ ADD R1,R2,R3   │ R1 = R2 + R3
0010   │ SUB R1,R2,R3   │ R1 = R2 - R3
0011   │ AND R1,R2,R3   │ R1 = R2 & R3
0100   │ OR R1,R2,R3    │ R1 = R2 | R3
0101   │ XOR R1,R2,R3   │ R1 = R2 ^ R3
0110   │ NOT R1,R2      │ R1 = ~R2
0111   │ MOV R1,R2      │ R1 = R2
1000   │ LOAD R1,[R2]   │ R1 = memory[R2]
1001   │ STORE [R1],R2  │ memory[R1] = R2
1010   │ JMP addr       │ PC = addr
1011   │ JZ addr        │ if (Z) PC = addr
1100   │ JNZ addr       │ if (!Z) PC = addr
1101   │ HALT           │ Stop

Instruction decoder

Converts opcode to control signals:

Instruction[15:12] ──[Decoder]──┬─ ALU_Op[2:0]
  (Opcode)                       ├─ Reg_Write_Enable
                                 ├─ Mem_Read
                                 ├─ Mem_Write
                                 ├─ PC_Inc
                                 ├─ PC_Load
                                 └─ Halt

Truth table (simplified):

Opcode │ Operation │ ALU_Op │ RegWrite │ MemRead │ MemWrite
───────┼───────────┼────────┼──────────┼─────────┼─────────
0001   │ ADD       │ 000    │ 1        │ 0       │ 0
0010   │ SUB       │ 001    │ 1        │ 0       │ 0
1000   │ LOAD      │ ---    │ 1        │ 1       │ 0
1001   │ STORE     │ ---    │ 0        │ 0       │ 1

Built entirely from gates!

Program Counter (PC)

Holds address of next instruction:

        ┌─────────────┐
        │  Register   │
        │  (16 bits)  │
        └─────────────┘
             │
             ├──── Address[15:0]
             │
PC_Inc ──┬───┤
         │   │
PC_Load ─┼───┴─[Adder +1]
         │      [Mux]
         │        │
Jump_Addr────────┘

Operations:

  • PC_Inc: PC = PC + 1 (next instruction)
  • PC_Load: PC = Jump_Addr (jump/branch)

│           │                                       │

│                                    │              │
└────────────────────────────────────┼──────────────┘
                                     ▼
                               Output/Memory

Data path

How data flows:

1. Fetch:

PC → Memory → Instruction Register
PC = PC + 1

2. Decode:

Instruction Register → Decoder → Control Signals

3. Execute:

Register File → (read registers) → ALU → (compute)
ALU → Register File (write back)

4. Memory (if LOAD/STORE):

Register → Memory Address
Memory → Register (LOAD)
Register → Memory (STORE)

Control signals

Decoder generates these signals each cycle:

SignalPurpose
RegWriteEnableWrite ALU result to register
MemReadRead from memory
MemWriteWrite to memory
ALU_Op[2:0]Which ALU operation
PC_Src0=PC+1, 1=Jump address

Example - ADD R1, R2, R3:

1. Fetch: Read instruction from memory[PC], PC++
2. Decode:
   - Opcode = 0001 (ADD)
   - Reg1 = 001 (R1, dest)
   - Reg2 = 010 (R2, src1)
   - Reg3 = 011 (R3, src2)
3. Execute:
   - Read R2 → ALU input A
   - Read R3 → ALU input B
   - ALU_Op = 000 (ADD)
   - ALU computes A + B
4. Writeback:
   - RegWriteEnable = 1
   - Write ALU result → R1

All happens in one clock cycle! (Simple CPU)

6. Running a program

Example program: Sum two numbers

Assembly code:

Machine code (binary):

Address │ Binary                │ Instruction
────────┼───────────────────────┼──────────────
0000    │ 0111 001 000 00000101 │ MOV R1, 5
0001    │ 0111 010 000 00000011 │ MOV R2, 3
0002    │ 0001 011 001 010 0000 │ ADD R3,R1,R2
0003    │ 1101 000 000 00000000 │ HALT

Load into memory:

Memory:
[0000] = 0111 001 000 00000101
[0001] = 0111 010 000 00000011
[0002] = 0001 011 001 010 0000
[0003] = 1101 000 000 00000000

Execution trace

Clock cycle 0:

PC = 0000
Fetch: Instruction = 0111 001 000 00000101
Decode: MOV R1, 5
Execute: R1 = 5
PC = 0001

Clock cycle 1:

PC = 0001
Fetch: Instruction = 0111 010 000 00000011
Decode: MOV R2, 3
Execute: R2 = 3
PC = 0002

Clock cycle 2:

PC = 0002
Fetch: Instruction = 0001 011 001 010 0000
Decode: ADD R3, R1, R2
Execute:
  Read R1 = 5 → ALU input A
  Read R2 = 3 → ALU input B
  ALU: 5 + 3 = 8
  Write R3 = 8
PC = 0003

Clock cycle 3:

PC = 0003
Fetch: Instruction = 1101 000 000 00000000
Decode: HALT
Execute: Stop clock

Final state:

R1 = 5
R2 = 3
R3 = 8  ✓ Correct!

Program executed in 4 clock cycles!

More complex example: Loop

Assembly code:

Execution:

Cycle  │ R1 │ R2 │ R3 │ Z │ Action
───────┼────┼────┼────┼───┼────────────
0      │ 0  │ 5  │ -  │ - │ MOV R1, 0
1      │ 0  │ 5  │ -  │ - │ MOV R2, 5
2      │ 1  │ 5  │ -  │ - │ ADD R1 (1)
3      │ 1  │ 5  │ -4 │ 0 │ SUB R3
4      │ 1  │ 5  │ -4 │ 0 │ JNZ (jump!)
5      │ 2  │ 5  │ -4 │ 0 │ ADD R1 (2)
6      │ 2  │ 5  │ -3 │ 0 │ SUB R3
7      │ 2  │ 5  │ -3 │ 0 │ JNZ (jump!)
...
18     │ 5  │ 5  │  0 │ 1 │ SUB R3
19     │ 5  │ 5  │  0 │ 1 │ JNZ (don't jump, Z=1)
20     │ 5  │ 5  │  0 │ 1 │ HALT

Loop executed 5 times!

This is programming at the hardware level.

7. Making it faster

The problem: Everything is slow

Our simple CPU:

  • 1 instruction per clock cycle
  • Slow clock (gates take time)
  • Memory is slow

Modern CPUs:

  • 5+ GHz clock (5 billion cycles/second)
  • 4+ instructions per cycle
  • Out-of-order execution
  • Branch prediction

How?

Pipelining

Idea: Overlap instructions like assembly line.

Without pipeline:

Instruction 1: [Fetch][Decode][Execute][Memory][Writeback]
Instruction 2:                                             [Fetch][Decode][Execute][Memory][Writeback]

Total: 10 cycles for 2 instructions

With pipeline:

Instruction 1: [Fetch][Decode][Execute][Memory][Writeback]
Instruction 2:        [Fetch][Decode][Execute][Memory][Writeback]
Instruction 3:               [Fetch][Decode][Execute][Memory][Writeback]
Instruction 4:                      [Fetch][Decode][Execute][Memory][Writeback]

Total: 8 cycles for 4 instructions

Throughput: 1 instruction per cycle (after filling pipeline)!

Problem: Hazards (dependencies, branches)

See: CPU-Pipeline

Caching

Problem: Memory (DRAM) is 100× slower than CPU.

Solution: Cache (SRAM) - small, fast memory on CPU.

Hierarchy:

L1 Cache:  32 KB, 4 cycles,  per-core
L2 Cache: 256 KB, 12 cycles, per-core
L3 Cache:  16 MB, 40 cycles, shared
RAM:       16 GB, 100 cycles
Disk:       1 TB, 10,000,000 cycles

Locality:

  • Temporal: Access again soon → Keep in cache
  • Spatial: Access nearby → Fetch cache line (64 bytes)

Example:

See: Cache, Memory-Hierarchy

Superscalar execution

Idea: Execute multiple instructions at once.

Modern CPU has multiple ALUs:

         Fetch
          │
        Decode
          │
     ┌────┼────┐
     │    │    │
   ALU1 ALU2 Load/Store
     │    │    │
     └────┼────┘
          │
      Writeback

Example:

Both execute simultaneously!

See: Superscalar

Branch prediction

Problem: Branches stall pipeline.

Don’t know if branch taken until Execute stage!

Solution: Predict!

Simple predictor:

  • Predict taken: If branch usually loops
  • Predict not taken: If branch usually falls through

Modern: Complex predictors (98% accuracy!)

Wrong prediction: Flush pipeline, restart (expensive)

See: Branch-Prediction

Out-of-order execution

Idea: Execute instructions in different order for efficiency.

Program order:

Out-of-order:

1. Issue LOAD R1, [R2]  (starts memory read)
2. Issue MUL R5, R6, R7 (execute while waiting!)
3. Wait for LOAD to finish
4. Issue ADD R3, R1, R4

Executes faster!

See: Out-of-Order-Execution

The big picture

From gates to programs

The complete stack:

Physics (semiconductors)
  ↓
Transistors (switches)
  ↓
Logic Gates (AND, OR, NOT)
  ↓
Combinational Logic (Adders, Muxes)
  ↓
Sequential Logic (Registers, Memory)
  ↓
Functional Units (ALU, Control)
  ↓
CPU (Fetch-Decode-Execute)
  ↓
Machine Code (binary instructions)
  ↓
Assembly Language (mnemonics)
  ↓
C/C++/Rust (compiled)
  ↓
Python/JavaScript (interpreted)
  ↓
Your Program

Every layer hides complexity of layer below.

Transistor count

Our simple 8-bit CPU:

  • 8 registers × 8 bits = 64 flip-flops × 6 transistors = ~400
  • ALU: ~1000 transistors
  • Control logic: ~500 transistors
  • Total: ~2000 transistors

Intel 4004 (1971):

  • First commercial CPU
  • 2,300 transistors
  • 4-bit
  • 740 kHz
  • Could execute our simple programs!

Apple M2 Ultra (2023):

  • 134 billion transistors
  • 24 cores × 64-bit
  • 3.5 GHz
  • 67,000,000× more transistors!

But fundamentally the same design.

Clock speed evolution

1971: Intel 4004       740 kHz
1985: Intel 386          16 MHz
1993: Pentium           60 MHz
2000: Pentium 4        1.5 GHz
2006: Core 2 Duo       2.4 GHz
2020: Ryzen 9 5950X    4.9 GHz
2024: Intel Core Ultra 5.8 GHz

~10,000× faster in 50 years

Why not 100 GHz?

  • Power consumption (P ∝ f × V²)
  • Heat dissipation
  • Speed of light (signals can’t cross chip in 1 cycle!)

Solution: More cores, not faster clock.

Key takeaways

It’s all gates

Every operation - addition, memory access, control flow - is implemented with logic gates.

Gates are built from transistors.

Transistors are just switches.

Programming:

Reality:

1. Compiler → ADD instruction
2. CPU fetches instruction from memory (gates)
3. Decoder activates control signals (gates)
4. Registers output A and B (flip-flops = gates)
5. ALU adds A + B (full adders = gates)
6. Result written to register (flip-flops = gates)

All gates. All switches. All in nanoseconds.

The abstraction tower

Each layer provides interface to layer above:

  • Gates: AND, OR, NOT
  • Arithmetic: ADD, SUB, MUL
  • CPU: Instructions (fetch-decode-execute)
  • Assembly: Mnemonics (ADD, MOV, JMP)
  • C: Functions, variables, control flow
  • Python: Objects, methods, high-level

You write Python. CPU executes gates.

The tower connects them.

The answer

”But How Do It Know?”

It doesn’t know anything.

It’s switches, following rules:

  • IF input A AND input B are high, output is high
  • IF opcode is 0001, enable ALU add operation
  • IF clock pulse, store data in flip-flop

No understanding. No intelligence. Just rules.

But from billions of simple rules, emerges:

  • Operating systems
  • Web browsers
  • Video games
  • Artificial intelligence
  • This text you’re reading

All from switches.

All from gates.

All from sand.

The miracle

Modern computers:

  • Billions of transistors
  • Switching billions of times per second
  • Running millions of programs simultaneously
  • All perfectly synchronized
  • 99.9999% reliable
  • Fits in your pocket

And it all works because:

  1. We can make tiny switches (transistors)
  2. We can combine switches into gates
  3. We can combine gates into circuits
  4. We can combine circuits into CPUs
  5. We can program CPUs with instructions

Each layer builds on the last.

Each layer hides complexity.

That’s how computers work.

That’s how it knows.

It’s switches all the way down.
