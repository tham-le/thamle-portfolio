WTF are Safe IT Practices?

Lessons written in blood (and compromised credentials)

After Sha1-Hulud compromised 700+ npm packages and stole credentials from 25,000+ repositories, the security community asked: “How do we prevent this?” The uncomfortable answer: you can’t prevent it completely. But you can make it much harder, detect it faster, and limit the damage when it happens.

Safe IT practices aren’t about perfect security. They’re about defense in depth, where multiple layers of protection ensure that no single failure cascades into total compromise.

This guide synthesizes lessons from real attacks: Sha1-Hulud, SolarWinds, Log4Shell, CodeCov, and dozens more. These aren’t theoretical recommendations. They’re battle-tested strategies that would have stopped or contained actual breaches.

The Fundamental Principles

1. Assume Breach

Traditional mindset: “Our perimeter is secure, so we’re safe.”

Reality: Attackers are already inside. They compromised a dependency, a developer’s laptop, or a CI/CD token. Now what?

Defense in depth:

Layer 1: Prevention (stop attacks) ✗ Attacker bypassed
Layer 2: Detection (find attacks) ✗ Attacker evaded
Layer 3: Containment (limit damage) ← This is where you win
Layer 4: Recovery (restore service)
Layer 5: Learning (improve defenses)

If any single layer stops the attack, you win. Attackers must defeat every layer.

2. Principle of Least Privilege

Every system, process, and person should have the minimum permissions needed, no more.

Why this matters: Sha1-Hulud stole tokens and used them to create repositories, publish packages, and establish persistence. Narrow tokens would have limited this.

3. Defense in Depth

Single points of failure are unacceptable. Every critical operation requires multiple independent checks:

Deploying to production requires:
✓ Code review approved
✓ Tests passing
✓ Security scan passed
✓ Manual approval from senior engineer
✓ Deployment during business hours only
✓ Gradual rollout with monitoring

If attackers compromise one step, others still protect you.

4. Zero Trust

Trust nothing by default. Verify everything constantly:

Traditional: "This request comes from inside our network, so it's trusted"
Zero Trust: "This request must prove its identity, authorization, and legitimacy regardless of origin"

Pre-commit Protection

Separate Environments

Development:
  ├─ Uses fake/test credentials
  ├─ No access to production data
  └─ No access to production systems

Staging:
  ├─ Uses production-like credentials
  ├─ Access to sanitized production data copy
  └─ No access to actual production systems

Production:
  ├─ Uses real credentials
  ├─ Access restricted to minimal number of people
  └─ All access logged and monitored

Why: If Sha1-Hulud compromises a developer’s laptop, it gets dev credentials, not production.

Defense Checklist

Immediate (Do Today)

  • Enable 2FA on all critical accounts
  • Audit dependencies (npm audit, snyk test)
  • Check for Sha1-Hulud IOCs (bun_environment.js, suspicious repos)
  • Review GitHub Actions runners
  • Rotate credentials if any suspicion of compromise

Short-Term (This Week)

  • Install pre-commit hooks for secret detection
  • Lock dependency versions (commit package-lock.json)
  • Set up automated security scanning in CI/CD
  • Document incident response procedures
  • Review and minimize dependencies

Medium-Term (This Month)

  • Implement OIDC for CI/CD (no static tokens)
  • Set up private npm registry with scanning
  • Migrate to short-lived credentials
  • Implement network restrictions on build environments
  • Conduct supply chain security training

Long-Term (This Quarter)

  • Full supply chain security audit
  • Implement comprehensive monitoring and alerting
  • Establish credential rotation automation
  • Deploy hardware security keys for critical access
  • Regular penetration testing and red team exercises

The Hard Truths

  1. Perfect security doesn’t exist. You will be compromised. Focus on detection and containment.

  2. Convenience fights security. Every security measure makes development slightly harder. This is the price of safety.

  3. Developers are not security experts. They need good defaults, clear guidelines, and automated tools.

  4. Trust is a vulnerability. Every dependency, every maintainer, every service is a potential attack vector.

  5. Attackers only need to win once. You need to win every time.

  6. Security is ongoing. Not a checklist, not a one-time fix. Continuous vigilance.

Key Takeaways

  • Defense in depth: Multiple independent layers of protection
  • Assume breach: Plan for compromise, focus on containment
  • Least privilege: Minimal permissions, short-lived credentials
  • Continuous monitoring: Detect compromise quickly
  • Incident response: Have a plan before you need it
  • Learn from real attacks: Sha1-Hulud teaches many lessons
  • Security is a team responsibility: Every developer must understand basics

The fundamental reality: Modern software development is built on trust at scale. Managing that trust through technical controls, monitoring, and rapid response is how we stay secure in an insecure world.
