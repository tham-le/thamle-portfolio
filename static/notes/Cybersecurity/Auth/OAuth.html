WTF is OAuth 2.0?

The protocol that lets you say “yes, this app can read my calendar” without handing it your password.

You click “Sign in with Google” on some random website. Google pops up, you approve, and suddenly that website has your email address. How did that happen without the website ever touching your password?

That’s OAuth 2.0. And once you see the system design behind it, you’ll realize it’s solving a genuinely elegant problem.

The Core Analogy

Think of OAuth 2.0 like a hotel key card system.

You check in at the front desk (the authorization server). You show your passport (your actual credentials). The front desk doesn’t give the valet your passport. They give you a key card (an access token) that only opens specific doors, only works for a limited time, and can be revoked if lost.

The valet (the third-party app) gets the key card. They can park your car. They cannot open your room safe. They cannot check you out. The key card is all they get, scoped to exactly what they need.

That’s the entire mental model. Now let’s unpack the architecture.

The Problem It Solves

Before OAuth, the only way for a third-party app to access your data elsewhere was to give it your username and password directly. The app would log in as you. Full access. Forever.

This is a beautiful dumpster fire:

  • The third-party app can do anything your account allows, not just what you intended
  • If you change your password, every integration breaks
  • If the app gets compromised, attackers have your real credentials
  • You have no mechanism to revoke access to just one app without nuking all of them

OAuth 2.0 replaces “give the app your password” with “give the app a scoped, expiring, revocable token.” The password never leaves the authorization server.

The Four Actors

RFC 6749 defines four roles. You need these before the flows make sense.

Resource Owner is you, the user. You own the data. You decide who gets access to it.

Client is the third-party application that wants access. A web app, a mobile app, a backend service. The client never sees your password.

Authorization Server is the trusted gatekeeper. It verifies your identity, gets your consent, and issues tokens. Google’s auth server, GitHub’s auth server, your company’s Keycloak instance. This is where authentication logic lives.

Resource Server is the API holding your protected data. Google Calendar API, GitHub API, your company’s user service. It validates incoming tokens and decides whether to serve the request.

One auth server, many resource servers

A single authorization server can issue tokens accepted by multiple resource servers. Google’s auth server mints tokens for Gmail, Drive, Calendar, YouTube. The services are separate. The trust chain runs through the auth server.

Here’s the system at a glance:

You (Resource Owner)
    |
    | (1) You trust the Auth Server. You log in there.
    v
Authorization Server  <-----> Client (Third-party App)
    |                              |
    | (2) Issues access token      | (3) Presents token
    v                              v
                          Resource Server
                          (your protected data)

The client never asks you for credentials directly. You authenticate with the auth server, which delegates access to the client via a token.

The Token Vocabulary

Access Token is the key card. Short-lived (often 1 hour). The client presents this to the resource server with every request. The resource server validates it and serves data. RFC 6749 intentionally does not specify the format. Implementations use opaque strings or JSON Web Tokens.

Refresh Token is a token for getting new access tokens without making the user log in again. Longer-lived. Never sent to the resource server, only back to the authorization server. Think of it as a “reissue key card” voucher stored safely server-side.

Authorization Code is a short-lived, single-use intermediate code. A claim ticket you exchange for the actual access token. It exists for a very specific security reason explained below.

The Flows (Grant Types)

OAuth 2.0 is a framework, not a single protocol. It defines multiple “grant types”, each for a different context.

Authorization Code Flow (The One You Should Use)

Use this for web apps, mobile apps, SPAs. It’s designed so the access token never touches the browser.

User     Browser     Your App (Client)     Auth Server     Resource Server
 |          |               |                   |                |
 |-click--->|               |                   |                |
 |          |---redirect--->|                   |                |
 |          |               |--build auth URL-->|                |
 |          |               |  client_id,       |                |
 |          |<--redirect----|  scope,            |                |
 |          |   to auth     |  redirect_uri,     |                |
 |          |   server      |  state (CSRF)      |                |
 |          |                                    |                |
 |<---login page---------------------------------|                |
 |--creds-->|------------------------------------------login---->|
 |          |<---------redirect with auth code-------(front ch.)--|
 |          |               |                   |                |
 |          |---code------->|                   |                |
 |          |               |--POST /token----->| (back channel) |
 |          |               |  code,            |                |
 |          |               |  client_secret    |                |
 |          |               |<--access_token----|                |
 |          |               |   refresh_token   |                |
 |          |               |                   |                |
 |          |               |--GET /data, Bearer token---------->|
 |          |               |<--protected data------------------|

The auth code travels through the browser (front channel, exposed). The access token is exchanged server to server (back channel, never exposed to the browser).

Why the two-step? Why not get the token directly?

Because browsers leak things. History, logs, referrer headers, browser extensions. The auth code is useless without the client_secret, which only lives on your server. Even if an attacker intercepts the auth code, they can’t exchange it without the secret.

The state parameter is CSRF protection. Your app generates a random value, sends it with the auth request, and verifies it comes back unchanged on the callback. If it doesn’t match, abort.

Client Credentials Flow (Machine to Machine)

No user involved. A backend service authenticates as itself.

Your Service (Client)           Auth Server
        |                            |
        |--POST /token------------->|
        |  client_id                |
        |  client_secret            |
        |  grant_type=client_credentials
        |<--access_token------------|
        |                            |
        |--API call with token------>Resource Server

Use this for microservice-to-microservice calls, batch jobs, CI/CD pipelines. Any time there’s no human in the loop.

No refresh tokens in this flow

Why would you need one? The service has its own credentials and can get a new access token directly whenever needed.

PKCE (The Mobile App Fix)

Public clients like mobile apps and SPAs can’t keep a client_secret secret. The binary can be decompiled. The JavaScript can be read. So how do you use the Authorization Code flow safely?

PKCE (Proof Key for Code Exchange, pronounced “pixie”) replaces the static client_secret with a per-request dynamic secret:

1. App generates random code_verifier (43-128 random chars)
2. App hashes it: code_challenge = base64url(SHA-256(code_verifier))
3. App sends code_challenge in the auth request (safe to expose)
4. Auth server stores code_challenge
5. App sends code_verifier when exchanging the auth code for a token
6. Auth server verifies SHA-256(code_verifier) == stored code_challenge

Even if an attacker steals the auth code, they don’t have the code_verifier. And the code_verifier is never transmitted until the exchange step, so it’s never in browser history.

PKCE is now recommended for all clients

Not just mobile. The OAuth 2.0 Security Best Current Practice recommends PKCE for all Authorization Code flows, even confidential web apps with client secrets. It’s layered defense.

Scopes: The Access Control Layer

Scopes are how OAuth limits what a token can do. When the client builds the auth URL, it declares what it wants:

https://accounts.google.com/authorize?
  client_id=my-app
  &scope=email profile calendar.readonly
  &response_type=code
  &redirect_uri=https://myapp.com/callback
  &state=abc123

The user sees a consent screen listing exactly what the app is requesting. The issued token is then limited to those scopes. The resource server enforces them on every request.

This is the “limited doors” part of the hotel analogy. A token with calendar.readonly scope cannot modify your calendar. The resource server checks the scope claim and rejects any out-of-scope request.

The Token Lifecycle in Production

Access Token  ------>  short-lived (15 min to 1 hour)
Refresh Token ------>  long-lived (days to weeks), rotated on use
Auth Code     ------>  single-use, max ~10 minutes

When the access token expires, the client silently uses the refresh token to get a new one. No user interaction needed. When the refresh token is used, a new refresh token is issued and the old one is invalidated (this is called rotation). If the refresh token itself expires or gets revoked, the user logs in again.

Bearer tokens: possession equals access

OAuth access tokens are bearer tokens. Whoever holds one gets access. No additional proof required. This is why TLS is non-negotiable for all OAuth communication. A token leaked over plain HTTP is game over.

What OAuth 2.0 Does NOT Do

Here’s the most important architectural boundary in OAuth: it is an authorization framework, not an authentication protocol.

OAuth answers: “Does this client have permission to access these resources?”

It does not answer: “Who is the user?”

There is no standard field in an OAuth access token that says “this is Alice.” There is no standard endpoint in RFC 6749 for fetching user identity. This is by design. Authorization and authentication are different concerns, and keeping them separate keeps the spec clean.

If you need identity (“who is this person?”), you need something layered on top of OAuth. That something is OpenID Connect.

Common Misconceptions

”OAuth is for login”: No. OAuth is for delegation. It lets apps act on your behalf. OpenID Connect is the layer that adds login/identity on top.

”The access token is always a JWT”: No. RFC 6749 says access tokens are opaque strings to the client. Many providers issue JWTs because it lets resource servers validate tokens locally without calling back to the auth server. But it’s not required.

”Implicit flow is fine for SPAs”: No. Implicit flow is deprecated. It returns the access token directly in the URL fragment, which is exposed to the browser. Use Authorization Code with PKCE instead.

What’s Next

You now understand OAuth 2.0 as a system: four actors, three token types, two main flows, one scope system.

But there’s still the identity question. OAuth gives the client an access token. It says nothing about who approved it. For that, we layer OpenID Connect on top, which adds an ID token and a standardized way to ask “who is this user?”
