WTF is OpenID Connect?

OAuth 2.0 tells you what a token can do. OpenID Connect tells you who the person is.

If you read OAuth 2.0, you know it deliberately doesn’t answer the identity question. You can have a valid OAuth access token and still have no idea which human being approved it. That’s fine for pure API delegation. It’s a problem the moment you need to build a login system.

OpenID Connect (OIDC) is the thin layer on top of OAuth 2.0 that fixes this. It adds exactly one new token, a handful of standard claims, and a discovery mechanism. The OAuth machinery underneath stays exactly the same.

The Core Analogy

If OAuth 2.0 is the hotel key card system, OpenID Connect is the hotel ID badge they put on the same key card.

The key card (access token) still opens doors. But now there’s a badge attached (the ID token) that says “this person is Alice, room 412, checked in on Monday.” The concierge can read the badge directly without calling the front desk. The badge is cryptographically signed, so it can’t be forged.

Same infrastructure. New data. Different purpose.

The Problem It Solves

Say you’re building a web app and you want “Log in with Google.” You implement the OAuth Authorization Code flow. You get an access token. Now what?

You could call the Google People API with your token to fetch the user’s profile. But:

  • That’s an extra HTTP round trip on every login
  • Google’s People API response format is Google-specific. GitHub’s is different. Okta’s is different. Every provider has a different schema.
  • You’re using an access token (authorization) to answer an identity question. That’s a category error.

OIDC solves this by standardizing the identity response. Every OIDC provider returns the same thing: a JWT called an ID token with a standardized set of claims. You decode it locally. No extra API call needed.

The One New Thing: The ID Token

Everything in OIDC flows from one addition: the ID token.

It’s a JWT returned alongside the access token during the Authorization Code flow. It’s always a JWT (unlike access tokens, which can be opaque strings). It’s cryptographically signed by the authorization server, so you can verify it locally using the server’s public key.

Let’s decode the claims that matter architecturally:

iss (Issuer) is the URL of the authorization server that issued this token. Your app verifies this matches the provider you trust.

sub (Subject) is the stable, unique identifier for this user at this provider. This is the user’s “ID” in the OIDC world. It’s opaque and persistent. Use sub as your primary key when storing users, not email (emails change).

aud (Audience) is your app’s client_id. This ensures the token was issued for your app specifically. You verify this to prevent token reuse across apps.

exp (Expiration) is a Unix timestamp. Reject any token past this time.

nonce is a random value your app generates and includes in the auth request. The server echoes it back in the ID token. You verify it matches what you sent. This prevents replay attacks.

Validate all of these, every time

Don’t just decode the JWT and trust it. Verify the signature. Check iss, aud, exp, nonce. A JWT that fails any of these checks is invalid, full stop.

How OIDC Plugs Into OAuth

OIDC doesn’t replace the OAuth flow. It extends it with one change: add openid to the scope.

GET /authorize?
  client_id=my-app
  &response_type=code
  &scope=openid email profile    <-- the magic word is "openid"
  &redirect_uri=https://myapp.com/callback
  &state=random-csrf-value
  &nonce=random-replay-prevention-value

The openid scope is the signal. When the authorization server sees it, it knows this is an OIDC request, not a plain OAuth request. The token endpoint response now includes both an access_token and an id_token:

The id_token is a JWT you decode locally to get the user’s identity. The access_token is still what you use to call APIs. They serve different purposes.

The Actor Renaming

OIDC introduces a naming shift that’s worth knowing:

OAuth 2.0 termOIDC term
Authorization ServerOpenID Provider (OP)
ClientRelying Party (RP)

The functionality is identical. The names reflect the shifted focus. In OIDC, the emphasis is on the identity provider relationship, not just token issuance.

The UserInfo Endpoint

Sometimes the ID token doesn’t contain all the user data you need. Either because the provider limits token size, or because you requested claims that aren’t in the default set.

Discovery: The .well-known Endpoint

OIDC mandates that providers publish their configuration at a standardized URL:

https://{provider}/.well-known/openid-configuration

For Google: https://accounts.google.com/.well-known/openid-configuration

This returns a JSON document with everything a client needs to talk to this provider:

The jwks_uri is especially important: it’s where you fetch the public keys needed to verify ID token signatures. Well-behaved clients fetch this, cache it, and refresh it when a new kid (key ID) appears in a token header.

This discovery mechanism means you can write a single OAuth/OIDC client library that works with any OIDC-compliant provider just by knowing the issuer URL. That’s the killer feature for SSO integrations.

The Clean Separation

Here’s the architectural insight that makes OIDC elegant:

The ID token answers “who is this user?” for the client (your app). You use it to establish a session, render “Welcome, Alice”, store a user record.

The access token answers “what can this client do?” for the resource server. The resource server doesn’t care about Alice’s name. It cares whether this token has the right scopes.

The ID token is never sent to the resource server. The access token is never used to identify the user on the client. Each token has one job.

Real production pattern

User logs in with Google via OIDC. Your app gets an id_token with sub=110169484474386276334. You look up or create a user record in your database keyed on that sub. You create your own session cookie (a random token you control). For the rest of the user’s session, you use your own session, not the OAuth tokens. The OIDC flow is only needed at login time.

What OIDC Doesn’t Do

OIDC handles authentication (“who is this person?”) and hands you claims about them. It does not handle:

  • Authorization inside your app. Whether Alice can see a specific page is still your problem.
  • Session management across apps. OIDC defines logout specs, but SSO session invalidation across multiple relying parties is notoriously painful in practice.
  • Fine-grained permissions. OIDC scopes (openid, email, profile) are coarse. For row-level or resource-level permissions, you need a separate authorization system.

OAuth vs OIDC: The One-Line Summary

OAuth 2.0: “Here’s a token. It lets you do these things.”

OpenID Connect: “Here’s a token. It lets you do these things. And here’s a signed certificate proving which human approved it.”

OIDC is OAuth with identity. Same pipes. New data. Different question answered.

What’s Next

You now understand the full authentication and authorization picture at the protocol level. OAuth 2.0 handles delegation and access control. OIDC layers identity on top.

The next question in real systems: what format should access tokens use? Opaque strings require the resource server to call back to the auth server to validate them (introspection). JSON Web Tokens let resource servers validate locally using public key cryptography. That trade-off has real performance and security implications worth understanding.
