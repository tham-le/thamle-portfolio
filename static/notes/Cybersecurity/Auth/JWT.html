WTF is JWT?

A tamper-evident envelope that anyone can open but nobody can fake.

You’ve seen JWTs. That opaque blob starting with eyJ in your Authorization header, your browser’s cookie jar, or your OAuth token response. Everyone uses them. Most explanations stop at “it’s a signed JSON object.” That’s like saying TCP is “a way to send bytes.” Technically true, useless in practice.

Let’s go deeper. Structure, cryptography, verification, and the attacks that have burned real production systems, including a cluster of CVEs in early 2026 that are still being patched.

The Core Analogy

Think of a JWT like a government-issued ID card.

The card has your name, photo, expiry date, and a holographic seal. Anyone can read it. Nobody can forge the seal without the government’s secret equipment. The bouncer at the door doesn’t call the government to verify every ID. They check the seal themselves, right there, in milliseconds.

That’s the entire value proposition of a JWT. Self-contained. Verifiable locally. No phone-home required.

The Three-Part Structure

A JWT is always three base64url-encoded chunks separated by dots:

HEADER.PAYLOAD.SIGNATURE

None of this is encrypted by default. The dots in your eyJ blob are just base64url. Decode them and you get readable JSON.

JWTs are NOT secret by default

Base64url is encoding, not encryption. Anyone who gets your JWT can decode and read the payload. Do not put passwords, PII, or secrets in a JWT unless you’re using JWE (JSON Web Encryption). In practice, most JWTs you encounter are signed but not encrypted.

Part 1: The Header

alg declares the signing algorithm. typ is always JWT. Sometimes you’ll also see kid (Key ID), which tells the verifier which specific key to use when the issuer rotates keys:

The header is attacker-controlled input

The header is not signed. It is part of what gets signed, but before verification happens, your library reads it to decide how to verify. This is the source of the most devastating JWT attack class. More on this below.

Part 2: The Payload (Claims)

RFC 7519 defines seven registered claims, the standard fields every JWT library understands:

ClaimNameMeaning
issIssuerWho created this token
subSubjectWho this token is about (usually a user ID)
audAudienceWho this token is intended for
expExpirationUnix timestamp after which the token is invalid
iatIssued AtUnix timestamp when the token was created
nbfNot BeforeUnix timestamp before which the token is invalid
jtiJWT IDUnique identifier for this specific token

Everything beyond those seven is a custom claim, like scope, email, or role. You can add anything. The receiver will either use them or ignore them.

All timestamps are Unix seconds, not milliseconds. A common bug.

Part 3: The Signature

This is where the tamper-evidence comes from. The signature is computed over the encoded header and payload:

signature = sign(
  algorithm,
  secret_or_private_key,
  base64url(header) + "." + base64url(payload)
)

If you change a single character in the header or payload, the signature becomes invalid. The receiver recomputes this check and rejects anything that doesn’t match.

The Two Signing Modes

This is the most important thing to understand about JWT cryptography.

Symmetric: HS256 (HMAC + SHA-256)

One secret key. Used for both signing and verification. Both parties must share it in advance.

sign:   HMAC-SHA256(secret, header + "." + payload)  →  signature
verify: HMAC-SHA256(secret, header + "." + payload)  →  must match signature

When to use it: when the same system that issues tokens also verifies them, like a monolith where your auth service and your API live in the same process. The secret never needs to leave your own infrastructure.

When not to use it: when multiple independent services need to verify tokens. Every service needs the same secret, which means the secret has to be distributed. Distributing secrets is a security and operations nightmare.

Asymmetric: RS256 (RSA + SHA-256)

Two keys. Private key signs. Public key verifies. Only the issuer needs the private key.

sign:   RSA-sign(private_key, header + "." + payload)  →  signature
verify: RSA-verify(public_key, header + "." + payload, signature)  →  valid/invalid

When to use it: in distributed systems where many services need to verify tokens but only one service (the auth server) should be able to issue them. The public key is published at a JWKS endpoint (/.well-known/jwks.json). Any service can fetch it and verify tokens locally. The private key stays on the auth server, and only the auth server can create valid tokens.

ES256 (ECDSA + SHA-256) is the modern alternative to RS256. Same asymmetric model, smaller signatures, faster verification. Use it for new systems if your library supports it well.

The Attacks (This Is Where It Gets Interesting)

JWT has a remarkable attack history. Not because the spec is broken, but because the spec left enough flexibility that implementations got it badly wrong, repeatedly, across many languages.

Attack 1: alg: none

The spec defines an algorithm value "none" for “unsecured JWTs.” The idea was for use in contexts where integrity has been verified by other means. In practice, many early libraries accepted alg: none and skipped signature verification entirely.

The attack is trivial. Take any valid JWT. Decode the header. Change "alg": "RS256" to "alg": "none". Modify the payload however you like ("role": "admin"). Remove the signature. Send it.

A vulnerable library reads alg: none, thinks “no signature needed,” skips verification, and trusts your payload.

Fix: never accept alg: none in production. Pin the algorithm at the library configuration level. Do not let tokens dictate their own verification method.

Attack 2: RS256 to HS256 Algorithm Confusion

This one is subtle and devastating. It was discovered in 2015, and variants are still appearing in new CVEs. In Q1 2026 alone, Hono (CVE-2026-22817, CVSS 8.2), HarbourJwt (CVE-2026-23993), and jose-swift were all found vulnerable.

The setup: your server uses RS256. It verifies tokens using its RSA public key. The public key is, well, public. Attackers can fetch it from your JWKS endpoint.

The attack: an attacker crafts a token with "alg": "HS256" in the header, signs it with HS256 using your RSA public key as the HMAC secret, and sends it. Your library reads alg: HS256, switches to HMAC verification, uses the key argument (your public key) as the HMAC secret, and the signature validates. The attacker’s forged token is accepted as legitimate.

The math works because HMAC with a known key is trivially computable. The RSA public key was meant for verification only, but it becomes the HMAC signing secret. Since you published the public key, the attacker has everything they need.

Fix: never trust the alg field from the token. Pin the algorithm in your configuration. Every modern library has a parameter for this:

In every language: pass an explicit algorithm allowlist. Never let the token header decide.

Attack 3: Weak HMAC Secrets

If you use HS256, the security of every token you ever issue depends entirely on the strength of your secret. A weak secret can be brute-forced offline: an attacker captures one valid token, runs it through hashcat with a wordlist, and recovers the secret. Then they can forge any token they want.

Fix: use a cryptographically random secret of at least 256 bits. Never use a human-readable string, a company name, “secret”, “password”, or anything from a wordlist. If you can, prefer RS256 so there’s no shared secret to protect.

Attack 4: Missing Claim Validation

The signature proves the token wasn’t tampered with. It does not prove the token is meant for you, is still valid, or came from the issuer you trust. Those checks are your job, and they must happen after signature verification.

Missing aud check: a token issued for api.payments.example.com gets accepted by api.analytics.example.com. An attacker with a valid token for one service can access another.

Missing exp check: expired tokens are accepted forever.

Missing iss check: tokens from a different issuer (maybe a test environment, maybe a compromised auth server) are accepted. Keycloak CVE-2026-23552 was exactly this: missing iss validation caused cross-realm token acceptance.

Fix: validate all of them, every time. iss, aud, exp, nbf. No exceptions.

Attack 5: kid Injection

The kid (Key ID) header tells your library which key to use for verification. Some naive implementations use kid directly in a file path or database query:

An attacker sets kid to "../../../etc/passwd" or "../../../dev/null". Path traversal. Or they use a SQL injection payload if kid is used in a query.

Fix: kid should be treated as an opaque identifier looked up in a fixed key registry, never interpolated into file paths or queries.

Local Validation vs Introspection

JWTs enable local validation: the resource server verifies the token without calling back to the auth server. This is the core performance benefit, no network round trip on every request.

The trade-off is revocation. If you revoke a JWT at the auth server, resource servers using local validation won’t know until the token expires naturally. A stolen 1-hour token is valid for up to 1 hour even after revocation.

Two ways to handle this in production:

Short lifetimes: issue access tokens with 5-15 minute expiry. Stolen tokens have a narrow window. Pair with refresh tokens so users aren’t constantly re-authenticating.

Hybrid approach: use local validation for the common case (99.9% of requests). On cache miss or high-value operations, fall back to introspection (RFC 7662) to check the current token state. Revocation takes effect within one cache TTL.

The right expiry for your system

There’s no universal answer. 15 minutes is a common default for access tokens. If you have strict revocation requirements (security incident response, medical systems, financial systems), you need either short-lived tokens or introspection on every request. If you’re building a remote desktop SDK, an attacker with a stolen session token could hijack an active desktop session. Short tokens plus introspection at session start is the right call.

The JOSE Family (JWT’s Siblings)

JWT is part of the JOSE (JSON Object Signing and Encryption) family of RFCs:

RFC 7515 (JWS): JSON Web Signature. The signing mechanism that makes JWTs tamper-evident. When you sign a JWT, you’re actually creating a JWS with a JWT payload.

RFC 7516 (JWE): JSON Web Encryption. Encrypted JWTs. The payload is actually encrypted, not just encoded. Use when you need confidentiality, not just integrity.

RFC 7517 (JWK): JSON Web Key. The format for publishing cryptographic keys. The JWKS endpoint your auth server exposes is a JSON object containing an array of JWK objects, one per key.

RFC 7518 (JWA): JSON Web Algorithms. The catalog of algorithm identifiers used in the alg header: RS256, HS256, ES256, and so on.

RFC 7519 (JWT): JSON Web Token. The claims layer that sits on top of JWS or JWE.

Most developers only ever interact with RFC 7519. The others are what your JWT library implements underneath.

A Complete Verification Checklist

Before trusting a JWT in your system, verify every single one of these:

[ ] Signature valid (using pinned algorithm, never token's alg field)
[ ] exp > now (not expired, allow a few seconds of clock skew)
[ ] nbf <= now (not used before valid date, if present)
[ ] iss == expected issuer (your specific auth server URL)
[ ] aud contains your service's identifier
[ ] alg matches your configured algorithm (not whatever the token says)
[ ] kid resolves to a known, trusted key (not a path or query input)
[ ] jti not seen before (if you need replay protection)

Skip any of these and you have a vulnerability. The attacks above each correspond to one of these missing checks.

What’s Next

You now understand JWT from the format up through the cryptography and the attack surface. Every OAuth access token, every OpenID Connect ID token, and every session cookie that “uses JWTs” runs on this foundation.
