WTF is the OAuth RFC Zoo?

RFC 6749 is the skeleton. A dozen other RFCs are the organs. You need all of them.

If you’ve read OAuth 2.0, you know that RFC 6749 defines the framework: four actors, a few grant types, and access tokens. What it doesn’t tell you is how tokens should be formatted, how to validate them, how to revoke them, how to authenticate from a TV, or how to write a mobile app that isn’t trivially phishable.

All of that lives in the extension RFCs. The OAuth working group has been churning them out since 2012, and the result is a sprawling ecosystem where every RFC patches a specific gap left by the one before it.

Let’s map it.

The Core (You Already Know This)

RFC 6749 (2012): The OAuth 2.0 Authorization Framework. Defines the four roles, the three main grant types (Authorization Code, Client Credentials, Refresh Token), and the abstract token model. The skeleton everything else attaches to.

RFC 6750 (2012): Bearer Token Usage. Ships with 6749 but is its own RFC. Defines exactly how to present an access token in an HTTP request: Authorization: Bearer <token>. Also defines the WWW-Authenticate error response format that resource servers return.

RFC 6749 does not tell you how to send the token

That’s RFC 6750. If you’ve seen Authorization: Bearer ... headers, you’ve seen RFC 6750 in action.

These two together are “OAuth 2.0” in everyday speech. Everything below extends them.

Layer 1: Filling the Gaps in the Core

The core RFC intentionally left several things vague. These RFCs fill the gaps.

RFC 7519: JSON Web Token (JWT)

Not technically an OAuth RFC. It’s the base format spec for JSON Web Tokens and sits under the JOSE (JSON Object Signing and Encryption) family of RFCs. But you cannot understand the OAuth ecosystem without it.

A JWT is a base64-encoded JSON payload with a cryptographic signature attached. Three parts separated by dots:

HEADER.PAYLOAD.SIGNATURE

eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJ1c2VyMTIzIiwiZXhwIjoxNzM1Njg5NjAwfQ.abc123signature

The signature is verified using the issuer’s public key (for RS256) or a shared secret (for HS256). Anyone with the public key can verify the token without calling the auth server. This “local validation” property is why JWTs are so popular for access tokens in distributed systems.

RFC 7519 is the base. RFC 7515 (JWS, JSON Web Signature) and RFC 7516 (JWE, JSON Web Encryption) define the crypto. RFC 7517 (JWK, JSON Web Keys) defines the key format used to publish public keys.

RFC 7636: PKCE (2015)

Proof Key for Code Exchange. Already covered in the main OAuth article but worth placing in the timeline: PKCE was created specifically to fix a code interception attack possible on mobile platforms where multiple apps can register the same URI scheme.

The attack: malicious app registers myapp://callback, intercepts the auth code your legitimate app was expecting, exchanges it for a token. PKCE kills this because the attacker doesn’t have the code_verifier.

Originally “for mobile apps.” Now recommended for all clients.

RFC 7009: Token Revocation (2013)

The core spec has no way to invalidate a token before it expires. RFC 7009 adds a /revoke endpoint:

The auth server invalidates the token. Done. This is how “log out of all devices” and “disconnect this app” features work. Without RFC 7009, you’re stuck waiting for tokens to expire.

RFC 7662: Token Introspection (2015)

The core spec also doesn’t tell resource servers how to validate a token. If the token is opaque (a random string, not a JWT), the resource server has no idea what it means. It has to ask the auth server.

RFC 7662 standardizes that conversation:

The key field is active. If it’s false, the token is invalid, expired, or revoked. The resource server rejects the request.

Introspection vs JWT local validation: the trade-off

Introspection requires a network call from the resource server to the auth server on every request. JWT local validation is instant but doesn’t catch revocation in real time. If you revoke a JWT access token, the resource server won’t know until it tries to introspect or the token expires naturally. Production systems often use short-lived JWTs (15 min) to limit the revocation window.

Layer 2: Platform-Specific Best Practices

These RFCs exist because the core spec says nothing about the weird realities of mobile, browser, and browserless environments.

RFC 8252: OAuth 2.0 for Native Apps (2017)

Native apps (iOS, Android, desktop) have a specific phishing risk that web apps don’t: an embedded WebView is controlled by the app, not the OS browser. If your app opens a WebView for the OAuth consent screen, a malicious app could do the same and harvest credentials, because users can’t see the URL bar.

RFC 8252 says: use the system browser, not an embedded WebView.

On iOS this means ASWebAuthenticationSession. On Android this means Custom Tabs. On desktop this means the actual browser with a loopback redirect (http://127.0.0.1:PORT/callback).

The system browser is sandboxed from the app. The user can see the real URL. No credential harvesting possible.

The spec also mandates PKCE for all native apps, and bans the use of claimed HTTPS redirect URIs that could be intercepted.

RFC 8628: Device Authorization Grant (2019)

This is the one from the RFC you pasted. It solves a specific UX problem: what happens when the device has no browser?

Smart TVs. IoT sensors. CLI tools. Game consoles. These can’t open a browser window and can’t let users type a 50-character URL. Standard OAuth is unusable on them.

The Device Flow decouples the device from the authorization:

Smart TV (Device Client)              Auth Server              Your Phone (User Agent)
        |                                  |                             |
        |--POST /device_authorization----->|                             |
        |  client_id                       |                             |
        |<--device_code--------------------|                             |
        |   user_code: WDJB-MJHT          |                             |
        |   verification_uri:             |                             |
        |   https://example.com/activate  |                             |
        |                                  |                             |
        |  [display on TV screen]          |                             |
        |  "Go to example.com/activate"    |                             |
        |  "Enter code: WDJB-MJHT"        |                             |
        |                                  |                             |
        |  [polling every 5 seconds]       |<--user visits URL----------|
        |>--POST /token------------------->|<--user enters WDJB-MJHT----|
        |   device_code                    |<--user approves-------------|
        |<--authorization_pending----------|                             |
        |  [keep polling...]               |                             |
        |>--POST /token------------------->|                             |
        |<--access_token-------------------|                             |
        |  [TV is now authenticated]       |                             |

The device polls the token endpoint repeatedly until the user completes the flow on their phone. The auth server responds with authorization_pending until they do, then returns the access token.

The user_code is designed to be short and human-typeable: RFC 8628 recommends 8 characters using only letters that look distinct (no 0/O, 1/I confusion). Something like WDJB-MJHT.

You've seen this every time you log in to Netflix on a TV

”Visit netflix.com/activate and enter the code shown on your TV.” That is the Device Flow. Exactly as specified in RFC 8628.

The polling interval is enforced by the auth server (typically 5 seconds minimum). If the client polls too fast, the server returns slow_down and the client must back off. This prevents hammering the auth server.

Layer 3: Token Format Standardization

After years of every provider inventing their own JWT access token format, the working group finally published a standard.

RFC 9068: JWT Profile for OAuth 2.0 Access Tokens (2021)

RFC 6749 left the access token format completely unspecified. The result was a decade of every auth server using a different JWT claim set. Okta did it one way. Azure did it another. Keycloak had its own. Resource servers had to be custom-coded for each provider.

Key additions over a plain JWT:

jti (JWT ID) is a unique identifier for this specific token, used by revocation systems to mark specific tokens invalid.

client_id is which OAuth client issued this request (not just sub, which is the user).

aud must be the resource server’s URL. A JWT access token for api.example.com should be rejected by api.other.com. This prevents token reuse across APIs.

With RFC 9068, a resource server can validate any compliant JWT access token using the same logic, regardless of which auth server issued it.

Layer 4: Discovery and Registration

These RFCs let systems find each other automatically.

The Deprecation Zone

Two grant types from the original RFC 6749 are now officially deprecated by the OAuth 2.0 Security Best Current Practice (RFC 9700) and omitted from the draft OAuth 2.1 spec.

Implicit Grant: Returned the access token directly in the URL fragment after a redirect. Designed for SPAs before CORS existed. The token was visible in browser history, server logs, and referrer headers. PKCE with Authorization Code is strictly better. Dead.

Resource Owner Password Credentials (ROPC): The client collects the user’s username and password directly and sends them to the auth server. Completely defeats the purpose of OAuth (the whole point is the user authenticates with the auth server, not the client). It was meant for “trusted” first-party apps. In practice it created terrible habits and is impossible to migrate away from cleanly. Dead.

You will see both of these in the wild

Legacy systems love ROPC especially. If you’re reviewing an older codebase and see grant_type=password in a token request, that’s ROPC. It should be replaced with Authorization Code + PKCE.

The Full Map

Here’s everything layered by responsibility:

WHAT THE USER SEES
  └── RFC 8628  Device Flow (TV / CLI / no-browser)
  └── RFC 8252  Native Apps (system browser, no WebView)

CORE FRAMEWORK
  └── RFC 6749  OAuth 2.0 (roles, grant types, abstract tokens)
  └── RFC 6750  Bearer Token Usage (how to send a token)

TOKEN LIFECYCLE
  └── RFC 7009  Token Revocation (/revoke endpoint)
  └── RFC 7662  Token Introspection (/introspect endpoint)
  └── RFC 7519  JWT format (the token payload format)
  └── RFC 9068  JWT Access Token Profile (standardized claims)
  └── RFC 7636  PKCE (auth code security)

DISCOVERY & REGISTRATION
  └── RFC 8414  AS Metadata (/.well-known endpoints)
  └── RFC 7591  Dynamic Client Registration

IDENTITY LAYER (not OAuth, but always used with it)
  └── OpenID Connect Core  (ID token, UserInfo, OIDC discovery)

What OAuth 2.1 Does to All of This

OAuth 2.1 (still in draft) is not a new protocol. It’s a cleanup spec that folds all the lessons from the extension RFCs back into a single document. The major changes:

  • PKCE is required for all Authorization Code flows
  • Implicit Grant is gone
  • Resource Owner Password Credentials is gone
  • Redirect URIs must be exact string matches (no pattern matching)
  • Refresh tokens for public clients must be sender-constrained or one-time-use

If you’re building something new today, just implement what OAuth 2.1 will require. You’re future-proof and you’re already following current best practice.

Which RFCs You Actually Need

Not every project needs every RFC. Here’s a practical cut:

Building a web app with a backend? RFC 6749 (Auth Code flow) + RFC 6750 (Bearer tokens) + RFC 7636 (PKCE) + RFC 7009 (revocation). That’s your baseline.

Building a mobile app? Add RFC 8252 (Native Apps). System browser is mandatory.

Building microservices? RFC 6749 (Client Credentials flow) + RFC 9068 (JWT access tokens so resource servers can validate locally) + RFC 7009 (revocation).

Building for a TV or CLI? RFC 8628 (Device Flow). It’s the only sane option.

Building a platform where others integrate with you? RFC 8414 (discovery) + RFC 7591 (dynamic registration). You want integrators to be able to self-serve.

Need user identity on top of all this? OpenID Connect. Not an RFC, but it’s built on the same foundation and standardized by the OpenID Foundation.

What’s Next

You now have a map of the entire OAuth ecosystem. Each RFC solves a specific problem that the one before it left open.

The most important unresolved problem in the current spec: once you issue a JWT access token, revocation is painful. If a token is stolen and you revoke it, resource servers using local JWT validation won’t know until the token expires naturally. This is why short token lifetimes (15 minutes is common) are recommended even when it means more refresh token usage.

The emerging answer involves sender-constrained tokens via DPoP (Demonstration of Proof-of-Possession), currently in RFC 9449. That’s the next frontier.
