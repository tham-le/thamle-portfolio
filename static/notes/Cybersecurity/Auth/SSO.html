WTF is SSO?

One login. Every door opens. Until logout, which is where things get complicated.

You log into Gmail. You open a new tab and navigate to Google Drive. You’re already logged in. You open Google Calendar. Still logged in. You never entered a password for Drive or Calendar. You only authenticated once.

That’s SSO. Single Sign-On. And it sounds simple until you try to implement it across apps you don’t fully control, on different domains, with different session lifetimes. Then it’s anything but.

The Core Analogy

Think of SSO like a theme park wristband.

You buy a ticket at the main gate (the Identity Provider). They verify your identity and attach a wristband. Every ride in the park (each application) checks the wristband, not your ticket. You don’t re-buy your ticket for each ride. The wristband is your proof.

When you leave the park, you cut off the wristband. Now no rides will let you in. But here’s the catch: if the park has six different roller coasters and each one has its own separate door locked to your wristband, cutting it off at the gate doesn’t automatically unlock all six doors. Someone has to go notify each ride.

That notification problem is what makes SSO logout genuinely hard.

The Problem SSO Solves

Without SSO, each application manages its own authentication entirely:

  • App A has its own username/password database
  • App B has its own username/password database
  • App C has its own username/password database

Users maintain separate credentials for each. Password reuse is inevitable. When an employee leaves a company, IT has to revoke access in every system separately. When a user changes their password, it only changes it for that one app. The security audit is a spreadsheet nightmare.

SSO centralizes authentication to a single Identity Provider (IdP). Applications delegate “is this user authenticated?” to the IdP. The IdP answers with a signed assertion. Applications trust the assertion. No local credential databases needed for each app.

The Three Sessions

This is the architectural insight most explanations skip. SSO doesn’t magically eliminate sessions. It creates a layered session model with three distinct levels, each with its own lifetime:

IdP Session: The session between the user and the Identity Provider. Created when the user authenticates at the IdP. Represented by a cookie set on the IdP’s domain (accounts.google.com, login.microsoftonline.com, your Okta tenant). This is the master session. As long as it exists, any app can get tokens without prompting the user.

Application Session: The session between the user and a specific application. Created after the app receives a valid ID token from the IdP and establishes its own session (usually a cookie on app.example.com). Each app has its own, with its own expiry.

IdP-issued Tokens: The short-lived access and ID tokens used to bootstrap the application session. These are transient artifacts of the OIDC flow, not the session itself.

                    IdP Session
                    (cookie at accounts.google.com)
                         |
          ┌──────────────┼──────────────┐
          |              |              |
    App A Session   App B Session   App C Session
    (cookie at      (cookie at      (cookie at
     app-a.com)      app-b.com)      app-c.com)

Each application session is independent. They can have different expiry times. They all depend on the IdP session being valid to get bootstrapped, but once established, they live independently.

How OIDC Implements SSO

The mechanism is the OIDC Authorization Code flow with one key addition: the IdP sets a session cookie on its own domain when the user authenticates.

Here’s what happens when you log into your first app:

Browser          App A           Identity Provider
   |               |                    |
   |--visit------->|                    |
   |               |--redirect auth---->|
   |               |  request           |
   |<--redirect----|                    |
   |--follow-------|------------------>|
   |               |         [no IdP session cookie]
   |               |         show login form
   |<--login page-----------           |
   |--enter creds-->                   |
   |               |         authenticate user
   |               |         SET IdP SESSION COOKIE
   |               |         issue auth code
   |<--redirect with code--------------|
   |--code-------->|                   |
   |               |--exchange code--->|
   |               |<--id_token--------|
   |               |   access_token    |
   |               |                   |
   |               |  validate id_token
   |               |  SET APP SESSION COOKIE
   |<--session-----|                   |

Now you navigate to App B:

Browser          App B           Identity Provider
   |               |                    |
   |--visit------->|                    |
   |               |--redirect auth---->|
   |               |  request           |
   |<--redirect----|                    |
   |--follow-------|------------------>|
   |               |         [IdP session cookie PRESENT]
   |               |         skip login form
   |               |         issue auth code immediately
   |<--redirect with code--------------|
   |--code-------->|                   |
   |               |--exchange code--->|
   |               |<--id_token--------|
   |               |  SET APP SESSION COOKIE
   |<--session-----|                   |

The IdP cookie does all the work. The user never sees a login form for App B. The full OIDC flow still runs, but the IdP completes the authentication step silently because it already knows who you are.

The OIDC flow always runs, even for SSO

SSO doesn’t bypass the Authorization Code flow. It just means the IdP’s “authentication” step is instant because the IdP session cookie is present. The redirect still happens. The code exchange still happens. The tokens are still issued. The browser just doesn’t pause to show a login form.

Silent Authentication (prompt=none)

SPAs and apps that want to check session status without a full redirect can use prompt=none:

GET /authorize?
  client_id=my-app
  &scope=openid email
  &response_type=code
  &redirect_uri=https://app.example.com/callback
  &prompt=none      <-- do NOT show UI, fail if not already authenticated

If the IdP session is active, the IdP completes the flow silently and redirects back with an auth code. If not, it redirects back with error=login_required. The app catches the error and shows its own “session expired” message.

This is how SPAs implement background token refresh without forcing a visible redirect.

The Logout Problem

Login is easy. Logout is where SSO gets genuinely messy.

The question: when a user logs out of App A, what should happen?

There are two distinct operations:

Local logout: clear App A’s session cookie. The user is logged out of App A. The IdP session is still active. If the user navigates to App B, they’re still logged in. If they navigate to App A, the OIDC flow runs again silently and logs them back in instantly. This is usually wrong from a user expectation standpoint.

Single Logout (SLO): terminate the IdP session AND notify all apps that had sessions to invalidate theirs too. The user is logged out everywhere. This is what users expect when they click “Log out of all devices.”

OIDC defines four separate specifications for logout, which tells you something about how complicated it is:

RP-Initiated Logout

The app (Relying Party) initiates logout by redirecting the user to the IdP’s end_session_endpoint:

GET https://idp.example.com/logout?
  id_token_hint=eyJ...    (the user's ID token)
  &post_logout_redirect_uri=https://app.example.com/logged-out
  &state=random-csrf-value

The IdP terminates the IdP session and redirects back. This is the simple case: one app, one logout. But it doesn’t notify other apps.

Front-Channel Logout

The IdP renders hidden iframes in the logout page, one per app that had a session. Each iframe loads the app’s registered logout URL:

Each app receives the request in the iframe, clears its session, returns 200. The user gets logged out of all apps simultaneously, all through the browser.

The problem: modern browsers increasingly block third-party cookies and cross-origin iframe requests for tracking prevention. Front-channel logout is dying. It relies on the browser sending session cookies to third-party iframes, which browsers are restricting.

Back-Channel Logout

The IdP sends HTTP POST requests directly from its servers to each app’s registered backchannel_logout_uri. No browser involved:

The logout token is a signed JWT (similar to an ID token) containing the session ID (sid) and user (sub) to log out. The app validates the token, finds the corresponding server-side session, and terminates it.

Back-channel logout is browser-independent. It works even if the user has already closed their browser. It requires the app to have accessible server-side session storage (not just a client-side cookie) and requires the IdP to be able to reach the app’s server.

Which logout mechanism to use

For new implementations: back-channel logout. It’s the most reliable, unaffected by third-party cookie restrictions, and works when users close their browser without clicking logout. Front-channel logout is on the way out. RP-initiated logout alone is only acceptable if you genuinely don’t care about propagating logout to other apps.

SAML vs OIDC for SSO

SSO predates OIDC. The older enterprise standard is SAML (Security Assertion Markup Language). You’ll encounter both in production.

PropertyOIDCSAML
Token formatJWTSigned XML
TransportHTTP redirects, JSONHTTP redirects, XML POST
Primary targetWeb apps, mobile, APIsEnterprise web apps
AdoptionModern cloud, consumerLegacy enterprise, B2B
ComplexityModerateHigh

OIDC is generally the right choice for new systems. SAML is what you deal with when integrating with enterprise customers who run Okta, Azure AD, or similar in SAML mode. Many organizations run both: OIDC for their own apps, SAML for integrating with enterprise customers.

The Identity Provider Landscape

In practice, SSO means choosing or running an IdP. The main options:

Hosted/SaaS IdPs: Okta, Auth0, Azure Entra ID, Google Identity Platform. Managed for you. Expensive at scale but operational overhead is minimal.

Self-hosted IdPs: Keycloak, Authentik, Dex. Open source. You run the infrastructure. Full control.

Social login: “Sign in with Google/GitHub/Apple.” The social platform acts as the IdP. Simple for consumer apps. You don’t control the IdP or its policies.

Enterprise federation: your customer’s Okta/Azure AD is the IdP. You’re the SP (Service Provider). Common in B2B SaaS where enterprise customers require SSO via their own identity management system.

What SSO Does NOT Solve

SSO handles authentication centrally. It doesn’t handle:

Authorization: whether Alice can see a specific record in App A is still App A’s problem. The IdP tells you who Alice is, not what she’s allowed to do. Authorization lives in each app.

Session consistency: if App A has a 24-hour session lifetime and App B has a 1-hour session lifetime, they’ll expire at different times. A user might be “logged in” to the IdP and App A but automatically logged out of App B. Your system needs to handle this gracefully.

Credential security: SSO makes the IdP a high-value target. If the IdP is compromised, every app is compromised. This is the “blast radius” trade-off of centralization. Multi-factor authentication at the IdP becomes non-negotiable.

What’s Next

SSO manages authentication across apps by centralizing the session at the IdP. Every app builds its own session on top of the IdP’s session, and logout propagation across all of those sessions is the genuinely hard part.

The sessions that apps create (their own cookies) need to be managed correctly regardless of whether SSO is involved. That brings us to the next article: how session management actually works at the HTTP level, what makes a session cookie secure, and the attacks that happen when you get it wrong.
