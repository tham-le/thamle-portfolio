WTF is Session Management?

OAuth logs you in. OIDC tells you who you are. Session management is how you stay logged in.

You’ve authenticated. The OpenID Connect flow completed. You have an ID token. Your app knows you’re Alice.

Now what? You can’t attach a JWT to every page load. You can’t re-run the full OIDC flow on every click. The user needs to stay logged in for hours, maybe days. The mechanism that makes that work is session management, and it’s where the majority of auth bugs in production actually live.

The Core Analogy

Think of session management like a coat check at a restaurant.

When you arrive, you hand over your coat (authenticate). They give you a numbered ticket (session ID). For the rest of the evening, you don’t bring your coat to the table. You just have the ticket. When you leave, you hand in the ticket and get your coat back.

The ticket is useless without the coat check’s records. The records are what give the ticket meaning. And if someone steals your ticket mid-dinner, they can walk out with your coat at the end of the night. The security of the system depends entirely on keeping that ticket private.

What a Session Actually Is

A session is a server-side record associating a random identifier with a user’s state:

Session Store (server-side)
┌─────────────────┬──────────┬───────────────────────────────┐
│ session_id      │ user_id  │ data                          │
├─────────────────┼──────────┼───────────────────────────────┤
│ a3f9c8d2e1b7... │ user_123 │ {role: "admin", cart: [...]} │
│ 7b2e4a1c9f3d... │ user_456 │ {role: "user", prefs: {...}} │
└─────────────────┴──────────┴───────────────────────────────┘

The session ID is a random, unguessable string. It lives in the client’s browser as a cookie. On every request, the browser sends the cookie, the server looks up the session ID in the store, finds the associated user and state, and proceeds. The session ID means nothing on its own. It’s only meaningful because the server has a record for it.

This is fundamentally different from a JWT-based approach, where all the state lives in the token itself and the server validates it locally without a lookup. Both patterns coexist in production systems, and choosing between them has real trade-offs:

PropertyServer-side SessionJWT / Token
StorageServer (DB, Redis)Client (cookie, localStorage)
RevocationInstant (delete record)Delayed (wait for expiry)
ScaleRequires shared session storeStateless, scales horizontally
Payload sizeTiny (just an ID)Can be large
Invalidation on compromiseImmediateRequires short lifetimes or introspection

Most production systems use both: a server-side session to track the authenticated user within your own app, and short-lived JWTs for calling external APIs.

The Cookie: Your Session’s Delivery Mechanism

The session ID travels between browser and server as an HTTP cookie. The Set-Cookie response header creates it:

Each attribute in that header matters. Get any of them wrong and you have a vulnerability.

Secure

The cookie is only sent over HTTPS. Never over plain HTTP. Without this, anyone on the same network (coffee shop Wi-Fi, corporate proxy, ISP) can read the cookie in transit and hijack the session. This is not optional. The Firesheep browser extension from 2010 demonstrated this attack against major sites like Facebook in a trivially simple way.

HttpOnly

JavaScript cannot read this cookie. document.cookie will not return it. This is your primary defense against XSS-based session hijacking. If an attacker injects JavaScript into your page, they cannot steal the session cookie and exfiltrate it. They can still make authenticated requests from within the XSS context, but they can’t take the cookie offline.

SameSite

Controls whether the browser sends the cookie on cross-site requests. Three values:

Strict: Cookie is never sent on any cross-site request. Strongest CSRF protection. Can break legitimate cross-site navigation (e.g., clicking a link from an external site to a page that requires auth).

Lax: Cookie is sent on top-level navigations (user clicks a link), but not on cross-site sub-resource requests (images, iframes, AJAX). This is the modern default and the right choice for most session cookies. Protects against CSRF while preserving normal link-following behavior.

None: Cookie is always sent cross-site. Required for cross-site cookies (third-party embeds, SSO cookies used across different domains). Must be paired with Secure.

SameSite alone does not fully prevent CSRF

It’s a partial defense. You still need CSRF tokens for state-changing requests on sensitive operations, especially if you use SameSite=Lax (which allows cookies on top-level GET navigation). The OWASP recommendation is to use SameSite as one layer among several.

__Host- and __Secure- Prefixes

Browser-enforced security guarantees on top of the standard attributes:

__Secure-: Browser enforces that the Secure attribute is set. Prevents cookies from being set by non-HTTPS responses.

__Host-: Even stricter. Browser enforces Secure, Path=/, and no Domain attribute. The no Domain part is key: without a domain, the cookie only goes to the exact host, not to subdomains. A __Host- cookie set by app.example.com won’t be sent to evil.app.example.com.

Use __Host- for your session cookies in production. It’s the most secure option and browsers enforce the constraints automatically.

Session Lifetime: The Expiry Trade-Off

How long should a session live? It’s always a security-usability trade-off.

Absolute timeout: the session expires N hours after creation, regardless of activity. Protects against forgotten logged-in sessions on shared computers. Annoying for active users.

Idle timeout: the session expires N minutes after the last activity. Resets on every request. More user-friendly. Common for sensitive applications (banking: 15 min idle, consumer apps: 30-60 min idle).

Sliding window: like idle timeout, but the cookie’s Max-Age is refreshed on every response. The session extends as long as the user stays active. Can result in very long-lived sessions for power users.

In practice: use an idle timeout with an absolute maximum. A session shouldn’t live longer than 24 hours no matter how active the user is. For high-security contexts (financial, medical), tighten both to much shorter windows.

When a session expires, always:

  1. Delete it from the server-side session store
  2. Clear the cookie with Max-Age=0
  3. Redirect to login

The Attacks

Session Hijacking

An attacker obtains a valid session ID and uses it to impersonate the user. This is the abstract threat that all the cookie attributes above are designed to prevent. The methods:

Network interception: capturing the cookie from unencrypted HTTP traffic. Prevented by Secure flag and enforcing HTTPS.

XSS cookie theft: injecting JavaScript that reads document.cookie and sends it to an attacker’s server. Prevented by HttpOnly.

Cross-site request forgery (CSRF): tricking the user’s browser into making an authenticated request to your app from an attacker-controlled page. The browser automatically sends the session cookie. Prevented by SameSite=Lax or Strict, plus CSRF tokens for sensitive operations.

Session Fixation

A more subtle attack. The attacker sets a known session ID in the victim’s browser before authentication. The victim logs in. If the server doesn’t generate a new session ID on login and instead keeps the pre-auth ID, the attacker now knows the authenticated session ID.

The fix is simple and mandatory: always regenerate the session ID immediately after successful authentication. The pre-authentication session (if any) is discarded. A brand new session ID is issued for the authenticated session.

Every web framework has a method for this. In Express.js it’s req.session.regenerate(). In Django it’s request.session.cycle_key(). In Rails it’s reset_session. Use it. Always.

Session ID in URLs

Never put session IDs in URLs. They end up in browser history, server access logs, Referer headers, and shared links. A URL like https://app.example.com/dashboard?session=abc123 is a session waiting to be hijacked.

Cookies exist precisely so session state doesn’t travel in URLs. Use them.

Token-Based Sessions: When JWTs Replace Cookies

Some architectures store the JWT itself in a cookie instead of a server-side session ID. The browser sends the JWT with every request. The server validates it locally, no session store lookup needed.

This is stateless and horizontally scalable. No shared session store needed across multiple server instances.

The trade-off: instant revocation is gone. If you need to log out this specific user right now (security incident, account deletion, force-logout-all-devices), you can’t. The JWT is valid until it expires. Your options are short token lifetimes (15-30 minutes) paired with refresh tokens, or a token revocation list (which reintroduces server-side state).

JWTs in localStorage are worse. localStorage is accessible to JavaScript, which means XSS can steal the token and take it offline. An HttpOnly cookie prevents this. If you’re storing auth tokens client-side, always use HttpOnly cookies, never localStorage.

The "JWTs in localStorage are fine" myth

This comes up constantly. It is not fine. If your site has any XSS vulnerability (even in a third-party script you load), an attacker can steal tokens from localStorage and use them from their own server until they expire. An HttpOnly cookie cannot be read by JavaScript at all. The tradeoff is that HttpOnly cookies require SameSite and CSRF handling, which is more work but is the correct approach.

Logout: The Often-Forgotten Step

Logout must do three things:

  1. Delete the server-side session record
  2. Clear the cookie on the client (Max-Age=0)
  3. Redirect to a page that requires authentication (so the browser can’t navigate “back” to a cached authenticated page)

Forgetting step 1 is the most common mistake. Clearing only the client-side cookie leaves the server-side session alive. If an attacker already copied the session ID, they can still use it. The session store is the authoritative source of truth.

For JWT-based sessions where there’s no server-side record, step 1 means either adding the token to a revocation list, or accepting that “logout” only clears the client-side cookie and the token remains technically valid until expiry. Short token lifetimes make this acceptable in practice.

A Production Session Cookie

Putting it all together, here’s what a production-grade session cookie looks like:

Breaking it down:

  • __Host-session: __Host- prefix enforces Secure, Path=/, no Domain
  • <128-bit-random-hex>: cryptographically random, at least 64 bits of entropy (OWASP recommends 128 bits)
  • Max-Age=86400: 24-hour absolute maximum
  • Path=/: applies to all paths on this host
  • Secure: HTTPS only
  • HttpOnly: no JavaScript access
  • SameSite=Lax: CSRF protection without breaking link navigation

And on login, before setting this cookie: session_store.regenerate_id().

What’s Next

Session management is the machinery that keeps users logged in after authentication. It’s where OIDC and OAuth hand off to your own application layer. Get the cookie attributes wrong and even perfect OAuth implementation doesn’t protect you.

The final piece in the auth picture is the frontier of bearer token security. Regular session cookies and JWTs are “possession-based”: whoever has the token, has access. The next evolution is sender-constrained tokens, where a stolen token is cryptographically useless without the corresponding private key. That’s mTLS and DPoP.
