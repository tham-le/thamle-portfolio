When your package manager becomes a weapon

You run npm install on a Tuesday morning. Your build passes. Your tests pass. Everything looks normal. But in the background, a 10MB obfuscated JavaScript file just scanned your entire home directory, exfiltrated your AWS credentials to a public GitHub repository, and infected every npm package you maintain with a copy of itself.

Welcome to Sha1-Hulud, the most aggressive Supply Chain Attack of 2025.

What Just Happened?

Between November 21-23, 2025, attackers compromised over 700 npm packages, including popular libraries from Zapier, PostHog, Postman, and AsyncAPI. The malware spread to more than 25,000 GitHub repositories in just a few hours. It’s a worm, a credential harvester, and a time bomb all in one.

The name comes from Dune’s giant sandworms. The malware’s creator calls this wave “The Second Coming” because it’s actually the second iteration. The first Shai-Hulud attack in September 2025 stole approximately $50 million in cryptocurrency. This version is worse.

Current Status

As of November 26, 2025, 99% of malicious package versions have been removed from npm. However, if you installed any compromised packages between November 21-26, you need to rotate all credentials immediately.

The Core Mechanism: Preinstall Scripts

The attack exploits a feature of npm called lifecycle scripts. When you install a package, npm can run code at different stages:

The preinstall script runs before the package even finishes installing. This means:

  • It executes even if installation fails
  • It runs in CI/CD pipelines automatically
  • Most developers never see it happen
  • No user interaction required

Traditional malware used postinstall, which only ran after successful installation. By moving to preinstall, Sha1-Hulud dramatically increased its attack surface.

How It Works: The Attack Chain

Stage 1: The Loader

The malware injects two files into compromised packages:

  • setup_bun.js (the loader, looks innocent)
  • bun_environment.js (the actual malware, 10MB of obfuscated code)

It downloads Bun (a legitimate JavaScript runtime) and uses it to execute the payload. This serves two purposes:

  1. Bun is less monitored than Node.js in most environments
  2. The download looks like a normal developer tool installation

The malware downloads TruffleHog, a legitimate security tool designed to find secrets in code repositories, and runs it against your entire home directory:

TruffleHog finds secrets by:

  • Pattern matching (looks for things like ghp_[a-zA-Z0-9]{36})
  • High entropy detection (random-looking strings that might be keys)
  • Git history scanning (even deleted commits!)

Stage 4: Exfiltration via GitHub

Here’s where it gets clever. Instead of sending data to a traditional command and control server, the malware uses GitHub itself:

The genius of this approach:

  1. Creates public repositories with description “Sha1-Hulud: The Second Coming.”
  2. Uploads stolen credentials as normal-looking commits
  3. If it can’t create a repo, searches for other victims’ repos and borrows their tokens
  4. All traffic looks like legitimate GitHub API usage

This creates a resilient botnet where infected systems share access tokens with each other.

What Makes This Attack So Effective?

  1. Timing: Executes during preinstall before installation completes
  2. Stealth: Uses legitimate tools (Bun, TruffleHog) and GitHub API
  3. Resilience: Infected systems share tokens with each other
  4. Scale: Each compromised developer infects 100 more packages
  5. Deterrent: Dead man’s switch prevents simple takedowns

Immediate Actions

  1. Rotate ALL credentials:

Long-Term Practices

  1. Disable lifecycle scripts in CI/CD:

  1. Use short-lived tokens:

    • GitHub: Fine-grained tokens with minimal scope and expiration
    • Cloud providers: Temporary credentials via IAM roles
    • Never use classic tokens that never expire

  2. Never store secrets in files:

  1. Separate environments:
    • Run builds in containers or VMs
    • Use OIDC for CI/CD instead of static tokens
    • Apply principle of least privilege

Key Takeaways

  • Sha1-Hulud is a self-replicating worm that spreads through npm packages
  • It executes during preinstall, before installation completes
  • Uses legitimate tools (Bun, TruffleHog) to evade detection
  • Exfiltrates to GitHub instead of traditional C2 servers
  • Contains a destructive payload if containment is attempted
  • Compromised 700+ packages and 25,000+ repositories in hours

The fundamental lesson: Supply chain attacks exploit trust. When you run npm install, you’re trusting hundreds of packages and their maintainers. One compromised account can cascade into thousands of infected systems.

This is why Supply Chain Security isn’t just about checking for vulnerabilities. It’s about understanding and managing trust relationships in your entire dependency tree.
