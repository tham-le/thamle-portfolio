The waiter who serves code from a menu you didn’t write

You run npm install express and suddenly 50+ packages appear in your node_modules folder. You wanted one package. You got 50. Where did they come from? Who wrote them? Are they safe? WTF is npm?

npm (Node Package Manager) is a combination of three things:

  1. A registry (a giant warehouse of JavaScript packages)
  2. A CLI tool (the npm command you run)
  3. A package format (rules for how packages are structured)

Think of npm like a restaurant supply chain:

  • The registry is the food distributor’s warehouse
  • The CLI is the delivery truck
  • Your package.json is your order form
  • node_modules is your kitchen’s pantry

Lifecycle Scripts: Useful Feature or Security Nightmare?

Lifecycle scripts were designed for legitimate use cases:

Legitimate Use: Compiling Native Code

bcrypt needs to compile C++ code for your specific system. The install script builds it.

Legitimate Use: Downloading Binary Dependencies

Puppeteer downloads a version of Chrome. The postinstall script fetches the right binary for your OS.

Legitimate Use: Setup Configuration

Husky sets up git hooks. The postinstall script configures them.

Malicious Use: Sha1-Hulud

Same mechanism, different intent:

  • Downloads Bun runtime
  • Executes 10MB obfuscated payload
  • Steals credentials
  • Propagates to other packages
  • Exfiltrates to GitHub

The problem: npm can’t tell the difference between legitimate and malicious scripts.

The Lifecycle Script Order

When you run npm install, scripts execute in this order:

1. preinstall    (runs before package is installed)
2. install       (runs during installation)
3. postinstall   (runs after package is installed)
4. prepublish    (deprecated, but still runs)
5. prepare       (runs after installation finishes)

Sha1-Hulud used preinstall because:

  • It runs before installation completes
  • It runs even if installation fails
  • It executes in CI/CD pipelines automatically
  • Developers rarely see the output

This is more aggressive than previous attacks that used postinstall.

Version Resolution: Semantic Versioning

npm uses semver (semantic versioning):

MAJOR.MINOR.PATCH
  5  . 2   . 1

MAJOR: Breaking changes (5.x.x → 6.0.0)
MINOR: New features, backwards compatible (5.2.x → 5.3.0)
PATCH: Bug fixes, no new features (5.2.1 → 5.2.2)

The security problem:

Today, this installs 1.0.0 (safe). Tomorrow, maintainer publishes 1.0.1 with malicious code. Your next npm install gets the compromised version automatically.

Lock Files: Freezing Versions

package-lock.json locks exact versions:

Key fields:

  • version: Exact version installed
  • resolved: Where it was downloaded from
  • integrity: SHA-512 hash to verify contents

With a lock file, running npm install installs exactly the same versions every time, even if newer versions are published.

But this only helps if:

  1. You commit the lock file to git
  2. You don’t run npm update
  3. New dependencies don’t pull in new transitive dependencies

npm Security Features

npm has added security features over time:

1. Two-Factor Authentication (2FA)

Requires 2FA for publishing packages. Sha1-Hulud succeeded because many maintainers hadn’t enabled this.

2. npm audit

Checks dependencies against a database of known vulnerabilities:

# npm audit report

lodash  <4.17.21
Severity: high
Prototype Pollution - https://github.com/advisories/GHSA-xxx
fix available via `npm audit fix`

Limitation: Only finds known vulnerabilities. Zero-days and novel attacks like Sha1-Hulud aren’t detected.

3. Package Signatures (Experimental)

Checks cryptographic signatures on packages. Not widely adopted yet.

4. Install Scripts Prompt (npm 7+)

Prompts before running scripts. Easy to click through without thinking.

Why npm Is an Attractive Target

From an attacker’s perspective, npm is perfect:

  1. Massive reach: 2+ million packages, billions of downloads per week
  2. Auto-execution: Scripts run automatically during install
  3. Deep trust: Developers rarely audit dependencies
  4. Transitive dependencies: One compromise reaches many projects
  5. CI/CD integration: Runs in production build pipelines
  6. Full permissions: Scripts run with user’s permissions

One compromised popular package reaches thousands of applications instantly.

Alternatives and Competitors

Yarn

Facebook’s alternative to npm. Uses the same registry, different CLI. Added features:

  • Better caching
  • Deterministic installs
  • Workspaces for monorepos

Same security model as npm.

pnpm

More efficient storage using hard links. Stricter about dependency access. Slightly better security model but same fundamental issues.

Deno

Explicit permissions model:

Must explicitly grant network, filesystem, and environment access. Scripts can’t steal credentials without permissions.

The Fundamental Tension

npm faces an impossible balance:

Convenience vs. Security

  • Make it easy to share code → More attack surface
  • Auto-run scripts → Malicious code executes automatically
  • Thousands of dependencies → One compromise cascades

Openness vs. Trust

  • Anyone can publish → No vetting process
  • Democratic access → No quality control
  • Low barrier to entry → Easy to inject malicious packages

There’s no perfect solution. The ecosystem chose convenience and openness. Security is your responsibility.

Key Takeaways

  • npm is a package manager, registry, and CLI tool for JavaScript
  • Lifecycle scripts run arbitrary code during installation
  • Modern apps depend on hundreds of packages transitively
  • Each dependency is a trust relationship and potential attack vector
  • Lock files freeze versions but don’t prevent initial compromises
  • npm chose convenience over security; vigilance is your job

The hard truth: Every npm install is an act of faith. You’re trusting hundreds of people you’ve never met with access to your system. Understanding this is the first step to managing that risk.
